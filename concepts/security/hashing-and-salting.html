Hashing & Salting

Hashing is a one-way function: easy to compute, infeasible to reverse. Salting makes each hash unique. Together they're how you store passwords without storing passwords.

#security #hashing #passwords #cryptography

The idea in plain English

A hash maps input to a fixed-size value and is one-way — you can't feasibly recover the input from the output, and the same input always yields the same hash. That's why you store the hash of a password, not the password: at login you hash the attempt and compare. Unlike encryption, there's no key to steal that reverses it.

Plain hashing isn't enough, because identical passwords hash identically and attackers precompute huge rainbow tables of common-password → hash. A salt — a unique random value per user, stored alongside the hash — is mixed in before hashing, so two users with the same password get different hashes and precomputed tables are useless.

Speed is the other dimension. Fast hashes (MD5, SHA-256) are designed for throughput, so a stolen database is brute-forced at billions of guesses per second on a GPU. Password hashes are deliberately slow and memory-hard — bcrypt, scrypt, Argon2 — with a tunable work factor, so each guess is expensive.

So the rule for passwords: a slow, salted hash (Argon2id today, bcrypt as a solid default), never plain SHA, never reversible encryption. For non-password uses (integrity checks, deduplication), a fast cryptographic hash like SHA-256 is exactly right — match the tool to the job.

Real-world example

🏢 How this works in production

Reputable systems store passwords as salted bcrypt/Argon2 hashes; ASP.NET Core Identity uses PBKDF2 with a per-user salt and many iterations by default. The mega-breaches that expose millions of accounts are invariably the ones that used unsalted MD5 or SHA-1 — those dumps are cracked within hours because the hashes are fast and identical passwords share a hash.

Salt + slow hash = uncrackable dump

In C#

C# 
// Store a salted, slow hash — never the password, never reversible encryption.
string hash = BCrypt.Net.BCrypt.HashPassword(password, workFactor: 12);
// the salt and cost are embedded in the hash string itself.

// Verify at login (constant-time compare inside):
bool ok = BCrypt.Net.BCrypt.Verify(attempt, hash);

// ❌ Don't: SHA256(password) — fast + unsalted = cracked in hours.
// ✅ For integrity/dedup (not passwords), a fast SHA-256 IS the right tool.

Key takeaways

🎯 Remember these three

  • Hashing is one-way (no key to reverse it) — store the hash, compare at login; encryption is the wrong tool for passwords.
  • A per-user salt makes identical passwords hash differently, defeating rainbow tables.
  • Use a slow, memory-hard password hash (Argon2/bcrypt) with a work factor — never fast MD5/SHA for passwords.

🗣️ When this comes up

Three points win it: hashing is one-way (vs reversible encryption), salt defeats rainbow tables, and password hashes must be slow (Argon2/bcrypt). Knowing SHA-256 is right for integrity but wrong for passwords shows real understanding.
