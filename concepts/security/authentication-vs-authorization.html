Authentication vs Authorization

Two words one letter apart that people constantly mix up. Authentication = who are you? Authorization = what are you allowed to do? Different steps, different failures.

#security #auth #identity #access-control

The idea in plain English

Authentication (authn) verifies identity — proving you are who you claim to be. It's the login: a password, a one-time code, a biometric, a token. Its output is a trusted answer to 'who is this?'

Authorization (authz) happens after authentication and decides permissions — what this now-known identity may access or do. Being logged in (authenticated) doesn't mean you can delete another user's data (that's an authorization check).

The order matters: authenticate first, then authorize each action. They fail differently too — a failed authentication is '401 Unauthorized' (we don't know who you are), while a failed authorization is '403 Forbidden' (we know you, but you can't do this). The HTTP names are historically confusing, but that's the mapping.

Authorization models include role-based (RBAC) — permissions by role (admin, editor) — and attribute/claims-based — decisions from attributes (owner of this resource, in this department). The most common real-world bug is checking authentication but forgetting per-resource authorization, letting a logged-in user act on objects that aren't theirs.

Real-world example

🏢 How this works in production

Every app does both: you log in (authn) once, then each action is authorized — can this user edit that document? The classic breach (OWASP's #1 API risk, BOLA/IDOR) is an authorization failure: the app authenticates you, then trusts an object ID from the request without checking you own it, so changing /orders/123 to /orders/124 exposes someone else's order.

Identify first, then permit

In C#

C# 
// Authentication: establish identity (who you are).
[Authorize]                                  // must be authenticated at all
public class OrdersController : ControllerBase
{
    [HttpGet("{id}")]
    public async Task<IActionResult> Get(int id)
    {
        var order = await _db.Orders.FindAsync(id);
        // Authorization: does THIS identity own THIS resource? (the missing check)
        if (order is null || order.UserId != User.GetId())
            return Forbid();                 // 403 — authenticated, not permitted
        return Ok(order);
    }
}

Key takeaways

🎯 Remember these three

  • Authentication = who you are (login); authorization = what you're allowed to do (permissions) — authn first, then authz.
  • Failure codes: 401 (not authenticated) vs 403 (authenticated but forbidden).
  • Most real breaches are authorization failures — check per-resource ownership, not just 'is logged in.'

Interview tip

🗣️ When this comes up

Define both in one breath ('who you are vs what you can do') and map them to 401 vs 403. The depth signal: most vulnerabilities are missing authorization (per-object) checks, not authentication.
