Encryption Basics

Encryption makes data unreadable without a key. The two families — symmetric (one shared key) and asymmetric (a key pair) — solve different problems and are usually combined.

#security #encryption #tls #cryptography

The idea in plain English

Symmetric encryption uses one secret key to both encrypt and decrypt (AES is the standard). It's fast and ideal for bulk data — but both parties need the same key, which raises the hard question: how do you share that key securely over an untrusted network?

Asymmetric encryption solves key distribution with a key pair: a public key (shareable) and a private key (secret). Anything encrypted with the public key can only be decrypted with the private key, so anyone can send you a secret without a pre-shared key. It's slow, so it's used for small things — like exchanging a symmetric key, or signatures.

TLS (HTTPS) combines them perfectly: the handshake uses asymmetric crypto to authenticate the server (via its certificate) and agree on a fresh symmetric session key, then all the actual traffic is encrypted with fast symmetric crypto. Best of both worlds.

Two scopes to keep distinct: in transit (TLS protects data on the wire) and at rest (disk/database encryption protects stored data if media is stolen). And the cardinal rule: don't roll your own crypto — use vetted libraries and standard algorithms; subtle mistakes break security completely.

Real-world example

🏢 How this works in production

Every HTTPS connection is this hybrid: your browser uses the site's certificate (asymmetric) to verify identity and negotiate an AES session key (symmetric) for the page data. Databases and cloud storage add encryption at rest so a stolen disk or backup is useless. The recurring security failure isn't broken AES — it's home-grown crypto, reused keys, or missing TLS, which is why 'use TLS, use a library, don't invent algorithms' is the practical advice.

Asymmetric to share a key, symmetric for the data

In C#

C# 
// Use the platform's vetted primitives — never hand-rolled crypto.
// Symmetric (bulk data): AES via the framework.
using var aes = Aes.Create();                // strong defaults, random IV
aes.Key = key;                               // 256-bit key from a KDF / vault

// Asymmetric (key exchange / signatures): RSA or ECDSA.
using var rsa = RSA.Create(2048);

// In practice you rarely touch these directly — TLS handles transit,
// and a KMS/key vault manages keys for encryption at rest.

Key takeaways

🎯 Remember these three

  • Symmetric (one shared key, e.g. AES) is fast for bulk data; asymmetric (public/private pair) solves key distribution and identity.
  • TLS combines them: asymmetric handshake to exchange a symmetric session key.
  • Encrypt in transit (TLS) and at rest; never roll your own crypto — use vetted libraries.

Interview tip

🗣️ When this comes up

Contrast symmetric (fast, shared key) and asymmetric (key pair, solves distribution), then explain TLS as the hybrid. Add 'in transit vs at rest' and 'don't roll your own crypto' to round out a complete, practical answer.
