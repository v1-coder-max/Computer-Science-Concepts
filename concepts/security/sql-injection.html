SQL Injection

SQL injection happens when user input is glued into a query as code instead of data. The fix is old, simple, and absolute: parameterized queries.

#security #sql-injection #owasp #validation

The idea in plain English

An injection attack tricks an interpreter into running attacker-supplied input as commands. With SQL injection, a value from the user is concatenated straight into a SQL string, so input like ' OR '1'='1 changes the query's meaning — bypassing a login, dumping a table, or (with ; DROP TABLE) destroying data.

The root cause is mixing code and data in the same string. The database can't tell which characters were meant as a literal value and which as SQL syntax, because you handed it one blended string.

The fix is parameterized queries (prepared statements): you send the SQL with placeholders (@id) and the values separately. The database treats parameters strictly as data — never as executable SQL — so no input can alter the query structure. ORMs like EF Core parameterize by default, which is why using them correctly is itself a defense.

Escaping input or blocklisting bad words is not a reliable fix — attackers find encodings you missed. Parameterize everything; additionally apply least-privilege database accounts (so a breach can't drop tables) and validate input for good measure. But parameterization is the one that actually closes the hole.

Real-world example

🏢 How this works in production

SQL injection has topped vulnerability lists for two decades and still causes breaches — the famous ones dump entire user tables through a single unsanitized search box or login form. It's also entirely preventable: every mainstream data library supports parameters, and ORMs parameterize automatically. The breaches almost always trace back to one place where someone built a query by string concatenation 'just this once.'

Code vs data: concatenation vs parameters

In C#

C# 
// ❌ VULNERABLE — input becomes part of the SQL.
var sql = "SELECT * FROM Users WHERE Name = '" + name + "'";   // ' OR '1'='1

// ✅ Parameterized — the value can never change the query structure.
using var cmd = new SqlCommand(
    "SELECT * FROM Users WHERE Name = @name", conn);
cmd.Parameters.AddWithValue("@name", name);    // bound as DATA, not code

// ✅ EF Core parameterizes automatically:
db.Users.Where(u => u.Name == name);           // safe by construction

Key takeaways

🎯 Remember these three

  • SQL injection = untrusted input concatenated into a query, so it executes as code, not data.
  • The fix is parameterized queries / prepared statements — they bind input strictly as data; ORMs do this by default.
  • Escaping/blocklists are not reliable; add least-privilege DB accounts and input validation as defense in depth.

Interview tip

🗣️ When this comes up

Explain it as confusing code with data, then give the single correct fix: parameterized queries (not escaping). Noting that ORMs parameterize by default — and that the bug appears wherever someone hand-builds SQL — shows practical grounding.
