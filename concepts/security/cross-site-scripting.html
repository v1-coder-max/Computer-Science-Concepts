Cross-Site Scripting (XSS)

XSS is SQL injection's front-end cousin: untrusted input rendered into a page as markup/script runs the attacker's JavaScript in your users' browsers.

#security #xss #owasp #frontend

The idea in plain English

Cross-Site Scripting (XSS) occurs when an app includes untrusted data in a web page without properly encoding it, so the browser executes attacker-supplied <script> as if it were yours. That script runs with the user's session — it can steal cookies/tokens, log keystrokes, make requests as the user, or rewrite the page.

There are three flavors. Stored XSS: the payload is saved (a comment, a profile field) and served to everyone who views it — the most dangerous. Reflected XSS: the payload is in the request (a URL parameter) and echoed back in the response, triggered via a crafted link. DOM-based XSS: client-side JavaScript writes untrusted input into the DOM unsafely, with no server involved.

The primary defense is output encoding: when you render user data, escape it for its context (HTML, attribute, JavaScript, URL) so <script> becomes inert text. Modern frameworks (React, Razor, Angular) auto-encode by default — most XSS today comes from deliberately bypassing that (e.g. dangerouslySetInnerHTML, @Html.Raw).

Add defense in depth: a Content Security Policy that blocks inline and unknown-origin scripts, sanitize any HTML you must allow (a vetted library, never a regex), and mark session cookies HttpOnly so a successful XSS still can't read the token.

Real-world example

🏢 How this works in production

XSS is a perennial OWASP Top 10 entry. A single stored XSS in a comment field or username can compromise every visitor to that page — which is why platforms invest heavily in auto-encoding frameworks and strict CSPs. The bugs that slip through are the deliberate escape hatches: raw HTML insertion of user content, or client-side code that builds the DOM from URL fragments without encoding.

Three ways the script gets in

In C#

JavaScript 
// ❌ Injecting raw user content → XSS.
element.innerHTML = userComment;          // a &lt;script&gt; here executes
// @Html.Raw(userInput)  /  dangerouslySetInnerHTML — same danger.

// ✅ Encode for context (frameworks do this by default).
element.textContent = userComment;        // rendered as inert text
// Razor: @userInput  (auto HTML-encoded)

// ✅ Defense in depth: a Content Security Policy header.
// Content-Security-Policy: default-src 'self'; script-src 'self'
// + sanitize allowed HTML with a vetted library, + HttpOnly cookies.

Key takeaways

🎯 Remember these three

  • XSS = untrusted input rendered as script, running the attacker's JS in the victim's browser (session theft, actions-as-user).
  • Three types: stored (worst), reflected, DOM-based.
  • Defend with output encoding (frameworks auto-encode), a CSP, sanitizing allowed HTML, and HttpOnly cookies.

Interview tip

🗣️ When this comes up

Define XSS (attacker JS in your page), name the three types, and lead the defense with context-aware output encoding. Mentioning that frameworks auto-encode and most XSS comes from bypassing it — plus CSP and HttpOnly as depth — is the strong answer.
