The Middleware Pipeline

Every ASP.NET Core request flows through an ordered chain of middleware — each one can act on the way in, the way out, or stop the request entirely.

#middleware #aspnetcore #pipeline #request

The idea in plain English

ASP.NET Core doesn’t hand a request straight to your controller. It runs the request through a pipeline you assemble in Program.cs: a sequence of middleware components. Each one receives the request, optionally does something, and calls next() to pass control to the component behind it. When the inner components finish, control unwinds back out — so each middleware can also touch the response on the way back.

That’s the “onion” model: the request travels inward through each layer to a terminal endpoint (your controller / minimal API), then the response travels back outward through the same layers in reverse.

Two consequences you must internalize:

  • Order matters. Put exception handling first (so it wraps everything), authentication before authorization, and routing before endpoints.
  • Middleware can short-circuit. If it doesn’t call next() — say auth rejects the request — the rest of the pipeline never runs.

Real-world example

🏢 How this works in production

A typical API request to a service like GitHub’s passes through layers of cross-cutting concerns before any business code runs: a request-logging/correlation-ID middleware, HTTPS redirection, rate limiting, CORS, authentication, then authorization — and only then the endpoint. Each is a reusable middleware, decoupled from the controllers. If rate limiting decides you’re over budget, it short-circuits with 429 and the expensive downstream work never executes.

The onion: in, then back out

In C#

The pipeline is just an ordered list of Use… calls — order is the program:

C# — Program.cs 
var app = builder.Build();

app.UseExceptionHandler("/error"); // outermost: wraps everything below
app.UseHttpsRedirection();
app.UseStaticFiles();
app.UseRouting();
app.UseAuthentication();           // must come before authorization
app.UseAuthorization();

// Inline middleware: do work, call next, then touch the response on the way out.
app.Use(async (ctx, next) =>
{
    var sw = Stopwatch.StartNew();
    await next();                  // hand off to the rest of the pipeline
    ctx.Response.Headers["X-Elapsed-ms"] = sw.ElapsedMilliseconds.ToString();
});

app.MapControllers();              // terminal: the endpoint
app.Run();

Reusable middleware is usually a small class with an InvokeAsync method:

C# 
public class CorrelationIdMiddleware(RequestDelegate next)
{
    public async Task InvokeAsync(HttpContext ctx)
    {
        ctx.Response.Headers["X-Correlation-ID"] =
            ctx.Request.Headers["X-Correlation-ID"].FirstOrDefault() ?? Guid.NewGuid().ToString();
        await next(ctx);           // omit this call to short-circuit the request
    }
}
// app.UseMiddleware<CorrelationIdMiddleware>();

⚠️ Order bugs are silent

Putting UseAuthorization before UseAuthentication, or anything after UseRouting that needs to run before it, fails quietly — requests just behave wrong. When something “isn’t being authenticated/CORS’d,” suspect pipeline order first.

Key takeaways

🎯 Remember these three

  • A request flows through ordered middleware (the onion): inward to the endpoint, then back out through each layer.
  • Order matters — exception handling first, auth before authz, routing before endpoints.
  • Middleware that skips next() short-circuits the request (e.g. auth rejection, rate limiting).
