Minimal APIs

Define HTTP endpoints as lambdas right on the app — no controllers, no attributes, far less ceremony. Ideal for small services.

#minimal-apis #endpoints #aspnetcore #microservices

The idea in plain English

Classic ASP.NET Core MVC needs a controller class, base types, and attributes to expose a route. Minimal APIs (introduced in .NET 6, matured since) cut that to a single call: app.MapGet("/products/{id}", handler). The handler is a plain lambda or method, and the framework figures out the rest.

The clever part is parameter binding. For each parameter the framework infers the source: route values (matching the template), query string, the request body (for complex types, deserialized from JSON), and — crucially — the DI container for your services. You return a value (serialized to JSON) or an IResult (Results.Ok, Results.NotFound, Results.Created) to control the status code.

You still get the full pipeline (auth, middleware), plus route groups (MapGroup) to share a prefix and policies, and endpoint filters for cross-cutting logic like validation. Reach for controllers when you want their richer conventions (model binding attributes, filters, large surface area); reach for minimal APIs for lean microservices and lower startup overhead.

Real-world example

🏢 How this works in production

A focused microservice — say a URL-shortener or a pricing service with a handful of endpoints — is a perfect fit. There’s no need for the MVC machinery; a dozen Map… calls grouped under /api/v1 give a clean, fast service with minimal boilerplate and a smaller, quicker-to-start app. Teams building lots of small services lean on minimal APIs precisely because there’s less to read and less to boot.

From route to result

In C#

C# — Program.cs 
var app = builder.Build();

// Group shares a prefix + policies + filters.
var products = app.MapGroup("/api/v1/products").RequireAuthorization();

// id ← route, catalog ← DI; return an IResult to control the status code.
products.MapGet("/{id:int}", async (int id, ICatalog catalog) =>
{
    Product? p = await catalog.FindAsync(id);
    return p is null ? Results.NotFound() : Results.Ok(p);
});

// Complex type ← JSON body; returns 201 Created with a Location header.
products.MapPost("/", async (CreateProduct dto, ICatalog catalog) =>
{
    int id = await catalog.AddAsync(dto);
    return Results.Created($"/api/v1/products/{id}", new { id });
})
.AddEndpointFilter<ValidationFilter<CreateProduct>>(); // cross-cutting validation

app.Run();

Key takeaways

🎯 Remember these three

  • Minimal APIs define endpoints as lambdas via MapGet/MapPost — no controllers, minimal boilerplate.
  • Parameters bind automatically from route, query, JSON body, and DI; return a value or an IResult for the status code.
  • Use MapGroup for shared prefixes/policies and endpoint filters for cross-cutting concerns; pick controllers when you want MVC’s richer conventions.

Interview tip

🗣️ When this comes up

“Minimal APIs vs controllers — when would you choose each?” Minimal APIs for small, focused services (less ceremony, faster startup); controllers when you want MVC conventions, filters, and a large, structured surface. Show you understand parameter binding from DI — it’s the part that surprises people.
