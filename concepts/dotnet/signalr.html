SignalR

Real-time, two-way communication for .NET: the server can push to clients the instant something happens — no polling.

#signalr #real-time #websockets #hubs

The idea in plain English

Plain HTTP is request/response — the client always asks first. For live features (chat, notifications, dashboards) you want the server to push. SignalR is the .NET library that makes that easy. Its central abstraction is the Hub: an RPC-style class where clients call server methods and the server calls client methods by name.

Under the hood SignalR negotiates the best transport available and falls back gracefully:

  • WebSockets — a persistent, full-duplex connection. The default and best option.
  • Server-Sent Events — server→client streaming when WebSockets aren’t available.
  • Long polling — the universal fallback.

It also gives you groups (broadcast to a subset, e.g. everyone in a chat room), stable per-client connection IDs, and automatic reconnection. To run across multiple servers you add a backplane (Redis), so a message published on server A reaches clients connected to server B.

Real-world example

🏢 How this works in production

A Slack-style chat or a live ops dashboard is the canonical use. When a teammate sends a message, the server pushes it to everyone in that channel’s group over their open WebSocket — milliseconds, no refresh. Because the app runs on many instances behind a load balancer, a Redis backplane fans each message out to all servers so a user on instance 7 sees the message published from instance 2. Stock tickers, sports scores, and collaborative editors work the same way.

Hubs, groups, and a backplane

In C#

C# 
public class ChatHub : Hub
{
    // Clients call this; the server pushes to others by calling a CLIENT method.
    public async Task SendToRoom(string room, string user, string text)
    {
        await Clients.Group(room).SendAsync("messageReceived", user, text);
    }

    public Task JoinRoom(string room) =>
        Groups.AddToGroupAsync(Context.ConnectionId, room); // ConnectionId = this client
}

// Program.cs
builder.Services.AddSignalR()
    .AddStackExchangeRedis("localhost:6379"); // backplane for multi-server scale-out

var app = builder.Build();
app.MapHub<ChatHub>("/hubs/chat");

A browser client subscribes to the server’s pushes and calls hub methods:

JavaScript 
const conn = new signalR.HubConnectionBuilder().withUrl("/hubs/chat").build();
conn.on("messageReceived", (user, text) => render(user, text)); // server → client push
await conn.start();
await conn.invoke("JoinRoom", "general");
await conn.invoke("SendToRoom", "general", "vishal", "hello!");

Key takeaways

🎯 Remember these three

  • SignalR gives real-time, bidirectional push via Hubs (RPC both ways) over a persistent connection.
  • It negotiates the best transport (WebSockets → SSE → long polling) automatically.
  • Scaling to multiple servers requires a backplane (Redis) so messages reach clients on every instance.

Interview tip

🗣️ When this comes up

“How would you build live notifications/chat?” Name SignalR and that it prefers WebSockets with graceful fallback. The senior follow-up is scale: when you mention multiple servers, immediately bring up the Redis backplane and sticky-session/affinity considerations behind a load balancer.
