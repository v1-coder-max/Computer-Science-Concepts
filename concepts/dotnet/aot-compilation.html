Native AOT

Compile the whole app to a native binary at build time — no JIT, no IL at runtime. Instant startup and tiny memory, in exchange for some flexibility.

#aot #native #startup #trimming

The idea in plain English

Normally a .NET app ships IL and the JIT compiles it to native code on first use (see the JIT page). Native AOT (Ahead-Of-Time) does all that compilation before you ship: the toolchain produces a single, self-contained native executable with the runtime and your code baked in. There’s no IL and no JIT at runtime — the OS just runs machine code.

What you gain:

  • Near-instant startup — nothing to JIT, so cold start drops dramatically.
  • Lower memory and a smaller, dependency-free deployable (no shared runtime needed).

What you give up — because there’s no runtime code generation:

  • Limited reflection and no Reflection.Emit / runtime IL.
  • Trimming is mandatory: unused code is cut to shrink the binary, which breaks libraries that reflect over types not seen at build time.
  • Builds are platform-specific (you compile per OS/architecture) and take longer.

This is why source generators matter for AOT: they replace reflection-based JSON, regex, and DI wiring with build-time code that survives trimming. Native AOT is opt-in and best for specific workloads — not every app needs (or can use) it.

Real-world example

🏢 How this works in production

Serverless functions (AWS Lambda, Azure Functions) and autoscaling containers live and die by cold-start time and memory footprint — you pay for both, and users feel the latency on a cold invocation. A trimmed Native AOT build of a small ASP.NET Core API or a CLI tool starts in single-digit milliseconds and ships as one small binary with no runtime to install, which is exactly what a “scale to zero” function or a distributed CLI wants. Heavy, reflection-laden apps (some ORMs, plugin hosts) stay on the JIT.

JIT vs AOT

In code

XML — .csproj 
<PropertyGroup>
  <PublishAot>true</PublishAot>          <!-- opt in to Native AOT -->
  <InvariantGlobalization>true</InvariantGlobalization>
</PropertyGroup>
Bash 
# Produces one self-contained native binary for the target platform.
dotnet publish -c Release -r linux-x64

# The output is an executable with no .NET runtime dependency — just run it.
./bin/Release/net8.0/linux-x64/publish/myapp

⚠️ Reflection-heavy code won’t survive trimming

Anything that reflects over types the trimmer can’t see at build time (some serializers, DI auto-scanning, dynamic plugin loading) can fail at runtime under AOT. The fix is to use source-generated equivalents (e.g. System.Text.Json source-gen) and heed the AOT/trim analyzer warnings — don’t flip PublishAot on blindly.

Key takeaways

🎯 Remember these three

  • Native AOT compiles to a self-contained native binary at build — no JIT, instant startup, low memory.
  • The cost: limited reflection, mandatory trimming, and platform-specific builds.
  • Use source generators to replace reflection; ideal for serverless, containers, and CLIs — not reflection-heavy apps.

Interview tip

🗣️ When this comes up

“JIT vs AOT — when would you choose Native AOT?” AOT for fast cold start and small footprint (serverless, containers, CLIs); accept the loss of runtime reflection and the trimming requirement. Tie it together: source generators are what make modern libraries AOT-compatible by removing reflection.
