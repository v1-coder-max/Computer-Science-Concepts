Assemblies & Metadata

An assembly is the unit of deployment and versioning in .NET — a .dll or .exe that bundles IL, metadata, and a manifest.

#assemblies #metadata #reflection #manifest

The idea in plain English

When you build a project, the output isn’t loose machine code — it’s an assembly: a single file (usually a .dll) that contains four things:

  • IL — the compiled instructions for every method.
  • Metadata — a complete, structured description of every type, member, and signature. This is why .NET needs no header files and why reflection, IntelliSense, and serializers can inspect your types at runtime.
  • Manifest — the assembly’s identity (name, version, culture, optional public key) and the list of other assemblies it references.
  • Resources — embedded files like images or localized strings.

The assembly is also the boundary for versioning and access: internal members are visible within the same assembly. At runtime, the CLR loads assemblies on demand into an AssemblyLoadContext (ALC) — the modern .NET replacement for the old AppDomain — which is how plugins and isolated load/unload scenarios work.

Real-world example

🏢 How this works in production

Editors like Visual Studio and CI tools load plugins as assemblies at runtime: each extension ships as a .dll, discovered and loaded into its own AssemblyLoadContext so it can be isolated (and even unloaded) without restarting the host. The same mechanism powers the [Authorize]-style attribute scanning, MVC controller discovery, and source generators — all of which lean on assembly metadata to find types and attributes without you wiring them up by hand.

What’s inside an assembly

In C# — reading metadata via reflection

Because the description travels with the code, you can inspect an assembly at runtime:

C# 
using System.Reflection;

Assembly asm = Assembly.GetExecutingAssembly();
AssemblyName id = asm.GetName();
Console.WriteLine($"{id.Name} v{id.Version}");      // identity from the manifest

// Metadata lets us discover types without knowing them at compile time:
foreach (Type t in asm.GetTypes().Where(t => t.IsClass))
    Console.WriteLine(t.FullName);

// Load a plugin assembly into its own isolated, unloadable context:
var ctx = new System.Runtime.Loader.AssemblyLoadContext("plugin", isCollectible: true);
Assembly plugin = ctx.LoadFromAssemblyPath(@"C:\plugins\Acme.Reporting.dll");
// ... use it via an interface, then ctx.Unload();

Strong naming (a public/private key in the manifest) gives an assembly a unique identity; NuGet packages are just the distribution wrapper around one or more assemblies plus their dependency metadata.

Key takeaways

🎯 Remember these three

  • An assembly (.dll/.exe) = IL + metadata + manifest; it’s the unit of deployment, versioning, and internal access.
  • Metadata is self-describing, which is what makes reflection, serialization, and DI scanning possible — no header files needed.
  • The CLR loads assemblies into an AssemblyLoadContext (the modern AppDomain), enabling plugins and isolated unload.
