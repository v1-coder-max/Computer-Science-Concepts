Records

A record is a type built for data: value-based equality, a concise declaration, and easy immutability — all generated by the compiler.

#records #immutability #value-equality #dto

The idea in plain English

A normal class uses reference equality: two objects are equal only if they’re the same instance. For data — a DTO, a coordinate, a money amount — that’s usually the wrong default; you want two objects with the same values to be equal. A record gives you exactly that. From a one-line declaration the compiler generates Equals, GetHashCode, ToString, a Deconstruct, and a copy constructor — all based on the record’s properties.

Records are designed around immutability:

  • Positional properties are init-only — settable when constructing, read-only after.
  • To “change” one, you don’t mutate it; you make a modified copy with a with expression: order with { Status = "Shipped" }.

record (or record class) is a reference type; record struct is a value type with the same conveniences. Reach for records for immutable data carriers; keep class for things with identity and behavior (services, entities with a lifecycle).

Real-world example

🏢 How this works in production

In event-driven systems (think the events flowing through a Kafka pipeline or a CQRS app), events and commands are immutable data — once OrderPlaced happened, it never changes. Modeling them as records gives free value equality (handy for deduplication and tests), safe sharing across threads (nothing can mutate them), and clean with-based transformations. The same applies to API DTOs and domain value objects like Money(decimal Amount, string Currency).

Value equality & non-destructive copy

In C#

C# 
// Positional record: one line generates ctor, init-only props, Equals,
// GetHashCode, ToString, and Deconstruct.
public record Person(string First, string Last, int Age);

var a = new Person("Ada", "Lovelace", 36);
var b = new Person("Ada", "Lovelace", 36);

Console.WriteLine(a == b);          // True  — value equality, not reference
Console.WriteLine(a);               // Person { First = Ada, Last = Lovelace, Age = 36 }

// Non-destructive mutation: 'a' is untouched, you get a new copy.
var older = a with { Age = 37 };

// Deconstruction falls out for free:
var (first, last, age) = a;

// A struct variant when you want value-type semantics + the conveniences:
public readonly record struct Money(decimal Amount, string Currency);

⚠️ Value equality is shallow

Generated equality compares each member with its own Equals. If a record holds a mutable reference (like a List<T>), equality compares the references, not the contents — two records with different list instances of equal items won’t be equal. Keep records made of immutable members to avoid surprises.

Key takeaways

🎯 Remember these three

  • Records give compiler-generated value equality plus ToString/Deconstruct/copy from a concise declaration.
  • They’re built for immutable data: init-only props and non-destructive with copies.
  • Use records for DTOs/value objects/events; use class for entities with identity and behavior.

Interview tip

🗣️ When this comes up

“When would you use a record instead of a class?” Records for immutable data where value equality is what you want (DTOs, value objects, events); classes for objects with identity/behavior. Mention with expressions and that equality is value-based but shallow — the nuance that shows you’ve actually used them.
