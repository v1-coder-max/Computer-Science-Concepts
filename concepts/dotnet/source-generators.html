Source Generators

Code that writes code at compile time: a generator inspects your program via Roslyn and emits extra C# that’s compiled alongside it — no runtime reflection.

#source-generators #roslyn #compile-time #performance

The idea in plain English

A source generator is a compiler plug-in. During the build, Roslyn hands it your syntax trees and semantic model; it looks for things it cares about (a marker attribute, a partial method) and adds new C# source to the compilation. That generated code is compiled right along with yours — you can call into it as if you’d written it.

Why bother? It replaces work that used to happen at runtime via reflection. Building a JSON serializer or a regex by reflecting over types is flexible but slow (and pulls in metadata that breaks under trimming/AOT). A generator does that analysis once, at build time, and emits straight-line code. The payoff: faster startup, lower allocations, and AOT compatibility. The .NET BCL itself ships generators for System.Text.Json, [GeneratedRegex], and high-performance logging (LoggerMessage).

Modern ones are incremental generators: they cache and only re-run the affected part when you edit, so they don’t slow the IDE. Generators are purely additive — they can add code, never modify or delete your existing source.

Real-world example

🏢 How this works in production

A serverless function or a trimmed container that must start in milliseconds can’t afford reflection-based JSON. Switching System.Text.Json to its source-generated mode emits a bespoke serializer for your DTOs at build time — measurably faster first-call serialization, fewer allocations, and it survives Native AOT trimming (which strips the metadata reflection would need). The same pattern ([GeneratedRegex], generated logging) is how high-throughput .NET services shave startup and steady-state cost.

Compile time, not run time

In C# — consuming built-in generators

C# 
// 1) Regex: the engine is generated at build time into the partial method —
//    no runtime parsing/compiling of the pattern.
public partial class Validators
{
    [GeneratedRegex(@"^\d{3}-\d{4}$")]
    public static partial Regex PhoneRegex();
}
bool ok = Validators.PhoneRegex().IsMatch("123-4567");

// 2) System.Text.Json source generation: a serializer is emitted for these
//    types, so no reflection at runtime (AOT- and trim-friendly).
[JsonSerializable(typeof(Order))]
public partial class AppJsonContext : JsonSerializerContext { }

string json = JsonSerializer.Serialize(order, AppJsonContext.Default.Order);

// 3) High-performance logging: the LoggerMessage source generator emits
//    an allocation-free, strongly-typed log method.
public static partial class Log
{
    [LoggerMessage(Level = LogLevel.Information, Message = "Order {OrderId} placed")]
    public static partial void OrderPlaced(ILogger logger, int orderId);
}

Key takeaways

🎯 Remember these three

  • Source generators inspect your code at compile time and emit additional C# compiled with it — purely additive.
  • They replace runtime reflection (JSON, regex, logging), giving faster startup, fewer allocations, and AOT/trim safety.
  • Incremental generators cache their work so they don’t bog down the IDE.

Interview tip

🗣️ When this comes up

“How do source generators differ from reflection?” Reflection inspects types at runtime (flexible, slower, AOT-hostile); a source generator does the work at compile time and emits concrete code (fast, allocation-free, AOT-friendly). Cite System.Text.Json source-gen and [GeneratedRegex] as the canonical examples.
