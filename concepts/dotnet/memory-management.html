Memory Management

Where your data lives — stack or heap, value or reference — decides how fast it is and how much pressure it puts on the GC.

#memory #stack-vs-heap #boxing #allocations

The idea in plain English

.NET has two places to put data. The stack is a fast, per-thread scratchpad that grows and shrinks with method calls — allocation is a pointer move and cleanup is automatic when the method returns. The heap is the shared, GC-managed pool where objects live until they’re collected.

Which one is used depends on the type:

  • Value types (int, bool, struct, enum) hold their data inline. A local value type sits on the stack; a value-type field lives wherever its container lives.
  • Reference types (class, string, arrays, delegates) live on the heap; the variable holds a reference (a pointer) to it.

That distinction drives copy semantics (values copy the data; references copy the pointer) and the cost of boxing — wrapping a value type in a heap object so it can be treated as object. Boxing allocates and pressures the GC, and it’s easy to do by accident.

Real-world example

🏢 How this works in production

Kestrel — the web server inside ASP.NET Core that powers high-traffic sites — is engineered to allocate almost nothing on the request hot path. It parses HTTP using Span<T> over pooled buffers (ArrayPool, System.IO.Pipelines) instead of slicing strings, so millions of requests don’t bury the GC in gen-0 garbage. Less allocation → fewer collections → lower, more predictable tail latency.

Stack, heap, and boxing

In C# — copy semantics, boxing, low-alloc tools

C# 
// Value semantics: 'b' is an independent copy.
struct Point { public int X, Y; }
var a = new Point { X = 1, Y = 2 };
var b = a;            // copies the data
b.X = 99;             // a.X is still 1

// Accidental boxing: the int is wrapped in a heap object to become 'object'.
int n = 42;
object boxed = n;     // ← allocation
int back = (int)boxed; // unboxing (a copy back out)

// Avoid allocations on hot paths:
Span<byte> buffer = stackalloc byte[256];   // on the stack, no GC involvement
byte[] rented = ArrayPool<byte>.Shared.Rent(4096); // reuse instead of new[]
try { /* ... use rented ... */ }
finally { ArrayPool<byte>.Shared.Return(rented); }

⚠️ “Value type = stack” is a myth

It’s not that simple. A value type that’s a field of a class lives on the heap with that object; a captured local can be hoisted into a closure object. The reliable rule is about type semantics (copy vs reference), not a fixed memory location.

Key takeaways

🎯 Remember these three

  • Value types copy their data and are often stack-allocated; reference types live on the heap and copy a pointer.
  • Boxing turns a value type into a heap allocation — avoid it on hot paths (watch object/non-generic APIs).
  • Cut allocations with struct, Span<T>/stackalloc, and ArrayPool<T> to lighten the GC.
