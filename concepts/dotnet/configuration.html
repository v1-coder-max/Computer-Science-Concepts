The Configuration System

.NET merges settings from many sources into one IConfiguration — later sources override earlier ones — and binds them to strongly-typed classes.

#configuration #options-pattern #appsettings #environments

The idea in plain English

Configuration in .NET is a layered key/value system. At startup the host stacks several providers in order, and reading a key returns the value from the last provider that supplied it. The default stack is roughly:

  1. appsettings.json (base)
  2. appsettings.{Environment}.json (e.g. .Production.json)
  3. User secrets (Development only)
  4. Environment variables
  5. Command-line arguments

So the same build runs in dev, staging, and prod by overriding a few keys via the environment — the 12-factor approach, no recompile. Keys are hierarchical ("Logging:LogLevel:Default"); in environment variables the separator is a double underscore (Logging__LogLevel__Default).

Reading raw strings everywhere is brittle, so .NET gives you the Options pattern: bind a config section to a typed class and inject it as IOptions<T>. Secrets (connection strings, API keys) should come from environment variables or a vault — never be committed in appsettings.json.

Real-world example

🏢 How this works in production

A service deployed across environments keeps non-secret defaults in appsettings.json, environment-specific overrides in appsettings.Staging.json / .Production.json, and pulls secrets at runtime from Azure Key Vault or environment variables injected by Kubernetes. The app code just reads IOptions<StripeOptions> — it has no idea whether the API key came from a file in dev or a vault in prod. That separation is what lets the identical container image ship to every environment.

Sources layered into one view

In C# — the Options pattern

JSON — appsettings.json 
{
  "Stripe": {
    "ApiBase": "https://api.stripe.com",
    "TimeoutSeconds": 30
  }
}
C# 
public class StripeOptions
{
    public string ApiBase { get; set; } = "";
    public int TimeoutSeconds { get; set; }
}

// Bind the "Stripe" section to the typed class:
builder.Services.Configure<StripeOptions>(builder.Configuration.GetSection("Stripe"));

// Inject it anywhere — no magic strings:
public class StripeClient(IOptions<StripeOptions> options)
{
    private readonly StripeOptions _cfg = options.Value;
    // use _cfg.ApiBase, _cfg.TimeoutSeconds
}

// Reload-aware variants:
//   IOptionsSnapshot<T>  → re-read per request (scoped)
//   IOptionsMonitor<T>   → live updates + change callbacks (singleton-friendly)

⚠️ Don’t commit secrets to appsettings.json

Connection strings and API keys belong in environment variables, user secrets (dev), or a vault — not in a JSON file in source control. Use IOptionsSnapshot/ IOptionsMonitor when values can change at runtime; plain IOptions is a singleton captured at startup.

Key takeaways

🎯 Remember these three

  • Config is layered providers merged into one IConfiguration; later sources (env vars, CLI) override earlier (JSON files).
  • The Options pattern binds a section to a typed class injected as IOptions<T> — no magic strings.
  • IOptions = startup snapshot; IOptionsSnapshot = per request; IOptionsMonitor = live reload.

Interview tip

🗣️ When this comes up

“How does configuration precedence work, and how do you handle secrets?” Explain the layered providers (JSON → env-specific → env vars → CLI) and that the last writer wins, then name the Options pattern and a vault for secrets. Knowing IOptions vs IOptionsSnapshot vs IOptionsMonitor seals it.
