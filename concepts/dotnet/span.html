Span<T> & Memory<T>

A Span<T> is a type-safe window over memory you already have — slice and read it without allocating or copying.

#span #performance #zero-allocation #ref-struct

The idea in plain English

Before Span<T>, working with “part of” a buffer usually meant copying: Substring, Array.Copy, a new array. Each copy is an allocation the GC later has to clean up. Span<T> removes the copy. It’s a tiny value (a pointer + a length) that points into existing contiguous memory — a managed array, a stackalloc block, or even unmanaged memory — and lets you read/write it with bounds checking.

Slicing is free. span.Slice(10, 4) doesn’t copy four elements; it returns another span pointing at the same memory with a new offset and length. That’s why high-performance parsing chops a buffer into fields with zero allocations.

The catch is the safety mechanism: Span<T> is a ref struct, so it must live on the stack. That means you can’t store it in a class field, capture it in a lambda, use it across an await, or box it. When you need those (especially async), reach for its heap-friendly sibling, Memory<T>, and get a span from it (memory.Span) at the point of use.

Real-world example

🏢 How this works in production

System.Text.Json and Kestrel are built on spans. When Kestrel parses an incoming HTTP request, it slices the raw byte buffer into method, path, and headers as ReadOnlySpan<byte> windows — no intermediate strings, no garbage — which is a big part of how ASP.NET Core handles millions of requests with minimal GC pressure. Any time you replace Substring in a hot loop with AsSpan().Slice(), you’re doing the same thing at small scale.

A window, not a copy

In C#

C# 
// Slice without copying:
int[] data = { 10, 20, 30, 40, 50 };
Span<int> middle = data.AsSpan(1, 3);  // {20,30,40} — a window, not a copy
middle[0] = 99;                         // writes through: data[1] is now 99

// Stack buffer + span = zero heap allocation on a hot path:
Span<byte> tmp = stackalloc byte[16];

// Parse a substring with no Substring allocation:
ReadOnlySpan<char> date = "2026-06-06".AsSpan();
int year  = int.Parse(date.Slice(0, 4));
int month = int.Parse(date.Slice(5, 2));

// Need it across await / in a field? Use Memory<T> and get the span at use-site.
async Task ConsumeAsync(Memory<byte> buffer)
{
    await Task.Yield();
    Span<byte> span = buffer.Span;       // materialize the span here, after the await
    span[0] = 1;
}

⚠️ Span<T> is stack-only

Because it’s a ref struct, you can’t put a Span<T> in a class field, capture it in a lambda/closure, use it as a generic type argument, or hold it across an await or yield. The compiler enforces this. When any of those is needed, use Memory<T> (heap-friendly) and slice to a span only where you actually touch the bytes.

Key takeaways

🎯 Remember these three

  • Span<T> is a zero-copy, bounds-checked window over array/stack/unmanaged memory — slicing allocates nothing.
  • It’s a ref struct (stack-only): no fields, closures, or use across await.
  • For async or stored scenarios use Memory<T> and get its .Span at the point of use.

Interview tip

🗣️ When this comes up

“What is Span<T> and why does it matter?” Say it’s a no-copy view over contiguous memory that eliminates allocations in parsing/slicing hot paths. Then prove depth: it’s a ref struct, so stack-only — and that’s exactly why Memory<T> exists for async/stored cases.
