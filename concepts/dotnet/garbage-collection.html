Garbage Collection

The GC automatically frees objects you’re no longer using — so you never free(), but you do need to know how it decides what’s garbage.

#gc #memory #generations #idisposable

The idea in plain English

Every object you new lives on the managed heap. The GC periodically figures out which objects are still reachable — traceable from a root (a local variable, a static field, a CPU register) — and reclaims everything else. Unreachable means unusable, so it’s safe to collect.

The key insight that makes it fast is the generational hypothesis: most objects die young. So the heap is split into generations:

  • Gen 0 — brand-new, small objects. Collected very often and very cheaply.
  • Gen 1 — survived one GC; a buffer between short- and long-lived.
  • Gen 2 — long-lived objects (caches, singletons). Collected rarely because gen2 collections are expensive (they scan the whole heap).

Objects that survive a collection get promoted to the next generation. A collection does mark (find reachable), sweep (reclaim the rest), and compact (slide survivors together so allocation stays a cheap pointer bump). Very large objects (≥ 85,000 bytes) go on the separate Large Object Heap (LOH), which isn’t compacted by default.

Real-world example

🏢 How this works in production

A high-throughput Amazon-scale API allocates a storm of tiny, short-lived objects per request (strings, DTOs, JSON nodes). Because almost all of them die in gen 0, the GC clears them in sub-millisecond sweeps and rarely touches gen 2. These services run Server GC (a heap and a dedicated GC thread per core) for maximum throughput, and engineers cut allocations with pooling and Span<T> to keep gen-2 pauses out of their tail latency.

The generational heap

In C# — the part you control

The GC handles managed memory. It does not promptly release unmanaged resources (file handles, sockets, DB connections). For those, implement IDisposable and use using so cleanup is deterministic:

C# 
// 'using' calls Dispose() the moment the scope ends — deterministic cleanup,
// independent of when the GC next runs.
using (var stream = new FileStream("data.bin", FileMode.Open))
{
    // ... use the file handle (an unmanaged resource) ...
} // stream.Dispose() runs here, releasing the OS handle now

// C# 8+ 'using declaration': disposed at the end of the enclosing scope.
using var conn = new SqlConnection(cs);
await conn.OpenAsync();

// ❌ Don't do this in app code — you almost never need to force a collection:
// GC.Collect();  // hurts more than it helps; the GC self-tunes.

Want fewer collections? Allocate less. Reuse buffers with ArrayPool<T>, prefer structs and Span<T> on hot paths, and avoid boxing — covered in the memory-management page.

Key takeaways

🎯 Remember these three

  • The GC reclaims unreachable objects; it’s generational because most objects die young — gen 0 is cheap, gen 2 is costly.
  • The GC does not free unmanaged resources promptly — use IDisposable + using for those.
  • Reduce GC pressure by reducing allocations (pooling, Span<T>, structs), not by calling GC.Collect().
