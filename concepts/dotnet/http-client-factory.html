HttpClientFactory

HttpClient looks simple and is famously easy to misuse. The factory fixes both the “new it every time” bug and the “singleton forever” bug.

#httpclient #socket-exhaustion #resilience #polly

The idea in plain English

HttpClient is meant to be reused, but it sits on top of an HttpMessageHandler that owns the underlying TCP connections. That creates a trap with two jaws:

  • New one per request (using var c = new HttpClient()) → each disposed handler leaves sockets stuck in TIME_WAIT. Under load you hit socket exhaustion (SocketException).
  • One static singleton forever → it never picks up DNS changes (failovers, blue/green), happily sending traffic to an IP that moved.

IHttpClientFactory threads the needle. It maintains a pool of handlers and rotates them on a lifetime (default ~2 minutes): connections are reused for efficiency, but handlers are recycled often enough to respect DNS. On top of that it gives you named and typed clients (pre-configured base address, headers) and a clean place to plug in Polly resilience policies — retries, timeouts, circuit breakers.

Real-world example

🏢 How this works in production

A checkout service calling Stripe and a shipping API can’t afford SocketExceptions during a flash sale, nor can it keep hammering an instance that just failed over. With a typed client per downstream and a Polly retry + circuit-breaker policy attached, transient blips are retried with backoff, a struggling dependency is given breathing room by the open circuit, and connection reuse keeps latency low — all without the calling code knowing any of it happened.

Pooled, rotated handlers

In C# — a typed client with resilience

C# 
// A typed client encapsulates the base address, headers, and calls.
public class CatalogClient(HttpClient http)
{
    public Task<Product?> GetAsync(int id) =>
        http.GetFromJsonAsync<Product>($"/products/{id}");
}

// Register it + attach resilience (Microsoft.Extensions.Http.Resilience / Polly):
builder.Services.AddHttpClient<CatalogClient>(c =>
{
    c.BaseAddress = new Uri("https://catalog.internal");
    c.Timeout = TimeSpan.FromSeconds(10);
})
.AddStandardResilienceHandler(); // retries, timeout, circuit breaker out of the box

// Consumers just inject CatalogClient — no HttpClient lifecycle to worry about.

⚠️ Stop writing new HttpClient()

The per-request new HttpClient() in a using is the canonical bug — it exhausts sockets under load. Don’t replace it with one eternal static instance either (stale DNS). Inject a typed/named client from the factory and let it manage handler lifetime.

Key takeaways

🎯 Remember these three

  • new HttpClient() per request → socket exhaustion; one forever-singleton → stale DNS. The factory avoids both by pooling and rotating handlers.
  • Prefer typed clients (base address + calls in one class) injected via DI.
  • Attach Polly/standard resilience (retry, timeout, circuit breaker) at registration — transparent to callers.

Interview tip

🗣️ When this comes up

“What’s wrong with newing up an HttpClient per request?” Answer: socket exhaustion from handlers stuck in TIME_WAIT. Then show you know the other half — a permanent singleton ignores DNS changes — and that IHttpClientFactory solves both via handler pooling and rotation, with a natural home for Polly policies.
