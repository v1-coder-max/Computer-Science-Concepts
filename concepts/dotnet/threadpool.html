The Thread Pool

Creating threads is expensive, so .NET keeps a pool of reusable worker threads and feeds them a queue of short work items.

#thread-pool #starvation #scalability #concurrency

The idea in plain English

Spinning up an OS thread costs memory and time, and a thread that finishes a tiny job and dies has wasted most of that cost. The thread pool fixes this: it keeps a set of long-lived worker threads alive and hands them work items from a queue. When a worker finishes one item, it grabs the next instead of dying.

Almost everything concurrent rides this pool: Task.Run, async continuations, Timer callbacks, and I/O completions. It has two kinds of threads — general worker threads and I/O completion threads — and it self-tunes: a “hill-climbing” heuristic slowly injects more threads if the queue backs up and trims them when demand falls.

The catch is that injection is deliberately slow (roughly one new thread at a time). So if you block pool threads — e.g. calling .Result on async work — they can’t pick up new items, the queue grows faster than threads are added, and you get thread-pool starvation: latency spikes, then a stall.

Real-world example

🏢 How this works in production

ASP.NET Core dispatches every incoming request as a work item on the thread pool. Under a traffic spike, a service that does db.GetAsync().Result (sync-over-async) blocks pool threads waiting on I/O; new requests pile up in the queue while the pool adds threads one-per-~half-second. The symptom teams see on the dashboard is a sudden cliff in throughput and a climbing ThreadPool queue length — the classic starvation incident. Going fully async keeps those threads free and the pool small.

Queue in, workers out

In C#

C# 
// Queue work directly (rarely needed; Task.Run is the friendly wrapper):
ThreadPool.QueueUserWorkItem(_ => DoBackgroundWork());

// The everyday way to offload CPU work onto a pool thread:
await Task.Run(() => CrunchNumbers());

// Inspect the pool (useful when diagnosing starvation):
ThreadPool.GetAvailableThreads(out int worker, out int io);
ThreadPool.GetMinThreads(out int minWorker, out int minIo);
Console.WriteLine($"available workers: {worker}, min: {minWorker}");

// ❌ The starvation trigger — blocking a pool thread on async work:
// var data = httpClient.GetStringAsync(url).Result;  // never do this on the hot path

⚠️ Don’t “fix” starvation by raising min threads

ThreadPool.SetMinThreads can paper over a spike, but the real fix is to stop blocking: go async all the way down so workers are never parked on I/O. Bumping min threads just trades one bottleneck for higher memory and context-switching.

Key takeaways

🎯 Remember these three

  • The pool reuses a small set of worker threads for short work items — Task.Run, continuations, timers, I/O callbacks all use it.
  • It grows slowly under load, so blocking pool threads causes thread-pool starvation (latency cliff).
  • The cure is async-all-the-way, not raising min threads.

Interview tip

🗣️ When this comes up

“What is thread-pool starvation and how do you avoid it?” Explain that blocking pool threads (sync-over-async) while the pool injects threads slowly drains available workers. The fix interviewers want to hear: async end-to-end — not cranking SetMinThreads.
