Blazor Basics

Build interactive web UI in C# instead of JavaScript, using reusable components — running either on the server or right in the browser via WebAssembly.

#blazor #components #webassembly #spa

The idea in plain English

Blazor is a UI framework built around components — .razor files that mix HTML markup with C# logic (in an @code block), with data binding, events, and a lifecycle. The headline is that you write the interactivity in C#, sharing models and validation with your backend, and skipping most JavaScript.

The same component can run under different hosting models:

  • Blazor Server — the component runs on the server. UI events are sent up over a SignalR connection, the server computes the DOM diff, and just the changes are pushed back. Tiny download, instant start, full server power — but every interaction is a network round trip and it needs a live connection.
  • Blazor WebAssembly — your C# is compiled to WASM and runs in the browser. Works offline, no per-click latency, scales like static files — but a bigger initial download and the browser’s sandbox limits.

.NET 8 unified these under render modes: a page can be statically server-rendered (SSR) for speed/SEO, then “light up” interactivity per-component as InteractiveServer, InteractiveWebAssembly, or InteractiveAuto (start on the server, switch to WASM once it’s downloaded).

Real-world example

🏢 How this works in production

Blazor shines for line-of-business apps and internal tools — admin portals, dashboards, back-office systems — where a .NET team wants one language top to bottom and doesn’t want to staff a separate JavaScript front-end. A back-office portal might use Blazor Server for an instant-start, server-secured experience on the corporate network, while a customer-installable tool uses WebAssembly so it runs offline. The component model and shared C# DTOs mean validation rules live in exactly one place.

Two ways to run the same component

In code — a component

Razor — Counter.razor 
@page "/counter"
@rendermode InteractiveServer   @* or InteractiveWebAssembly / InteractiveAuto *@

<h3>Counter</h3>
<p>Current count: @count</p>

@* @onclick wires a DOM event straight to a C# method *@
<button class="btn" @onclick="Increment">Click me</button>

@code {
    private int count;                 // component state

    private void Increment() => count++; // re-renders automatically after the event
}

Key takeaways

🎯 Remember these three

  • Blazor builds web UI from C# components (.razor) — markup plus an @code block, with binding and events.
  • Server runs UI on the server over SignalR (tiny download, needs a live connection); WebAssembly runs C# in the browser (offline-capable, bigger download).
  • .NET 8 unifies them with per-component render modes (Server / WebAssembly / Auto) on top of static SSR.

Interview tip

🗣️ When this comes up

“Blazor Server vs WebAssembly — trade-offs?” Server = small download, instant start, server-side secrets, but latency per interaction and a stateful SignalR connection (scaling cost). WebAssembly = offline, no round-trips, scales like static hosting, but larger initial load and browser sandbox. Mentioning .NET 8 render modes (InteractiveAuto) shows you’re current.
