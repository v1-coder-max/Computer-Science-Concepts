Dependency Injection

Don’t let a class build its own dependencies — hand them in. The .NET container does that for you, wiring whole object graphs from your registrations.

#di #ioc #lifetimes #testability

The idea in plain English

Dependency Injection is a form of Inversion of Control: instead of a class doing new SqlOrderRepository() inside itself (welding it to a concrete type), it declares what it needs in its constructor — IOrderRepository — and something else supplies it. That “something else” is the DI container.

You register your services once at startup (“IOrderRepository → SqlOrderRepository”), and when the container builds a controller it walks the constructor parameters, resolves each one (and their dependencies, recursively), and injects the finished graph. You get loose coupling, swappable implementations, and trivially testable classes (inject a fake in unit tests).

The one thing you must get right is the lifetime of each service:

  • Singleton — one instance for the whole app. Use for stateless/shared services (caches, config).
  • Scoped — one instance per request (per DI scope). The default for DbContext and most app services.
  • Transient — a new instance every time it’s resolved. Use for cheap, stateless helpers.

Real-world example

🏢 How this works in production

ASP.NET Core is a DI framework — the whole stack runs on it. On every request to a service like Shopify’s admin API, the container builds a fresh scope: a scoped DbContext shared by all the repositories in that one request, singleton caches and HTTP-client factories shared across requests, and transient validators created on demand. Because everything is injected through interfaces, the same code runs against real SQL in production and in-memory fakes in tests.

Resolving a graph

In C#

C# — Program.cs 
// Register services with explicit lifetimes:
builder.Services.AddSingleton<IClock, SystemClock>();          // one for the app
builder.Services.AddScoped<IOrderRepository, SqlOrderRepository>(); // one per request
builder.Services.AddTransient<IEmailValidator, EmailValidator>();   // new each time
builder.Services.AddDbContext<AppDbContext>();                 // scoped by default

// Consume via constructor injection — the class never news up its dependencies:
public class OrderService(IOrderRepository repo, IClock clock)
{
    public async Task PlaceAsync(Order o)
    {
        o.PlacedUtc = clock.UtcNow;
        await repo.AddAsync(o);
    }
}

⚠️ The captive dependency

Injecting a scoped (or transient) service into a singleton captures it for the app’s entire life — a stale, possibly thread-unsafe instance. The container will actually throw on scoped-into-singleton in development. When a singleton needs a scoped service per operation, inject IServiceScopeFactory and create a scope.

Key takeaways

🎯 Remember these three

  • DI hands a class its dependencies (via constructor) instead of it creating them — loose coupling and testability.
  • Three lifetimes: singleton (app), scoped (per request), transient (per resolve).
  • Never inject a shorter-lived service into a longer-lived one — that’s the captive-dependency bug.

Interview tip

🗣️ When this comes up

The guaranteed question is “explain the service lifetimes.” Define all three, then prove depth by explaining why a DbContext is scoped (not thread-safe, one unit of work per request) and the captive-dependency trap. Tie DI back to the Dependency Inversion principle for bonus credit.
