Authentication & Authorization

Two different jobs that always get confused: authentication proves who you are; authorization decides what you’re allowed to do.

#authentication #authorization #claims #jwt

The idea in plain English

In ASP.NET Core these are two middleware steps that run in order:

  • UseAuthentication reads the incoming credential — a cookie, or a Bearer JWT — validates it against the configured scheme, and if valid builds a ClaimsPrincipal onto HttpContext.User. A claim is just a key/value statement about the user (sub=42, role=Admin, email=…).
  • UseAuthorization then checks that principal against the endpoint’s requirements — a [Authorize] attribute, a role, or a named policy — and lets the request through or blocks it.

The two failure modes map to two status codes: 401 Unauthorized means “I don’t know who you are” (authentication failed/missing); 403 Forbidden means “I know who you are, but you’re not allowed” (authorization failed). Authorization is expressed as policies built from requirements and evaluated by handlers, which is far more flexible than hard-coding role checks everywhere.

Real-world example

🏢 How this works in production

A GitHub-style API uses JWT bearer authentication: you sign in through an OAuth/OIDC flow, receive a signed token, and every request carries Authorization: Bearer <jwt>. The auth middleware validates the signature and expiry and turns the token’s claims into a ClaimsPrincipal. An endpoint that deletes a repo is guarded by a policy like “must own the repo or be an org admin” — evaluated from the user’s claims — returning 403 if you’re a known user without that right, and 401 if your token is missing or expired.

Authenticate, then authorize

In C#

C# — Program.cs 
builder.Services.AddAuthentication(JwtBearerDefaults.AuthenticationScheme)
    .AddJwtBearer(o =>
    {
        o.Authority = "https://login.example.com"; // validates signature via the issuer's keys
        o.Audience  = "devlens-api";
    });

builder.Services.AddAuthorization(o =>
{
    o.AddPolicy("AdminsOnly", p => p.RequireRole("Admin"));
    o.AddPolicy("Over18", p => p.RequireClaim("age_verified", "true"));
});

var app = builder.Build();
app.UseAuthentication(); // 1) who are you?  → builds HttpContext.User
app.UseAuthorization();  // 2) are you allowed? → enforces policies

app.MapGet("/admin/stats", () => "secret")
   .RequireAuthorization("AdminsOnly");      // 403 if authenticated but not Admin
C# — reading claims 
[Authorize]
[HttpGet("me")]
public IActionResult Me()
{
    string? userId = User.FindFirst("sub")?.Value;     // a claim from the token
    bool isAdmin   = User.IsInRole("Admin");
    return Ok(new { userId, isAdmin });
}

Key takeaways

🎯 Remember these three

  • Authentication = who you are (builds a ClaimsPrincipal); authorization = what you may do (policies/roles/claims).
  • UseAuthentication must run before UseAuthorization in the pipeline.
  • 401 = unknown/invalid identity; 403 = known but not permitted. Prefer policies over scattered role checks.

Interview tip

🗣️ When this comes up

“Difference between authentication and authorization — and 401 vs 403?” Nail the one-liner (who you are vs what you can do), map them to 401 vs 403, and mention that ASP.NET Core models authorization as claims-based policies evaluated after the identity is built. Don’t forget pipeline order.
