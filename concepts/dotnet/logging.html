Logging (Serilog / NLog)

Log structured events, not formatted strings. The ILogger abstraction captures named properties you can later search and aggregate.

#logging #structured-logging #serilog #observability

The idea in plain English

.NET ships a logging abstraction — ILogger<T> — that your code writes to without caring where the logs go. Behind it sit providers/sinks (Console, Debug, and third-party Serilog or NLog targets like files, Seq, or Elasticsearch). Swap the sink without touching your code.

The big idea is structured logging. Instead of baking values into a string, you log a message template with named placeholders and pass the values separately:

  • logger.LogInformation("Order {OrderId} placed by {User}", id, user) — captures OrderId and User as queryable fields.
  • logger.LogInformation($"Order {id} placed by {user}") — flattens to text; you lose the fields.

With structured logs you can later ask “show me all events where OrderId = 5512” across millions of lines. Log levels (Trace → Debug → Information → Warning → Error → Critical) let you filter by severity, and scopes attach context (like a correlation ID) to every log inside a block.

Real-world example

🏢 How this works in production

At observability-heavy shops like Netflix, services emit structured logs enriched with a request/correlation ID and ship them to a central store (Elasticsearch/Kibana or Seq). When something breaks, an engineer filters by CorrelationId to follow a single request across services, or aggregates by StatusCode to spot a spike — none of which is possible if the values were smeared into free-text strings. Serilog’s enrichers add machine name, environment, and trace IDs automatically.

One API, many sinks

In C#

C# 
public class OrderService(ILogger<OrderService> logger)
{
    public async Task PlaceAsync(Order o)
    {
        // ✅ Structured: OrderId & Total are captured as fields you can query.
        logger.LogInformation("Order {OrderId} placed, total {Total:C}", o.Id, o.Total);

        // Scopes attach context to every log inside the block:
        using (logger.BeginScope("Correlation:{CorrelationId}", o.CorrelationId))
        {
            try { await _repo.SaveAsync(o); }
            catch (Exception ex)
            {
                // Always pass the exception as the first arg — captures the stack trace.
                logger.LogError(ex, "Failed to save order {OrderId}", o.Id);
                throw;
            }
        }

        // ❌ Avoid: interpolation flattens to text and loses the fields.
        // logger.LogInformation($"Order {o.Id} placed");
    }
}

Wiring Serilog as the sink takes a few lines and replaces the default providers:

C# — Program.cs (Serilog) 
builder.Host.UseSerilog((ctx, cfg) => cfg
    .ReadFrom.Configuration(ctx.Configuration)  // levels/sinks from appsettings
    .Enrich.FromLogContext()                    // pulls in scope properties
    .WriteTo.Console()
    .WriteTo.Seq("http://localhost:5341"));

Key takeaways

🎯 Remember these three

  • Use message templates with named placeholders — structured properties are queryable; string interpolation is not.
  • ILogger is the abstraction; providers/sinks (Serilog, NLog) decide where logs go and can be swapped freely.
  • Pass the exception object to LogError(ex, …) and use scopes for correlation context.

Interview tip

🗣️ When this comes up

“What is structured logging and why does it matter?” Contrast a message template ("User {UserId} …") with string interpolation, and explain that captured properties make logs searchable and aggregatable in a central store — essential for debugging distributed systems. Mention correlation IDs via scopes.
