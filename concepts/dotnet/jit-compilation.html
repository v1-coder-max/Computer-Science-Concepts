JIT Compilation

The IL in your assembly can’t run on a CPU. The JIT compiles it to native code just in time — the first time each method is called.

#jit #tiered-compilation #readytorun #performance

The idea in plain English

IL is portable on purpose — the same assembly runs on x64, Arm64, Windows, or Linux. The price is that something must translate IL into instructions for the actual chip. That something is the Just-In-Time compiler. It works method-by-method: the first time Foo() is called, the JIT compiles its IL to native, caches the result, and every later call jumps straight to the cached native code.

Compiling everything fully optimized up front would make startup crawl. So modern .NET uses tiered compilation:

  • Tier 0 — a quick, lightly-optimized compile so the app starts fast.
  • Tier 1 — once a method proves “hot” (called often), the JIT recompiles it with full optimizations in the background and swaps it in.

ReadyToRun (R2R) goes further: it pre-JITs much of your code at publish time and embeds native code in the assembly, so startup skips most Tier 0 work. (At the far end sits Native AOT, which removes the JIT entirely — see the related page.)

Real-world example

🏢 How this works in production

A Netflix-style microservice that autoscales sees pods spin up and down constantly, so cold-start latency matters. Teams publish with ReadyToRun so a fresh pod serves its first requests without waiting on the JIT, while tiered compilation keeps optimizing the hot request path (serialization, routing) to Tier 1 for steady-state throughput. It’s the classic trade: fast to start and fast once warm.

How tiered compilation flows

Controlling the JIT

Tiered compilation is on by default. You tune it in runtimeconfig (or env vars) — for example, a short-lived CLI tool might disable tiering to avoid background recompilation:

JSON — runtimeconfig.json 
{
  "runtimeOptions": {
    "configProperties": {
      "System.Runtime.TieredCompilation": true,
      "System.Runtime.TieredPGO": true        // profile-guided opt for even better Tier 1
    }
  }
}

Or publish with ReadyToRun so most code is pre-JITted before it ever ships:

Bash 
# Pre-compile IL to native at publish time → faster cold start
dotnet publish -c Release -r linux-x64 -p:PublishReadyToRun=true

⚠️ Don’t micro-benchmark cold methods

The first call to a method includes its JIT time, and it may still be Tier 0. If you measure performance, warm up first (or use BenchmarkDotNet, which handles this) — otherwise you’re timing the compiler, not your code.

Key takeaways

🎯 Remember these three

  • The JIT compiles IL → native per method on first call, then caches it.
  • Tiered compilation starts fast (Tier 0) and re-optimizes hot methods (Tier 1) in the background.
  • ReadyToRun pre-JITs at publish for faster cold start; Native AOT removes the JIT entirely.

Interview tip

🗣️ When this comes up

“Why is the first request to a .NET service slower?” The answer is JIT compilation (often Tier 0) on the cold path. Mention tiered compilation, ReadyToRun, and warm-up — and contrast JIT (optimizes using runtime info) with AOT (faster start, no runtime re-optimization).
