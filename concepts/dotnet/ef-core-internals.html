EF Core Internals

EF Core maps objects to tables — but to use it well you need to see the two engines inside: the change tracker and the LINQ-to-SQL translator.

#ef-core #orm #change-tracking #n+1

The idea in plain English

A DbContext is a unit of work. When you query, EF materializes rows into entity objects and, by default, starts tracking them: it keeps a snapshot and notes each entity’s state — Unchanged, Modified, Added, or Deleted. You mutate plain C# objects; when you call SaveChanges, EF diffs against the snapshots and emits exactly the INSERT/UPDATE/DELETE statements needed, in one transaction.

The other engine is query translation. A LINQ query over IQueryable isn’t run in memory — EF builds an expression tree and translates it into SQL that runs on the database. The crucial distinction:

  • IQueryable — composable, translated to SQL, filtered server-side. Where runs in the database.
  • IEnumerable — once you’ve pulled data into memory (e.g. .ToList()), further LINQ runs in your process.

Get that boundary wrong and you accidentally pull a whole table into memory. The other classic trap is the N+1 problem: lazily loading a related entity inside a loop fires one query per row. Include (eager loading) or a projection fixes it. For read-only queries, AsNoTracking skips the snapshot overhead.

Real-world example

🏢 How this works in production

An orders dashboard that lists 50 orders and each customer’s name is the textbook N+1 disaster: 1 query for the orders + 50 more for the customers = 51 round trips, and it gets worse under load. The fix is one Include (or a projection to a DTO) so EF emits a single joined query. Read-heavy API endpoints add AsNoTracking to drop change-tracking cost — exactly how high-traffic .NET services keep their read paths cheap.

Track, translate, save

In C#

C# 
// ❌ N+1: lazy-loading Customer inside the loop → 1 + N queries.
var orders = await db.Orders.Where(o => o.IsOpen).ToListAsync();
foreach (var o in orders)
    Console.WriteLine(o.Customer.Name); // each access may hit the DB

// ✅ Eager-load in one round trip, read-only (no tracking), projected to a DTO.
var rows = await db.Orders
    .Where(o => o.IsOpen)               // translated to SQL WHERE (IQueryable)
    .Include(o => o.Customer)           // JOIN, single query
    .AsNoTracking()                     // skip snapshot cost for reads
    .Select(o => new OrderRow(o.Id, o.Customer.Name, o.Total))
    .ToListAsync();

// Writes: mutate tracked entities, then one SaveChanges = one transaction.
var order = await db.Orders.FindAsync(42);
order!.Status = "Shipped";             // tracker marks it Modified
db.Orders.Add(new Order { /* ... */ }); // marked Added
await db.SaveChangesAsync();            // emits UPDATE + INSERT together

⚠️ Watch the IQueryable → IEnumerable boundary

Calling .ToList() (or any non-translatable method) before your filters pulls the whole table into memory, then filters in C#. Keep Where/Select on the IQueryable so they become SQL. And a DbContext is not thread-safe — one per unit of work (scoped per request).

Key takeaways

🎯 Remember these three

  • The DbContext is a unit of work; the change tracker diffs entity states and SaveChanges emits batched SQL in a transaction.
  • LINQ over IQueryable becomes SQL on the server — don’t cross into IEnumerable too early.
  • Avoid N+1 with Include/projection, and use AsNoTracking for read-only queries.

Interview tip

🗣️ When this comes up

“What’s the N+1 problem and how do you fix it?” is near-guaranteed. Explain it, fix it with Include/projection, and add depth: AsNoTracking for reads, the IQueryable vs IEnumerable translation boundary, and that the change tracker is what makes SaveChanges know what to write.
