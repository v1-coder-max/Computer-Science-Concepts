Nullable Reference Types

Make the compiler hunt the “billion-dollar mistake” for you: a reference type’s nullability becomes part of its type, checked at compile time.

#nullable #null-safety #compiler #flow-analysis

The idea in plain English

For most of C#’s life, any reference could be null, and dereferencing a null gave you a NullReferenceException at runtime — the most common .NET crash. Nullable reference types (NRT, C# 8+, on by default in new projects) flip the default. With the feature enabled:

  • string means “not null” — the compiler warns if you might assign null to it.
  • string? means “maybe null” — the compiler warns if you dereference it without checking.

The compiler does flow analysis: after if (name is not null), it knows name is safe inside that branch and stops warning. Nullability also shows up in your API signatures, so callers can see at a glance whether a method can return null.

One crucial caveat: NRT is compile-time only. It’s static analysis (annotations + warnings), not a runtime guard — nothing stops a null arriving from old code, reflection, or JSON. The null-forgiving operator ! (value!) tells the compiler “trust me, it’s not null here” — use it sparingly, because you’re overriding the safety net.

Real-world example

🏢 How this works in production

When a team turns on <Nullable>enable</Nullable> across a large codebase, the compiler immediately surfaces hundreds of places where a value could be null but was dereferenced anyway — exactly the latent NullReferenceExceptions that would otherwise page someone at 2am. Library authors (including the .NET BCL itself) annotate their public APIs so every consumer gets accurate “can this return null?” information at the call site, turning a whole class of runtime bugs into compile-time squiggles.

The compiler narrows null away

In C#

C# 
#nullable enable   // usually set project-wide via <Nullable>enable</Nullable>

public string Greet(string name)          // 'name' is non-nullable by contract
{
    return $"Hi {name}, {name.Length} chars"; // safe: callers can't pass null
}

public int Length(string? maybe)          // '?' = may be null
{
    // return maybe.Length;               // ⚠ CS8602: possible null dereference
    if (maybe is null) return 0;          // narrows; below is now safe
    return maybe.Length;
}

// Required members force initialization, satisfying non-null without warnings:
public class User
{
    public required string Email { get; init; }   // must be set at construction
    public string? Nickname { get; init; }         // optional
}

string? fromJson = Deserialize();
var len = fromJson!.Length;   // null-forgiving '!': "trust me" — use rarely

Key takeaways

🎯 Remember these three

  • string = non-null, string? = maybe-null; the compiler warns on unsafe dereferences and tracks nullability through flow analysis.
  • It’s compile-time only — static analysis, not a runtime guard; nulls can still arrive from untyped sources.
  • The ! null-forgiving operator silences the compiler — use it sparingly; prefer real checks and required members.

Interview tip

🗣️ When this comes up

The gotcha question: “do nullable reference types prevent null at runtime?” No — they’re compile-time static analysis. A null can still slip in via reflection, old libraries, or deserialization. Mention flow analysis and that ! opts out of the checking (so overusing it defeats the point).
