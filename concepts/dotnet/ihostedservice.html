IHostedService & BackgroundService

The built-in way to run background work tied to your app’s lifetime — started on boot, stopped gracefully on shutdown.

#hosted-service #background-worker #generic-host #graceful-shutdown

The idea in plain English

Sometimes you need work that runs alongside request handling: draining a message queue, sending queued emails, refreshing a cache, running a scheduled cleanup. The .NET Generic Host manages these through IHostedService, an interface with two methods — StartAsync (called once when the app starts) and StopAsync (called on graceful shutdown).

You rarely implement the interface directly. Instead you derive from BackgroundService, which gives you a single method to override — ExecuteAsync(CancellationToken) — for a long-running loop. The host starts it, and on shutdown signals the cancellation token so your loop can finish the current item and exit cleanly.

It runs in-process with your web app (or as a standalone Worker Service). The host even waits, up to a timeout, for hosted services to stop — that’s what makes rolling deploys drain in-flight work instead of dropping it.

Real-world example

🏢 How this works in production

An order system at Amazon-style scale accepts an order on the web tier (fast), then a BackgroundService consumes an order-processing queue — charging payment, reserving inventory, emailing confirmation — off the request path. When Kubernetes sends SIGTERM during a deploy, the host cancels the token; the worker stops pulling new messages, finishes the one in flight, and exits within the shutdown window. No half-processed orders.

The hosted-service lifecycle

In C#

C# 
public class QueueWorker(
    IServiceScopeFactory scopeFactory,
    ILogger<QueueWorker> logger) : BackgroundService
{
    protected override async Task ExecuteAsync(CancellationToken stoppingToken)
    {
        // Loop until the host signals shutdown via the token.
        while (!stoppingToken.IsCancellationRequested)
        {
            try
            {
                Message msg = await _queue.DequeueAsync(stoppingToken);

                // A BackgroundService is a SINGLETON — create a scope for scoped
                // dependencies like a DbContext, per unit of work.
                using IServiceScope scope = scopeFactory.CreateScope();
                var handler = scope.ServiceProvider.GetRequiredService<IOrderHandler>();
                await handler.HandleAsync(msg, stoppingToken);
            }
            catch (OperationCanceledException) { break; } // normal on shutdown
            catch (Exception ex)
            {
                logger.LogError(ex, "Failed to process message"); // don't let the loop die
            }
        }
    }
}

// Registration:
builder.Services.AddHostedService<QueueWorker>();

⚠️ Don’t inject scoped services into the constructor

A hosted service is a singleton, so injecting a scoped service (like DbContext) directly captures it for the whole app lifetime — a classic captive dependency bug. Inject IServiceScopeFactory and create a scope per work item, as above. Also: never let an unhandled exception escape ExecuteAsync — it stops the service silently.

Key takeaways

🎯 Remember these three

  • IHostedService (StartAsync/StopAsync) runs background work tied to the host lifetime; BackgroundService wraps it in an ExecuteAsync loop.
  • On shutdown the host cancels the token — honor it to drain in-flight work gracefully.
  • It’s a singleton: create a DI scope per work item for scoped dependencies, and never let the loop throw out.
