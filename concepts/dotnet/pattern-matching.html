Pattern Matching

Test a value’s shape and pull data out of it in one expression — replacing towers of if/else, casts, and null checks.

#pattern-matching #switch-expression #csharp #readability

The idea in plain English

Pattern matching lets you ask “does this value match this shape?” and, if so, bind parts of it to variables — all in a compact, expression-oriented way. It started small (the is type check with a variable) and has grown into a rich mini-language:

  • Type pattern — obj is Customer c (test + cast + bind in one).
  • Property pattern — order is { Status: "Open", Total: > 100 }.
  • Relational & logical — >, <=, and and/or/not: age is >= 13 and < 20.
  • List pattern (C# 11) — arr is [1, _, .., 9].
  • Positional pattern — deconstructs records/tuples: p is (0, 0).

The headline syntax is the switch expression — it returns a value, so mapping an input to an output becomes one clean block instead of a mutable variable assigned across branches. The compiler also warns when a switch isn’t exhaustive, nudging you to handle every case.

Real-world example

🏢 How this works in production

Anywhere you map a “kind of thing” to a decision — pricing tiers, shipping rules, a small state machine, handling a discriminated set of events — pattern matching collapses sprawling if/else if chains into a readable table. A payments service that picks a fee based on method, amount, and region reads almost like a spec when written as a switch expression with property and relational patterns, and the exhaustiveness warning catches the case someone forgot when a new payment type is added.

Input → matched pattern → result

In C#

C# 
// Switch expression with property + relational + logical patterns.
string Shipping(Order o) => o switch
{
    { Total: > 100 }                         => "free",
    { Region: "EU" or "UK" }                 => "eu-rate",
    { Weight: >= 0 and < 1 }                  => "light",
    null                                     => "no order",
    _                                        => "standard"    // default arm
};

// Type pattern: test, cast, and bind in one go.
if (shape is Circle { Radius: var r })
    Console.WriteLine(Math.PI * r * r);

// List pattern (C# 11) with slice and discards.
static string Describe(int[] a) => a switch
{
    []            => "empty",
    [var only]    => $"one: {only}",
    [var first, .., var last] => $"{first}…{last}",
};

Key takeaways

🎯 Remember these three

  • Patterns test shape and bind data at once — type, property, relational/logical, positional, and list patterns.
  • The switch expression returns a value, turning branchy mapping logic into one readable block.
  • The compiler warns on non-exhaustive switches — lean on it, and use _ for the default arm.

Interview tip

🗣️ When this comes up

You may be asked to refactor an if/else chain or a switch statement. Reach for a switch expression with property/relational patterns — it’s shorter and the compiler checks exhaustiveness. Pairing records (deconstruction) with positional patterns is a clean way to show fluency.
