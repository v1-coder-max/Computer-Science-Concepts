Health Checks

A standard endpoint that tells orchestrators and load balancers, in one HTTP call, whether your app is alive and ready to serve.

#health-checks #liveness #readiness #kubernetes

The idea in plain English

In a cloud deployment, something is constantly asking your app “are you OK?” — Kubernetes, a load balancer, a monitoring system. ASP.NET Core’s health-check feature gives them a consistent answer at an endpoint like /health, returning Healthy, Degraded, or Unhealthy.

There are two questions that look similar but mean different things:

  • Liveness — “is the process alive and not deadlocked?” If this fails, the orchestrator restarts the pod.
  • Readiness — “can it actually serve requests right now (DB reachable, cache up, warmed)?” If this fails, the orchestrator stops sending traffic but leaves the pod running.

Keeping them separate matters: if your database blips, you want readiness to fail (drain traffic) without liveness failing (which would pointlessly restart a perfectly fine process). You register checks for each dependency, tag them, and expose them at different paths.

Real-world example

🏢 How this works in production

On Kubernetes, every pod has a livenessProbe and a readinessProbe hitting your health endpoints. During a deploy, a new pod reports not ready until its DB connection pool and caches are warm — so Kubernetes holds traffic until it’s genuinely able to serve, giving zero-downtime rollouts. If a dependency like Redis goes down, readiness flips to unhealthy and the load balancer routes around that instance until it recovers.

Probe → check dependencies → verdict

In C#

C# — Program.cs 
builder.Services.AddHealthChecks()
    .AddDbContextCheck<AppDbContext>("sql", tags: ["ready"])
    .AddRedis(redisConn, name: "redis", tags: ["ready"])
    .AddCheck("self", () => HealthCheckResult.Healthy(), tags: ["live"]);

var app = builder.Build();

// Liveness: just "is the process up?" — only the lightweight self check.
app.MapHealthChecks("/health/live", new()
{
    Predicate = r => r.Tags.Contains("live")
});

// Readiness: dependencies must be reachable.
app.MapHealthChecks("/health/ready", new()
{
    Predicate = r => r.Tags.Contains("ready")
});

A custom check is one small class:

C# 
public class QueueDepthCheck(IMessageQueue queue) : IHealthCheck
{
    public async Task<HealthCheckResult> CheckHealthAsync(
        HealthCheckContext ctx, CancellationToken ct = default)
    {
        int depth = await queue.GetDepthAsync(ct);
        return depth < 10_000
            ? HealthCheckResult.Healthy($"depth {depth}")
            : HealthCheckResult.Degraded($"backlog {depth}");
    }
}

Key takeaways

🎯 Remember these three

  • Health checks expose a standard verdict (Healthy/Degraded/Unhealthy) for orchestrators and load balancers.
  • Liveness failing → restart the pod; readiness failing → stop sending traffic (don’t restart). Keep them separate.
  • Register a check per dependency, tag them, and expose /health/live and /health/ready separately.

Interview tip

🗣️ When this comes up

“What’s the difference between a liveness and a readiness probe?” Liveness failing restarts the container; readiness failing just removes it from rotation. The senior insight: a transient dependency outage should fail readiness only, so you drain traffic without thrashing healthy processes with restarts.
