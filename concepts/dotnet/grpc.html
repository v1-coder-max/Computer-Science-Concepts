gRPC

A high-performance, schema-first RPC framework: define a service in a .proto file, generate strongly-typed clients and servers, talk over HTTP/2.

#grpc #protobuf #http2 #rpc

The idea in plain English

Instead of designing URLs and JSON shapes, with gRPC you write a contract in a .proto file — the service, its methods, and the message types. Tooling generates a typed client stub and a server base class in C#, so calling a remote method feels like calling a local one: await client.GetPriceAsync(request).

Two technologies make it fast:

  • Protocol Buffers — a compact binary serialization. Much smaller and quicker to encode/decode than JSON, with a strict schema and built-in versioning.
  • HTTP/2 — multiplexed streams over one connection, header compression, and native support for streaming.

That streaming gives four call shapes:

  • Unary — one request, one response (like a normal call).
  • Server streaming — one request, a stream of responses.
  • Client streaming — a stream of requests, one response.
  • Bidirectional — both stream at once.

The trade-off: it’s binary and uses HTTP/2 trailers, so browsers can’t call it directly without gRPC-Web. That makes gRPC ideal for internal service-to-service calls, and REST/JSON still the easy choice for public, browser-facing APIs.

Real-world example

🏢 How this works in production

Google created gRPC, and it’s the lingua franca of internal microservice meshes there and at companies like Netflix. When dozens of services call each other millions of times a second, the binary payloads and HTTP/2 multiplexing cut latency and bandwidth dramatically versus JSON over HTTP/1.1, and the generated contracts eliminate a whole class of “the field was renamed” integration bugs. The browser-facing edge still speaks REST; the chatter behind the gateway is gRPC.

Contract → codegen → typed calls

In code

Protobuf — catalog.proto 
syntax = "proto3";

service Catalog {
  rpc GetPrice (PriceRequest) returns (PriceReply);          // unary
  rpc StreamPrices (PriceRequest) returns (stream PriceReply); // server streaming
}

message PriceRequest { int32 product_id = 1; }
message PriceReply   { double amount = 1; string currency = 2; }
C# — server & client 
// Server: override the generated base class.
public class CatalogService : Catalog.CatalogBase
{
    public override Task<PriceReply> GetPrice(PriceRequest req, ServerCallContext ctx)
        => Task.FromResult(new PriceReply { Amount = 9.99, Currency = "USD" });
}
// Program.cs:  builder.Services.AddGrpc();   app.MapGrpcService<CatalogService>();

// Client: a typed stub — calling remote feels local.
var channel = GrpcChannel.ForAddress("https://catalog.internal");
var client  = new Catalog.CatalogClient(channel);
PriceReply reply = await client.GetPriceAsync(new PriceRequest { ProductId = 42 });

Key takeaways

🎯 Remember these three

  • gRPC is schema-first RPC: a .proto generates typed clients/servers; messages are binary Protocol Buffers over HTTP/2.
  • It’s faster and smaller than JSON/REST and supports four streaming modes — ideal for internal service-to-service traffic.
  • Browsers can’t call it directly (need gRPC-Web); keep REST for public, browser-facing APIs.

Interview tip

🗣️ When this comes up

“gRPC vs REST — when would you use each?” gRPC for low-latency, high-volume internal calls (binary protobuf, HTTP/2, typed contracts, streaming); REST for public/browser-facing APIs (human-readable, cache-friendly, no gRPC-Web needed). Naming the four streaming modes shows real depth.
