Async / Await Internals

async/await isn’t about running things in parallel — it’s about not blocking a thread while you wait. Here’s the machinery underneath.

#async #tasks #threading #state-machine

The idea in plain English

Most of what a web app does is wait — for a database, an HTTP call, a file. With old blocking code, the thread handling your request just sits there doing nothing until the I/O returns. Threads are expensive (~1 MB of stack each, plus scheduling), so a server that blocks a thread per in-flight request falls over under load.

await changes the deal. When you await an operation that hasn’t finished, your method pauses and returns the thread to the pool so it can serve someone else. When the I/O completes, the runtime grabs a thread (not necessarily the same one) and resumes your method right where it left off. No thread was held hostage during the wait.

The magic is a compiler trick. An async method is rewritten into a state machine: each await becomes a checkpoint. Your local variables are lifted into a struct so they survive the pause, and a “continuation” is registered to fire when the awaited Task completes. Async is about concurrency (juggling waits), not parallelism (using many cores).

Real-world example

🏢 How this works in production

A Netflix-style API gateway fans a single page request out to dozens of backend microservices. If each of those calls blocked a thread while waiting on the network, a handful of slow downstream services would exhaust the thread pool and stall everyone. With async/await, those threads are released during every network wait, so one box can keep tens of thousands of requests in flight on a small pool of threads. This is exactly why ASP.NET Core controllers, EF Core, and HttpClient are async to the core.

What happens to the thread

The difference isn’t speed of the I/O — it’s whether a thread is wasted standing guard over it.

In C#

Read the comments as the timeline: synchronous until the first await, paused during the I/O, resumed afterward with locals intact.

C# 
public async Task<decimal> GetPriceAsync(int productId)
{
    // 1. Runs synchronously on the caller's thread up to the first await.
    HttpClient http = _httpFactory.CreateClient("catalog");

    // 2. The Task isn't done yet → this method RETURNS to its caller here.
    //    The thread goes back to the pool. Nothing is blocked.
    HttpResponseMessage res = await http.GetAsync($"/price/{productId}");

    // 3. When the network responds, the runtime resumes HERE (the "continuation").
    //    productId and http were saved in the state machine, so they're still valid.
    string body = await res.Content.ReadAsStringAsync();

    return decimal.Parse(body);
}

Roughly, the compiler turns that into a state machine the runtime drives:

C# — generated (simplified) 
// The compiler lifts locals into a struct and tracks a 'state' (-1, 0, 1, ...).
// On each await it: stores state, hooks a continuation via awaiter.OnCompleted,
// and returns. MoveNext() is called again when the awaited task completes.
struct GetPriceStateMachine : IAsyncStateMachine
{
    public int _state;
    public int productId;                 // hoisted local
    private TaskAwaiter<HttpResponseMessage> _awaiter;

    public void MoveNext()
    {
        if (_state == 0) goto resumePoint; // jump back in on resume
        // ... start GetAsync, get awaiter ...
        if (!_awaiter.IsCompleted)
        {
            _state = 0;
            _awaiter.OnCompleted(MoveNext); // resume later, don't block now
            return;
        }
        resumePoint:
        var res = _awaiter.GetResult();      // throws here if the task faulted
        // ... continue ...
    }
}

⚠️ Never block on async (the classic deadlock)

Calling .Result or .Wait() on a Task blocks the current thread and, in UI/legacy SynchronizationContexts, the continuation needs that very thread to resume — so it never can. Result: deadlock, or thread-pool starvation under load. Go async all the way down, and use ConfigureAwait(false) in library code that doesn’t need the original context.

Key takeaways

🎯 Remember these three

  • await frees the thread during I/O — it’s concurrency, not parallelism, and uses no thread while waiting.
  • The compiler rewrites each async method into a state machine; locals are hoisted so they survive the pause and resume.
  • Never .Result/.Wait() on async code — go async end-to-end and use ConfigureAwait(false) in libraries.

Interview tip

🗣️ When this comes up

The trap question is “does await create a new thread?” — the answer is no. For pure I/O there’s no thread during the wait at all; the OS notifies completion and a pool thread runs the continuation. Say “state machine,” “the thread is returned to the pool,” and “async ≠ parallel” and you’re ahead of most candidates.
