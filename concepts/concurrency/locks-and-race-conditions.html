Locks & Race Conditions

When two threads touch the same mutable data at once, you get a race condition — results that depend on timing. A lock restores order by letting only one thread in at a time.

#concurrency #locks #race-condition #thread-safety

The idea in plain English

A race condition happens when the correctness of your program depends on the interleaving of threads. The classic case: count++ is actually read, increment, write — three steps. Two threads can both read the same value, both increment, and both write back, so two increments produce one. The data is now wrong, intermittently and unreproducibly.

A lock enforces mutual exclusion: only one thread can hold it at a time, so the protected critical section runs atomically with respect to other threads. In C#, lock(obj){ ... } is the everyday tool (sugar over Monitor).

Locks have costs. Contention — threads queuing for the same lock — serializes your program and kills the benefit of multiple cores, so keep the critical section as small as possible. And taking multiple locks in different orders across threads causes deadlock (each waits for a lock the other holds); always acquire locks in a consistent order.

Prefer alternatives when you can: Interlocked for simple atomic counters, concurrent collections for shared containers, and immutability (no shared mutable state = no races) above all.

Real-world example

🏢 How this works in production

Every shared in-memory cache, counter, or connection pool wrestles with this. A bank-transfer bug where a balance occasionally goes wrong under load is almost always an unprotected read-modify-write. Production systems lean on ConcurrentDictionary, Interlocked, and careful lock ordering; high-throughput code goes further and avoids shared mutable state entirely (per-thread state, immutable messages) because the fastest lock is the one you never take.

Why count++ loses updates

In C#

C# 
private readonly object _gate = new();
private int _count;

// ✅ Mutual exclusion: one thread in the critical section at a time.
public void Increment()
{
    lock (_gate) { _count++; }     // keep this section tiny to reduce contention
}

// ✅ Even better for a simple counter — lock-free atomic:
Interlocked.Increment(ref _count);

// Deadlock rule: if you must hold two locks, always acquire them in the SAME order.

Key takeaways

🎯 Remember these three

  • A race condition = correctness depends on thread timing; count++ is not atomic.
  • A lock gives mutual exclusion — keep the critical section tiny to limit contention.
  • Prefer Interlocked, concurrent collections, or immutability; acquire multiple locks in a consistent order to avoid deadlock.
