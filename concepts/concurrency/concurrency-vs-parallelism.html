Concurrency vs Parallelism

Two words people use interchangeably and shouldn't: concurrency is about dealing with many things at once; parallelism is about doing many things at once.

#concurrency #parallelism #threads #async

The idea in plain English

Concurrency is a way of structuring a program so that multiple tasks can be in progress and interleave — a single barista taking several orders, switching between them while each drink brews. You can have concurrency on one CPU core: the core rapidly switches between tasks, so they all make progress even though only one runs at any instant.

Parallelism is executing multiple things literally simultaneously, which needs multiple cores — two baristas making two drinks at the same moment. Parallelism is a way to speed up CPU-bound work; concurrency is a way to stay responsive while waiting (I/O).

The practical split: use async concurrency for I/O-bound work (network, disk) — a thread isn't burned while waiting. Use parallelism (multiple threads/cores) for CPU-bound work you want to finish faster. Confusing the two leads to thread-pool starvation (blocking on I/O) or pointless parallelism (parallelizing a network call).

Real-world example

🏢 How this works in production

A web server is the canonical example: it handles thousands of concurrent requests with a small thread pool because most requests are waiting on a database or API (I/O), and async frees the thread during the wait. When that same server needs to resize 10,000 images, it reaches for parallelism (Parallel.ForEach across cores) because that work is CPU-bound. Mixing them up — say, blocking a pool thread on I/O — is what causes the dreaded thread starvation outage.

One core vs many cores

In C#

C# 
// I/O-bound → concurrency with async: no thread is blocked while waiting.
var results = await Task.WhenAll(urls.Select(u => httpClient.GetStringAsync(u)));

// CPU-bound → parallelism: spread the work across cores to finish faster.
Parallel.ForEach(images, img => Resize(img));   // uses multiple threads/cores

// Wrong: parallelizing I/O wastes threads; blocking on I/O starves the pool.

Key takeaways

🎯 Remember these three

  • Concurrency = structure (tasks interleave, even on one core); parallelism = simultaneous execution (needs multiple cores).
  • Use async for I/O-bound work, parallelism for CPU-bound work.
  • They compose — a server is concurrent across requests and may be parallel within one.

Define both and give the I/O-vs-CPU rule: async/concurrency for waiting, parallelism for crunching. The line that lands: 'you can be concurrent on a single core; parallelism needs more than one.'
