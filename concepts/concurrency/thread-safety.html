Thread Safety & Atomicity

Code is thread-safe if it behaves correctly when called from multiple threads at once. There are only a few ways to get there — and they're cheaper than locking everything.

#concurrency #thread-safety #atomic #immutability

The idea in plain English

Thread safety isn't one technique; it's a set of strategies, roughly in order of preference. Immutability: if data never changes after creation, no thread can corrupt it — sharing is free. This is why record types, string, and immutable collections are so valuable in concurrent code.

Confinement: don't share at all — give each thread its own state (e.g., ThreadLocal, or a per-request scope). No sharing means no synchronization.

Atomic operations: for the unavoidable shared counter or flag, Interlocked provides operations (increment, compare-and-swap) that complete as a single indivisible step the hardware guarantees — no lock needed.

Concurrent collections: ConcurrentDictionary, ConcurrentQueue, etc. are built to be safely used from many threads. And don't forget visibility: without a memory barrier (which locks and Interlocked and volatile provide), one thread may not see another's writes due to caching/reordering — atomicity and visibility are both required.

Real-world example

🏢 How this works in production

High-concurrency .NET services lean hard on immutability and concurrent collections: an in-memory cache backed by ConcurrentDictionary, request state confined to the request scope, and configuration passed as immutable records. The bugs that slip through are usually visibility ones — a flag written by one thread that another never observes because there's no barrier — which is why volatile and Interlocked exist alongside locks.

Four routes to thread safety

In C#

C# 
// 1. Immutable — safe to share with zero synchronization.
public record Money(decimal Amount, string Currency);

// 2. Atomic — lock-free compare-and-swap for shared state.
int original, updated;
do { original = _value; updated = original + 1; }
while (Interlocked.CompareExchange(ref _value, updated, original) != original);

// 3. Concurrent collection — safe multi-threaded access built in.
var cache = new ConcurrentDictionary<string, byte[]>();
cache.GetOrAdd(key, k => Load(k));

Key takeaways

🎯 Remember these three

  • Prefer, in order: immutability → confinement → atomics (Interlocked) → concurrent collections → locks.
  • Thread safety needs both atomicity and visibility (a memory barrier), or threads won't see each other's writes.
  • The cheapest synchronization is none — immutable, unshared data can't race.

Interview tip

🗣️ When this comes up

List the strategies in preference order and note that visibility matters as much as mutual exclusion — a write without a barrier may never be seen. That second point separates a deep answer from 'use ConcurrentDictionary.'
