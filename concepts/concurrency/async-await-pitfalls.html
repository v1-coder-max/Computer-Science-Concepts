Async/Await Pitfalls

async/await is easy to write and easy to misuse. A handful of well-known traps cause deadlocks, swallowed exceptions, and thread-pool starvation.

#async #deadlock #concurrency #csharp

The idea in plain English

Don't block on async code. Calling .Result or .Wait() on a Task blocks the current thread until it finishes. In a context with a synchronization context (classic ASP.NET, UI apps), the continuation needs that same thread — which is blocked — so you get a deadlock. The rule: async all the way down; never mix blocking and async.

Avoid async void. An async void method can't be awaited and its exceptions can't be caught by the caller — they crash the process. The only legitimate use is event handlers. Everywhere else return Task.

Fire-and-forget is a trap. Calling an async method without awaiting it discards the Task, so exceptions vanish and you have no idea when (or if) it finished. If you truly want background work, hand it to a proper background service, not a dropped Task.

ConfigureAwait(false) in library code avoids capturing the context, which both prevents the deadlock above and is a small performance win. Application code in modern ASP.NET Core doesn't need it (no sync context), but libraries should still use it.

Real-world example

🏢 How this works in production

Nearly every team has hit the .Result deadlock: a synchronous controller action or a constructor calls SomeAsync().Result, works fine under light load, then deadlocks under load or in a UI app and the thread pool fills with blocked threads. The fix is always the same — make the call chain async end to end. Library authors (e.g., the BCL, Polly) sprinkle ConfigureAwait(false) everywhere precisely to be safe inside any caller's context.

How the .Result deadlock happens

In C#

C# 
// ❌ Deadlock risk + thread starvation: blocking on async.
var data = GetDataAsync().Result;            // or .Wait()

// ✅ Async all the way down.
var data = await GetDataAsync();

// ❌ async void: exceptions escape and crash the process.
async void Save() { await db.SaveAsync(); }  // only OK for event handlers

// ✅ Return Task so callers can await and catch.
async Task SaveAsync() { await db.SaveAsync(); }

// In LIBRARY code, don't capture the context:
await SomeAsync().ConfigureAwait(false);

Key takeaways

🎯 Remember these three

  • Never block on async (.Result/.Wait()) — go async all the way to avoid deadlocks and starvation.
  • Avoid async void (except event handlers); return Task so exceptions are observable.
  • Use ConfigureAwait(false) in libraries; don't fire-and-forget Tasks you care about.
