Partitioning & Sharding

When one database can’t hold or serve all your data, you split it. Sharding spreads the rows across many servers — powerful, but it changes the rules.

#sharding #partitioning #scaling #shard-key

The idea in plain English

You scale a single box vertically (more CPU/RAM) only so far. Past that you partition the data:

  • Vertical partitioning — split by columns (rarely-used or huge columns into their own table).
  • Horizontal partitioning — split by rows (e.g. orders by date range), still within one server.
  • Sharding — horizontal partitioning spread across multiple servers/nodes, each holding a disjoint subset. Now you scale writes and storage, not just reads.

Each row goes to a shard based on its shard key (e.g. user_id), via range, hash, or directory mapping. The shard key is the most important decision you’ll make:

  • It must distribute evenly — a bad key creates a hotspot (one shard swamped while others idle). Hashing the key spreads load; range keys can hotspot on “recent.”
  • It should match your access pattern, because cross-shard queries are expensive: joins and transactions across shards are hard or unsupported, and a query without the shard key may have to fan out to every node.

Sharding adds real operational complexity (rebalancing/resharding, no easy global joins), so it’s a last resort after indexing, caching, and read replicas. Consistent hashing is the usual technique for distributing keys so adding/removing a node moves minimal data.

Real-world example

🏢 How this works in production

Instagram famously sharded its Postgres by user ID across thousands of logical shards, so each user’s data lives on one shard and the system scales horizontally with the user base. Multi-tenant SaaS often shards by tenant_id so each customer maps cleanly to a shard. The recurring lesson everyone learns the hard way: pick the shard key for your dominant query and even distribution — getting it wrong means painful resharding and a hot node taking all the traffic.

In practice

C# — shard routing (conceptual) 
// Pick the connection by the shard key — queries WITH the key hit one shard.
string ConnFor(int userId) => _shards[Math.Abs(userId.GetHashCode()) % _shards.Count];

await using var conn = new SqlConnection(ConnFor(userId));   // single-shard, fast

// ❌ A report across ALL users must fan out to every shard and merge — expensive,
//    and there's no cross-shard JOIN or transaction. Design the key to avoid this.
SQL — single-server table partitioning (a lighter step) 
-- Horizontal partitioning by date keeps one server but prunes scans to one partition.
-- (PostgreSQL declarative partitioning)
CREATE TABLE orders (id bigint, created date, ...) PARTITION BY RANGE (created);
CREATE TABLE orders_2026 PARTITION OF orders
    FOR VALUES FROM ('2026-01-01') TO ('2027-01-01');

Key takeaways

🎯 Remember these three

  • Sharding = horizontal partitioning across multiple servers, scaling writes and storage, not just reads.
  • The shard key is everything: even distribution (no hotspots) and alignment with your main query.
  • Cross-shard joins/transactions are hard — shard only after indexing, caching, and replicas fall short.

Interview tip

🗣️ When this comes up

In a scaling question, distinguish partitioning (within a server) from sharding (across servers), and make the shard-key choice the centerpiece — even distribution to avoid hotspots, plus the pain of cross-shard queries. Mention consistent hashing for rebalancing, and that sharding is a last resort.
