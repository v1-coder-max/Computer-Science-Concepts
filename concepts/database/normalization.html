Normalization vs Denormalization

Normalize to keep data correct; denormalize to make reads fast. Knowing when to break the rules is the real skill.

#normalization #schema-design #denormalization #trade-offs

The idea in plain English

Normalization organizes a schema so each fact lives in exactly one place, eliminating redundancy. The normal forms build on each other:

  • 1NF — atomic columns, no repeating groups.
  • 2NF — no partial dependency on part of a composite key.
  • 3NF — no column depends on a non-key column (no transitive dependencies).

The payoff is integrity: change a customer’s name in one row and it’s correct everywhere — no update anomalies. The cost is that answering a question often needs joins across many tables.

Denormalization deliberately reintroduces redundancy — duplicating columns or pre-joining data — to avoid those joins and make reads faster. The trade-off flips: faster reads, but now you must keep the copies in sync on every write (more complex writes, risk of drift). The rule of thumb: normalize for OLTP (transactional, write-heavy); denormalize for read-heavy/analytics (OLAP) via materialized views, read models, or wide tables. Normalize first, then denormalize deliberately where a real read bottleneck proves it’s worth it.

Real-world example

🏢 How this works in production

Amazon’s order-writing system is normalized — customers, orders, and order-items in separate tables — so a price or address is stored once and stays consistent. But the product page you read serves from a denormalized, pre-joined read model (often cached or a separate store) so it doesn’t join six tables on every view. This split — normalized source of truth + denormalized read views — is exactly the CQRS idea, and it’s how high-traffic systems get both correctness and speed.

Joins vs one wide row

In SQL

SQL 
-- Normalized: customer name lives once, in Customers.
SELECT o.Id, c.Name, o.Total
FROM Orders o JOIN Customers c ON c.Id = o.CustomerId;   -- a join per read

-- Denormalized read model: name copied onto the order row → no join.
SELECT Id, CustomerName, Total FROM OrderReadModel;      -- fast, but...
-- ...a customer rename must now update every OrderReadModel row (or rebuild it).

-- A materialized view is a managed way to denormalize for reads:
CREATE MATERIALIZED VIEW OrderSummary AS
SELECT o.Id, c.Name AS CustomerName, o.Total
FROM Orders o JOIN Customers c ON c.Id = o.CustomerId;   -- refreshed on a schedule

Key takeaways

🎯 Remember these three

  • Normalization (1NF–3NF) stores each fact once — write integrity, no update anomalies, but more joins.
  • Denormalization duplicates/pre-joins for read speed — faster reads, harder writes, risk of drift.
  • Normalize OLTP; denormalize read-heavy paths deliberately (read models, materialized views).

Interview tip

🗣️ When this comes up

“When would you denormalize?” Show you default to normalized (correctness) and denormalize only to fix a proven read bottleneck — and that you then own keeping the duplicates in sync. Naming update anomalies and materialized views/read models signals real experience.
