Deadlocks

Two transactions, each holding the lock the other needs, each waiting forever. The database breaks the tie by killing one — and your job is to make that rare and recoverable.

#deadlock #locking #concurrency #retry

The idea in plain English

Transactions take locks on rows they read/write so others don’t interfere. A deadlock is a circular wait: transaction T1 locks row A and wants row B; T2 locks row B and wants row A. Neither can proceed, and neither will give up its lock. Unlike a slow query, this never resolves on its own.

Databases handle it automatically with a deadlock detector (a background process that looks for cycles in the wait-for graph). When it finds one, it picks a victim — usually the cheaper transaction to roll back — aborts it with an error (SQL Server 1205), and lets the other proceed. The victim’s app sees a failure and should retry.

Most deadlocks trace to a few causes, each with a fix:

  • Inconsistent lock order → always access tables/rows in the same order everywhere.
  • Long transactions holding locks → keep transactions short; do slow work outside them.
  • Excessive locking / missing indexes (scans lock more rows) → index the predicates; use the lowest correct isolation.
  • It will still happen occasionally → add retry logic for the deadlock error.

Real-world example

🏢 How this works in production

The textbook case is two money transfers running at once: one transfers A→B, the other B→A, and each locks its first account then waits on the second — instant deadlock. A payments team fixes it by always locking accounts in ascending ID order, so a cycle can’t form, and wraps the operation in a retry so the rare remaining deadlock just re-runs transparently. High-write tables (inventory, counters) are the usual hot spots, which is why short transactions and good indexing matter so much there.

The circular wait

In SQL & C#

SQL — the trap vs the fix 
-- ❌ Different order in two sessions → deadlock risk
-- Session 1: UPDATE Accounts SET ... WHERE Id = 1;  then  WHERE Id = 2;
-- Session 2: UPDATE Accounts SET ... WHERE Id = 2;  then  WHERE Id = 1;

-- ✅ Always touch rows in a consistent order (e.g. ascending Id) in BOTH paths.
BEGIN TRAN;
    UPDATE Accounts SET Balance = Balance - 100 WHERE Id = 1;   -- lower id first
    UPDATE Accounts SET Balance = Balance + 100 WHERE Id = 2;
COMMIT;
C# — retry the victim 
for (int attempt = 1; attempt <= 3; attempt++)
{
    try { await DoTransferAsync(); break; }
    catch (SqlException ex) when (ex.Number == 1205)   // chosen as deadlock victim
    {
        if (attempt == 3) throw;
        await Task.Delay(50 * attempt);                // brief backoff, then retry
    }
}
// EF Core: builder.UseSqlServer(cs, o => o.EnableRetryOnFailure()); also retries deadlocks.

Key takeaways

🎯 Remember these three

  • A deadlock is a circular lock wait; the DB detects it and aborts a victim transaction.
  • Prevent it with consistent lock ordering, short transactions, and good indexes (less locking).
  • They can’t be eliminated entirely — wrap write transactions in retry-on-deadlock logic.

Interview tip

🗣️ When this comes up

“What causes a deadlock and how do you prevent it?” Draw the circular wait, say the DB kills a victim, and give the prevention trio: consistent lock ordering, short transactions, and retry logic. Bonus: missing indexes cause scans that lock extra rows and make deadlocks more likely.
