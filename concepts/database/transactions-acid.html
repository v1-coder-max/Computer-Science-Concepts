Transactions & ACID

A transaction turns several operations into one all-or-nothing unit. ACID is the four guarantees that make that safe even when things crash.

#transactions #acid #durability #consistency

The idea in plain English

A bank transfer is two steps: debit one account, credit another. If the power dies between them, you must never end up with money debited but not credited. A transaction guarantees both happen or neither does. You wrap them in BEGIN … COMMIT (or ROLLBACK to undo). The four properties — ACID:

  • Atomicity — all operations succeed or all are rolled back. No partial transactions.
  • Consistency — the database moves from one valid state to another; constraints, foreign keys, and triggers always hold.
  • Isolation — concurrent transactions don’t step on each other (the degree is set by the isolation level).
  • Durability — once committed, the change survives a crash/power loss.

The mechanism behind atomicity + durability is the write-ahead log (WAL): changes are written to a durable log before being applied to the data files. On commit the log entry is flushed to disk; after a crash, the engine replays committed entries (redo) and discards uncommitted ones (undo). That’s how a committed order is still there after the server reboots.

Real-world example

🏢 How this works in production

Placing an order at Amazon touches several tables — create the order, decrement inventory, record the payment intent. Those run in one transaction: if the payment authorization fails, the whole thing rolls back so you never have an order with stock removed but no payment. And once you see “Order confirmed,” durability means that record survives even if the database node crashes a millisecond later — it’s already in the write-ahead log on disk.

All, or nothing

In SQL & C#

SQL 
BEGIN TRANSACTION;
    UPDATE Accounts SET Balance = Balance - 100 WHERE Id = 1;  -- debit
    UPDATE Accounts SET Balance = Balance + 100 WHERE Id = 2;  -- credit
    -- a CHECK constraint (Balance >= 0) enforces Consistency
COMMIT;          -- both survive a crash (Durability); ROLLBACK undoes both (Atomicity)
C# — EF Core 
// A single SaveChanges is already one atomic transaction.
using var tx = await db.Database.BeginTransactionAsync();   // explicit, multi-step
try
{
    db.Accounts.Find(1)!.Balance -= 100;
    db.Accounts.Find(2)!.Balance += 100;
    await db.SaveChangesAsync();
    await tx.CommitAsync();        // all or nothing
}
catch
{
    await tx.RollbackAsync();      // any failure → undo everything
    throw;
}

Key takeaways

🎯 Remember these three

  • A transaction is an all-or-nothing unit: COMMIT applies everything, ROLLBACK undoes it.
  • ACID = Atomicity, Consistency, Isolation, Durability — the four guarantees relational DBs give you.
  • The write-ahead log delivers atomicity + durability: log to disk before applying, replay/undo on recovery.

Interview tip

🗣️ When this comes up

Define ACID with the bank transfer example for atomicity, then go one level deeper: durability is delivered by the write-ahead log, and the “I” (isolation) is tunable via isolation levels. Knowing distributed systems often trade ACID for BASE/eventual consistency is a strong follow-up.
