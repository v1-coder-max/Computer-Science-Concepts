Stored Procedures vs ORMs

Two ways your app talks to the database: hand-written SQL living in the database, or an object-mapper that generates SQL for you. Each wins somewhere.

#stored-procedures #orm #ef-core #trade-offs

The idea in plain English

A stored procedure is named SQL stored and executed inside the database. Pros: the query plan is reused, it’s network-efficient (send one call, not a big query + round trips), it can encapsulate complex set-based logic, and you can lock down table access. Cons: business logic lives in the database (harder to version-control, unit-test, and refactor), it’s tied to that DB’s dialect, and it splits your logic across two codebases.

An ORM like EF Core maps tables to objects and turns LINQ into SQL. Pros: huge productivity, type-safety, maintainable code, migrations, easy testing, and DB portability. Cons: it can emit inefficient SQL (the N+1 problem, over-fetching), the abstraction leaks for complex queries, and you’re one step removed from the exact SQL.

In modern apps the answer is usually “ORM by default, raw SQL where it pays.” Use EF Core for the 90% of CRUD where productivity matters, and drop to raw SQL or a micro-ORM (Dapper) — or a stored procedure — for the hot, complex, or performance-critical 10%. EF Core can even call stored procedures and run raw SQL, so it’s not all-or-nothing.

Real-world example

🏢 How this works in production

A typical team builds the bulk of an app on EF Core — entities, migrations, LINQ queries — because it ships features fast and stays testable. Then the nightly reconciliation report that joins eight tables over millions of rows gets a hand-tuned stored procedure or a Dapper query, because there the exact execution plan and minimal allocations matter more than developer convenience. The mistake juniors make is treating it as a holy war; seniors use the right tool per query.

Two paths to the data

In C#

C# 
// ORM (EF Core): productive, type-safe, testable — great for everyday CRUD.
var open = await db.Orders
    .Where(o => o.Status == "Open")
    .Select(o => new { o.Id, o.Total })
    .ToListAsync();

// Hot path: call a stored procedure (tuned plan, set-based logic in the DB).
var report = await db.Database
    .SqlQuery<ReportRow>($"EXEC GetSalesReport @from={from}, @to={to}")
    .ToListAsync();

// Or a micro-ORM (Dapper) for fast, explicit raw SQL without full EF overhead:
var rows = await conn.QueryAsync<ReportRow>(
    "SELECT region, SUM(total) AS total FROM Orders GROUP BY region");

Key takeaways

🎯 Remember these three

  • Stored procedures: reused plans, network-efficient, encapsulated — but logic in the DB, harder to test/version, DB-specific.
  • ORMs: productivity, type-safety, maintainability, migrations — but can emit inefficient SQL (N+1) and leak on complex queries.
  • Pragmatic answer: ORM by default, drop to raw SQL / Dapper / a stored proc for the hot, complex 10%.

Interview tip

🗣️ When this comes up

“Stored procedures or an ORM?” Avoid dogma. Lay out the trade-offs (control/perf vs productivity/testability), then give the senior answer: ORM by default, raw SQL/Dapper/sprocs for performance-critical or complex queries. Mention the N+1 problem as the ORM pitfall to watch.
