Reading Execution Plans

SQL says what you want; the optimizer decides how to get it. The execution plan is that decision — and reading it is how you fix slow queries.

#execution-plan #query-optimization #statistics #performance

The idea in plain English

SQL is declarative — you describe the result, not the steps. The query optimizer takes your query and produces an execution plan: a tree of physical operators (scans, seeks, joins, sorts) it’ll run. It’s cost-based — it estimates how many rows each step produces using statistics (histograms of column values) and picks the cheapest plan.

What to look for when you read one:

  • Index Seek vs Scan — a seek jumps to the rows (good); a scan reads the whole table/index (often the smoking gun).
  • Join type — nested loops (small inputs), hash join (large unsorted), merge join (sorted inputs).
  • Key Lookups — the index found the row but had to go back to the table for more columns (a covering index removes these).
  • Estimated vs actual rows — a big mismatch means bad cardinality estimates, usually from stale statistics.

Two classic traps: stale statistics mislead the optimizer into a bad plan, and parameter sniffing — the plan cached for one parameter value performs terribly for another with a very different data distribution. The fix flow is always: read the plan first, then act.

Real-world example

🏢 How this works in production

When a dashboard query that ran in 50 ms suddenly takes 8 seconds, a DBA pulls the actual execution plan before touching anything. Nine times out of ten the story is there: a Table Scan where a seek should be (missing/!sargable index), a fat Key Lookup loop, or estimated-vs-actual rows off by 1000× because statistics went stale after a bulk load. Sometimes it’s parameter sniffing — fast for one customer, slow for the whale account. The plan turns guesswork into a diagnosis.

SQL → optimizer → plan

In SQL

SQL 
-- SQL Server: see I/O and the actual plan
SET STATISTICS IO, TIME ON;          -- logical reads per table
-- (SSMS) enable "Include Actual Execution Plan", then run:
SELECT o.Id, o.Total, c.Name
FROM Orders o
JOIN Customers c ON c.Id = o.CustomerId
WHERE o.CreatedUtc >= '2026-01-01';
-- Look for: Index Seek vs Table Scan, Key Lookups, estimated vs actual rows.

-- PostgreSQL: EXPLAIN ANALYZE shows estimated AND real timings/rows
EXPLAIN (ANALYZE, BUFFERS)
SELECT * FROM orders WHERE customer_id = 42;

-- Keep stats fresh so estimates stay accurate:
UPDATE STATISTICS Orders;            -- SQL Server   ( ANALYZE orders;  in Postgres )

Key takeaways

🎯 Remember these three

  • The optimizer is cost-based and statistics-driven; the execution plan is the operator tree it chose.
  • Hunt for scans-instead-of-seeks, key lookups, and big estimated-vs-actual row gaps.
  • Stale statistics and parameter sniffing are the usual culprits behind a suddenly-slow query.

Interview tip

🗣️ When this comes up

“A query is slow — how do you debug it?” The senior answer leads with read the execution plan, not “add an index.” Mention seek vs scan, statistics/cardinality estimates, and parameter sniffing. Only then propose a covering index — and verify it changed the plan.
