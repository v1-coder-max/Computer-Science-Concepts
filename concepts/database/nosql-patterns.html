NoSQL Patterns

“NoSQL” isn’t one thing — it’s four families of non-relational stores, each shaped around a different access pattern. Pick by how you’ll read the data.

#nosql #document #key-value #graph

The idea in plain English

NoSQL (“not only SQL”) stores drop the rigid relational schema to gain flexible schemas, easy horizontal scaling, and high throughput — usually by denormalizing data to match queries and relaxing consistency (often BASE: eventual consistency rather than strict ACID). The four families:

  • Document (MongoDB, Cosmos DB) — JSON-like documents. Flexible, nested; great when an entity is read/written as a whole (catalogs, user profiles, CMS).
  • Key-value (Redis, DynamoDB) — a giant hash map: get/put by key. Blazing fast for caches, sessions, feature flags.
  • Wide-column (Cassandra, HBase) — rows with dynamic columns, optimized for huge write volume and queries by partition key (time-series, event logs, feeds).
  • Graph (Neo4j) — nodes + edges; built for traversing relationships (social networks, recommendations, fraud rings).

The trade-off: NoSQL shines when your access pattern is known and singular, but it’s weak at ad-hoc joins and multi-record transactions. Many systems use polyglot persistence — SQL for transactional data, a document store for catalogs, key-value for cache, graph for relationships.

Real-world example

🏢 How this works in production

Big systems mix them deliberately. Netflix uses Cassandra (wide-column) for massive viewing-history writes; nearly everyone uses Redis (key-value) for caching and sessions; an e-commerce product catalog fits a document store because each product is a self-contained blob with varying attributes; and LinkedIn- style “people you may know” is a graph traversal. The relational database still holds the orders and payments where ACID is non-negotiable.

The four families

In code

JSON — a document (MongoDB) 
// One self-contained document — no joins to render a product page.
{
  "_id": "p-42",
  "name": "Mechanical Keyboard",
  "price": 129.99,
  "specs": { "switches": "brown", "layout": "75%" },   // nested, schema-flexible
  "reviews": [ { "user": "ada", "stars": 5 } ]          // embedded, denormalized
}
Key-value (Redis-style) 
SET session:abc123 "{userId:42,role:admin}"   # O(1) put
GET session:abc123                              # O(1) get — perfect for cache/sessions

Key takeaways

🎯 Remember these three

  • Four families: document (whole-entity), key-value (cache/sessions), wide-column (huge writes/feeds), graph (relationships).
  • NoSQL favors flexible schemas, horizontal scale, and denormalization — often BASE/eventual consistency over ACID.
  • Choose by access pattern; real systems mix stores (polyglot persistence).

Interview tip

🗣️ When this comes up

“Name the types of NoSQL and when you’d use each.” Document / key-value / wide-column / graph, each with a use case. The senior framing: choose by access pattern, accept BASE/eventual consistency, and use polyglot persistence rather than forcing one store to do everything.
