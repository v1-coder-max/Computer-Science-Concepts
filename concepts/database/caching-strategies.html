Caching Strategies

A cache trades a little staleness for a lot of speed. How you read and write through it — the strategy — decides the trade-offs you live with.

#caching #cache-aside #write-through #invalidation

The idea in plain English

A cache is a fast store (usually in-memory, like Redis) in front of a slower one (the database). The strategies differ in who loads the cache and how writes are handled:

  • Cache-aside (lazy loading) — the app checks the cache; on a miss it reads the DB, stores the result, and returns it. Most common. Only requested data is cached; a miss costs an extra round trip, and you must invalidate on writes.
  • Read-through — the cache itself fetches from the DB on a miss (the app only talks to the cache). Same effect, logic moved into the cache layer.
  • Write-through — writes go to the cache and DB synchronously. Cache always fresh; writes are slower.
  • Write-back (write-behind) — write to the cache, flush to the DB asynchronously. Fastest writes, but you can lose data if the cache dies before flushing.
  • Write-around — writes skip the cache and go straight to the DB (cache fills on later reads). Good when written data isn’t read soon.

The eternal hard part is invalidation — keeping the cache from serving stale data. Tools: a TTL (expire after N seconds) and/or explicit eviction on write. Watch for the thundering herd: a popular key expiring sends a flood of misses straight to the DB (mitigate with locks/“request coalescing” or staggered TTLs).

Real-world example

🏢 How this works in production

Nearly every high-traffic app puts Redis cache-aside in front of its database: a product page first checks Redis, and only on a miss hits SQL and caches the result for a few minutes — cutting database reads by an order of magnitude. Write-through suits data that must always be fresh (a config flag); write-back suits high-volume metrics where losing the last few seconds is acceptable for speed. Twitter/Facebook scale their reads almost entirely on cache hit rates.

In C#

C# — cache-aside with IDistributedCache (Redis) 
public async Task<Product?> GetProductAsync(int id)
{
    string key = $"product:{id}";

    // 1. Try the cache.
    var cached = await _cache.GetStringAsync(key);
    if (cached is not null) return JsonSerializer.Deserialize<Product>(cached);

    // 2. Miss → load from the DB.
    var product = await _db.Products.FindAsync(id);
    if (product is null) return null;

    // 3. Populate the cache with a TTL (freshness vs load trade-off).
    await _cache.SetStringAsync(key, JsonSerializer.Serialize(product),
        new() { AbsoluteExpirationRelativeToNow = TimeSpan.FromMinutes(5) });
    return product;
}

// On update: write the DB, then INVALIDATE the key (or rewrite it).
public async Task UpdateAsync(Product p)
{
    _db.Update(p); await _db.SaveChangesAsync();
    await _cache.RemoveAsync($"product:{p.Id}");      // avoid stale reads
}

Key takeaways

🎯 Remember these three

  • Cache-aside (app loads on miss) is the default; read-through moves that into the cache layer.
  • Write-through keeps the cache fresh (slower writes); write-back is fastest but can lose data.
  • Invalidation is the hard part — use TTLs + eviction on write, and guard against thundering herds.

Interview tip

🗣️ When this comes up

“How would you cache this, and how do you keep it fresh?” Default to cache-aside + TTL, then compare write-through (fresh, slower) and write-back (fast, lossy). Naming the thundering herd and cache invalidation as the hard problem signals you’ve run a cache in production.
