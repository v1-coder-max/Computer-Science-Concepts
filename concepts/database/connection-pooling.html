Connection Pooling

Opening a database connection is slow. A pool keeps a handful open and hands them out and takes them back, so requests reuse connections instead of building new ones.

#connection-pool #performance #ado-net #exhaustion

The idea in plain English

Establishing a DB connection isn’t free: a TCP handshake, TLS, authentication, and server-side session setup — easily milliseconds each. Under web load (hundreds of requests/second) opening a fresh connection per request would crush both the app and the database.

A connection pool solves it. The driver keeps a set of already-open connections. When your code “opens” a connection it actually borrows one from the pool; when you “close”/dispose it, the connection is returned (reset, not torn down) for the next caller. In ADO.NET / EF Core this is on by default, keyed by the exact connection string — so reusing the same string is what lets connections be shared.

The things that bite:

  • Pool exhaustion — if all connections are checked out (often because someone forgot to dispose, or a slow query holds one too long), new requests wait and then time out. Symptom: “Timeout expired… all pooled connections were in use.”
  • Always dispose — a using returns the connection promptly. A leaked connection is gone from the pool until GC finalizes it.
  • Size it sanely — Max Pool Size caps concurrency to the DB; bigger isn’t always better (the database has its own connection limits).

Real-world example

🏢 How this works in production

A classic outage: a code path opens a SqlConnection but an exception skips the Close(), so under traffic the pool drains and every request starts timing out — the app looks down though the database is fine. The fix is boring and reliable: wrap connections in using (or let EF Core’s scoped DbContext manage it), keep transactions short so connections return fast, and size Max Pool Size to what the database can actually handle. This is the DB-tier cousin of thread-pool starvation.

Borrow and return

In C#

C# 
// Pooling is ON by default; tune it in the connection string.
var cs = "Server=db;Database=Shop;Pooling=true;Max Pool Size=100;Min Pool Size=5;";

// 'using' RETURNS the connection to the pool at the end of scope — don't skip it.
await using var conn = new SqlConnection(cs);
await conn.OpenAsync();          // borrow from the pool (cheap if one's free)
await using var cmd = new SqlCommand("SELECT 1", conn);
await cmd.ExecuteScalarAsync();
// conn disposed here → returned to the pool, ready for the next request

// EF Core pools automatically; a scoped DbContext borrows/returns per request.
// Heavy multi-threaded apps can also pool DbContext instances: AddDbContextPool<T>().

Key takeaways

🎯 Remember these three

  • Opening connections is expensive; the pool reuses open ones, keyed by the connection string.
  • Forgetting to dispose (or holding connections in long operations) causes pool exhaustion and timeouts.
  • Always using/dispose, keep work short, and size Max Pool Size to the DB’s limits.

Interview tip

🗣️ When this comes up

“Why pool connections, and what is pool exhaustion?” Opening a connection is costly, so the driver reuses them; exhaustion happens when all are checked out (often a leak from a missing dispose) and new requests time out. The fix: using, short transactions, sane max pool size.
