Replication

Keep copies of your data on multiple servers — for read scaling and high availability. The catch is keeping the copies in sync.

#replication #read-replicas #high-availability #consistency

The idea in plain English

Replication maintains copies of the same data across nodes. The dominant model is leader–follower (a.k.a. primary/replica or master/slave): all writes go to the leader, which streams its changes to followers that serve reads. This scales reads horizontally and gives you a standby to promote if the leader dies.

The crucial dial is how changes propagate:

  • Synchronous — the leader waits for a follower to confirm before acking the write. No data loss on failover, but slower writes and a stalled leader if a follower lags.
  • Asynchronous — the leader acks immediately and ships changes in the background. Fast, but there’s replication lag: a follower can be milliseconds-to-seconds behind, so a read right after a write may return stale data (eventual consistency).

Multi-leader (master–master) lets you write to several nodes (e.g. per region) for write availability, but introduces write conflicts you must resolve. The everyday gotchas: read-your-own-writes (route a user’s reads to the leader right after they write), and failover (promoting a follower, ideally automatically). Note replication ≠ sharding: replication copies the whole dataset; sharding splits it.

Real-world example

🏢 How this works in production

A read-heavy app like a news site or Reddit puts a leader behind several read replicas: comments and votes (writes) hit the leader; the millions of page views (reads) fan out to replicas. Async replication keeps writes fast, so occasionally you post a comment and a replica hasn’t caught up — the classic “I don’t see my own comment” bug, fixed by reading your own writes from the leader. For HA, a managed service (RDS, Cloud SQL) auto-promotes a replica when the primary fails.

Writes to leader, reads from followers

In practice

C# — read/write split (conceptual) 
// Route writes to the leader, reads to a replica to scale reads.
await using var write = new SqlConnection(_leaderConnString);   // INSERT/UPDATE/DELETE
await using var read  = new SqlConnection(_replicaConnString);  // SELECT

// Read-your-own-writes: just after a user writes, read THEM from the leader to
// avoid showing stale data from a lagging replica.
var conn = justWroteByThisUser ? write : read;

Key takeaways

🎯 Remember these three

  • Leader–follower: writes to the leader, reads from followers — scales reads and provides HA standbys.
  • Async replication is fast but creates replication lag (eventual consistency); sync avoids loss but is slower.
  • Watch read-your-own-writes and failover; replication copies the whole dataset (vs sharding, which splits it).

Interview tip

🗣️ When this comes up

“How do you scale reads?” → read replicas. Then show depth: async replication causes lag/eventual consistency, so explain read-your-own-writes and the sync-vs-async trade-off. Distinguish replication (copies all data) from sharding (splits data) — a common mix-up.
