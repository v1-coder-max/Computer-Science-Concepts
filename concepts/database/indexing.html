Database Indexing

An index is the book’s index for your table: instead of reading every page to find a row, the database jumps almost straight to it.

#indexing #b-tree #query-performance #sql

The idea in plain English

Without an index, finding WHERE CustomerId = 42 means a full table scan: the engine reads every row and checks each one. That’s O(n) — fine for 1,000 rows, catastrophic for 50 million.

An index is a separate, sorted structure — almost always a B-tree — that maps column values to row locations. Because it’s sorted and balanced, the engine navigates from the root down through a few branch levels to the exact leaf, turning the lookup into a O(log n) seek. For 50 million rows that’s roughly 3–4 page reads instead of millions.

The flavors you’ll actually use:

  • Clustered — defines the physical order of the table itself; the leaf level is the row data. One per table (usually the primary key).
  • Non-clustered — a separate structure whose leaves point back to the row (via the clustered key). Many per table.
  • Composite — indexes several columns in order; great for multi-column filters and sorts (order matters — leftmost-prefix rule).
  • Covering — INCLUDEs the extra columns a query needs so it’s answered entirely from the index, never touching the table.

Real-world example

🏢 How this works in production

When you open your Amazon orders page, the query is essentially SELECT … FROM Orders WHERE CustomerId = @you ORDER BY CreatedUtc DESC. Across billions of orders, a composite index on (CustomerId, CreatedUtc) lets the database seek straight to your slice already sorted — no scan, no sort step. The flip side: every index must be updated on each insert/update, so Amazon doesn’t index everything — write- heavy tables get only the indexes their read patterns actually justify.

How a B-tree seek works

To find key 50, the engine compares at each level and follows one pointer down — three hops, not a scan of every row.

In SQL & EF Core

Create a covering, non-clustered index so the hot query never touches the table:

SQL 
-- Without an index: a full scan, O(n) over the whole Orders table.
SELECT Total, CreatedUtc FROM Orders WHERE CustomerId = 42;

-- A covering, non-clustered index turns it into a seek, O(log n).
CREATE NONCLUSTERED INDEX IX_Orders_CustomerId
    ON Orders (CustomerId)          -- the key the WHERE filters on
    INCLUDE (Total, CreatedUtc);    -- carried in the leaf → query answered from the index alone

-- Inspect what the optimizer actually chose:
SET STATISTICS IO ON;               -- watch logical reads drop after the index exists
-- (SQL Server) view the plan: look for "Index Seek" instead of "Table Scan".

The same index, declared in EF Core so it travels with your migrations:

C# — EF Core 
protected override void OnModelCreating(ModelBuilder b)
{
    b.Entity<Order>()
        .HasIndex(o => o.CustomerId)               // key column
        .IncludeProperties(o => new { o.Total, o.CreatedUtc }) // covering
        .HasDatabaseName("IX_Orders_CustomerId");
}

⚠️ Indexes aren’t free

Every index is extra storage and a write tax — each INSERT, UPDATE, and DELETE must keep it in sync. A function on the column (WHERE YEAR(CreatedUtc) = 2024) makes the index non-sargable and forces a scan. Index for your real query patterns, then verify with the execution plan — don’t guess.

Key takeaways

🎯 Remember these three

  • An index is a sorted B-tree that turns a scan (O(n)) into a seek (O(log n)).
  • Clustered = the table’s physical order (one per table); non-clustered = a side structure pointing back to rows (many).
  • Indexes speed reads but tax writes and storage — and a function on the column disables them. Index deliberately.

Interview tip

🗣️ When this comes up

The classic is “this query is slow — what do you do?” Don’t jump to “add an index.” Say you’d read the execution plan first, look for a scan vs seek, check whether the predicate is sargable, and only then add a covering composite index in the right column order — while weighing the write cost.
