Redis Internals

Redis is an in-memory, single-threaded data-structure store. That combination is why it’s absurdly fast — and why it’s used for far more than caching.

#redis #in-memory #data-structures #cache

The idea in plain English

Two design choices explain Redis’s speed. First, it keeps data in RAM, so reads/writes avoid disk entirely. Second, it processes commands on a single thread via an event loop — which sounds limiting but means no locks, no race conditions, and each command is atomic. Most operations are O(1) or O(log n), so it handles hundreds of thousands of ops/second per node.

It’s a data-structure server, not just a key→string map. The values can be:

  • Strings — caching, counters (INCR).
  • Hashes — objects (a user’s fields).
  • Lists — queues/stacks.
  • Sets — uniqueness, membership.
  • Sorted sets — leaderboards/ranking by score.
  • Streams — append-only logs for messaging.

Though in-memory, Redis can be durable: RDB point-in-time snapshots and/or AOF (append-only file logging every write). When memory fills, an eviction policy (e.g. LRU/LFU) decides what to drop. Keys can carry a TTL, which makes it a natural cache and session store. It also offers pub/sub and atomic Lua scripts.

Real-world example

🏢 How this works in production

Redis quietly powers a huge amount of the web. It’s the cache in front of databases, the session store shared across a server fleet (so any node can resolve your session), the engine behind rate limiting (INCR + TTL per user), and real-time leaderboards via sorted sets (a game ranking millions of players queries a single ZRANK). Its single-threaded atomicity is exactly what makes a distributed counter or rate limiter correct without locks.

In-memory structures, one loop

In commands & C#

Redis CLI 
SET user:42:name "Ada"          # string
EXPIRE user:42:name 3600        # TTL → auto-evict in 1h (great for cache/sessions)
INCR ratelimit:42               # atomic counter → rate limiting
ZADD leaderboard 950 "ada"      # sorted set → leaderboard
ZREVRANK leaderboard "ada"      # O(log n) rank lookup
C# — StackExchange.Redis 
var redis = ConnectionMultiplexer.Connect("localhost:6379"); // reuse this (singleton)
IDatabase db = redis.GetDatabase();

await db.StringSetAsync("product:42", json, TimeSpan.FromMinutes(5)); // cache + TTL
long n = await db.StringIncrementAsync("ratelimit:42");               // atomic
await db.SortedSetAddAsync("leaderboard", "ada", 950);                // leaderboard

Key takeaways

🎯 Remember these three

  • Redis is fast because it’s in-memory and single-threaded — no locks, and every command is atomic.
  • It’s a data-structure store (strings, hashes, lists, sets, sorted sets, streams), not just key→string.
  • Optional persistence (RDB/AOF) + TTL + eviction make it ideal for cache, sessions, rate limits, and leaderboards.

Interview tip

🗣️ When this comes up

“Why is Redis so fast, and what would you use it for?” In-memory + single-threaded (atomic, lock-free). Then list concrete uses tied to data structures: cache (strings + TTL), rate limiting (INCR), leaderboards (sorted sets), sessions. Mention RDB/AOF so you’re not the person who thinks Redis can’t persist.
