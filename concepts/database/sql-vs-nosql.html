SQL vs NoSQL

Not a religious war — a fit question. Relational databases and NoSQL optimize for different things; pick by data shape, consistency needs, and scale.

#sql-vs-nosql #acid-vs-base #scaling #decision

The idea in plain English

Relational (SQL) databases — PostgreSQL, SQL Server, MySQL — store structured rows with a fixed schema and enforce relationships. Their strengths: ACID transactions, powerful ad-hoc queries and joins, and strong consistency. They traditionally scale vertically (and via read replicas), and changing the schema is a deliberate migration.

NoSQL stores (document, key-value, wide-column, graph) trade some of that for flexible schemas, easy horizontal scaling, and very high throughput — usually with denormalized data and BASE (eventual) consistency rather than strict ACID. They’re fast when the access pattern is known and singular, but weak at joins and multi-record transactions.

Decide with a few questions:

  • Relationships & transactions matter? (orders, payments, inventory) → SQL.
  • Schema varies / evolves fast, entity read as a whole? (catalog, profile, CMS) → document NoSQL.
  • Massive scale / one known query pattern / huge write volume? (feeds, time-series, cache) → key-value or wide-column.
  • Deeply connected data? (social graph, recommendations) → graph.

The mature answer is rarely “one or the other.” It’s polyglot persistence: SQL for the transactional core, plus the right NoSQL store for catalogs, caches, feeds, and relationships.

Real-world example

🏢 How this works in production

Amazon keeps orders and payments in relational/ACID systems (you cannot “eventually” charge a card), serves the product catalog from a denormalized document/NoSQL store, caches hot data in Redis, and runs recommendations on graph-like systems. The takeaway from real architectures isn’t “NoSQL replaced SQL” — it’s that each workload got the database that fits it, with the relational store still guarding the data that must be exactly right.

Key takeaways

🎯 Remember these three

  • SQL: structure, relationships, ACID, joins, strong consistency — choose for transactional/relational data.
  • NoSQL: flexible schema, horizontal scale, high throughput, BASE — choose by a known access pattern.
  • It’s usually both (polyglot persistence), with SQL guarding data that must be exact.

Interview tip

🗣️ When this comes up

“Would you use SQL or NoSQL for X?” Never answer dogmatically. Ask about relationships, transactions, query patterns, and scale, then justify — and mention polyglot persistence (SQL for the money, NoSQL for catalog/cache/feed). Tie consistency to ACID vs BASE and the CAP theorem.
