Isolation Levels

The “I” in ACID is a dial: how much one transaction sees of others’ in-flight work. Turning it up buys consistency; it costs concurrency.

#isolation #concurrency #mvcc #acid

The idea in plain English

When many transactions run at once, weaker isolation lets them see each other’s partial work, causing three read phenomena:

  • Dirty read — you read another transaction’s uncommitted change (which may be rolled back).
  • Non-repeatable read — you read a row twice and get different values because someone committed an update in between.
  • Phantom read — you re-run a range query and new rows have appeared.

The four standard levels each forbid more of them:

  • Read Uncommitted — allows all three (fastest, dirty reads possible).
  • Read Committed — no dirty reads (the common default).
  • Repeatable Read — also no non-repeatable reads.
  • Serializable — forbids all three; behaves as if transactions ran one at a time (safest, lowest concurrency).

Higher levels traditionally mean more locking, which reduces concurrency and risks deadlocks. Modern engines use MVCC / snapshot isolation instead: readers see a consistent snapshot without blocking writers. (PostgreSQL is MVCC-first; SQL Server offers Read Committed Snapshot.) The skill is picking the lowest level that’s still correct for the job.

Real-world example

🏢 How this works in production

An e-commerce checkout reserving the last item in stock needs strong isolation (or explicit locking) so two shoppers can’t both buy it — a non-repeatable/phantom read here means overselling. Meanwhile a big analytics report runs under snapshot isolation so it sees a consistent point-in-time view without blocking the thousands of orders being written concurrently. Same database, two very different isolation choices — matched to how wrong it is to be wrong.

Levels vs phenomena

In SQL & C#

SQL 
SET TRANSACTION ISOLATION LEVEL SERIALIZABLE;   -- strongest
BEGIN TRANSACTION;
    SELECT Stock FROM Products WHERE Id = 42;    -- consistent for the whole txn
    UPDATE Products SET Stock = Stock - 1 WHERE Id = 42;
COMMIT;

-- Non-blocking consistent reads (SQL Server):
ALTER DATABASE Shop SET READ_COMMITTED_SNAPSHOT ON;  -- MVCC-style reads
C# — EF Core 
using var tx = await db.Database.BeginTransactionAsync(
    System.Data.IsolationLevel.Serializable);   // pick the level explicitly
// ... work ...
await tx.CommitAsync();

Key takeaways

🎯 Remember these three

  • Three anomalies — dirty read, non-repeatable read, phantom — are prevented by progressively stronger levels.
  • Read Uncommitted → Committed → Repeatable Read → Serializable: more consistency, less concurrency.
  • MVCC/snapshot isolation gives consistent reads without blocking writers; choose the lowest correct level.

Interview tip

🗣️ When this comes up

Be able to name the three phenomena and which level stops each, and state the common default (Read Committed). The senior nuance: modern DBs use MVCC/snapshot so readers don’t block writers — and you should pick the lowest isolation that’s still correct.
