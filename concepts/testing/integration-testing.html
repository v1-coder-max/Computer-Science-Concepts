Integration Testing

Unit tests prove each part works alone; integration tests prove they work together — against a real database, a real HTTP pipeline, real serialization.

#testing #integration #database #api

The idea in plain English

An integration test exercises several components wired together, crossing a boundary your unit tests faked — typically your code plus the database, or a full request through the HTTP stack. It catches the bugs that live between units: a wrong SQL query, a broken DI registration, a serialization mismatch, a misconfigured route.

These are the failures mocks hide. A repository unit test with a mocked database can't catch a malformed query or a migration that didn't run — only a test against a real database can. That realism is the whole point.

Make them realistic but repeatable. In .NET, WebApplicationFactory spins up your app in-memory and lets you send real HTTP requests through the actual middleware pipeline. For the database, prefer a real engine in a disposable container (Testcontainers) over an in-memory substitute, because in-memory providers don't behave like the real SQL you ship.

They're slower and fewer than unit tests (the pyramid's middle), so reserve them for the wiring and the critical paths, and keep each one isolated — a fresh database per test or transaction rollback so tests don't bleed into each other.

Real-world example

🏢 How this works in production

Teams run a suite of integration tests that boot the API with WebApplicationFactory against a Postgres container started by Testcontainers, then hit real endpoints and assert on real database rows. This catches the classic 'works in unit tests, 500s in production' bugs — a missing migration, a JSON casing mismatch, an EF Core query that throws only against real SQL — none of which a mocked test would ever surface.

Test across the real boundary

In C#

C# 
// WebApplicationFactory boots the real app in-memory.
public class OrdersTests : IClassFixture<WebApplicationFactory<Program>>
{
    private readonly HttpClient _client;
    public OrdersTests(WebApplicationFactory<Program> f) => _client = f.CreateClient();

    [Fact]
    public async Task Post_Order_Persists_And_Returns_201()
    {
        var resp = await _client.PostAsJsonAsync("/orders", new { item = "book" });
        Assert.Equal(HttpStatusCode.Created, resp.StatusCode);   // real pipeline + real DB
    }
}

Key takeaways

🎯 Remember these three

  • Integration tests catch the bugs between units: wiring, SQL, serialization, configuration — what mocks hide.
  • Use WebApplicationFactory for the real HTTP pipeline and a real DB in a container (Testcontainers), not an in-memory fake.
  • They're slower and fewer (the pyramid's middle) — keep each isolated with a fresh DB or transaction rollback.

Interview tip

🗣️ When this comes up

Explain what integration tests catch that units can't, and argue for a real database in a container over an in-memory provider — the in-memory one passes tests that the real SQL would fail. That nuance shows production scars.
