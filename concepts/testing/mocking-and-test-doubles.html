Mocking & Test Doubles

To test a unit in isolation you replace its real dependencies with test doubles — stand-ins you control. The art is knowing which kind, and not mocking everything.

#testing #mocking #test-doubles #isolation

The idea in plain English

A test double is any stand-in for a real dependency. The family (Gerard Meszaros's taxonomy): a dummy is a placeholder that's never used; a stub returns canned answers (so you control inputs); a fake is a working but lightweight implementation (an in-memory repository); a spy records how it was called; and a mock is pre-programmed with expectations and verifies them.

The key distinction is state vs interaction testing. A stub supports state verification (you assert on the result). A mock supports interaction verification (you assert that a method was called with certain arguments). Use mocks when the interaction is the behavior — e.g., 'on success, it publishes an event.'

Doubles let you isolate the unit: replace the database, the clock, the email sender, the HTTP call — anything slow or nondeterministic — so the test is fast and repeatable. In C#, libraries like Moq or NSubstitute generate doubles from interfaces.

Don't over-mock. Mocking everything couples tests to implementation details and produces tests that pass while the real code is broken. Mock at architectural boundaries (I/O, external services); for pure logic and value objects, use the real thing.

Real-world example

🏢 How this works in production

A service test typically stubs the repository to return a known order, then verifies (mock) that on payment success an OrderShipped event was published — interaction testing where the publish is the behavior. Mature teams favor fakes (an in-memory store) over piles of mocks for readability, and reserve strict mocks for genuine boundaries, because heavily-mocked tests are the ones that break on every refactor and lull you into false confidence.

The family of test doubles

In C#

C# 
// Stub: control the input (state-based test).
var repo = new Mock<IOrderRepository>();
repo.Setup(r => r.Find(1)).Returns(new Order(1, paid: true));

// Mock: verify the interaction IS the behavior.
var bus = new Mock<IEventBus>();
var sut = new ShippingService(repo.Object, bus.Object);

sut.Ship(1);

bus.Verify(b => b.Publish(It.IsAny<OrderShipped>()), Times.Once);  // assert it published

Key takeaways

🎯 Remember these three

  • Test doubles (dummy, stub, fake, spy, mock) stand in for real dependencies to isolate the unit.
  • Stubs/fakes support state verification; mocks/spies verify interactions — use mocks when the call is the behavior.
  • Don't over-mock: double real boundaries (I/O, services), use the real thing for pure logic.

Know the taxonomy, but the real signal is the state-vs-interaction distinction and the over-mocking warning. 'Mock the boundaries, not the logic' shows you've felt the pain of brittle, over-mocked suites.
