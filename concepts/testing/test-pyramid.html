The Test Pyramid

A simple shape that tells you how many of each kind of test to write: lots of fast unit tests, fewer integration tests, very few slow end-to-end tests.

#testing #strategy #unit #integration

The idea in plain English

The test pyramid stratifies tests by scope and speed. The wide base is unit tests: tiny, fast, isolated, testing one class or function. You write thousands; they run in seconds and pinpoint exactly what broke.

The middle is integration tests: a few components together — your code plus a real database or HTTP call — catching the wiring bugs units can't. Fewer of these, because they're slower and touch more.

The narrow top is end-to-end (E2E) tests: the whole system through the UI or API. They give the most realistic confidence but are slow, brittle, and hard to debug, so you write only a handful covering critical user journeys.

The point is proportions. Inverting the pyramid into an 'ice-cream cone' — mostly slow E2E tests, few units — gives a test suite that's slow, flaky, and tells you that something broke but not where. Push tests down the pyramid: prefer a fast unit test to a slow E2E one whenever it can catch the same bug.

Real-world example

🏢 How this works in production

Healthy engineering orgs report exactly this distribution — thousands of unit tests running in under a minute on every commit, a few hundred integration tests, and a small, curated E2E suite for checkout/login-style flows. Teams that skipped units and leaned on E2E end up with a CI pipeline that takes an hour and 'flaky test' Slack threads; the cure is always to rebalance toward fast, isolated unit tests.

Many fast, few slow

In C#

C# 
// Unit (base): pure logic, no I/O — runs in microseconds.
[Fact]
public void Discount_Over100_Is10Percent()
    => Assert.Equal(90m, Pricing.Apply(100m));

// Integration (middle): real DB via a test container / in-memory host.
[Fact]
public async Task Saves_And_Reads_Back() { /* spin up DB, round-trip */ }

// E2E (top): drive the running app through its API/UI — few of these.

Key takeaways

🎯 Remember these three

  • Write many fast unit tests, fewer integration tests, very few slow E2E tests.
  • Inverting it (the ice-cream cone) gives a slow, flaky suite that can't localize failures.
  • Push tests down: prefer a unit test to an E2E test whenever it catches the same bug.

🗣️ When this comes up

Sketch the pyramid and name the anti-pattern (ice-cream cone). The strong answer explains why proportions matter: fast feedback and precise failure localization, not just 'more tests good.'
