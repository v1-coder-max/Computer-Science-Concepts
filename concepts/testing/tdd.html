Test-Driven Development

TDD inverts the usual order: you write a failing test first, make it pass with the simplest code, then refactor — repeating in tiny cycles. It's a design tool wearing a testing hat.

#testing #tdd #design #red-green-refactor

The idea in plain English

The loop is red-green-refactor. Red: write a small test for behavior that doesn't exist yet and watch it fail (proving the test works). Green: write the minimum code to make it pass — even something crude. Refactor: clean up the code now that a passing test protects you. Repeat in minutes-long cycles.

Writing the test first forces you to use your code before you build it, which surfaces awkward APIs and tight coupling early — so TDD is really a design discipline. Code written test-first tends to be more modular and dependency-injected, because untestable designs are painful to test-first and you feel it immediately.

It also yields a comprehensive regression suite as a by-product, and it keeps you focused: you only write code a test demands, which curbs gold-plating. The discipline is the small steps — a failing test, the simplest pass, a tidy-up — rather than writing a mountain of code and testing at the end.

TDD isn't a silver bullet. It shines for logic-rich code with clear inputs/outputs and less so for exploratory or UI-heavy work. The valuable habit even without strict TDD: write the test close to the code, and let testability pressure improve your design.

Real-world example

🏢 How this works in production

Teams practicing TDD on domain logic — pricing, validation, state machines — report fewer regressions and more confident refactoring, because every behavior has a test that was watched failing first (so it genuinely tests something). Even teams that don't do strict TDD adopt its core lesson: code that's hard to test is usually badly designed, so writing tests early pushes toward dependency injection and small, focused units.

Red → Green → Refactor

In C#

C# 
// RED — write the test for behavior that doesn't exist yet; watch it fail.
[Fact] public void Fizz_For_Multiples_Of_3() => Assert.Equal("Fizz", FizzBuzz.Say(3));

// GREEN — simplest code that passes.
public static string Say(int n) => n % 3 == 0 ? "Fizz" : n.ToString();

// REFACTOR — generalize safely (Buzz, FizzBuzz…) with the test guarding you,
// adding a new failing test for each new rule first.

Key takeaways

🎯 Remember these three

  • TDD = red-green-refactor: failing test first, simplest code to pass, then clean up — in tiny cycles.
  • It's a design discipline: writing the test first surfaces coupling and pushes toward testable, modular code.
  • Best for logic-rich code; even without strict TDD, 'hard to test ⇒ probably badly designed' is the lesson worth keeping.

Interview tip

🗣️ When this comes up

Describe red-green-refactor, then make the deeper point: TDD's main payoff is design feedback, not just tests. 'Writing the test first makes untestable, tightly-coupled designs hurt immediately' is the insight interviewers want.
