Unit Testing

A unit test checks one unit of behavior in isolation, fast and deterministically. Good ones are your safety net; bad ones are a tax you pay on every refactor.

#testing #unit #xunit #aaa

The idea in plain English

A unit test exercises a single unit — usually one class or method — with its dependencies controlled, so a failure points to exactly one place. It runs in milliseconds with no database, network, or file system, which is what lets you run thousands on every save.

Structure each test as Arrange-Act-Assert (AAA): set up the inputs, perform the one action under test, then assert on the outcome. One logical assertion per test keeps failures unambiguous.

Good unit tests are FIRST: Fast, Isolated (no shared state or ordering), Repeatable (same result every run — no clocks/randomness leaking in), Self-validating (pass/fail, no manual checking), and Timely (written with the code).

Test behavior, not implementation. Assert on observable outputs and effects, not private internals, so the test survives a refactor. A test that breaks every time you rename a private method is testing the wrong thing — it impedes the very refactoring tests are meant to enable.

Real-world example

🏢 How this works in production

Every CI pipeline runs the unit suite first because it's the fastest, most precise feedback available — a red unit test names the broken class in seconds. Teams use xUnit/NUnit with a mocking library for dependencies, and the well-tested codebases are the ones developers refactor fearlessly, because the behavior-focused unit tests catch regressions without breaking on internal changes.

Arrange · Act · Assert

In C#

C# 
[Fact]                                   // xUnit
public void Withdraw_MoreThanBalance_Throws()
{
    // Arrange
    var account = new Account(balance: 100m);

    // Act + Assert (the one behavior under test)
    var ex = Assert.Throws<InvalidOperationException>(
        () => account.Withdraw(150m));

    Assert.Equal(100m, account.Balance);   // state unchanged — behavior, not internals
}

Key takeaways

🎯 Remember these three

  • A unit test checks one unit in isolation, fast and deterministic, so failures localize precisely.
  • Structure as Arrange-Act-Assert; keep tests FIRST (Fast, Isolated, Repeatable, Self-validating, Timely).
  • Test behavior, not implementation, so tests survive refactors instead of obstructing them.

Interview tip

🗣️ When this comes up

AAA and FIRST are table stakes; the differentiator is 'test behavior, not implementation' — explain that coupling tests to internals breaks the refactoring safety net they're supposed to provide.
