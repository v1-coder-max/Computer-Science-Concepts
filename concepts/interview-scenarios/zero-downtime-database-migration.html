Zero-Downtime Database Migration

You can't take a maintenance window, and a single 'big bang' migration risks an outage. The trick is to make every step backward-compatible so old and new code run side by side.

#migrations #databases #deployment #backward-compatibility

❓ The interview question

“You need to rename a column / change a schema / move to a new database — with zero downtime, while the app keeps running. How?”

How to think about it

State the constraint that drives everything: during a rolling deploy, old and new code run at the same time against the same database. So no single migration step may break the version that's still running. That rules out destructive, all-at-once changes.

Expand–migrate–contract. The standard recipe. Expand: add the new column/table/field additively — nothing reads it yet, old code is unaffected. Migrate: deploy code that writes to both old and new (dual write) and backfill existing rows in batches. Contract: once everything reads the new shape and the backfill is verified, switch reads over, then drop the old column in a later release.

Backfill safely. Copy historical data in small batches with pauses, so you don't lock tables or saturate I/O. Make the backfill idempotent and resumable — it will get interrupted.

Cut over behind a flag. Gate the read switch with a feature flag so you can flip to the new path — and instantly flip back — without a deploy. Verify the two sources agree (shadow reads / comparison) before trusting the new one.

For a whole-database move, the same shape applies: dual-write to old and new, backfill, run them in parallel comparing results, then cut reads over and decommission the old store. Every step is reversible until the final contract.

Real-world example

🏢 How this works in production

This is how teams run schema changes on live systems without a maintenance page: tools like gh-ost and pt-online-schema-change build the new table and backfill in the background, and frameworks structure migrations as additive-then-cleanup. The cautionary tale is a single migration that renames a column in place — the old app instances still querying the old name start throwing errors the moment it runs. Expand/contract avoids that by never breaking the currently-deployed code.

Expand → migrate → contract

A safe rename, in three releases

SQL — additive now, destructive much later 
-- Release 1 (Expand): add the new column, nothing reads it yet.
ALTER TABLE users ADD COLUMN full_name text;

-- Release 2 (Migrate): app writes BOTH columns; backfill in batches.
UPDATE users SET full_name = name
WHERE full_name IS NULL AND id BETWEEN :lo AND :hi;   -- repeat, small ranges

-- Release 3 (Contract): all code now reads full_name → drop the old one.
ALTER TABLE users DROP COLUMN name;
-- Each release is independently deployable and the old app never breaks.

Key takeaways

🎯 Remember these three

  • During a rolling deploy old + new code share the DB, so every step must be backward-compatible — no big-bang destructive change.
  • Use expand → migrate → contract: add additively, dual-write + backfill in batches, then switch reads and drop the old shape later.
  • Gate the cutover behind a feature flag and verify with shadow reads; keep each step reversible until the final drop.

Interview tip

🗣️ When this comes up

The phrase to say is 'expand/contract — every migration step is backward-compatible because old and new code run together.' Dual-write + batched backfill + a flagged read cutover is the concrete recipe; emphasize reversibility until the final drop.
