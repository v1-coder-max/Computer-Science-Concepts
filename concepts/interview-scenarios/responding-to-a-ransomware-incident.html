Ransomware Hit Your Cloud Storage: The First 15 Minutes

The 3 AM call is testing your process under pressure, not your heroics. You contain, assess, and escalate first — you do not start 'fixing' a security incident alone.

#security #incident-response #backups #cloud

❓ The interview question

“Your CTO calls at 3 AM: the entire S3 bucket just got encrypted, with a ransom note in the object metadata. What do you do in the first 15 minutes?”

How to think about it

Open with the meta-point: this is an incident-response question. The wrong answer is to start poking at the bucket; the right answer is to follow a playbook — contain, assess, communicate, recover — and to recognize that 'ransom note in object metadata' means someone had write credentials, so a compromised key is the likely entry point.

Contain the blast radius (minutes 0–5). Stop the bleeding: revoke/rotate the credentials and access keys that could touch the bucket, lock down the bucket policy (deny writes/deletes), and isolate any compromised IAM principals or instances. The goal is to ensure the attacker can't encrypt or exfiltrate more.

Assess scope (minutes 5–10). What exactly is affected — one bucket or many? Is data encrypted in place, deleted, or exfiltrated (a copy stolen)? Crucially: are backups / object versions intact and isolated? With S3 versioning + Object Lock, the 'encryption' may just be new object versions over recoverable originals.

Communicate and escalate (minutes 10–15). Declare an incident, assign an incident commander, and pull in security, legal, and leadership. Don't go solo — ransomware has legal/regulatory and PR dimensions. Preserve evidence (logs, the ransom note) for forensics; do not destroy the crime scene while flailing.

Then recover — don't pay if you can avoid it. Restore from immutable/versioned backups (paying funds crime and doesn't guarantee a key). Afterward: root-cause the entry (leaked key? over-broad permissions?), rotate everything, enable MFA, tighten least-privilege, and turn on versioning + Object Lock so it can't recur. Test the restore.

Real-world example

🏢 How this works in production

Mature orgs run exactly this: an incident-response runbook, an on-call incident commander, and cloud guardrails that make recovery boring — S3 Versioning keeps prior object versions, Object Lock (WORM) makes them undeletable for a retention period, and MFA-delete blocks destructive calls. The breaches that turn catastrophic are the ones where a single long-lived access key had broad write permissions and backups lived in the same account with no versioning — so 'just restore' wasn't an option. The fix is always least-privilege credentials plus immutable, isolated backups.

Contain → Assess → Communicate → Recover

Make recovery boring, before the incident

config — versioning + object lock + MFA delete 
# Immutable, recoverable backups turn ransomware into an inconvenience.
aws s3api put-bucket-versioning --bucket prod-data \
    --versioning-configuration Status=Enabled,MFADelete=Enabled

# Object Lock (WORM): versions can't be deleted/overwritten for the retention period.
aws s3api put-object-lock-configuration --bucket prod-data \
    --object-lock-configuration 'ObjectLockEnabled=Enabled,Rule={DefaultRetention={Mode=COMPLIANCE,Days=30}}'

# Principle: short-lived, least-privilege credentials — no single key should be
# able to overwrite an entire bucket. Keep backups in a separate, locked-down account.

Key takeaways

🎯 Remember these three

  • It's an incident-response question: contain → assess → communicate → recover, and never start fixing a breach solo.
  • Contain first (revoke keys, lock the bucket); the entry point is almost certainly a compromised write credential.
  • Recovery depends on immutable, isolated backups (versioning + Object Lock) — set those up before the incident; don't pay the ransom.
