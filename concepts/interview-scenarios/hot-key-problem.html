The Hot Key (Celebrity) Problem

Sharding assumes load spreads evenly — but one celebrity user or one viral item can send all the traffic to a single shard. The fix is to spread the hot key's load even though sharding can't.

#scaling #sharding #caching #fan-out

❓ The interview question

“You shard by user ID, but one celebrity with 100M followers makes their shard melt while others idle. How do you handle the hot key?”

How to think about it

Frame it: sharding distributes by key, so an unusually popular key concentrates load on one node — a hot partition. Adding more shards doesn't help, because the hot key still lives on exactly one. You need a different lever.

Replicate the hot data for reads. Most hot keys are read-heavy (everyone views the celebrity's profile). Keep many read replicas of that data and fan reads across them, or — most effective — cache it aggressively so reads never reach the shard at all.

Cache hot keys hard. A hot key is the ideal cache candidate: high hit rate, read-mostly. A local in-process cache on each app server plus Redis means the storm is absorbed before it touches the database. (Watch for stampedes when it expires — coalesce the rebuild.)

Split the key for writes. If the hotness is writes (a viral post's like counter), shard the key itself: keep N sub-counters across nodes and sum them on read. This turns one hot row into N warm ones.

Rethink fan-out. The celebrity case is exactly why Twitter/Instagram use a hybrid timeline: fan-out-on-write for normal users, but fan-out-on-read for celebrities (don't push to 100M inboxes; pull their posts at read time). Detect hot keys and treat them on a special path.

Real-world example

🏢 How this works in production

Twitter's timeline is the canonical example: pushing a celebrity's tweet to 100M follower timelines on every post is infeasible, so their posts are merged in at read time instead. Sharded counters (split a viral like-count across many keys and sum them) are standard at scale, and CDNs/edge caches exist precisely to absorb the read storm on a single hot object. The unifying idea: detect the hot key and give it its own strategy.

Spread the hot key

Split a hot counter across sub-keys

C# — N warm counters instead of one hot row 
// ❌ One row for a viral post → every like contends on the same key.
// ✅ Spread writes across N shards of the counter, sum on read.
const int Shards = 20;

Task IncrementLikes(long postId) {
    int shard = Random.Shared.Next(Shards);
    return redis.StringIncrementAsync($"likes:{postId}:{shard}");
}

async Task<long> GetLikes(long postId) {
    var keys = Enumerable.Range(0, Shards).Select(i => (RedisKey)$"likes:{postId}:{i}");
    var vals = await redis.StringGetAsync(keys.ToArray());
    return vals.Sum(v => (long)v);            // N warm reads, none contended
}

Key takeaways

🎯 Remember these three

  • Sharding spreads keys, not load — one popular key concentrates on a single shard (hot partition), and more shards don't help.
  • Hot reads: replicate + cache aggressively (it's the ideal cache target); hot writes: split the key into N sub-counters and sum.
  • For social fan-out, treat celebrities specially — fan-out-on-read instead of pushing to millions of inboxes.

Interview tip

🗣️ When this comes up

Make the distinction crisp: sharding balances keys, not load, so a hot key defeats it. Split the answer by read-hot (cache/replicate) vs write-hot (split the key), and cite the celebrity fan-out-on-read pattern for the social-graph version.
