Designing a Rate Limiter

A rate limiter caps how fast a client can call you. The interesting parts are the algorithm (bursts vs smoothness) and making it work across many servers with one shared counter.

#api #rate-limiting #distributed-systems #algorithms

❓ The interview question

“Design a rate limiter that allows N requests per user per minute. How does it work across a fleet of servers?”

How to think about it

Start with the algorithms and their trade-offs. Fixed window (count per minute bucket) is trivial but allows a 2× burst at the window boundary. Sliding window smooths that by weighting the previous window. Token bucket — the most common — refills tokens at a steady rate and lets each request spend one, allowing controlled bursts up to the bucket size. Leaky bucket enforces a smooth output rate. Pick token bucket unless you need strictly smooth output.

Make it distributed. On a fleet, an in-memory counter per server is wrong (10 servers = 10× the limit). The counter must live in a shared store — typically Redis — so every instance sees the same state. Use an atomic operation (INCR with a TTL, or a small Lua script for token bucket) so concurrent requests can't race past the limit.

Key it correctly. Limit per user, per API key, or per IP depending on the threat — usually a composite like user:endpoint. Put the limiter at the edge / API gateway so bad traffic is rejected before it costs you backend work.

Respond properly. Over-limit requests get HTTP 429 Too Many Requests with a Retry-After header and ideally X-RateLimit-Remaining, so well-behaved clients back off instead of hammering.

Mention the trade-offs: Redis adds a network hop per request (acceptable; can be optimized with local pre-checks or approximate counting), and you decide between hard limits (reject) and soft limits (throttle/queue). For resilience, fail open or closed deliberately if Redis is down.

Real-world example

🏢 How this works in production

Every public API — Stripe, GitHub, Twitter — rate-limits with exactly these pieces: a token-bucket-style limit keyed per API key, enforced at the gateway, returning 429 + Retry-After and rate-limit headers. At scale the counter lives in Redis (or a purpose-built service) using atomic scripts so the limit holds across hundreds of instances. The classic bug is a per-instance in-memory limiter that silently allows N× the intended rate once you scale horizontally.

Token bucket on a shared counter

Atomic per-key limit in Redis

C# — fixed window with INCR + TTL 
// One shared, atomic counter per key+window — correct across all instances.
public async Task<bool> AllowAsync(string key, int limit, TimeSpan window)
{
    var bucket = $"rl:{key}:{DateTime.UtcNow:yyyyMMddHHmm}";   // per-minute window
    long count = await _redis.StringIncrementAsync(bucket);     // atomic
    if (count == 1) await _redis.KeyExpireAsync(bucket, window); // set TTL once
    return count <= limit;
}

// On reject:
//   return Results.StatusCode(429) with header Retry-After: <seconds>.
// (Token bucket is a small Lua script so the check + decrement is one atomic step.)

Key takeaways

🎯 Remember these three

  • Know the algorithms: fixed/sliding window (simple) vs token/leaky bucket (bursts vs smooth) — token bucket is the common default.
  • Across a fleet the counter must be a shared, atomic store (Redis INCR/Lua), or each instance allows N× the limit.
  • Key per user/API-key/IP, enforce at the gateway, and respond with 429 + Retry-After so clients back off.

Interview tip

🗣️ When this comes up

Compare the algorithms (token bucket is the safe default), then make the distributed point explicitly: the counter lives in Redis with atomic ops, not in process memory. Finish with the 429 + Retry-After contract and the fail-open/closed decision if the store is down.
