Debugging a Memory Leak in Production

A leak shows up as memory that climbs and never comes back down, ending in OOM restarts. You find it by capturing heap dumps over time and asking what's holding these objects alive.

#debugging #performance #memory #incident-response

❓ The interview question

“A service's memory keeps climbing until it gets OOM-killed and restarts. How do you find and fix the leak?”

How to think about it

Confirm the pattern first. Plot memory (RSS / managed heap) over time: a leak is a steady upward trend across requests that GC can't reclaim, often sawtoothing up to an OOM and restart. Rule out 'just high load' (which plateaus) and a too-small heap.

Capture heap snapshots over time. Take dumps at intervals (e.g. via dotnet-gcdump/dotnet-dump) and diff them. The object types that keep growing between snapshots are your suspects — that comparison is the single most useful technique.

Find what roots them. A leak in a GC language isn't unfreed memory — it's objects still referenced, so GC can't collect them. Follow the retention path to the root. The usual culprits: a static/long-lived collection that only ever grows, event handlers never unsubscribed (the subscriber is pinned by the publisher), IDisposable not disposed (streams, connections, timers), an unbounded cache, or a captured closure holding a big object.

Mitigate, then fix. Buy time by recycling the process / rolling restarts and, if needed, raising the limit — but that's a band-aid. The real fix is at the root: bound caches (size/TTL, or weak references), unsubscribe from events, dispose resources (using), and stop unbounded accumulation.

Prevent recurrence: add a memory alert on the growth trend, load/soak-test to surface slow leaks before production, and code-review for the classic patterns. Note GC tuning won't fix a true leak — if it's rooted, no collector can reclaim it.

Real-world example

🏢 How this works in production

The .NET diagnostics flow is exactly this: dotnet-counters to spot the trend, dotnet-gcdump to capture, and a diff in Visual Studio / PerfView to find the growing type and its GC roots. The textbook .NET leak is an event handler — a long-lived publisher holds a reference to a short-lived subscriber that forgot to unsubscribe, so neither is collected. Unbounded in-memory caches (a Dictionary that only grows) and undisposed HttpClient/connection patterns are the other repeat offenders.

The classic leak — and the fix

C# — unsubscribe and bound 
// ❌ LEAK: the long-lived publisher pins every subscriber that never unsubscribes.
publisher.OnTick += handler;            // subscriber can't be collected while subscribed

// ✅ Unsubscribe when done (or use weak event patterns):
publisher.OnTick -= handler;

// ❌ LEAK: an unbounded cache that only ever grows.
static readonly Dictionary<string, byte[]> _cache = new();

// ✅ Bound it — size/TTL eviction so it can't grow forever:
var cache = new MemoryCache(new MemoryCacheOptions { SizeLimit = 10_000 });
// And dispose resources: using var stream = File.OpenRead(path);

Key takeaways

🎯 Remember these three

  • Confirm a steady, non-recovering growth trend (not just high load), then capture and diff heap dumps to find the growing type.
  • A managed leak = objects still referenced (can't be GC'd); find the root — static collections, unremoved event handlers, undisposed resources, unbounded caches.
  • Mitigate with restarts, but fix at the root: unsubscribe, dispose, and bound caches — GC tuning won't reclaim a rooted object.
