Scaling a Web Application

The classic open-ended question. There's a well-worn path from one box to millions of users — walk it tier by tier and explain why each step is forced.

#scaling #architecture #load-balancing #caching

❓ The interview question

“You have a web app on a single server. Traffic is growing fast. How do you scale it to handle millions of users?”

How to think about it

Start by naming the journey, not a silver bullet: you scale incrementally, and each step is forced by a specific bottleneck. Step 1 — split the tiers. Move the database off the web server onto its own box so the two can be sized and scaled independently.

Step 2 — go stateless and scale out the app tier. Put a load balancer in front of several identical app servers. This only works if the app holds no local state, so push sessions and uploads to shared stores (Redis, object storage). Now you add or remove app instances freely.

Step 3 — add caching. Most traffic is reads. A cache (Redis/Memcached) in front of the database, plus a CDN for static assets, removes the majority of load before it ever reaches your data tier.

Step 4 — scale the data tier. Add read replicas for read-heavy load; introduce sharding when a single primary can't hold the write volume or dataset. Step 5 — make work asynchronous. Push slow tasks (emails, image processing, analytics) onto a message queue so requests return fast and spikes are absorbed.

Close with the meta-point: measure first. You don't sharding-and-queue a 10k-user app — you scale the tier that's actually the bottleneck, and the database is almost always the hardest one.

Real-world example

🏢 How this works in production

Practically every large product is some version of this stack: a CDN (CloudFront/Cloudflare) for static assets, a fleet of stateless app servers behind an L7 load balancer (ALB / nginx), Redis for sessions and hot data, a primary database with read replicas, and a queue (SQS/Kafka/RabbitMQ) for background work. Teams grow into it one bottleneck at a time — a startup runs everything on one box until response times climb, then peels off the database, then adds a cache, and so on.

The scaling stack

Make the app tier stateless

C# — the one change that unlocks horizontal scaling 
// ❌ Local state pins a user to one server — breaks load balancing.
private static readonly Dictionary<string, Cart> _carts = new();

// ✅ Shared state: any app instance can serve any request.
builder.Services.AddStackExchangeRedisCache(o => o.Configuration = "redis:6379");
builder.Services.AddDataProtection().PersistKeysToStackExchangeRedis(redis); // shared auth keys

// Offload slow work instead of doing it in the request:
await _queue.PublishAsync(new SendWelcomeEmail(userId));   // returns immediately
// → add/remove app servers behind the load balancer at will.

Key takeaways

🎯 Remember these three

  • Scale incrementally, each step forced by a real bottleneck: split tiers → stateless app fleet + LB → cache + CDN → replicas/sharding → async queue.
  • Horizontal scaling of the app tier requires statelessness — sessions, files, and caches live in shared stores.
  • The database is the hardest tier; caching and read replicas defer the pain, sharding eventually divides it.

Interview tip

🗣️ When this comes up

Lead with the journey and the bottleneck-driven reasoning, not a buzzword list. The two points that signal seniority: app servers must be stateless, and the database is the hard part — which opens the door to replicas, sharding, and caching.
