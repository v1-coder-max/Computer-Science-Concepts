Defending Against XSS & CSRF

XSS and CSRF are different attacks with different fixes that interviewers love to conflate. XSS runs their script on your page; CSRF makes the browser send a request the user didn't intend.

#security #frontend #xss #csrf

❓ The interview question

“What's the difference between XSS and CSRF, and how do you defend against each?”

How to think about it

XSS (Cross-Site Scripting) = the attacker gets their JavaScript to run in your page (via an unescaped comment, username, URL param). That script runs with full access to the DOM, cookies readable by JS, and the user's session — it can steal tokens, keylog, or act as the user.

Defend XSS by never trusting data as code. Output-encode/escape everything you render (frameworks like React/Razor auto-escape by default), sanitize any HTML you must allow, and add a Content Security Policy that blocks inline and unknown-origin scripts. Mark session cookies HttpOnly so even a successful XSS can't read the token.

CSRF (Cross-Site Request Forgery) = a malicious site causes the user's browser to send an authenticated request to your site, riding on cookies that are sent automatically. The attacker can't read the response, but the state change (transfer money, change email) already happened.

Defend CSRF by proving intent. SameSite cookies (Lax/Strict) stop the cookie being sent on cross-site requests — the single biggest mitigation. Add an anti-forgery token (synchronizer or double-submit) that a forging site can't know, and verify the Origin/Referer on state-changing requests.

Tie them together: this is exactly why the JWT-storage debate matters. HttpOnly cookies remove the XSS token-theft, but because cookies are auto-sent you then owe CSRF defenses. The two attacks pull on opposite ends of the same cookie, which is why you address both, not one.

Real-world example

🏢 How this works in production

Frameworks bake these in: React escapes interpolated values, ASP.NET Razor HTML-encodes by default and ships antiforgery tokens, and modern browsers default cookies to SameSite=Lax. Real incidents still happen through the gaps — a dangerouslySetInnerHTML with unsanitized input (XSS), or a state-changing GET endpoint with no token and a permissive cookie (CSRF). Defense-in-depth (CSP + HttpOnly + SameSite + tokens) is standard because any single layer can be bypassed.

Two different attacks

Layer the defenses

C# — CSP, antiforgery, SameSite 
// XSS: a Content-Security-Policy that blocks inline/unknown scripts.
app.Use(async (ctx, next) => {
    ctx.Response.Headers["Content-Security-Policy"] =
        "default-src 'self'; script-src 'self'";      // no inline scripts
    await next();
});

// CSRF: anti-forgery token + SameSite cookies on state-changing requests.
builder.Services.AddAntiforgery(o => o.Cookie.SameSite = SameSiteMode.Strict);
app.MapPost("/transfer", [ValidateAntiForgeryToken] (TransferDto d) => Move(d));
// Session cookies: HttpOnly + Secure so XSS can't read them, SameSite so CSRF can't ride them.

Key takeaways

🎯 Remember these three

  • XSS = attacker's script runs on your page → output-encode, CSP, HttpOnly cookies, sanitize.
  • CSRF = forged authenticated request via auto-sent cookies → SameSite cookies, anti-forgery tokens, Origin checks.
  • They're distinct threats needing distinct fixes — and HttpOnly cookies (anti-XSS) still require CSRF defense.
