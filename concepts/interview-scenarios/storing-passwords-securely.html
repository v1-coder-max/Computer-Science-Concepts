Storing Passwords Securely

You never store passwords — you store a slow, salted one-way hash of them. The salt defeats precomputed tables; the slowness defeats brute force.

#security #hashing #auth #cryptography

❓ The interview question

“How do you store user passwords in a database?”

How to think about it

Lead with the negatives, because each is a real breach pattern. Never plaintext. Never reversible encryption — if you can decrypt it, so can an attacker who steals the key. And never a fast general-purpose hash (MD5, SHA-256): GPUs compute billions per second, so a stolen hash dump is cracked quickly.

Use a purpose-built password hash: Argon2 (preferred today), bcrypt, or scrypt. These are deliberately slow and (for Argon2/scrypt) memory-hard, so brute-forcing is expensive even on specialized hardware.

Salt every password. A unique random salt per user means identical passwords produce different hashes, which defeats rainbow tables and stops an attacker cracking many accounts at once. bcrypt/Argon2 generate and embed the salt for you.

Tune the work factor. The cost parameter (bcrypt rounds, Argon2 time/memory) is a dial: set it so a hash takes ~100–250 ms on your hardware, slow enough to thwart brute force but fast enough for login. Because hardware improves, raise it over time and rehash on next login when you do.

Finish with the details that signal care: compare with a constant-time function (avoid timing leaks), optionally add an app-side pepper (a secret kept outside the DB), enforce MFA for sensitive accounts, and never log the password. Don't implement the hash yourself — use the platform's library.

Real-world example

🏢 How this works in production

ASP.NET Core Identity hashes with PBKDF2 by default and libraries for bcrypt/Argon2 are one line to adopt; every reputable auth system uses a slow salted hash with a tunable cost. The breaches that expose millions of accounts are the ones that used unsalted MD5/SHA1 — those dumps get cracked within hours of leaking. The modern recommendation (OWASP) is Argon2id with sensible memory/time parameters, falling back to bcrypt where Argon2 isn't available.

Password → salt + slow hash → store

Hash and verify with a slow algorithm

C# — never store the password itself 
// Registration: hash with a salted, slow algorithm (bcrypt shown; Argon2 preferred).
string hash = BCrypt.Net.BCrypt.HashPassword(password, workFactor: 12);  // ~150 ms
// store 'hash' (salt + cost are embedded in it); never store 'password'.

// Login: constant-time verify, and upgrade the cost factor over time.
bool ok = BCrypt.Net.BCrypt.Verify(password, hash);
if (ok && BCrypt.Net.BCrypt.PasswordNeedsRehash(hash, 12))
    await SaveAsync(BCrypt.Net.BCrypt.HashPassword(password, 12));  // rehash silently

Key takeaways

🎯 Remember these three

  • Never plaintext, never reversible encryption, never a fast hash (MD5/SHA) — use Argon2/bcrypt/scrypt.
  • A unique salt per user defeats rainbow tables; a tunable work factor makes brute force expensive (and raise it over time).
  • Use the platform's library, constant-time compare, optionally a pepper, and add MFA — don't roll your own.
