Scaling the Database Tier

The database is almost always the scaling wall, because it holds state. You climb it in a deliberate order — cache, replicate, partition, shard — because each step adds complexity you only want when forced.

#database #replication #sharding #caching

❓ The interview question

“Your single database can't keep up. Walk me through how you'd scale the data tier, in order.”

How to think about it

Set the order and the reason: each technique solves a different bottleneck and adds complexity, so you apply them only as needed. Tune before you scale — the right index, a fixed N+1 query, or a bounded connection pool often buys more headroom than any new server.

1. Cache. Most load is reads, and the cheapest read never reaches the database. A cache (Redis) in front absorbs the hot read path — frequently the single biggest relief, and the first thing to reach for.

2. Read replicas. Still read-bound? Add replicas and route reads to them, keeping writes on the primary. This scales reads near-linearly; the cost is replication lag — replicas are slightly stale, so route read-your-own-writes traffic to the primary.

3. Vertical scale / partition. A bigger box is the simplest write-side win until it hits a ceiling. Then partition within one database (split a huge table by range/time) to keep indexes and working sets small.

4. Shard. When one primary can't hold the write volume or dataset, split data across multiple databases by a shard key. This is the most powerful and most painful step: cross-shard queries and transactions get hard, and a bad key creates hot shards — so choose a high-cardinality, evenly-distributed key. CQRS (separate read/write models) is the natural companion once reads and writes diverge.

Real-world example

🏢 How this works in production

Almost every large system climbs this exact ladder: cache hot data in Redis, fan reads across replicas, and only shard when forced — because sharding is where joins, transactions, and ops complexity spike. Companies famously delay sharding as long as possible (bigger boxes, more caching, more replicas) precisely because of that cost. The recurring mistake is sharding too early (huge complexity for a problem caching would have solved) or choosing a low-cardinality shard key that just recreates a single hot database.

// Writes go to the primary; cache is invalidated on change.
public async Task UpdateAsync(Product p) {
    await _primary.SaveChangesAsync();
    await _cache.RemoveAsync($"product:{p.Id}");
}

Key takeaways

🎯 Remember these three

  • Tune first (indexes, N+1, connection pool) — it often beats adding hardware.
  • Climb in order: cache → read replicas → partition → shard; each rung targets a different bottleneck and adds complexity.
  • Sharding is the powerful-but-painful last resort — cross-shard joins/transactions get hard, and the shard key must be high-cardinality and evenly distributed.

Interview tip

🗣️ When this comes up

Give the ordered ladder — cache, replicas, partition, shard — and stress that you tune (indexes, pools) before scaling and shard last. The shard-key point (high cardinality, even distribution, or you get hot shards) is what separates a real answer from a buzzword list.
