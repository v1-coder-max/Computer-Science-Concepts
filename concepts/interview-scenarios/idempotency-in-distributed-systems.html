Idempotency in Distributed Systems

Networks make 'did it succeed?' unanswerable, so clients retry — and retries cause duplicates. Idempotency makes repeating an operation harmless, turning at-least-once delivery into effectively exactly-once.

#distributed-systems #idempotency #retries #consistency

❓ The interview question

“A payment request times out, so the client retries. How do you make sure the customer isn't charged twice?”

How to think about it

Start with why this is unavoidable: when a call times out, the client cannot know whether the server processed it (response lost) or not (request lost). The only safe move is to retry, which means the server must tolerate duplicate requests without doing the work twice.

Idempotency key. The client attaches a unique key to the operation (one per logical attempt, reused across retries). The server records processed keys: the first time it does the work and stores the result; every retry with that key returns the stored result without re-executing. One charge, no matter how many retries.

Dedup table / outbox. Persist (idempotencyKey → result) transactionally with the side effect. Because the insert and the work commit together, you can't end up having charged without having recorded the key.

Natural idempotency where possible. Some operations are inherently safe to repeat: SET status = 'paid' is idempotent; balance = balance - 10 is not. Designing operations as 'set to a value' rather than 'apply a delta' removes the problem at the source.

Tie it to messaging: queues deliver at-least-once, so consumers must be idempotent too. Idempotent consumers + retries = effectively-once processing, which is how real systems get 'exactly once' without the impossible guarantee.

Real-world example

🏢 How this works in production

Stripe's API takes an Idempotency-Key header on every mutating request — retry with the same key and you get the original charge back, never a second one. Kafka and SQS consumers are built to be idempotent because redelivery is normal, not exceptional. The recurring production bug is a retried webhook or a redelivered message that double-creates an order; the fix is always a dedup key persisted alongside the effect.

Key takeaways

🎯 Remember these three

  • Timeouts make success unknowable, so clients must retry — and the server must make repeats harmless.
  • An idempotency key + a persisted (key → result) dedup record turns retries into a replay of the stored result, not a second effect.
  • Prefer naturally idempotent operations (set a value, not apply a delta); queue consumers are at-least-once, so they must be idempotent too.

Interview tip

🗣️ When this comes up

Lead with why retries are mandatory (timeout = unknown outcome), then idempotency keys + a dedup record committed with the effect. Mentioning that queues are at-least-once — so 'exactly once' is really 'idempotent consumer + retry' — is the staff-level note.
