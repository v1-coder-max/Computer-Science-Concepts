Where to Store a JWT on the Frontend

This is a threat-model question, not a storage one. localStorage exposes the token to XSS; cookies expose you to CSRF. The right answer names both threats and picks the lesser evil.

#security #auth #jwt #frontend

❓ The interview question

“Where should you store a JWT on the frontend — localStorage or HttpOnly cookies? And why?”

How to think about it

Frame the trade-off precisely. localStorage is convenient (read it in JS, attach it as a Bearer header), but any JavaScript on the page can read it — so a single XSS flaw (a malicious script, a compromised dependency) lets an attacker steal the token and impersonate the user. That's the dealbreaker.

HttpOnly cookies can't be read by JavaScript, so an XSS bug can't exfiltrate the token. The catch: cookies are sent automatically with every request to your domain, which opens CSRF — a malicious site can trigger an authenticated request using the user's cookie. So you must add CSRF defenses.

The recommended default: HttpOnly + Secure + SameSite cookies. HttpOnly blocks JS access (XSS-safe for the token), Secure forces HTTPS, and SameSite=Lax/Strict stops the cookie from being sent on cross-site requests — which neutralizes most CSRF. Add an anti-forgery token for the remaining cases.

The modern pattern splits the difference: keep a short-lived access token in memory (JS variable — gone on refresh, never persisted, not XSS-persistable) and a refresh token in an HttpOnly cookie. The app silently refreshes; a stolen-in-memory token expires in minutes.

Close with the honest caveat: if you have XSS, you've largely lost regardless — an attacker can ride the session even with HttpOnly cookies. So storage choice is damage-limitation; the real fix is preventing XSS (CSP, output encoding) in the first place.

Real-world example

🏢 How this works in production

Security teams and frameworks push toward HttpOnly, Secure, SameSite cookies for session tokens precisely because localStorage's XSS exposure is a recurring breach vector — a leaked token from a third-party script is game over. SPAs that need Bearer-header semantics commonly use the in-memory access token + HttpOnly refresh cookie pattern (e.g., libraries and BFF/'backend-for-frontend' setups), so the long-lived secret never touches JavaScript and the short-lived one is gone the moment the tab closes.

XSS-exposed vs CSRF-exposed

Issue the token as a hardened cookie

C# — HttpOnly, Secure, SameSite 
// Store the token where JavaScript can't read it, only over HTTPS, not cross-site.
Response.Cookies.Append("access_token", jwt, new CookieOptions
{
    HttpOnly = true,                       // JS cannot read it → XSS can't steal it
    Secure   = true,                       // sent only over HTTPS
    SameSite = SameSiteMode.Strict,        // not sent on cross-site requests → blunts CSRF
    Expires  = DateTimeOffset.UtcNow.AddMinutes(15)
});
// Pair a long-lived refresh token (also HttpOnly) with a short access-token TTL,
// and add an anti-forgery token for state-changing requests.

Key takeaways

🎯 Remember these three

  • It's an XSS vs CSRF trade-off: localStorage is XSS-readable; cookies are auto-sent (CSRF).
  • Default to HttpOnly + Secure + SameSite cookies (XSS can't read the token) plus a CSRF token — avoid localStorage for sensitive tokens.
  • Best practice: short-lived access token in memory + refresh token in an HttpOnly cookie; and remember XSS prevention (CSP) is the real fix.

Interview tip

🗣️ When this comes up

Don't just pick one — name both threat models. The strong answer: HttpOnly+Secure+SameSite cookie (so XSS can't read the token) plus CSRF protection, or the in-memory access + HttpOnly refresh pattern. The maturity signal: 'if you have XSS, storage choice only limits damage — prevent XSS.'
