Preventing Cascading Failures

One slow dependency can take down everything: callers pile up waiting, exhaust their threads, and fail too — failure cascades. You stop it by failing fast and isolating.

#resilience #circuit-breaker #timeouts #bulkhead

❓ The interview question

“One downstream service gets slow. Soon your whole system is down, even features that don't use it. Why, and how do you prevent it?”

How to think about it

Explain the cascade: service A calls slow service B. Without a timeout, A's threads block waiting on B; as requests arrive, A runs out of threads and stops serving everything — including endpoints that never touch B. The failure propagated through resource exhaustion.

Timeouts everywhere. Never wait indefinitely on a network call. A bounded timeout means a slow B costs you one timeout, not a hung thread forever. This is the most important and most often missing piece.

Circuit breaker. After B fails repeatedly, stop calling it for a cooldown — fail fast and return a fallback instead of hammering a sick service and tying up your own resources. The breaker periodically tests B and closes again when it recovers.

Bulkheads. Isolate resources so one dependency can't consume them all — a separate, bounded connection/thread pool per downstream. If B's pool is exhausted, calls to C and D are unaffected (like watertight compartments in a ship).

Retries done right + fallbacks. Retry only idempotent calls, with exponential backoff and jitter (naive retries amplify an overload into a retry storm). When all else fails, degrade gracefully: cached data, a default, or a partial response beats a cascading outage.

Real-world example

🏢 How this works in production

This is the entire reason resilience libraries exist — Polly in .NET, Hystrix/Resilience4j in the JVM, and Envoy/service-mesh policies wrap every cross-service call with timeouts, retries, circuit breakers, and bulkheads. The textbook incident is a recommendation service slowing down and taking the whole storefront with it because the product page blocked on it; the fix is a short timeout plus a breaker so the page renders without recommendations instead of not at all.

Fail fast and isolate

Wrap calls with a resilience pipeline

C# — Polly: timeout, breaker, backoff 
var pipeline = new ResiliencePipelineBuilder()
    .AddTimeout(TimeSpan.FromSeconds(2))                 // never hang on B
    .AddCircuitBreaker(new() {                           // stop calling a sick B
        FailureRatio = 0.5, MinimumThroughput = 20,
        BreakDuration = TimeSpan.FromSeconds(15) })
    .AddRetry(new() {                                    // idempotent calls only
        MaxRetryAttempts = 3, BackoffType = DelayBackoffType.Exponential,
        UseJitter = true })
    .Build();

// Fallback instead of cascading when B is open:
var recs = breaker.IsOpen ? CachedRecommendations() : await pipeline.ExecuteAsync(CallB);

Key takeaways

🎯 Remember these three

  • Cascades spread through resource exhaustion: callers block on a slow dependency until they run out of threads.
  • Timeouts (fail fast) + circuit breakers (stop calling the sick service) are the core defenses.
  • Bulkheads isolate per-dependency resources; retries need backoff + jitter, and a fallback degrades gracefully.
