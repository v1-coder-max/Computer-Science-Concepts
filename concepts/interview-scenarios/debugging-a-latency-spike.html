Debugging a Production Latency Spike

A latency spike question tests method, not luck. You narrow systematically: what changed, which signals moved, and which tier owns the time — before you touch anything.

#observability #debugging #performance #monitoring

❓ The interview question

“Production p99 latency just jumped from 100 ms to 3 seconds. Walk me through how you'd find and fix it.”

How to think about it

First, scope and correlate. Is it all endpoints or one? All users or one region? When exactly did it start — and what changed at that moment? A deploy, a config flip, a traffic surge, or a feature flag is the suspect ~80% of the time, so the fastest fix is often to roll back the recent change and confirm.

Read the golden signals — latency, traffic, errors, saturation. Did traffic spike (load problem) or did traffic hold steady while latency rose (something got slower)? Is a resource saturated (CPU, memory, connection pool, disk)? This splits 'we're overloaded' from 'a dependency degraded.'

Follow the telemetry from coarse to fine: metrics → traces → logs. Dashboards say which service/tier is slow; a distributed trace of a slow request shows which span ate the time (the DB call? an external API? GC?); logs explain why at that span.

Isolate the tier. Common culprits: a missing index or lock contention making queries slow, an exhausted connection pool (requests queue for a connection), a slow downstream dependency with no timeout, GC pauses, or a cache that stopped absorbing reads (a stampede). The trace points you straight at it.

Mitigate, then fix. Stop the bleeding first — roll back, scale out, add a timeout/circuit breaker, or shed load — then do the durable fix (add the index, bound the pool, cache the hot path) and a blameless post-mortem so it doesn't recur.

Real-world example

🏢 How this works in production

Mature teams wire up the three pillars of observability — metrics (Prometheus/Grafana), traces (OpenTelemetry/Jaeger), and structured logs — exactly so this investigation is minutes, not hours. The single most common root cause in practice is 'a deploy 10 minutes ago,' which is why 'what changed?' and a fast rollback path are the first tools you reach for. Right behind it: an N+1 query or an exhausted connection pool surfacing only under production load.

Narrow it down: metrics → traces → logs

Instrument so a trace can answer 'where'

C# — spans make the slow tier obvious 
// OpenTelemetry: trace each hop so a slow request shows where the time went.
builder.Services.AddOpenTelemetry().WithTracing(t => t
    .AddAspNetCoreInstrumentation()      // inbound request span
    .AddHttpClientInstrumentation()      // downstream HTTP spans
    .AddEntityFrameworkCoreInstrumentation()  // DB query spans
    .AddOtlpExporter());

// Custom span around a suspect section:
using var act = Activity.Current?.Source.StartActivity("rank-feed");
act?.SetTag("user.id", userId);
// A trace of one 3s request now points straight at the offending span.

Key takeaways

🎯 Remember these three

  • Start with “what changed?” — a deploy/flag/traffic change is the usual cause; a fast rollback is the fastest mitigation.
  • Read the golden signals (latency, traffic, errors, saturation) to split overload from a degraded dependency.
  • Drill metrics → traces → logs to localize tier → span → cause; mitigate first (rollback/scale/timeout), then fix durably.
