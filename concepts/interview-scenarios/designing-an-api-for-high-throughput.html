Designing an API for 1 Million Requests/Second

A million RPS isn't one giant server — it's many small ones, plus the discipline of not doing work you can avoid. Cache hard, stay stateless, and make writes asynchronous.

#api #scaling #caching #load-balancing

❓ The interview question

“How do you design an API that survives one million requests per second?”

How to think about it

Reframe the number: 1M RPS is impossible on one box but routine across a fleet. The job is to (a) spread load horizontally and (b) eliminate as much per-request work as possible. Start by doing the math — 1M RPS over, say, 5k RPS per instance is ~200 stateless instances behind a load balancer.

Stay stateless and scale out. Every API instance must be interchangeable, so put sessions/state in shared stores and front the fleet with an L7 load balancer (and DNS/anycast above it). Now capacity is a dial.

Cache at every layer. The cheapest request never reaches your origin: a CDN/edge cache serves read-heavy and static responses, and Redis fronts the database for dynamic data. At a million RPS, even a 90% cache hit rate cuts origin load tenfold.

Make the request path cheap. Use async, non-blocking I/O so a few threads serve thousands of in-flight requests; pool database connections (a million users is not a million connections); keep payloads small; and prefer fewer round trips (batching, HTTP/2 keep-alive, gRPC for internal calls).

Defer writes and protect the edge. Acknowledge writes fast by dropping them on a queue and processing asynchronously, so spikes don't hit the database synchronously. Put rate limiting at the gateway to cap abuse, and scale the data tier with read replicas and sharding. Then close with: measure first — you optimize the tier that's actually the bottleneck.

Real-world example

🏢 How this works in production

High-throughput APIs (think a CDN's edge API, an ad bidder, or a feed service) look identical in shape: anycast + edge caches absorb most reads, a large fleet of stateless instances behind L7 load balancers handles the rest, Redis caches hot data, and writes are buffered through Kafka/SQS so the database sees a steady stream rather than the raw spike. The teams that hit these numbers obsess over cache hit rate and p99 latency per request, because at a million RPS a 1 ms regression is a million milliseconds of work per second.

Spread load, cache everything

Cheap request path: async, cached, rate-limited

C# — minimal API tuned for throughput 
// Cacheable read: served from the edge/output cache, not the DB, most of the time.
app.MapGet("/api/products/{id}", async (int id, IDb db) =>
        await db.Products.FindAsync(id))
   .CacheOutput(p => p.Expire(TimeSpan.FromSeconds(30)).Tag("products"));

// Writes: accept fast (202), process off the request path.
app.MapPost("/api/events", async (Event e, IQueue q) =>
{
    await q.PublishAsync(e);                 // no synchronous DB write
    return Results.Accepted();
});

// Cap abuse at the edge so a million RPS can't all be one client.
builder.Services.AddRateLimiter(o =>
    o.AddTokenBucketLimiter("api", b => { b.TokenLimit = 100; b.TokensPerPeriod = 100;
        b.ReplenishmentPeriod = TimeSpan.FromSeconds(1); }));

Key takeaways

🎯 Remember these three

  • A million RPS = a stateless fleet behind a load balancer, not one big server — capacity becomes a dial.
  • Cache at every layer (edge/CDN + Redis); a high hit rate is what makes the origin survivable.
  • Keep the request path cheap (async I/O, pooled connections, small payloads) and queue writes; rate-limit at the gateway.

Interview tip

🗣️ When this comes up

Do the back-of-envelope math (RPS ÷ per-instance capacity = fleet size) to show it's a horizontal-scaling problem. Then the three levers that actually carry a million RPS: cache hit rate, a cheap async request path, and queued writes.
