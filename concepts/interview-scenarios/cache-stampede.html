Cache Stampede & Thundering Herd

A cache stampede is when a popular key expires and thousands of requests all miss at once and hammer the database simultaneously. The fix is to make sure only one of them does the work.

#caching #concurrency #resilience #performance

❓ The interview question

“A hot cached item expires and suddenly the database gets hammered by thousands of identical queries. What happened, and how do you prevent it?”

How to think about it

Diagnose it: a single hot key (a homepage, a trending post) was serving everyone from cache. The moment it expires, every concurrent request misses, and they all run the same expensive query against the database at once — the 'thundering herd.' The database, sized for cache-hit traffic, falls over.

Request coalescing (single-flight). The core fix: when many requests need the same missing key, let one recompute it while the others wait for that result. One database query serves the whole herd instead of thousands.

Locking on rebuild. Equivalent idea across instances: the first request to miss takes a short distributed lock and repopulates the cache; everyone else briefly waits or serves stale. Only the lock holder touches the database.

Stale-while-revalidate. Serve the slightly-stale cached value immediately and refresh it in the background just before expiry, so users never actually hit a cold miss. This removes the synchronous miss entirely.

Stagger TTLs (jitter). If many keys were cached together, they expire together. Add randomness to each TTL so expirations spread out instead of firing in lockstep.

Real-world example

🏢 How this works in production

Caching libraries and CDNs build this in: HTTP's stale-while-revalidate, Go's singleflight, and .NET's HybridCache all coalesce concurrent misses so a hot key triggers one origin fetch. The cautionary tale is a site that sets the same 60-second TTL on a popular fragment for every server — every minute, on the same second, the whole fleet stampedes the database. Adding jitter and single-flight turns that spike into a single quiet refresh.

One rebuild, not thousands

Coalesce concurrent misses

C# — one rebuild serves all waiters 
// HybridCache coalesces concurrent misses for the same key automatically:
var product = await cache.GetOrCreateAsync(
    $"product:{id}",
    async ct => await db.Products.FindAsync(id),     // runs ONCE for the herd
    new HybridCacheEntryOptions {
        Expiration = TimeSpan.FromMinutes(5),
        // add jitter so keys don't all expire on the same tick
        LocalCacheExpiration = TimeSpan.FromMinutes(4 + Random.Shared.Next(2))
    });
// The first caller rebuilds; concurrent callers await the same Task.

Key takeaways

🎯 Remember these three

  • A stampede = a hot key expires and every concurrent request misses and hits the DB at once.
  • Fix with request coalescing / single-flight (or a rebuild lock): exactly one request recomputes, the rest wait or serve stale.
  • Stale-while-revalidate removes the synchronous miss; TTL jitter stops many keys expiring in lockstep.

Interview tip

🗣️ When this comes up

Name the term ('thundering herd' / 'cache stampede') and go straight to single-flight as the fix — it's the answer interviewers want. Bonus depth: stale-while-revalidate to avoid the miss entirely, and TTL jitter so keys don't expire in lockstep.
