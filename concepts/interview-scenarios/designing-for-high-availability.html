Designing for High Availability

High availability is the discipline of having no single point of failure — every tier redundant, and failover that happens automatically, fast, and without losing data.

#availability #redundancy #failover #reliability

❓ The interview question

“How would you design this service for high availability — say, 99.99% uptime?”

How to think about it

Anchor it in a number. 99.9% is ~8.8 hours of downtime a year; 99.99% is ~52 minutes. Adding a nine usually means removing a class of single failure, so first hunt for single points of failure (SPOFs): one app server, one database primary, one load balancer, one availability zone.

Redundancy at every tier. Run N+1 instances of the app behind a load balancer; run the database as a primary with replicas; deploy the load balancer itself as a managed, redundant service. The rule: any single component can die and the system keeps serving.

Spread across failure domains. Two app servers in the same rack or availability zone still share a fate. Distribute across multiple availability zones (and regions for the highest tiers) so a data-center outage doesn't take you down.

Detect and fail over automatically. Health checks remove unhealthy instances from rotation; the database promotes a replica to primary on failure. The two metrics that matter are RTO (how fast you recover) and RPO (how much data you can lose) — synchronous replication gives RPO≈0 at a latency cost.

Finally, accept the CAP trade-off: during a network partition you choose availability or consistency. And remember availability multiplies — a request touching five tiers that are each 99.9% is only ~99.5% end to end, so redundancy has to be everywhere.

Real-world example

🏢 How this works in production

Cloud providers expose this directly: you deploy across multiple Availability Zones, run a Multi-AZ RDS database that auto-promotes a standby on failure, and front everything with a managed load balancer that health-checks targets. A well-run service survives losing an entire AZ with a blip, not an outage — because nothing critical existed in just one place. The famous outages are usually a hidden SPOF: a single config service, a shared cache, or a region-wide dependency.

No single point of failure

Health checks drive failover

C# / YAML — only healthy instances receive traffic 
// App exposes a real readiness probe (checks its own dependencies):
builder.Services.AddHealthChecks()
    .AddNpgSql(connString)        // can I reach the DB?
    .AddRedis(redisConn);         // can I reach the cache?
app.MapHealthChecks("/health/ready");

/* The orchestrator/LB removes a failing instance automatically:
   readinessProbe:
     httpGet: { path: /health/ready, port: 8080 }
     periodSeconds: 5
     failureThreshold: 3        # 3 strikes → pulled from rotation   */

Key takeaways

🎯 Remember these three

  • High availability = no single point of failure: redundancy at every tier, spread across availability zones.
  • Recovery must be automatic — health checks pull bad instances, replicas get promoted; size it by RTO (recovery time) and RPO (data loss).
  • Availability multiplies across tiers, and during a partition you trade availability vs consistency (CAP) — so redundancy has to be everywhere.

Interview tip

🗣️ When this comes up

Start by quantifying the nines, then systematically eliminate SPOFs tier by tier. Drop RTO/RPO and the fact that availability multiplies across the call chain — both show you think about HA quantitatively, not as 'just add a replica.'
