Keeping Data Consistent Across Services

Once data is split across services with their own databases, you can't wrap a change in one transaction. You trade ACID for sagas, the outbox, and eventual consistency.

#distributed-systems #saga #outbox #eventual-consistency

❓ The interview question

“An order touches the Order, Payment, and Inventory services — each with its own database. How do you keep them consistent without a distributed transaction?”

How to think about it

Explain why the obvious answer is wrong: a distributed transaction (two-phase commit) across services is slow, locks resources across the network, and a coordinator failure can leave everyone blocked. At scale it's avoided. Instead you accept eventual consistency and design for it.

Saga pattern. Model the business operation as a sequence of local transactions, each in one service, coordinated by events. If a later step fails, run compensating transactions to undo the earlier ones (refund the payment, release the inventory). The system is never globally locked — it converges to a consistent state.

Transactional outbox. The dangerous gap is 'commit my DB change and publish an event' — do them separately and a crash in between loses the event. The outbox writes the event into an outbox table in the same local transaction as the data change; a relay then publishes it. Now the event can't be lost (and consumers are idempotent to tolerate duplicates).

Idempotent, ordered consumers. Events are delivered at-least-once and possibly out of order, so each consumer must dedup (idempotency key) and handle reordering. This is what makes 'effectively once' processing real.

Set expectations: there's a window where Order says 'placed' but Inventory hasn't decremented yet. That's fine for most domains — design the UX around 'processing' states rather than pretending everything is instantly consistent.

Real-world example

🏢 How this works in production

This is the backbone of event-driven microservices: an order service writes the order and an OrderPlaced event in one transaction (outbox), the payment and inventory services react, and a failed payment triggers a compensating cancel. Real systems lean on this rather than 2PC because it's resilient and scalable. The classic bug it prevents is the dual-write problem — saving to the database and publishing to the broker as two steps, then losing the event when the process dies in between.

Saga + outbox, not 2PC

Write the event in the same transaction

C# — the transactional outbox 
// One local transaction: the state change AND the event, atomically.
await using var tx = await _db.BeginTransactionAsync();

_db.Orders.Add(order);                                  // business data
_db.Outbox.Add(new OutboxMessage("OrderPlaced", order.Id));  // event, same tx
await _db.SaveChangesAsync();
await tx.CommitAsync();           // both commit, or neither — no lost event

// A separate relay polls the outbox and publishes to the broker;
// consumers are idempotent because delivery is at-least-once.

Key takeaways

🎯 Remember these three

  • Avoid distributed transactions (2PC) across services — use sagas: local transactions coordinated by events, with compensations to undo on failure.
  • Use the transactional outbox to write state + event atomically, so the dual-write problem can't lose events.
  • Embrace eventual consistency: consumers are idempotent and the UX models 'processing' states between steps.
