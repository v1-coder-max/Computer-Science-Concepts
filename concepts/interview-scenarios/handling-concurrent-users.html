Handling Many Concurrent Users

“Concurrent users” is a question about where the bottleneck is — threads, connections, or CPU — and how you avoid blocking and falling over under load.

#concurrency #async #throughput #backpressure

❓ The interview question

“Your service needs to handle 50,000 concurrent users. How do you make a single instance handle that, and what breaks first?”

How to think about it

First, separate concurrency (connections in flight) from throughput (requests finished per second). Most “concurrent users” are idle between clicks, so the real question is how many in-flight requests you can hold without exhausting a resource.

Use async, non-blocking I/O. A thread-per-request model caps you at a few thousand threads (each costs ~1 MB of stack and context-switch overhead). With async/await, a thread is released while waiting on the database or network, so a handful of threads service thousands of concurrent requests. This is the single biggest lever.

Pool your scarce downstream resources. The bottleneck usually moves to the database: 50k users do not mean 50k DB connections. Use a bounded connection pool (say 20–100) and let requests queue briefly for one — a database melts if you open a connection per request.

Add backpressure and load shedding. When demand exceeds capacity, a bounded queue plus fast rejection (HTTP 429/503) keeps the service alive and predictable instead of collapsing into timeouts. Shedding 5% of load beats failing 100%.

Then scale out: stateless instances behind a load balancer multiply the ceiling. For huge fan-out of long-lived connections (chat, live feeds), reach for WebSockets/SignalR with a backplane so connections spread across instances.

Real-world example

🏢 How this works in production

ASP.NET Core, Node, and Go all default to async, event-loop or lightweight-thread models precisely so one box can hold tens of thousands of connections. The classic outage post-mortem is the opposite: a synchronous call or an unbounded connection pool exhausts threads or database connections during a spike, every request starts timing out, and retries amplify the overload. The fix is almost always 'make it async + bound the pool + shed load.'

Thread-per-request vs async

Async end-to-end + a bounded pool

C# — never block on I/O; cap downstream concurrency 
// ✅ Async all the way down — the thread is returned to the pool during the await.
app.MapGet("/orders/{id}", async (int id, OrderDb db) =>
    await db.Orders.FindAsync(id));           // no thread held while the DB works

// ❌ .Result / .Wait() block a pool thread → thread starvation under load.

// Bound concurrency to a slow dependency so 50k users != 50k DB connections:
var gate = new SemaphoreSlim(50);             // at most 50 in flight downstream
await gate.WaitAsync();
try { return await CallPaymentProviderAsync(); }
finally { gate.Release(); }                    // excess requests queue, then 429 if needed

Key takeaways

🎯 Remember these three

  • Separate concurrency from throughput; most concurrent users are idle, so the limit is in-flight requests, not total users.
  • Async, non-blocking I/O is the biggest lever — it frees the thread during waits; never block (.Result/.Wait()) on the request path.
  • Bound scarce resources (connection pools) and add backpressure / load shedding so the service degrades gracefully instead of collapsing.

Interview tip

🗣️ When this comes up

The senior answer is 'async to free threads, bounded pools so I don't overwhelm the database, and backpressure so I shed load instead of dying.' Mention that retries during overload amplify the problem — a subtle point most miss.
