Surviving Traffic Spikes

A spike is a capacity-vs-demand mismatch that appears faster than you can add servers. You win by absorbing, flattening, and, if needed, shedding load.

#scaling #autoscaling #rate-limiting #queues

❓ The interview question

“A flash sale (or a viral post) will 50× your traffic in seconds. How do you keep the site up?”

How to think about it

Acknowledge the core problem: autoscaling takes minutes (boot, warm up, join the pool) but a spike hits in seconds, so you cannot purely 'scale your way out' in real time. You combine four tactics.

Absorb with caching. The cheapest request is the one that never reaches your origin. Cache aggressively at the CDN and in Redis; for a flash sale, the product page is identical for everyone — serve it from cache so the database is untouched by the read storm.

Flatten with a queue. Convert synchronous spikes into a steady stream: accept the request, drop the work (order, signup) onto a queue, and process at the rate your backend can sustain. The user gets an instant 'we've got it' while workers drain the backlog.

Pre-scale and rate-limit. If the spike is known (a launch at noon), warm up capacity beforehand instead of reacting. Put rate limits in front to cap per-user load and blunt bots, and keep autoscaling on for the sustained tail.

Degrade gracefully. When you still exceed capacity, shed load deliberately: disable non-essential features, return cached/stale data, show a lightweight queue page. A fast 'try again' for some beats a total outage for all.

Real-world example

🏢 How this works in production

Ticketing sites and game launches live this every day: a waiting-room page (load shedding by design) holds users in a queue, the catalog is fully cached, and purchases are enqueued and processed steadily so the payment/database tier never sees the full spike. Retailers pre-warm capacity before a Black Friday drop rather than trusting reactive autoscaling, because cold-start latency during the surge is exactly when you can't afford it.

Flatten the spike with a queue

Accept fast, process steadily

C# — enqueue on the spike, drain in the background 
// On the hot path: validate, enqueue, return immediately (HTTP 202).
app.MapPost("/checkout", async (Order o, IQueue q) =>
{
    await q.PublishAsync(o);                  // milliseconds, no DB write yet
    return Results.Accepted($"/orders/{o.Id}", new { o.Id, status = "queued" });
});

// A background worker drains at the rate the database can sustain:
await foreach (var order in queue.ReadAllAsync(ct))
    await ProcessWithRetryAsync(order);       // steady, backpressured throughput

Key takeaways

🎯 Remember these three

  • You can't autoscale in seconds — combine caching (absorb), a queue (flatten), and pre-warming/rate limits (cap).
  • Turn synchronous spikes into asynchronous streams: accept fast (202), process at a sustainable rate.
  • Degrade gracefully — stale data, disabled extras, a waiting room — because partial service beats a full outage.

Interview tip

🗣️ When this comes up

Make the timing point explicitly: autoscaling is minutes, spikes are seconds, so caching + queue + pre-warm do the real work. Finishing with graceful degradation (a waiting room, stale reads) shows you'd rather control the failure than suffer it.
