Read-Heavy vs Write-Heavy Systems

The first question to ask about any system is the read/write ratio — because reads and writes are scaled with opposite tools, and optimizing the wrong one wastes effort.

#scaling #caching #replication #architecture

❓ The interview question

“How does your design change if the system is 99% reads versus 90% writes?”

How to think about it

Establish the principle: reads fan out, writes funnel in. A read can be served from any copy of the data, so you scale reads by making more copies. A write must update the source of truth consistently, so copies make writes harder. They pull in opposite directions.

Read-heavy (e.g. a news site, a profile page). Cache aggressively (CDN + Redis) so most reads never touch the database; add read replicas and send reads to them. Denormalize and precompute so a read is a single lookup. The accepted cost is replication lag — replicas are slightly stale (eventual consistency).

Write-heavy (e.g. metrics ingestion, event logging, IoT). Caching and replicas don't help the write path. Instead: shard to spread writes across nodes, batch small writes into bigger ones, buffer through a queue to absorb bursts, and pick storage built for writes — LSM-tree stores (Cassandra, RocksDB) beat B-trees on write throughput.

The replica trade-off flips. Replicas help reads but each one adds write cost (every write propagates to all of them), so a write-heavy system keeps replicas lean and leans on partitioning instead.

Many systems are split-brained: read-heavy serving + write-heavy ingestion. That's exactly when CQRS earns its keep — separate the write model (optimized for ingest) from the read model (optimized for queries), connected by a stream.

Real-world example

🏢 How this works in production

A media site is ~99% reads: it's mostly CDN and cache hits with a small database behind read replicas, and nobody minds if a view count is a few seconds stale. A telemetry pipeline is the inverse: millions of writes per second land in Kafka, get batched, and are stored in a write-optimized column/LSM store — reads are comparatively rare aggregations. Picking the cache-and-replica playbook for a write-heavy ingest (or sharding a read-mostly app) is the classic mismatch.

Opposite playbooks

Route reads to replicas, writes to primary

C# — split the two paths 
// Read-heavy: send queries to a replica, accept slight staleness.
public Task<Article> GetAsync(int id) =>
    _replica.Articles.AsNoTracking().FirstAsync(a => a.Id == id);   // cache in front

// Writes always go to the primary (source of truth):
public async Task PublishAsync(Article a) {
    _primary.Articles.Add(a);
    await _primary.SaveChangesAsync();
}

// Write-heavy ingest: buffer + batch instead of one-row-at-a-time.
await _channel.Writer.WriteAsync(metric);          // queue
// background: flush 5,000 metrics per bulk insert

Key takeaways

🎯 Remember these three

  • Reads and writes scale with opposite tools — ask the read/write ratio first.
  • Read-heavy → cache + read replicas + denormalize (accept replication lag); write-heavy → shard + batch + queue + LSM stores.
  • Replicas help reads but tax writes; when a system is both, split the models with CQRS.

Interview tip

🗣️ When this comes up

Open by asking for the read/write ratio — it signals you know the playbooks diverge. Reads → cache/replicas; writes → shard/batch/queue/LSM; and reach for CQRS when the system is heavy in both directions.
