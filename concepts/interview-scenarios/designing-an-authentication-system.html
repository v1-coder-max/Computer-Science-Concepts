Designing Authentication & Sessions

Authentication design is mostly one trade-off — stateful sessions vs stateless tokens — plus the details (refresh tokens, revocation, MFA, and not storing passwords yourself).

#auth #sessions #oauth #security

❓ The interview question

“Design the authentication system for a web application. Sessions or JWTs?”

How to think about it

Frame the core choice. Server sessions: on login you create a session, store its state server-side (Redis/DB), and hand the client an opaque session ID in a cookie. Easy to revoke (delete the session) and small cookies, but it's stateful — every request does a lookup, and the store must scale.

Stateless JWTs: the token itself carries signed claims, so any server validates it with no lookup — great for horizontal scale and microservices. The cost: you can't easily revoke a JWT before it expires (it's self-contained), so a stolen token is valid until expiry.

Resolve the revocation problem with short TTLs + refresh tokens. Issue a short-lived access token (minutes) plus a long-lived refresh token; the access token's brief life limits damage, and you revoke by invalidating the refresh token (a denylist or rotating refresh tokens). This is the common production pattern.

Store tokens safely (the JWT-storage question): access token in memory, refresh token in an HttpOnly+Secure+SameSite cookie. Don't store passwords if you can delegate — use OAuth2/OIDC (Google/GitHub/Auth0) so a provider handles credentials and you get an identity token.

Round it out with the security essentials: hash passwords properly if you do own them, offer MFA, build secure password-reset and email-verification flows, lock/throttle after failed attempts (credential stuffing), and log auth events. Match the choice to the system — sessions for a single web app, JWT/OIDC for APIs and microservices.

Real-world example

🏢 How this works in production

Most large systems land on a hybrid: OIDC/OAuth for login (delegating credentials to an identity provider), short-lived JWT access tokens for stateless API calls, and refresh tokens (HttpOnly cookie, often rotating) for longevity with revocation. Single classic web apps still happily use server-side sessions because revocation and simplicity matter more than statelessness. The anti-pattern is a long-lived JWT in localStorage with no refresh story — unrevocable and XSS-exposed.

Sessions vs tokens, and the refresh pattern

Cookie sessions vs JWT bearer

C# — pick the model deliberately 
// Stateful: server session, opaque ID in a hardened cookie — easy to revoke.
builder.Services.AddAuthentication().AddCookie(o => {
    o.Cookie.HttpOnly = true; o.Cookie.SecurePolicy = CookieSecurePolicy.Always;
    o.Cookie.SameSite = SameSiteMode.Strict; o.ExpireTimeSpan = TimeSpan.FromHours(8);
});

// Stateless: JWT bearer — no server lookup, scales across services.
builder.Services.AddAuthentication().AddJwtBearer(o => {
    o.TokenValidationParameters = new() {
        ValidateIssuer = true, ValidateAudience = true,
        ValidateLifetime = true };            // short TTL + refresh token for revocation
});
// Or delegate the whole thing: AddOpenIdConnect(...) → don't store passwords at all.

Key takeaways

🎯 Remember these three

  • Core trade-off: sessions (stateful, easy revoke, needs a store) vs JWTs (stateless, scalable, hard to revoke).
  • Square the circle with short-lived access tokens + refresh tokens; store access in memory, refresh in an HttpOnly cookie.
  • Delegate to OAuth2/OIDC to avoid storing passwords; add MFA, throttling, and secure reset flows.

🗣️ When this comes up

Frame it as sessions-vs-JWT and immediately name the JWT revocation weakness, then solve it with short access TTLs + refresh tokens. Mentioning OIDC ('don't store passwords if you can delegate') and MFA shows breadth beyond the token mechanics.
