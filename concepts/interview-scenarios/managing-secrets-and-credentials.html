Managing Secrets & Credentials

Secrets in source control are the breach that keeps happening. The rule: secrets live in a vault, are injected at runtime, are short-lived, and are least-privilege.

#security #secrets #devops #cloud

❓ The interview question

“How do you manage database passwords, API keys, and other secrets in your application?”

How to think about it

Start with the cardinal sin: never commit secrets to code or git, and never bake them into images or config files. Once a key is in git history it's effectively public — bots scan for exactly that. Even private repos leak.

Store them in a secrets manager. Use a dedicated vault — HashiCorp Vault, AWS Secrets Manager, Azure Key Vault, GCP Secret Manager. It encrypts secrets at rest, controls and audits access, and centralizes rotation. The app references a secret name, not the value.

Inject at runtime, prefer identities over keys. The app fetches secrets at startup via an identity it already has — a managed identity / IAM role — so there's no bootstrap secret to leak. Short-lived, automatically-rotated credentials beat long-lived static keys (the latter are what get found in breaches, like the ransomware case).

Least privilege and scope. Each service gets only the secrets and permissions it needs. A leaked credential should unlock as little as possible — not god-mode over the whole account.

Rotate and detect. Rotate regularly and immediately on suspected compromise (a vault makes this a non-event). Run secret-scanning in CI and on the repo to catch accidental commits, and audit access logs for anomalies. Encrypt in transit and at rest throughout.

Real-world example

🏢 How this works in production

Modern deployments wire the app to a vault via a managed identity: an Azure app reads connection strings from Key Vault using its managed identity, an AWS task assumes an IAM role and pulls from Secrets Manager — no secret is ever in the repo or the container image. The recurring headline breach is the opposite: a long-lived AWS key committed to a public GitHub repo, found by a bot within minutes, used to encrypt or exfiltrate data. Secret scanning, short-lived credentials, and least-privilege roles exist precisely to kill that pattern.

Secrets out of code, into a vault

Reference secrets, don't embed them

C# — pull from Key Vault via managed identity 
// ❌ Never: hard-coded secret in code or appsettings committed to git.
// var conn = "Server=...;Password=SuperSecret123";

// ✅ Fetch at runtime using the app's managed identity — no bootstrap secret.
builder.Configuration.AddAzureKeyVault(
    new Uri("https://my-vault.vault.azure.net/"),
    new DefaultAzureCredential());           // managed identity, not a key

var conn = builder.Configuration["Db:ConnectionString"];   // resolved from the vault
// Rotate in the vault → apps pick up the new value with no code change.

Key takeaways

🎯 Remember these three

  • Never in code or git — a committed secret is a public secret; bots find them in minutes.
  • Store in a secrets manager/vault, inject at runtime via a managed identity (no bootstrap secret to leak).
  • Prefer short-lived, least-privilege credentials, rotate regularly, and run secret-scanning in CI.
