Designing So You Never Lose Data

Durability means an acknowledged write survives anything — a crash, a disk, a whole data center. You buy it with redundancy and honest acknowledgments, and you pay for it in latency.

#durability #replication #backups #reliability

❓ The interview question

“How do you design a system so that once a user's action is confirmed, that data is never lost — even if a server dies a second later?”

How to think about it

Define the contract precisely: durability is about what an acknowledgment promises. The danger is acking before the data is safely persisted and replicated — then a crash loses 'confirmed' work. So make the ack honest.

Write-ahead log + fsync. Databases append the change to a durable log and flush it to disk before acknowledging. The WAL is the source of truth for crash recovery; an in-memory buffer that hasn't been fsynced is not durable.

Replicate before you ack. A single disk can fail. With synchronous replication, the primary waits for at least one (or a quorum of) replicas to persist the write before confirming. Now a single-node loss can't lose acknowledged data — the trade-off is added write latency.

Quorum math. With N replicas, requiring W acks on write and R on read with W + R > N guarantees a read sees the latest write. This is the dial distributed databases expose (e.g., Cassandra/Dynamo consistency levels).

Backups and point-in-time recovery guard against the failures replication can't: a bad deploy, an accidental DELETE, corruption that replicates. Keep periodic snapshots plus the log so you can restore to any moment — and test restores, because an untested backup isn't a backup.

Real-world example

🏢 How this works in production

PostgreSQL acks a commit only after its WAL record is flushed; configure synchronous_commit with a standby and the primary waits for the replica too. Kafka does the same with acks=all and min.insync.replicas — a producer's write is acknowledged only once a quorum of brokers has it. The classic data-loss incident is the opposite: async replication plus a primary failure loses the last few seconds of 'committed' orders, which is why payment systems pay the latency for synchronous, quorum-acked writes.

Acknowledge only after it's safe

Make the acknowledgment durable

config — wait for the log and the replicas 
# PostgreSQL: don't confirm a commit until WAL is flushed AND a standby has it
synchronous_commit = on
synchronous_standby_names = 'replica1'   # primary waits for the replica

# Kafka producer: ack only when a quorum of in-sync replicas has the record
acks = all
min.insync.replicas = 2                   # tolerate losing 1 of 3 brokers
# Trade-off: every write pays one network round trip for durability.

Key takeaways

🎯 Remember these three

  • Durability is about honest acknowledgments — never confirm a write until it's persisted (WAL + fsync) and replicated.
  • Synchronous/quorum replication (W + R > N) protects acknowledged data from single-node loss, at a latency cost.
  • Backups + point-in-time recovery cover what replication can't (bad deploys, accidental deletes); test the restore.

Interview tip

🗣️ When this comes up

The key sentence is 'don't acknowledge until it's durable and replicated.' Mention the durability-vs-latency trade-off and quorum (W + R > N), and that backups guard against logical errors replication will happily copy.
