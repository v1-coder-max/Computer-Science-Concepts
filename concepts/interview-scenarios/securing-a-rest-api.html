Securing a REST API

API security is layers, not a single switch. Encrypt the transport, prove who's calling, check what they're allowed to touch on every object, and validate everything they send.

#security #api #auth #owasp

❓ The interview question

“How would you secure a public REST API?”

How to think about it

Give a layered checklist rather than one trick. Transport: HTTPS/TLS everywhere — no exceptions — so credentials and data aren't sniffable; redirect HTTP to HTTPS and use HSTS.

Authentication — who are you? Use a proven scheme: OAuth2/OIDC with JWTs or opaque tokens, or API keys for service-to-service. Never invent your own auth or crypto. Validate token signature, issuer, audience, and expiry on every request.

Authorization — what are you allowed to do? This is the most-failed part. The #1 OWASP API risk is Broken Object-Level Authorization (BOLA/IDOR): an authenticated user requests /orders/123 that isn't theirs and gets it because you only checked 'logged in', not 'owns this object'. Check ownership/role on every resource access, server-side.

Validate and constrain input. Treat all input as hostile: validate types/ranges, use parameterized queries (no SQL injection), cap payload sizes, and encode output. Lock down CORS to known origins rather than *.

Operational layers: rate-limit to stop abuse and credential-stuffing, store secrets in a vault (never in code), log and monitor for anomalies (audit trail), and keep dependencies patched. Reference the OWASP API Security Top 10 as your map.

Real-world example

🏢 How this works in production

Real breaches rarely come from broken encryption — they come from missing authorization checks (a user iterating IDs to read others' data), leaked API keys in a public repo, or an unvalidated input enabling injection. That's why the OWASP API Top 10 leads with broken object- and function-level authorization. Production APIs put auth at a gateway, enforce per-object ownership checks in handlers, pull secrets from Key Vault/Secrets Manager, and ship audit logs to a SIEM for anomaly detection.

Defense in layers

Authenticate, then authorize the object

C# — JWT auth + per-resource ownership check 
app.UseHttpsRedirection();                 // TLS only
builder.Services.AddAuthentication().AddJwtBearer();   // validate signature/iss/aud/exp
builder.Services.AddAuthorization();

app.MapGet("/orders/{id}", async (int id, ClaimsPrincipal user, IDb db) =>
{
    var order = await db.Orders.FindAsync(id);
    // ❌ 'is the caller logged in?' is NOT enough.
    // ✅ 'does THIS user own THIS object?' — the BOLA/IDOR check.
    if (order is null || order.UserId != user.GetUserId())
        return Results.NotFound();          // don't leak existence
    return Results.Ok(order);
}).RequireAuthorization();

Key takeaways

🎯 Remember these three

  • Security is layers: TLS → authentication → authorization → input validation → rate limiting → secrets/audit.
  • The most common failure is broken object-level authorization (BOLA/IDOR) — check ownership on every resource, not just 'is logged in.'
  • Use proven auth (OAuth2/OIDC), never roll your own crypto, lock down CORS, and follow the OWASP API Top 10.

Interview tip

🗣️ When this comes up

List the layers, but spend your time on authorization — name BOLA/IDOR and the per-object ownership check. Saying 'authentication is who you are, authorization is what you can touch, and most APIs fail the second one' is what marks a security-aware engineer.
