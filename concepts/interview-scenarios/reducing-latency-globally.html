Reducing Latency for a Global Audience

Latency is dominated by the speed of light and round trips. You win by putting data and compute closer to users and by making fewer round trips.

#latency #cdn #edge #replication

❓ The interview question

“Users in Asia complain the app is slow, but it's fast in the US where your servers are. How do you fix global latency?”

How to think about it

Name the physics: a packet from Singapore to Virginia and back is ~250 ms minimum, before any processing. No amount of CPU fixes distance — you have to shorten it or avoid the trip.

Push static content to a CDN. Images, JS, CSS, video, and cacheable API responses should be served from an edge location near the user, not your origin. This alone fixes most 'the site feels slow abroad' complaints.

Route users to the nearest region. Use geo-DNS or anycast so an Asian user hits an Asia-Pacific deployment instead of crossing an ocean. Deploy the app tier in multiple regions; it's stateless, so this is the easy part.

Bring data closer with read replicas. Reads can be served from a regional replica at low latency. Writes are the hard part: either funnel them to a single primary region (accepting higher write latency for distant users) or go multi-primary and accept eventual consistency and conflict resolution.

Reduce round trips. Batch requests, use HTTP/2 multiplexing and keep-alive, compress, and prefer one chatty call over ten serial ones. Latency adds up per round trip, so collapsing N requests into one is often the biggest app-level win.

Real-world example

🏢 How this works in production

Netflix and every major site front their assets with a CDN that has points of presence worldwide, so the bytes travel the last mile, not the whole globe. Apps deploy the stateless tier to multiple regions with geo-routing, keep a single write region (or a globally-distributed database like Spanner/DynamoDB global tables) and serve reads locally. The recurring lesson: distant users are slow because of round trips and origin distance, and CDNs + regional replicas remove both.

Move data to the edge

Cache at the edge, set the right headers

C# — make responses CDN-cacheable 
// Let the CDN serve this for an hour and revalidate in the background:
app.MapGet("/api/catalog", async (Catalog c) =>
{
    var data = await c.GetAsync();
    return Results.Json(data);
}).WithMetadata(new ResponseCacheAttribute {
    Duration = 3600, Location = ResponseCacheLocation.Any
});
// Cache-Control: public, max-age=3600, stale-while-revalidate=60
// → edge nodes near users answer most requests; origin sees a trickle.

Key takeaways

🎯 Remember these three

  • Latency is round trips and distance — fix it by moving data/compute closer (CDN, edge, regional replicas), not by adding CPU.
  • Static and cacheable content belongs on a CDN; geo-route users to the nearest region (the stateless app tier makes this easy).
  • Writes are the hard part of going global — single write region (simpler) vs multi-primary (faster, but eventual consistency + conflicts).

Interview tip

🗣️ When this comes up

Open with the speed-of-light framing — it shows you understand why distance hurts. CDN + geo-routing + read replicas is the standard answer; the depth marker is naming the write-path trade-off (single region vs multi-primary consistency).
