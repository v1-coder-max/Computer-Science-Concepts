Hash Tables

The workhorse data structure: a hash table turns a key into an array index via a hash function, giving O(1) average insert, lookup, and delete.

#dsa #hash-table #hashing #collisions

The idea in plain English

A hash table stores key-value pairs in an array of buckets. A hash function maps each key to a bucket index, so to find a key you hash it and jump straight to its bucket — no scanning. That's the source of the magic O(1) average operations.

Different keys can hash to the same bucket — a collision — and handling them is the crux. Chaining stores a small list (or tree) per bucket. Open addressing probes for the next free slot. Either way, if too many keys land in one bucket, operations on it degrade toward a linear scan.

That's why the worst case is O(n): if every key collides (a bad hash, or an adversary crafting collisions), the table becomes one long list. Good hash functions spread keys uniformly, and the load factor (entries ÷ buckets) is kept low — when it crosses a threshold (~0.7), the table resizes and rehashes everything to keep buckets short. Resizing is O(n) but rare, so insertion stays amortized O(1).

Keys must have consistent equality and hashing: equal keys must produce equal hashes (in .NET, override Equals and GetHashCode together), and keys should be immutable — mutating a key after insertion changes its hash and 'loses' it in the table. This is the most common hash bug in practice.

Real-world example

🏢 How this works in production

Hash tables back Dictionary<K,V> and HashSet<T> in .NET, Python dicts, Java HashMaps — they're everywhere because O(1) lookup is transformative (it's how Two Sum drops from O(n²) to O(n)). Real systems also mind the adversarial worst case: web frameworks randomize hash seeds to prevent 'hash-flooding' denial-of-service attacks where an attacker sends keys engineered to collide and force O(n) behavior.

Key takeaways

🎯 Remember these three

  • A hash table maps keys to buckets via a hash function → O(1) average insert/lookup/delete.
  • Collisions (chaining or open addressing) and a high load factor degrade it; resizing keeps buckets short — worst case is O(n).
  • Keys need consistent Equals + GetHashCode and should be immutable (mutating a key loses it).
