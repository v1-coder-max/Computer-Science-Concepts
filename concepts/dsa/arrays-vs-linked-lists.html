Arrays vs Linked Lists

The two ways to store a sequence, with opposite strengths: arrays give instant indexing and cache-friendly memory; linked lists give cheap insertion/deletion — in theory.

#dsa #arrays #linked-list #memory

The idea in plain English

An array stores elements in one contiguous block of memory. That makes indexing O(1) — element i is at base + i × size, simple arithmetic — but inserting or deleting in the middle is O(n) because you must shift everything after it. A dynamic array (List<T>) adds amortized O(1) append by occasionally doubling capacity.

A linked list stores each element in a node that points to the next (and maybe previous). Inserting or deleting is O(1) — just rewire pointers — but only if you already hold the node. Finding that position is O(n) because there's no indexing; you walk the chain from the head.

So on paper they look complementary, but in practice arrays usually win. Linked lists scatter nodes across memory, so traversal causes constant cache misses, while arrays stream through contiguous memory the CPU prefetches. Plus each node carries pointer overhead (8–16 bytes). The O(1) insert rarely beats the array's cache advantage unless you're inserting/removing constantly at known positions.

Rule of thumb: default to a dynamic array (List<T>). Reach for a linked list only when you genuinely need frequent O(1) insertion/removal at ends or known nodes (e.g. an LRU cache's recency list, or a queue) — and even then, contiguous structures often outperform it.

Real-world example

🏢 How this works in production

Modern hardware tilts this hard toward arrays: CPU caches and prefetching reward contiguous access, so even algorithms that 'should' favor linked lists are often faster on arrays. That's why .NET's List<T> (a dynamic array) is the default collection and LinkedList<T> is rarely the right choice. Linked structures shine in specific spots — an LRU cache pairs a hash map with a doubly-linked list for O(1) move-to-front — but 'use a linked list for fast inserts' is usually outdated advice once cache effects dominate.

Contiguous vs pointer-chained

In C#

C# 
// Array / dynamic array — O(1) index, cache-friendly, the sensible default.
var list = new List<int> { 1, 2, 3 };
int x = list[2];                 // O(1)
list.Add(4);                     // amortized O(1) (occasional resize)
list.Insert(0, 0);               // O(n) — shifts everything

// Linked list — O(1) insert/remove GIVEN the node, but O(n) to find it.
var ll = new LinkedList<int>();
var node = ll.AddLast(10);
ll.AddAfter(node, 20);           // O(1) rewire — only because we held 'node'

Key takeaways

🎯 Remember these three

  • Array: O(1) index, O(n) middle insert/delete, contiguous + cache-friendly; linked list: O(1) insert/delete given the node, but O(n) to find it.
  • In practice arrays usually win — prefetching/cache locality beats pointer-chasing, and nodes add memory overhead.
  • Default to a dynamic array (List<T>); use linked lists only for genuine frequent O(1) end/known-node edits (e.g. LRU).

Interview tip

🗣️ When this comes up

Give the textbook trade-off, then add the modern twist: cache locality makes arrays win in practice far more often than the Big-O table suggests. That nuance — and knowing where linked lists genuinely help (LRU) — sets a strong answer apart.
