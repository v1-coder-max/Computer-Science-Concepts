Big-O Notation

Big-O describes how an algorithm's cost grows as the input gets bigger — the single most important lens for comparing approaches before you ever run them.

#dsa #big-o #complexity #algorithms

The idea in plain English

Big-O notation expresses the growth rate of an algorithm's running time (or memory) as a function of input size n, ignoring constants and lower-order terms. It answers 'if the input doubles, what happens to the work?' — not 'how many milliseconds', because hardware varies but growth doesn't.

It describes the asymptotic worst case (the dominant term as n grows). So 3n + 50 is O(n) and 2n² + 100n is O(n²) — constants and smaller terms vanish because for large n the highest-order term dominates everything.

The classes you must know, fastest-growing last: O(1) constant (hash lookup), O(log n) logarithmic (binary search), O(n) linear (a scan), O(n log n) (good sorts), O(n²) quadratic (nested loops), O(2ⁿ) exponential and O(n!) factorial (brute-force combinatorics — infeasible beyond tiny inputs).

Two cautions. The gap is enormous: at n = 1,000,000, O(n) is a million steps while O(n²) is a trillion — that's the difference between instant and never. But Big-O hides constants, so for small n a 'worse' Big-O with tiny constants can win; and always consider space complexity too, not just time.

Real-world example

🏢 How this works in production

Big-O is the lingua franca of technical interviews and real performance work alike. When a feature that was instant in testing crawls in production, it's almost always an accidental O(n²) (a nested loop, or a query inside a loop) that was invisible at small scale. Engineers reach for Big-O to reason about whether an approach will hold up at 10× or 100× the data before writing a line — and to explain in code review why one solution scales and another doesn't.

How the classes diverge

In C#

C# 
// O(1) — constant: one hash lookup regardless of size.
bool has = set.Contains(x);

// O(n) — linear: one pass over the input.
foreach (var v in nums) sum += v;

// O(n²) — quadratic: a loop inside a loop (watch for this in prod!).
for (int i = 0; i < n; i++)
    for (int j = 0; j < n; j++)
        Compare(nums[i], nums[j]);

// O(log n) — halve the search space each step (binary search).

Key takeaways

🎯 Remember these three

  • Big-O measures how cost grows with input size (worst-case, dropping constants/lower terms) — growth, not wall-clock time.
  • Know the ladder: O(1) < O(log n) < O(n) < O(n log n) < O(n²) < O(2ⁿ) < O(n!).
  • The gap is huge at scale (an accidental O(n²) is the classic prod slowdown) — but consider constants at small n, and analyze space too.

Interview tip

🗣️ When this comes up

Be able to give the class of any snippet on sight (nested loop → O(n²), halving → O(log n)) and explain that Big-O is about growth, not time. Mentioning that constants matter at small n, and analyzing space too, shows nuance beyond rote memorization.
