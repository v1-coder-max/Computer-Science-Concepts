Trees & Graphs

When data isn't a line but a hierarchy or a network, you reach for trees and graphs — and almost everything you do with them is a variation on two traversals: DFS and BFS.

#dsa #trees #graphs #traversal

The idea in plain English

A tree is a hierarchy: nodes with a single root, each node having children, no cycles. Special trees do specific jobs — a binary search tree keeps elements sorted for O(log n) search/insert (when balanced), a heap keeps the min/max at the root for O(log n) priority operations, a trie indexes strings by prefix. Trees model file systems, the DOM, org charts, and parse trees.

A graph generalizes a tree: nodes (vertices) connected by edges, with any structure — cycles, multiple paths, disconnected pieces. Edges can be directed or undirected, weighted or not. Graphs model social networks, maps/routes, dependencies, and the web. You store one as an adjacency list (each node's neighbors — compact for sparse graphs) or matrix.

The two traversals do most of the work. Depth-First Search (DFS) goes as deep as possible before backtracking — uses a stack (or recursion). It's natural for exploring all paths, detecting cycles, and topological sorting. Breadth-First Search (BFS) explores level by level — uses a queue — and finds the shortest path in an unweighted graph because it reaches nearer nodes first.

Most graph algorithms are DFS/BFS plus bookkeeping: track visited nodes (or you'll loop forever on cycles — the key difference from trees), and layer on extras for weighted shortest paths (Dijkstra), connectivity (union-find), or ordering (topological sort). Recognizing 'this is a graph, and I need DFS or BFS' is half the battle in interviews.

Real-world example

🏢 How this works in production

Graphs are everywhere at scale: a social network is a graph and 'people you may know' is a traversal; map routing is a weighted shortest-path (Dijkstra/A*); a build system topologically sorts a dependency graph; package managers detect cycles. Trees back databases (B-trees for indexes), priority scheduling (heaps), and autocomplete (tries). The traversal choice is load-bearing — BFS for shortest unweighted path, DFS for exhaustive exploration — which is why these two algorithms anchor a huge fraction of the coding-problem set.

Hierarchy vs network; DFS vs BFS

In C#

C# 
// Adjacency list — compact graph representation.
var adj = new Dictionary<int, List<int>>();

// BFS — level by level with a QUEUE → shortest path (unweighted).
var q = new Queue<int>(); var seen = new HashSet<int>();
q.Enqueue(start); seen.Add(start);
while (q.Count > 0) {
    int node = q.Dequeue();
    foreach (var nbr in adj[node])
        if (seen.Add(nbr)) q.Enqueue(nbr);     // 'visited' set prevents cycles looping
}

// DFS — go deep with a STACK or recursion → paths, cycles, topo-sort.

Key takeaways

🎯 Remember these three

  • Tree = hierarchy, no cycles (BST/heap/trie for sorted/priority/prefix); graph = network with arbitrary edges (adjacency list for sparse).
  • DFS (stack/recursion) goes deep — paths, cycles, topo-sort; BFS (queue) goes wide — shortest path in unweighted graphs.
  • Always track visited nodes on graphs (cycles!) — that's the key difference from tree traversal.

Interview tip

🗣️ When this comes up

Be able to say which traversal fits a problem — BFS for shortest unweighted path, DFS for exhaustive/path/cycle work — and why (queue vs stack). The detail that trips people: graphs need a visited set or they loop on cycles, unlike trees.
