The idea in plain English

This page is a quick reference, not a tutorial — bookmark it for interviews and performance work. The tables below give the average and worst-case time complexity of the operations on the core data structures, then the time/space and stability of the common sorting algorithms.

A few reminders while you read them. Hash tables are O(1) average but O(n) worst case (everything colliding into one bucket) — the average is what you rely on, the worst is what you guard against. A balanced BST guarantees O(log n); an unbalanced one degrades to O(n), which is why production trees self-balance (red-black, AVL).

For sorting, the practical default is an O(n log n) comparison sort — that's the proven lower bound for comparison-based sorting. Quicksort is usually fastest in practice (great cache behavior) but is O(n²) worst-case; mergesort guarantees O(n log n) and is stable but uses O(n) extra space; heapsort is O(n log n) in place but not stable. Non-comparison sorts (counting/radix) can beat O(n log n) for restricted integer keys.

Stable means equal elements keep their relative order — it matters when you sort by one key after another. Match the structure and algorithm to your access pattern and constraints; the tables tell you the costs of each choice.

Real-world example

🏢 How this works in production

These complexities are exactly what coding interviews probe ('what's the lookup cost?', 'why is your sort O(n log n)?') and what guides real engineering decisions — choosing a hash set for O(1) membership over a list's O(n) scan, or knowing a join inside a loop is quietly O(n·m). Language standard libraries pick their sort accordingly: .NET's Array.Sort uses an introsort (quicksort that falls back to heapsort to avoid the O(n²) worst case), and stable sorts where order must be preserved.

In C#

C# 
// Big-O ladder, slowest-growing first:
//   O(1) < O(log n) < O(n) < O(n log n) < O(n²) < O(2^n) < O(n!)

// Library sorts pick pragmatically — .NET Array.Sort uses introsort
// (quicksort + heapsort fallback) to get average speed without the O(n²) worst case:
Array.Sort(nums);                 // O(n log n) average, not stable
var stable = nums.OrderBy(x => x).ToArray();   // LINQ OrderBy is stable

Key takeaways

🎯 Remember these three

  • Hash table = O(1) average, O(n) worst; balanced BST = O(log n); array = O(1) index but O(n) search.
  • Comparison sorts bottom out at O(n log n) — quicksort fastest in practice, mergesort stable but O(n) space, heapsort in-place.
  • Stable sorts preserve equal-element order; counting/radix beat O(n log n) only for bounded integer keys.

Interview tip

🗣️ When this comes up

Have these numbers instant: hash O(1) avg, balanced tree O(log n), comparison sort O(n log n) floor. The follow-ups that catch people are hash's O(n) worst case and quicksort's O(n²) worst case — know why each happens.
