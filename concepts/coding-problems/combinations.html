Combinations

All k-element combinations of 1..n. Backtracking with a start index keeps them increasing, and a count check prunes dead branches.

#backtracking #recursion #medium #combinatorics #pruning #interview

The problem

Given two integers n and k, return all possible combinations of k numbers chosen from the range 1..n.

📥 Example

n=4, k=2 → [[1,2],[1,3],[1,4],[2,3],[2,4],[3,4]]

Brute force

Generating all subsets and keeping those of size k is wasteful — most subsets are the wrong size. Building only size-k combinations via backtracking avoids the waste.

Optimized approach

Backtrack with a start index: at each level, try numbers from start..n, append one, recurse from the next index, then un-choose. The start index guarantees increasing order so no combination repeats. Prune when the remaining numbers can’t fill the rest of the combination (n − i + 1 < k − path.Count). Record when the path reaches length k.

Time O(k · C(n,k)), Space O(k).

C# solution

C# 
public IList<IList<int>> Combine(int n, int k)
{
    var result = new List<IList<int>>();
    Backtrack(1, new List<int>());
    return result;

    void Backtrack(int start, List<int> path)
    {
        if (path.Count == k) { result.Add(new List<int>(path)); return; }
        // prune: not enough numbers left to reach size k
        for (int i = start; i <= n - (k - path.Count) + 1; i++)
        {
            path.Add(i);
            Backtrack(i + 1, path);
            path.RemoveAt(path.Count - 1);
        }
    }
}

Complexity

⏱️ Complexity

  • Time: O(k · C(n,k)) — each combination built in O(k).
  • Space: O(k) recursion depth.

Pattern & takeaways

🎯 Remember these three

  • Start index enforces increasing order (no dup combos).
  • Prune when remaining numbers can’t fill the combination.
  • It’s the subsets template constrained to a fixed size.

Interview tip

🗣️ When this comes up
