Single Element in a Sorted Array

Every element appears twice except one — find it in O(log n). Binary search on pair parity: the single element breaks the even-index pairing.

#binary-search #arrays #medium #parity #sorted #interview

The problem

Given a sorted array where every element appears exactly twice except one that appears once, return the single element in O(log n) time and O(1) space.

📥 Example

[1,1,2,3,3,4,4,8,8] → 2  ·  [3,3,7,7,10,11,11] → 10

Brute force

XOR all elements (O(n)) or scan pairs (O(n)) both find it but ignore the sorted order. Sortedness enables O(log n) via index parity.

Optimized approach

Before the single element, pairs start at even indices ((0,1), (2,3), …); after it, the pairing shifts to odd starts. Binary search on even midpoints: force mid to be even; if nums[mid] == nums[mid+1], the single element is to the right (pairing still intact), so lo = mid + 2; otherwise it’s at mid or to the left, so hi = mid. Convergence lands on the single element.

Time O(log n), Space O(1).

C# solution

C# 
public int SingleNonDuplicate(int[] nums)
{
    int lo = 0, hi = nums.Length - 1;
    while (lo < hi)
    {
        int mid = lo + (hi - lo) / 2;
        if (mid % 2 == 1) mid--;                  // align to even index
        if (nums[mid] == nums[mid + 1]) lo = mid + 2;   // pairing intact → right
        else hi = mid;                            // broken here or left
    }
    return nums[lo];
}

Complexity

⏱️ Complexity

  • Time: O(log n).
  • Space: O(1).

Pattern & takeaways

🎯 Remember these three

  • The single element shifts pair parity — binary search on it.
  • Align mid to an even index before comparing.
  • XOR is the O(n) fallback that ignores sortedness.

Interview tip

🗣️ When this comes up

The parity insight is the whole problem — explain why pairs start even before the single element and odd after. Aligning mid to even keeps the comparison valid.
