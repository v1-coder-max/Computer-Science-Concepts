Minimum Depth of Binary Tree

Depth to the nearest leaf. The catch: a node with only one child isn’t a leaf, so you can’t just take min of the two sides blindly.

#trees #bfs #recursion #easy #depth #interview

The problem

Given a binary tree, return its minimum depth: the number of nodes on the shortest path from the root down to the nearest leaf (a leaf has no children).

📥 Example

[3,9,20,null,null,15,7] → 2  ·  [2,null,3,null,4] → 4

Brute force

A naive 1 + min(left, right) is wrong: for a node with one missing child, the missing side reports depth 0, giving a too-small answer. You must special-case single-child nodes.

Optimized approach

Recurse, but handle the single-child case: if a node has only a left or only a right child, the minimum depth must go down the existing side, so return 1 + that side. Only when both children exist do you take 1 + min(left, right). Even better, a BFS returns the depth at the first leaf it reaches — naturally the shallowest — and can stop early.

Time O(n), Space O(h) or O(width).

C# solution

C# 
public int MinDepth(TreeNode root)
{
    if (root == null) return 0;
    if (root.left == null) return 1 + MinDepth(root.right);   // only right
    if (root.right == null) return 1 + MinDepth(root.left);   // only left
    return 1 + Math.Min(MinDepth(root.left), MinDepth(root.right));
}

Complexity

⏱️ Complexity

  • Time: O(n) recursion (BFS can stop early).
  • Space: O(h) recursion / O(width) BFS.

Pattern & takeaways

🎯 Remember these three

  • A node with one child is not a leaf — recurse down the present side.
  • Take min only when both children exist.
  • BFS returns at the first leaf — the shallowest — and stops early.

Interview tip

🗣️ When this comes up

The whole trap is the single-child case — the naive min fails there. Mention the BFS solution as the one that can return early, which matters for very unbalanced trees.
