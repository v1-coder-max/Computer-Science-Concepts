Search in Rotated Sorted Array

A sorted array got “rotated” at some pivot. You still want O(log n). The insight: at any midpoint, one side is always still sorted — use it to decide where to go.

#binary-search #arrays #rotation #medium

The problem

An ascending array was rotated at an unknown pivot (e.g. [0,1,2,4,5,6,7] → [4,5,6,7,0,1,2]). Given the rotated array and a target, return its index or -1. All values are distinct.

📥 Example

nums = [4,5,6,7,0,1,2], target = 0 → 4  ·  target = 3 → -1

Brute force

A linear scan finds it in O(n) but ignores the structure. Because each half around the midpoint is still (mostly) sorted, we can keep the logarithmic time of plain binary search.

Optimized approach

Do a modified binary search. At mid, check whether target == nums[mid]. Otherwise decide which half is sorted by comparing nums[lo] with nums[mid]:

  • If nums[lo] <= nums[mid], the left half is sorted. If the target lies within [nums[lo], nums[mid]), search left; otherwise search right.
  • Else the right half is sorted. If the target lies within (nums[mid], nums[hi]], search right; otherwise search left.

Time O(log n), Space O(1).

C# solution

C# 
public int Search(int[] nums, int target)
{
    int lo = 0, hi = nums.Length - 1;
    while (lo <= hi)
    {
        int mid = lo + (hi - lo) / 2;
        if (nums[mid] == target) return mid;

        if (nums[lo] <= nums[mid])              // left half is sorted
        {
            if (nums[lo] <= target && target < nums[mid]) hi = mid - 1;
            else                                         lo = mid + 1;
        }
        else                                    // right half is sorted
        {
            if (nums[mid] < target && target <= nums[hi]) lo = mid + 1;
            else                                         hi = mid - 1;
        }
    }
    return -1;
}

Complexity

⏱️ Complexity

  • Time: O(log n) — still halving each step.
  • Space: O(1).

Pattern & takeaways

🎯 Remember these three

  • In a rotated array, one half of every split is still sorted — that’s the lever.
  • Identify the sorted side, then a range check decides which half to keep.
  • With duplicates the worst case degrades to O(n); the distinct-values version stays O(log n).

Interview tip

🗣️ When this comes up

State the key invariant first — “at least one side of the split is sorted” — then derive the branches. That framing shows you understand why binary search still applies, which is what separates a memorized answer from a real one.
