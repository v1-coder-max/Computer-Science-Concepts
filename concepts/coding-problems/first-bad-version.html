First Bad Version

Versions are good up to a point, then all bad. Binary-search that boundary so you call the checker as few times as possible.

#binary-search #easy

The problem

Versions 1..n are good up to some point, then bad. Given an IsBadVersion(v) API, return the first bad version using the fewest calls.

📥 Example

n = 5, first bad = 4 → returns 4

Brute force

Scanning 1..n until the first bad version is O(n) API calls. Because the predicate is monotonic (false…false, true…true), binary search finds the boundary in O(log n).

Optimized approach

Binary-search for the boundary. If mid is bad, the first bad version is mid or to its left (hi = mid); otherwise it is to the right (lo = mid + 1). When lo == hi, that is the answer.

Time O(log n), Space O(1).

C# solution

C# 
public int FirstBadVersion(int n)
{
    int lo = 1, hi = n;
    while (lo < hi)
    {
        int mid = lo + (hi - lo) / 2;
        if (IsBadVersion(mid)) hi = mid;   // first bad is mid or left
        else lo = mid + 1;                 // first bad is to the right
    }
    return lo;
}

Complexity

⏱️ Complexity

  • Time: O(log n) — halving the range each step.
  • Space: O(1) — two bounds.

Pattern & takeaways

🎯 Remember these three

  • This is binary search on a monotonic predicate (false→true boundary).
  • Use lo + (hi - lo)/2 to avoid integer overflow.
  • Keep mid as a candidate (hi = mid) — don't discard it.
