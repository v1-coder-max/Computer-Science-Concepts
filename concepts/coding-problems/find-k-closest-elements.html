Find K Closest Elements

The k elements nearest to x, returned sorted. Binary search for the best window start by comparing the gaps at the window’s two ends.

#binary-search #arrays #medium #window #sorted #interview

The problem

Given a sorted array arr, an integer k, and a value x, return the k closest elements to x (sorted ascending; ties prefer the smaller element).

📥 Example

arr=[1,2,3,4,5], k=4, x=3 → [1,2,3,4]  ·  x=-1 → [1,2,3,4]

Brute force

Sort all elements by distance to x and take k — O(n log n), discarding the sorted order. Two pointers from the closest element is O(n). Binary search on the window start is O(log(n−k) + k).

Optimized approach

The answer is a contiguous window of length k. Binary search the window’s start index in [0, n−k]: at mid, compare x − arr[mid] (left gap) with arr[mid+k] − x (right gap). If the left gap is larger, the window should slide right (lo = mid + 1); otherwise hi = mid. Convergence gives the optimal start; return that length-k slice.

Time O(log(n−k) + k), Space O(k).

C# solution

C# 
public IList<int> FindClosestElements(int[] arr, int k, int x)
{
    int lo = 0, hi = arr.Length - k;
    while (lo < hi)
    {
        int mid = lo + (hi - lo) / 2;
        if (x - arr[mid] > arr[mid + k] - x) lo = mid + 1;   // window too far left
        else hi = mid;
    }
    return arr.Skip(lo).Take(k).ToList();
}

Complexity

⏱️ Complexity

  • Time: O(log(n−k) + k).
  • Space: O(k) for the output.

Pattern & takeaways

🎯 Remember these three

  • The result is a contiguous window; binary search its start.
  • Compare the two boundary gaps to decide the slide direction.
  • Two pointers expanding from the closest element is the O(n) alternative.

Interview tip

🗣️ When this comes up

Binary-searching the window start (not the element) is the elegant O(log) approach — explain the two-gap comparison. The two-pointer O(n) version is a fine fallback to mention.
