Evaluate Reverse Polish Notation

Postfix expressions need no parentheses and no precedence rules — just a stack. Push numbers; on an operator, pop two, combine, push back.

#stack #math #parsing #medium

The problem

Evaluate an arithmetic expression in Reverse Polish (postfix) Notation. Tokens are integers and the operators + − × ÷. Division truncates toward zero. Return the integer result.

📥 Example

["2","1","+","3","*"] → 9 ((2 + 1) × 3)  ·  ["4","13","5","/","+"] → 6

The idea

In postfix, an operator applies to the two most recently produced values — which is exactly the top of a stack. Scan left to right: a number gets pushed; an operator pops the top two operands (mind the order: the first popped is the right-hand operand), applies itself, and pushes the result. After the scan, the lone remaining value is the answer.

Time O(n), Space O(n).

C# solution

C# 
public int EvalRPN(string[] tokens)
{
    var stack = new Stack<int>();
    foreach (string tok in tokens)
    {
        switch (tok)
        {
            case "+": case "-": case "*": case "/":
                int b = stack.Pop(), a = stack.Pop();   // a is left, b is right
                stack.Push(tok switch
                {
                    "+" => a + b,
                    "-" => a - b,
                    "*" => a * b,
                    _   => a / b      // truncates toward zero
                });
                break;
            default:
                stack.Push(int.Parse(tok));
                break;
        }
    }
    return stack.Pop();
}

Complexity

⏱️ Complexity

  • Time: O(n) — one pass over the tokens.
  • Space: O(n) — operands on the stack.

Pattern & takeaways

🎯 Remember these three

  • Postfix evaluation is a stack — no precedence or parentheses needed.
  • Mind operand order for − and ÷: first popped is the right operand.
  • Infix → postfix conversion (shunting-yard) is the related compiler-flavored follow-up.
