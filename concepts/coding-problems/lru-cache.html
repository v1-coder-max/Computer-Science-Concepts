LRU Cache

A cache with O(1) get and put that evicts the least-recently-used entry. The combo: a hash map for lookup + a doubly linked list for recency order.

#linked-list #hash-map #design #medium #o1 #interview

The problem

Design an LRU cache with a fixed capacity supporting Get(key) and Put(key, value) in O(1). When full, Put evicts the least recently used entry.

📥 Example

cap=2; put(1,1); put(2,2); get(1)→1; put(3,3) evicts 2; get(2)→-1

Brute force

A single hash map gives O(1) lookup but no recency order, so finding the LRU entry to evict is O(n). An array/list ordered by recency makes reordering O(n). You need both O(1) lookup and O(1) reordering.

Optimized approach

Pair a hash map (key → node) with a doubly linked list ordered most-recent at the head, least-recent at the tail. Get: look up the node and move it to the head. Put: update or insert at the head; if over capacity, remove the tail node and its map entry. Doubly-linked so any node unlinks in O(1); the map makes lookup O(1). (C#’s LinkedList<T> nodes work, or roll your own.)

Time O(1) per op, Space O(capacity).

C# solution

C# 
public class LRUCache
{
    private readonly int _cap;
    private readonly Dictionary<int, LinkedListNode<(int k, int v)>> _map = new();
    private readonly LinkedList<(int k, int v)> _list = new();   // head = most recent

    public LRUCache(int capacity) => _cap = capacity;

    public int Get(int key)
    {
        if (!_map.TryGetValue(key, out var node)) return -1;
        _list.Remove(node); _list.AddFirst(node);        // mark most-recent
        return node.Value.v;
    }

    public void Put(int key, int value)
    {
        if (_map.TryGetValue(key, out var node)) _list.Remove(node);
        else if (_map.Count >= _cap)
        {
            var lru = _list.Last;                        // least recent
            _list.RemoveLast(); _map.Remove(lru.Value.k);
        }
        var fresh = new LinkedListNode<(int, int)>((key, value));
        _list.AddFirst(fresh); _map[key] = fresh;
    }
}

Complexity

⏱️ Complexity

  • Time: O(1) for both Get and Put.
  • Space: O(capacity).

Pattern & takeaways

🎯 Remember these three

  • Hash map + doubly linked list = O(1) lookup and O(1) reorder.
  • Head = most recent; evict the tail when full.
  • Doubly-linked is essential so any node unlinks in O(1).

Interview tip

🗣️ When this comes up

This is a top design question. State the two structures and why each is needed (map for lookup, doubly-linked for O(1) reorder/evict) before coding. Mention LinkedHashMap/OrderedDictionary as language shortcuts.
