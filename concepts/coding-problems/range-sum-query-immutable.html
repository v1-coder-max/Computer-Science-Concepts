Range Sum Query - Immutable

Answer many range-sum queries fast. Precompute prefix sums once, then each query is a single subtraction.

#arrays #prefix-sum #design #easy #preprocessing #interview

The problem

Design NumArray initialized with an array; SumRange(i, j) returns the sum of elements from index i to j inclusive. Expect many queries.

📥 Example

nums=[-2,0,3,-5,2,-1]; sumRange(0,2)→1; sumRange(2,5)→-1

Brute force

Summing the range on each query is O(n) per call — fine for a few queries, slow for many. Prefix sums shift the cost to a one-time O(n) build and make each query O(1).

Optimized approach

Precompute prefix[k] = sum of the first k elements, with prefix[0] = 0. Then SumRange(i, j) = prefix[j+1] − prefix[i] — the total up to j minus the total before i. Build is O(n); every query is O(1) thereafter.

Build O(n), Query O(1), Space O(n).

C# solution

C# 
public class NumArray
{
    private readonly int[] _prefix;

    public NumArray(int[] nums)
    {
        _prefix = new int[nums.Length + 1];
        for (int i = 0; i < nums.Length; i++)
            _prefix[i + 1] = _prefix[i] + nums[i];      // cumulative sums
    }

    public int SumRange(int left, int right) => _prefix[right + 1] - _prefix[left];
}

Complexity

⏱️ Complexity

  • Time: O(n) build, O(1) per query.
  • Space: O(n) for the prefix array.

Pattern & takeaways

🎯 Remember these three

  • Prefix sums turn range queries into one subtraction.
  • Use a size n+1 array with prefix[0]=0 to avoid edge cases.
  • The 2-D version uses an inclusion-exclusion of four corners.
