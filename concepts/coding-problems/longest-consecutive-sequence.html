Longest Consecutive Sequence

Find the longest run of consecutive integers — in O(n), without sorting. A HashSet plus the trick of only counting from a sequence’s start.

#hashing #arrays #set #medium

The problem

Given an unsorted array nums, return the length of the longest run of consecutive integers. The run’s elements need not be adjacent in the array. Aim for O(n).

📥 Example

[100,4,200,1,3,2] → 4 (the run 1, 2, 3, 4)

Brute force

Sorting then scanning runs is O(n log n) — and the problem explicitly asks for O(n). Sorting does more work than needed; membership tests are all we actually require.

Optimized approach

Put every number in a HashSet for O(1) lookups. Then, for each number, only begin counting if it’s a sequence start — i.e. num - 1 is not in the set. From a start, walk num+1, num+2, … while they exist, measuring the run. Because each number is visited at most twice (once in the loop, once during a walk it belongs to), the total is O(n), not O(n²).

Time O(n), Space O(n).

C# solution

C# 
public int LongestConsecutive(int[] nums)
{
    var set = new HashSet<int>(nums);
    int longest = 0;
    foreach (int n in set)
    {
        if (set.Contains(n - 1)) continue;   // not a sequence start — skip
        int length = 1;
        while (set.Contains(n + length)) length++;   // walk the run
        longest = Math.Max(longest, length);
    }
    return longest;
}

Complexity

⏱️ Complexity

  • Time: O(n) — the start-only guard makes each number part of one walk.
  • Space: O(n) — the HashSet.

Pattern & takeaways

🎯 Remember these three

  • A HashSet turns “is x present?” into O(1) — the enabler for O(n).
  • The start-only guard (n-1 ∉ set) is what prevents O(n²) re-walks.
  • “O(n) but it looks like it needs sorting” is a cue to reach for a set.

Interview tip

🗣️ When this comes up

The interviewer wants the O(n), not the sort. The make-or-break detail is the start-only check — without it your “O(n)” is secretly O(n²). Say explicitly why each element is touched a constant number of times.
