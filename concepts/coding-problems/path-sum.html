Path Sum

Is there a root-to-leaf path summing to a target? Recurse down, subtracting each value, and check the remainder at the leaves.

#trees #recursion #dfs #easy #root-to-leaf #interview

The problem

Given the root of a binary tree and a targetSum, return true if some root-to-leaf path’s node values add up to targetSum.

📥 Example

root=[5,4,8,11,null,13,4,7,2,null,null,null,1], targetSum=22 → true (5→4→11→2)

Brute force

Enumerate every root-to-leaf path and sum it — that’s essentially the same traversal, just with explicit path lists and extra space. Carrying a running remainder is cleaner.

Optimized approach

Pass the remaining target down the recursion, subtracting the current node’s value. At a leaf (no children), the path is valid iff the remaining target equals the leaf’s value (i.e. it hits 0 after subtracting). Otherwise recurse into the children with the reduced target; succeed if either branch does. Empty tree returns false.

Time O(n), Space O(h).

C# solution

C# 
public bool HasPathSum(TreeNode root, int targetSum)
{
    if (root == null) return false;
    if (root.left == null && root.right == null)        // leaf
        return targetSum == root.val;
    int remaining = targetSum - root.val;
    return HasPathSum(root.left, remaining)
        || HasPathSum(root.right, remaining);
}

Complexity

⏱️ Complexity

  • Time: O(n) — visit each node once.
  • Space: O(h) recursion depth.

Pattern & takeaways

🎯 Remember these three

  • Thread the remaining target down the recursion.
  • Validate only at leaves (both children null).
  • Path Sum II collects the actual paths via backtracking.

Interview tip

🗣️ When this comes up

The leaf check is the part people botch — it must require both children null, not just one. Carrying the remainder down is cleaner than building path lists.
