Merge K Sorted Lists

Merge k sorted lists into one. A min-heap of the current heads always surfaces the global smallest next node — O(N log k).

#linked-list #heap #divide-and-conquer #hard #merge #interview

The problem

Given an array of k sorted linked lists, merge them into one sorted list and return its head.

📥 Example

[[1,4,5],[1,3,4],[2,6]] → 1→1→2→3→4→4→5→6

Brute force

Merge the lists one by one (list1+list2, then +list3, …). Each merge re-walks the growing result, giving O(N·k) total. A heap or divide-and-conquer cuts the k factor to log k.

Optimized approach

Put the head of each list into a min-heap keyed by value. Pop the smallest, append it to the result, and push that node’s next (if any). The heap never holds more than k nodes, so each of the N nodes costs O(log k). Divide-and-conquer (pairwise-merge halves, like merge sort) achieves the same O(N log k) without a heap.

Time O(N log k), Space O(k).

C# solution

C# 
public ListNode MergeKLists(ListNode[] lists)
{
    var pq = new PriorityQueue<ListNode, int>();
    foreach (var node in lists) if (node != null) pq.Enqueue(node, node.val);

    var dummy = new ListNode();
    var tail = dummy;
    while (pq.Count > 0)
    {
        var node = pq.Dequeue();        // global smallest head
        tail.next = node; tail = node;
        if (node.next != null) pq.Enqueue(node.next, node.next.val);
    }
    return dummy.next;
}

Complexity

⏱️ Complexity

  • Time: O(N log k) — N nodes, heap of size ≤ k.
  • Space: O(k) for the heap.

Pattern & takeaways

🎯 Remember these three

  • k-way merge → a min-heap of the k current heads.
  • Sequential merging is O(N·k); the heap drops it to O(N log k).
  • Divide-and-conquer pairwise merge is the heap-free equivalent.

🗣️ When this comes up

Two strong answers: min-heap of heads, or divide-and-conquer pairwise merge — both O(N log k). Contrast with the naive O(N·k) sequential merge to justify the improvement.
