Diameter of Binary Tree

The longest path between any two nodes may not pass through the root. The trick: compute height bottom-up, and at every node ask “what if the path bends here?”

#trees #recursion #dfs #easy

The problem

Given the root of a binary tree, return its diameter: the length (number of edges) of the longest path between any two nodes. The path need not go through the root.

📥 Example

[1,2,3,4,5] → 3 (the path 4 → 2 → 1 → 3, or 5 → 2 → 1 → 3)

The idea

For any node, the longest path that bends at that node equals height(left) + height(right) (in edges). The tree’s diameter is the maximum of that quantity over all nodes. So run a single post-order pass that returns each subtree’s height, and as a side effect update a global best with leftH + rightH at every node. One traversal computes both.

Time O(n), Space O(h).

C# solution

C# 
public int DiameterOfBinaryTree(TreeNode root)
{
    int best = 0;
    Height(root);
    return best;

    int Height(TreeNode node)
    {
        if (node == null) return 0;
        int left  = Height(node.left);
        int right = Height(node.right);
        best = Math.Max(best, left + right);     // path bending at this node (edges)
        return 1 + Math.Max(left, right);        // height back to the parent
    }
}

Complexity

⏱️ Complexity

  • Time: O(n) — a single post-order pass computes height and diameter together.
  • Space: O(h) recursion depth.

Pattern & takeaways

🎯 Remember these three

  • Return one thing (height), update another (best diameter) as a side effect.
  • The optimal path may bend at a node, not pass through the root — check every node.
  • This “height + global update” shape also solves binary-tree-max-path-sum.

Interview tip

🗣️ When this comes up

The “aha” is computing height and diameter in one pass via a side-effect variable — naming that shows you won’t write an O(n²) double recursion. Clarify edges-vs-nodes for the length; off-by-one here is the usual trap.
