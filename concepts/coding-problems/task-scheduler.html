Task Scheduler

Identical tasks need n idle slots between them. The schedule is paced by the most frequent task — a neat formula, or a greedy heap.

#heap #greedy #math #medium #scheduling #interview

The problem

Given task labels and a cooldown n (minimum gap between two same tasks), return the least number of time units to finish all tasks (idling allowed).

📥 Example

tasks = ["A","A","A","B","B","B"], n = 2 → 8 (A B _ A B _ A B)

The idea

The bottleneck is the task with the highest frequency fmax. Lay out its copies with n gaps between them: that frames (fmax − 1) blocks of size (n + 1), plus the final group of all tasks that share the max frequency. So the answer is max(tasks.Length, (fmax − 1) · (n + 1) + countOfMaxFreq) — the max handles the case where there are so many distinct tasks that no idling is needed. A greedy max-heap simulation gives the same result.

Time O(n), Space O(1) (26 task types).

C# solution

C# 
public int LeastInterval(char[] tasks, int n)
{
    var counts = new int[26];
    foreach (char t in tasks) counts[t - 'A']++;

    int fmax = counts.Max();
    int maxCount = counts.Count(c => c == fmax);   // tasks tied at the top

    int framed = (fmax - 1) * (n + 1) + maxCount;
    return Math.Max(tasks.Length, framed);
}

Complexity

⏱️ Complexity

  • Time: O(total tasks) — counting plus O(1) arithmetic.
  • Space: O(1) — a fixed 26-count array.

Pattern & takeaways

🎯 Remember these three

  • The most frequent task dictates the minimum length.
  • Formula: max(len, (fmax−1)(n+1) + #tiesAtMax).
  • A greedy max-heap (run the most-remaining task each tick) is the simulation alternative.

Interview tip

🗣️ When this comes up

Two valid routes: the closed-form formula (fast, but justify the max with “len”) or the greedy max-heap simulation (more intuitive). Deriving the formula from the peak-frequency skeleton is the impressive answer.
