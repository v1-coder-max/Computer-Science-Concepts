Coin Change II

Count the ways to make an amount (not the fewest coins). The trick: iterate coin by coin so order doesn’t create duplicate combinations.

#dynamic-programming #unbounded-knapsack #counting #medium #combinations #interview

The problem

Given coin denominations coins (unlimited each) and an amount, return the number of distinct combinations that sum to amount.

📥 Example

coins = [1,2,5], amount = 5 → 4 (5; 2+2+1; 2+1+1+1; 1×5)

Brute force

Recursively branch on “use this coin or skip to the next” — exponential. Memoizing on (coinIndex, remaining) works, but a simple 1-D bottom-up table is cleaner and avoids double-counting if iterated correctly.

Optimized approach

Let dp[a] = number of ways to make amount a, starting with dp[0] = 1 (one way: use nothing). Loop coins on the outside and amounts inside, doing dp[a] += dp[a − coin]. Iterating coins outermost means each combination is built in one fixed coin order, so 1+2 and 2+1 aren’t counted twice (that ordering is the key difference from the min-coins version).

Time O(amount · coins), Space O(amount).

C# solution

C# 
public int Change(int amount, int[] coins)
{
    var dp = new int[amount + 1];
    dp[0] = 1;                              // one way to make 0
    foreach (int coin in coins)            // coin loop OUTSIDE → combinations
        for (int a = coin; a <= amount; a++)
            dp[a] += dp[a - coin];
    return dp[amount];
}

Complexity

⏱️ Complexity

  • Time: O(amount · number of coins).
  • Space: O(amount).

Pattern & takeaways

🎯 Remember these three

  • Counting combinations → coins loop on the outside.
  • Counting permutations → amounts outside, coins inside.
  • Base case dp[0] = 1: the empty combination.

Interview tip

🗣️ When this comes up

The make-or-break detail is loop order: coins outside counts combinations, amounts outside counts ordered sequences. Be able to explain why — interviewers love that distinction.
