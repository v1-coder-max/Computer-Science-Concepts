Find Peak Element

Find any element greater than its neighbors — in O(log n). Binary search toward the higher neighbor: going uphill always leads to a peak.

#binary-search #arrays #medium #peak #slope #interview

The problem

Given an array where no two adjacent elements are equal (treat out-of-bounds as −∞), return the index of any peak element (one greater than both neighbors). Must run in O(log n).

📥 Example

[1,2,3,1] → 2  ·  [1,2,1,3,5,6,4] → 5 (or 1)

Brute force

A linear scan finds a peak in O(n). But the slope structure lets binary search reach one in O(log n) without examining every element.

Optimized approach

Compare nums[mid] with nums[mid + 1]. If nums[mid] < nums[mid + 1], you’re on an upward slope, so a peak lies to the right — move lo = mid + 1. Otherwise a peak lies at mid or to its left — move hi = mid. Because the boundaries are effectively −∞, walking uphill must eventually crest, so the search converges on a peak.

Time O(log n), Space O(1).

C# solution

C# 
public int FindPeakElement(int[] nums)
{
    int lo = 0, hi = nums.Length - 1;
    while (lo < hi)
    {
        int mid = lo + (hi - lo) / 2;
        if (nums[mid] < nums[mid + 1]) lo = mid + 1;   // go uphill (right)
        else hi = mid;                                 // peak is here or left
    }
    return lo;
}

Complexity

⏱️ Complexity

  • Time: O(log n).
  • Space: O(1).

Pattern & takeaways

🎯 Remember these three

  • Binary search works on slope, not sorted order.
  • Always move toward the higher neighbor — uphill crests.
  • The −∞ boundaries guarantee a peak exists in the chosen half.

Interview tip

🗣️ When this comes up

The insight is that binary search needs monotonic decision, not a sorted array — here the slope provides it. Explain why walking uphill must hit a peak given the −∞ ends.
