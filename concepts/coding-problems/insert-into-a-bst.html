Insert into a Binary Search Tree

Inserting never reshapes existing nodes: you just walk down to where the value belongs and attach a new leaf.

#trees #bst #medium

The problem

Given the root of a BST and a value not already present, insert it and return the (possibly new) root.

📥 Example

Insert 5 into [4,2,7,1,3] → it becomes the left child of 7

Brute force

Collecting all values, adding the new one, and rebuilding a balanced BST is overkill and changes the tree's shape. A direct walk-and-attach is O(h).

Optimized approach

Walk down using BST order until you reach the empty child where the value belongs (left if smaller, right if larger), then create a new leaf there. An empty tree just becomes the single node.

Time O(h), Space O(1).

C# solution

C# 
public TreeNode InsertIntoBST(TreeNode root, int val)
{
    if (root == null) return new TreeNode(val);
    TreeNode cur = root;
    while (true)
    {
        if (val < cur.val)
        {
            if (cur.left == null) { cur.left = new TreeNode(val); break; }
            cur = cur.left;
        }
        else
        {
            if (cur.right == null) { cur.right = new TreeNode(val); break; }
            cur = cur.right;
        }
    }
    return root;
}

Complexity

⏱️ Complexity

  • Time: O(h) — one root-to-leaf walk.
  • Space: O(1) — iterative.

Pattern & takeaways

🎯 Remember these three

  • A valid BST always has exactly one correct empty slot for a new value.
  • Insertion adds a leaf; existing nodes are untouched.
  • Handle the empty-tree case by returning a fresh node.

Interview tip

🗣️ When this comes up

Stress that insertion only ever creates a leaf and never restructures — that is why it is O(h). Deletion (which can restructure) is the harder sibling.
