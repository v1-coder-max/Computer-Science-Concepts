Undoing Changes: reset, revert, reflog

Git rarely loses anything — you just need the right undo. The trio: revert for shared history, reset for local history, and reflog to rescue 'lost' commits.

#git #reset #revert #reflog

The idea in plain English

git revert undoes a commit by creating a new commit that applies the inverse change. History is preserved and moves forward, so it's safe on shared branches — everyone just sees an 'undo' commit. This is the right way to back out a bad change that's already pushed.

git reset moves the current branch pointer to a different commit, effectively rewriting local history. Its three modes differ in what they touch: --soft moves the branch but keeps your changes staged; --mixed (default) unstages them but keeps them in your working files; --hard discards them entirely. Use reset on local, unpushed work — it rewrites history, so it's dangerous on shared branches.

git reflog is the safety net: Git logs everywhere HEAD has been (commits, resets, rebases, checkouts) for ~90 days. So after a bad reset --hard or a botched rebase, the 'lost' commit is still findable — grab its hash from the reflog and git reset --hard or cherry-pick it back.

The decision rule: revert what's public, reset what's private, and reach for reflog when you think you've lost work — because you usually haven't.

Real-world example

🏢 How this works in production

When a deploy goes bad, teams git revert the offending merge so main rolls forward to a safe state with full history intact (and a clear audit trail). Individually, developers reset --soft HEAD~1 to recombine the last commit, or recover from a panicked reset --hard via reflog — the moment every developer learns that Git almost never truly deletes a committed change.

Three undos for three situations

In C#

bash 
# Undo a PUSHED commit safely (new inverse commit, history preserved):
git revert <bad-commit>

# Rewrite LOCAL history — three modes, increasingly destructive:
git reset --soft  HEAD~1     # keep changes STAGED
git reset --mixed HEAD~1     # keep changes in working dir (default)
git reset --hard  HEAD~1     # DISCARD changes — careful

# Recover after a bad reset/rebase — your commit is still in the reflog:
git reflog                   # find the lost commit's hash
git reset --hard <hash>      # restore it

Key takeaways

🎯 Remember these three

  • revert = new inverse commit, safe on shared branches; reset = move the branch pointer, for local history only.
  • reset modes: --soft (keep staged), --mixed (keep in working dir), --hard (discard).
  • reflog is the safety net — it remembers where HEAD has been (~90 days), so 'lost' commits are usually recoverable.

Interview tip

🗣️ When this comes up

The crisp answer: 'revert for public, reset for private, reflog to recover.' Knowing the three reset modes and that reflog makes most 'lost' commits recoverable shows you won't panic — or destroy shared history — under pressure.
