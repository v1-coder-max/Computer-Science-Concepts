How Git Works: Objects & Refs

Git stops being magic once you see the model: a few object types in a content-addressed store, and branches that are just pointers to commits.

#git #internals #commits #dag

The idea in plain English

Git stores four object types, each identified by the SHA-1/256 hash of its content (so identical content is stored once). A blob is a file's bytes; a tree is a directory listing (names → blobs/trees); a commit points to one tree (a full snapshot of the project) plus parent commit(s), an author, and a message.

Crucially, a commit is a snapshot, not a diff — it references the complete tree of the project at that moment. Git computes diffs on the fly for display, but stores snapshots, which is why history operations are fast and reliable.

Commits link to their parents, forming a directed acyclic graph (DAG) of history. A branch is nothing but a lightweight, movable pointer to one commit; HEAD points to your current branch. 'Creating a branch' just writes a 40-byte ref — which is why branching is instant in Git.

This model explains everything else: a merge creates a commit with two parents; a rebase rewrites commits onto a new base (new hashes); 'detached HEAD' means HEAD points straight at a commit instead of a branch. Once you picture commits-as-snapshots-in-a-DAG and branches-as-pointers, Git commands stop being incantations.

Real-world example

🏢 How this works in production

Every Git host — GitHub, GitLab — visualizes this DAG as the commit graph, and the speed you feel (instant branches, fast switches) comes directly from branches being mere pointers and objects being deduplicated by hash. Understanding it is what lets engineers confidently use rebase, cherry-pick, and reflog instead of fearing them, because they know each command just moves pointers and adds objects.

Commits in a DAG, branches as pointers

In C#

bash 
# A commit references a full tree snapshot + parent(s) + metadata.
$ git cat-file -p HEAD
tree 9c1f2a...            # the project snapshot at this commit
parent 4b7d3e...          # previous commit (the DAG edge)
author Jane <jane@x.io> 1700000000 +0000

# A branch is just a 40-char pointer in a file:
$ cat .git/refs/heads/main
3c2a9f8e...               # the commit 'main' points to — that's the whole branch

Key takeaways

🎯 Remember these three

  • Git stores snapshots, not diffs: blobs (files), trees (dirs), commits (tree + parents), all content-addressed by hash.
  • Commits form a DAG; a branch is just a movable pointer to a commit (so branching is instant).
  • Merge = a commit with two parents; rebase = replay commits onto a new base (new hashes).
