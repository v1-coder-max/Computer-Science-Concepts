Merge vs Rebase

Two ways to combine branches: merge records what really happened (with a merge commit); rebase rewrites your commits onto the latest base for a clean, linear story.

#git #merge #rebase #history

The idea in plain English

Merge takes the two branch tips and creates a new merge commit with both as parents. It's non-destructive — every commit stays exactly as it happened — but a busy repo accumulates merge commits and a tangled graph.

Rebase instead replays your branch's commits one by one on top of the target branch's latest commit, producing brand-new commits (new hashes) and a linear history with no merge commit. It's great for tidying your local feature branch before sharing — git rebase main so your work sits cleanly on top.

The trade-off is honesty vs cleanliness. Merge preserves the true, messy timeline; rebase fabricates a tidier one. Many teams combine them: rebase your feature branch to stay current and clean, then merge it into main (optionally squashed) so main reads as a sequence of features.

The golden rule of rebase: never rebase commits that others have already pulled. Rebasing rewrites history (new hashes), so a shared branch's history diverges from everyone else's, forcing painful force-pushes and duplicate commits. Rebase local/private work; merge shared work.

Real-world example

🏢 How this works in production

A common team policy: developers rebase their feature branch onto main while working (to integrate the latest and resolve conflicts incrementally), then open a PR that's squash-merged — giving main a clean, linear history of one commit per feature. The cardinal sin everyone learns once: rebasing or force-pushing a shared branch and breaking every teammate's checkout, which is why main/develop are protected from force-push.

Merge commit vs linear replay

In C#

bash 
# Rebase your private feature branch onto the latest main (clean, linear):
git switch feature
git rebase main                 # replays feature's commits on top of main

# Merge shared work (preserves history with a merge commit):
git switch main
git merge feature

# ⚠️ Golden rule: do NOT rebase commits already pushed and pulled by others.

Key takeaways

🎯 Remember these three

  • Merge = non-destructive, adds a merge commit (true history); rebase = replays commits for a clean linear history (new hashes).
  • Common workflow: rebase local branches to stay current, merge (or squash) into main.
  • Golden rule: never rebase commits others have already pulled — it rewrites shared history.

Interview tip

🗣️ When this comes up

Contrast honest-history (merge) vs clean-history (rebase), then state the golden rule unprompted — 'rebase local, merge shared.' Knowing rebase rewrites hashes (and why that breaks shared branches) is the depth check.
