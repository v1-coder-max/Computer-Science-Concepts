Resolving Merge Conflicts

A conflict just means Git can't decide between two edits to the same lines and is asking you to choose. They're routine — knowing the workflow turns panic into a two-minute task.

#git #conflicts #merge #workflow

The idea in plain English

Git auto-merges changes that don't overlap. A conflict arises only when two branches changed the same lines of the same file (or one edited a file the other deleted). Git pauses the merge/rebase and marks the spots, leaving the decision to you — it won't guess.

Inside a conflicted file, Git inserts markers: <<<<<<< HEAD shows your side, ======= separates, and >>>>>>> branch shows theirs. You edit the region to the correct final result, delete the markers, then stage the file.

The workflow: run the merge/rebase, list conflicts with git status, open each file and resolve it (often combining both sides, not just picking one), git add the resolved files, then git commit (merge) or git rebase --continue. A merge tool or your IDE's 3-way view makes this far easier than raw markers.

Prevention beats cure: integrate early and often (pull/rebase frequently so divergence stays small), keep changes small and focused, and agree on formatting (so whitespace doesn't manufacture conflicts). Big, long-lived branches are conflict factories — which is the practical argument for trunk-based development.

Real-world example

🏢 How this works in production

Conflicts are a daily, normal part of team development — two people touch the same function and Git asks who wins. Experienced developers resolve them calmly with their IDE's 3-way merge view and a quick test run; the teams that suffer 'merge hell' are the ones with long-lived branches that diverged for weeks. That pain is exactly why short-lived branches and frequent integration are recommended.

Two edits to the same lines

In C#

bash 
$ git merge feature
Auto-merging Pricing.cs
CONFLICT (content): Merge conflict in Pricing.cs

# The file now contains:
<<<<<<< HEAD
var price = 10;          // your side
=======
var price = 12;          // their side
>>>>>>> feature

# Edit to the correct result, remove the markers, then:
$ git add Pricing.cs
$ git commit             # (or: git rebase --continue)

Key takeaways

🎯 Remember these three

  • A conflict means two branches changed the same lines; Git pauses and asks you to choose — it never guesses.
  • Resolve by editing between the markers (<<</===/>>>), then git add and commit/continue; a 3-way merge tool helps.
  • Prevent conflicts by integrating early, keeping changes small, and standardizing formatting.
