Branching Strategies

How a team branches shapes how often it integrates and how it releases. The spectrum runs from heavy Git Flow to lightweight trunk-based development.

#git #workflow #trunk-based #git-flow

The idea in plain English

Git Flow is the heavyweight model: long-lived develop and main branches plus feature/*, release/*, and hotfix/* branches with strict rules. It suits versioned software with scheduled releases, but the long-lived branches mean code integrates late and merges hurt.

GitHub Flow simplifies to one rule: main is always deployable; you branch a short-lived feature branch, open a pull request, review, and merge back quickly. It fits continuous delivery and is the common default for web apps.

Trunk-based development goes furthest: everyone commits to main (trunk) at least daily, with very short-lived or no feature branches, and unfinished work hidden behind feature flags. It maximizes integration frequency and is what high-performing CI/CD teams (per DORA research) tend to use.

The core trade-off is branch lifetime vs merge pain: the longer a branch lives, the more it diverges and the worse 'merge hell' gets. Short-lived branches and frequent integration keep merges trivial — which is why the industry has drifted from Git Flow toward trunk-based.

Real-world example

🏢 How this works in production

Most modern web teams run GitHub Flow or trunk-based: small PRs merged to an always-deployable main, behind feature flags so incomplete features can ship dark. DORA's research links trunk-based development and small batch sizes to elite delivery performance. Git Flow still appears where there are real versioned releases (desktop apps, libraries, firmware) that need parallel maintenance of multiple versions.

From heavy to light

In C#

bash 
# GitHub Flow — the common default:
git switch -c feature/checkout-fix     # short-lived branch off main
# ...commit small changes...
git push -u origin feature/checkout-fix
# open a PR → review → merge → delete branch (main stays deployable)

# Trunk-based — integrate daily, hide unfinished work behind a flag:
if (FeatureFlags.IsEnabled("new-checkout")) { NewCheckout(); } else { OldCheckout(); }

Key takeaways

🎯 Remember these three

  • Spectrum: Git Flow (heavy, versioned releases) → GitHub Flow (short branches, always-deployable main) → trunk-based (commit to main daily + feature flags).
  • Core trade-off: branch lifetime vs merge pain — long branches diverge and cause merge hell.
  • High-performing CI/CD teams favor trunk-based development with small batches and feature flags.
