Event-Driven Architecture

Instead of services calling each other, they announce that something happened and let anyone interested react. Producers don’t know — or care — who’s listening.

#event-driven #pub-sub #choreography #eventual-consistency

The idea in plain English

An event is an immutable statement of fact about the past: OrderPlaced, PaymentFailed, UserSignedUp. In event-driven architecture, a service publishes events to a broker/event bus, and other services subscribe to the ones they care about (pub/sub). The publisher has no idea who consumes them — that’s the whole point: loose coupling.

The big wins:

  • Decoupling & extensibility — add a new consumer (analytics, fraud check, a new notification) without touching the producer.
  • Resilience & scale — consumers process independently, at their own pace, and survive each other’s outages.

Two ways to coordinate a multi-step business process:

  • Choreography — no central brain; each service reacts to events and emits its own. Decentralized and flexible, but the overall flow is implicit and harder to trace.
  • Orchestration — a coordinator explicitly drives the steps. Easier to see and change the flow, but the orchestrator is a focal point.

Two consequences to accept: the system is eventually consistent (reactions happen shortly after the fact, not atomically), and a related pattern, event sourcing, takes it further by storing the stream of events as the source of truth and deriving current state by replaying them.

Real-world example

🏢 How this works in production

When you place an order, the order service emits one OrderPlaced event. Independently, payment charges the card, inventory reserves stock, the shipping service schedules a pickup, the email service sends a confirmation, and analytics records the sale — all subscribers to the same event, none aware of the others. Months later, adding a “loyalty points” service is a new subscriber and zero changes to the order service. This is how Uber and LinkedIn build on Kafka: one event stream, many independent reactions.

One event, many reactions

In C# (publish & subscribe)

C# 
public record OrderPlaced(Guid OrderId, decimal Total, string Email);  // immutable fact

// Producer: announce what happened; don't call anyone directly.
await _bus.Publish(new OrderPlaced(order.Id, order.Total, order.Email));

// Consumers subscribe independently — each one a separate, isolated reaction.
public class EmailOnOrderPlaced : IConsumer<OrderPlaced>
{
    public Task Consume(ConsumeContext<OrderPlaced> ctx)
        => _email.SendConfirmationAsync(ctx.Message.Email, ctx.Message.OrderId);
}
public class StockOnOrderPlaced : IConsumer<OrderPlaced>
{
    public Task Consume(ConsumeContext<OrderPlaced> ctx)
        => _inventory.ReserveAsync(ctx.Message.OrderId);   // adding a new consumer = new class
}

Key takeaways

🎯 Remember these three

  • Services publish immutable events and subscribe to the ones they care about — producers don’t know consumers (loose coupling).
  • Extensible by design: add a new subscriber without changing the producer.
  • Choreography (decentralized reactions) vs orchestration (a coordinator); the system is eventually consistent; event sourcing stores events as the source of truth.

Interview tip

🗣️ When this comes up

Stress loose coupling (producer doesn’t know consumers) and extensibility (add a subscriber, change nothing upstream). Distinguish events from commands, and choreography vs orchestration, and acknowledge eventual consistency as the trade. Bonus: define event sourcing.
