Inter-Service Communication

Once you split into services, they have to talk over the network. The biggest design choice is synchronous (call and wait) vs asynchronous (fire an event).

#communication #sync-vs-async #rest #messaging

The idea in plain English

Synchronous communication (REST over HTTP, or gRPC) is request/response: service A calls service B and waits for the answer. It’s simple and intuitive, and right when you need an immediate result (a query). The danger is temporal coupling: if B is slow or down, A is stuck — and in a chain A→B→C→D, one slow link can cause cascading failures across the system. Within microservices, gRPC is often preferred over REST for internal calls (binary, HTTP/2, typed contracts, faster).

Asynchronous communication (message queues / events via a broker) is fire-and-forget: A publishes a message/event and moves on; B processes it when ready. This decouples the services in time — B can be down and catch up later — and adds resilience (retries, buffering). The cost is eventual consistency and the operational weight of a broker, plus harder debugging (no single call stack).

The rule of thumb: sync for queries that need an answer now; async for commands/events and anything that can happen in the background. And keep an eye on chatty synchronous chains — deep A→B→C→D request trees are a classic microservices anti-pattern that multiplies latency and failure surface.

Real-world example

🏢 How this works in production

At checkout, the order service might call the pricing service synchronously (you need the total right now), but it publishes an OrderPlaced event asynchronously so payment, inventory, shipping, and email all react on their own time — none of them block the user’s “order confirmed.” This split is why a slow email provider never delays a checkout. Internally, hot service-to-service calls (recommendations, catalog) often use gRPC for speed, while public-facing traffic stays REST.

Call-and-wait vs publish-and-forget

In C#

C# 
// SYNC: need the answer now (a query). Use resilient HttpClient or gRPC.
var price = await _pricingClient.GetTotalAsync(cart);   // waits for pricing service

// ASYNC: a command/event that can happen in the background. Publish and return.
await _bus.Publish(new OrderPlaced(order.Id, price));   // payment/inventory/email react later

// ⚠ Avoid deep synchronous chains: Order → Pricing → Tax → FX → ...
//   each hop adds latency and a new way to fail. Prefer events past the first hop.

Key takeaways

🎯 Remember these three

  • Synchronous (REST/gRPC) is simple and immediate but tightly couples services and risks cascading failures.
  • Asynchronous (events/queues) decouples services and adds resilience, at the cost of eventual consistency.
  • Sync for queries; async for commands/events. Avoid chatty deep sync chains; prefer gRPC for internal sync calls.

Interview tip

🗣️ When this comes up

Frame every comms decision as sync vs async: immediate answer → sync (REST/gRPC); background/decoupled → async (events). Raise cascading failures as the sync risk and eventual consistency as the async cost, and warn against deep synchronous call chains.
