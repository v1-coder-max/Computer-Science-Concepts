Outbox Pattern

You need to update your database and publish an event — atomically. But they’re two different systems. The transactional outbox makes it one atomic write.

#outbox #dual-write #reliable-messaging #at-least-once

The idea in plain English

A service often must do two things when something happens: save state to its database and publish an event to a broker. This is the dual-write problem — the DB and the message broker are separate systems with no shared transaction. If you save the order then crash before publishing, the event is lost (downstream never reacts). If you publish first then the DB save fails, you’ve sent a phantom event for something that didn’t happen. Either way, the system is inconsistent.

The outbox pattern removes the second system from the critical write. Instead of publishing directly, you insert the event into an outbox table in the same database, inside the same local transaction as the business data. Because it’s one transaction, the data change and the recorded event commit (or roll back) together — atomic, no dual write.

A separate relay/dispatcher process then reads unsent rows from the outbox and publishes them to the broker, marking each as sent. If it crashes mid-publish, it just retries on restart — so you get at-least-once delivery (consumers must be idempotent). A common variant uses Change Data Capture (Debezium) to tail the DB log instead of polling the table.

Real-world example

🏢 How this works in production

An order service saves the new order and an OrderPlaced row into its outbox in one transaction — so it’s impossible to have an order without a recorded event, or an event without an order. A background relay then publishes outbox rows to Kafka/RabbitMQ for payment, inventory, and email to consume. If the broker is briefly down or the relay restarts, nothing is lost: unsent rows sit safely in the database until they’re successfully published. This is the standard fix for “we sometimes lose events” bugs in event-driven systems, and it underpins reliable saga steps.

Atomic write, then relay

In C#

C# — EF Core 
// WRITE: order + event in ONE transaction → atomic, no dual write.
public async Task PlaceAsync(Order order)
{
    db.Orders.Add(order);
    db.Outbox.Add(new OutboxMessage {
        Type = nameof(OrderPlaced),
        Payload = JsonSerializer.Serialize(new OrderPlaced(order.Id, order.Total)),
        OccurredUtc = DateTime.UtcNow
    });
    await db.SaveChangesAsync();   // commits BOTH rows together — or neither
}

// RELAY (a BackgroundService): publish unsent rows, mark sent, retry on failure.
var batch = await db.Outbox.Where(m => m.SentUtc == null).Take(100).ToListAsync();
foreach (var m in batch)
{
    await _bus.Publish(m.Type, m.Payload);   // at-least-once → consumers must be idempotent
    m.SentUtc = DateTime.UtcNow;
}
await db.SaveChangesAsync();
// (MassTransit ships a built-in transactional outbox; Debezium/CDC is the log-tailing variant.)

Key takeaways

🎯 Remember these three

  • The dual-write problem: a DB write + a broker publish aren’t atomic, so events get lost or sent for nothing.
  • The outbox writes the event into an outbox table in the same transaction as the data — atomic by construction.
  • A relay publishes the outbox separately with retries → at-least-once delivery (consumers must be idempotent). CDC/Debezium is the log-tailing variant.

Interview tip

🗣️ When this comes up

“How do you reliably publish an event after a DB change?” Name the dual-write problem, then the outbox: same-transaction write + a relay that retries, yielding at-least-once (so consumers are idempotent). Mentioning CDC/Debezium as an alternative reader is a strong finish.
