Monolith vs Microservices

One deployable app or many small ones? It’s not about which is “modern” — it’s a trade between simplicity and independent scale/deploy, paid for in complexity.

#microservices #monolith #architecture #bounded-context

The idea in plain English

A monolith is one application — all features, one codebase, one deployable, usually one database. Microservices split the system into many small, independently deployable services, each owning a bounded context (a business capability) and its own data. They talk over the network.

Monolith — pros: simple to build, run, and test; one deploy; trivial in-process calls; easy ACID transactions. Cons: you scale the whole thing even if one module is hot; modules get tangled over time; every change is a big-bang deploy; one bug can take down everything.

Microservices — pros: deploy and scale each service independently; teams own services autonomously (Conway’s Law); pick the right tech per service; a failure can be isolated. Cons: you’ve traded code complexity for distributed-systems complexity — the network is unreliable, transactions span services (no easy ACID → sagas), and you need serious observability, deployment, and ops machinery.

The honest default: start with a monolith (ideally a modular monolith with clean internal boundaries), and extract services only when a real driver appears — a module that must scale separately, a team that needs to deploy on its own cadence, or a clear bounded context. Reaching for microservices on day one usually buys the complexity without the payoff.

Real-world example

🏢 How this works in production

Amazon and Netflix both started as monoliths and migrated to microservices once scale and org size demanded it — Netflix needed thousands of engineers to deploy independently and to scale streaming separately from billing. But for every famous migration there are countless startups that adopted microservices too early and drowned in distributed complexity (network failures, debugging across services, data consistency) before they had the traffic or team size to need it. The lesson everyone repeats: earn your microservices.

The decision factors

When microservices earn their cost 
choose_microservices_when:
  - independent scaling needed     # one part is hot (e.g. streaming vs billing)
  - many teams shipping in parallel # avoid stepping on each other (Conway's Law)
  - clear bounded contexts          # natural seams in the domain
  - fault isolation matters         # one feature must not sink the rest

stay_monolith_when:
  - small team / early stage        # simplicity > autonomy
  - domain still shifting           # boundaries aren't clear yet
  - you can't run distributed ops   # tracing, CI/CD, on-call maturity missing
# Pragmatic path: modular monolith → extract services as real drivers appear.

Key takeaways

🎯 Remember these three

  • Monolith = simple, one deploy, easy transactions; microservices = independent deploy/scale + team autonomy.
  • Microservices trade code complexity for distributed-systems complexity (network, consistency, observability, ops).
  • Start with a (modular) monolith; extract services only when a real driver — scale, teams, or a clear bounded context — demands it.

Interview tip

🗣️ When this comes up

“Would you use microservices here?” Never answer “yes, they’re modern.” Weigh the drivers (independent scale, team autonomy, bounded contexts) against the distributed tax, and say you’d start with a modular monolith and extract services when justified. Mention Conway’s Law and that data ownership (a DB per service) is the hardest part.
