Microservices with .NET

The patterns are universal; here’s the .NET toolbox that implements them — gateway, messaging, sidecar building blocks, resilience, and observability.

#dotnet #masstransit #dapr #aspire

The idea in plain English

.NET has a mature, batteries-included stack for microservices. Each tool maps to a pattern from this category:

  • API Gateway → YARP (Microsoft’s reverse proxy) or Ocelot — routing, auth, rate limiting at the edge.
  • Async messaging / events / saga / outbox → MassTransit — a broker abstraction over RabbitMQ, Azure Service Bus, and Amazon SQS, with first-class saga state machines, a transactional outbox, retries, and scheduling.
  • Service invocation, pub/sub, state, secrets → Dapr — a language-agnostic sidecar that runs next to your service and exposes portable building blocks over HTTP/gRPC, so you don’t hard-code Redis/Kafka/etc.
  • Local orchestration + dev-time observability → .NET Aspire — an opinionated stack to compose, run, and wire up multi-service apps (with dashboards, service discovery, and telemetry) locally and toward deploy.
  • Sync calls → gRPC / typed HttpClient; resilience → Polly (circuit breaker, retry, timeout, bulkhead); observability → OpenTelemetry + Health Checks; packaging → Docker + Kubernetes.

The mental model: Dapr abstracts infrastructure (don’t bind to a specific broker/store), MassTransit gives you a rich messaging/saga framework in C#, and Aspire ties the whole multi-service app together for development. You rarely need all of them — pick per need.

Real-world example

🏢 How this works in production

A .NET shop building an e-commerce backend fronts its services with YARP, has the order service publish OrderPlaced via MassTransit (with the outbox so no event is lost) and run the checkout saga as a MassTransit state machine, wraps downstream calls in Polly, and exports traces/metrics with OpenTelemetry to Grafana. During development, .NET Aspire spins up all the services plus Redis and RabbitMQ with one run and shows a live dashboard. In production it all ships as Docker images on Kubernetes. Every piece is a pattern from this category with a .NET implementation.

The stack, mapped to concerns

In C# (MassTransit)

C# 
// MassTransit over RabbitMQ, with the transactional outbox + consumers.
builder.Services.AddMassTransit(x =>
{
    x.AddConsumer<OrderPlacedConsumer>();
    x.AddEntityFrameworkOutbox<AppDbContext>(o => o.UsePostgres());  // outbox pattern, built in
    x.UsingRabbitMq((ctx, cfg) =>
    {
        cfg.Host("rabbitmq");
        cfg.ConfigureEndpoints(ctx);
        cfg.UseMessageRetry(r => r.Exponential(3,                       // resilience
            TimeSpan.FromSeconds(1), TimeSpan.FromSeconds(30), TimeSpan.FromSeconds(2)));
    });
});

// Publish an event — saga state machines and consumers react.
await _bus.Publish(new OrderPlaced(order.Id, order.Total));

Key takeaways

🎯 Remember these three

  • YARP/Ocelot (gateway), MassTransit (messaging/saga/outbox), Dapr (portable infra building blocks via a sidecar), Aspire (compose + dev-time observability).
  • gRPC + Polly + OpenTelemetry + Health Checks cover sync calls, resilience, and observability; Docker + K8s for packaging/running.
  • Each tool implements a pattern from this category — choose per concern, not all at once.
