Strangler Fig Pattern

You rarely get to rewrite a big system from scratch. The strangler fig pattern replaces a legacy app incrementally — one slice at a time — until the old one can be retired.

#microservices #migration #legacy #refactoring

The idea in plain English

The name comes from the strangler fig vine that grows around a tree, gradually replacing it until the original is gone. Applied to software, you grow a new system around the old one, moving functionality piece by piece, instead of attempting a risky big-bang rewrite (which routinely fails — it's a years-long parallel build that has to catch up to a moving target).

The mechanism is a façade / routing layer (often an API gateway or proxy) in front of the legacy app. Initially it forwards everything to the monolith. As you build a replacement for a slice of functionality, you reroute just that slice to the new service; everything else still hits the legacy. Over time more requests go to the new system and fewer to the old, until the monolith is 'strangled' and removed.

This makes migration incremental and reversible: each slice is a small, shippable change you can canary and roll back, delivering value continuously rather than after a multi-year freeze. You also avoid the rewrite trap of having to reproduce every quirk of the legacy before shipping anything.

The hard parts are data (the new and old systems may share or need to sync a database during the transition — often dual-writes or the same store temporarily) and discipline (it's tempting to leave the last awkward 10% on the legacy forever). But for evolving a monolith toward services, it's the standard low-risk approach.

Real-world example

🏢 How this works in production

Teams migrating a monolith to microservices almost always use this: put a gateway in front, then peel off bounded contexts (say, 'shipping', then 'payments') into their own services one at a time, rerouting traffic as each is ready. Companies have publicly described multi-year strangler migrations that shipped value the whole way through — the opposite of the doomed 'we'll freeze features and rewrite everything' project that the pattern exists to avoid.

Reroute slices until the monolith is gone

In C#

C# 
// The façade routes by slice — migrated paths go to new services,
// everything else falls through to the legacy monolith.
app.MapWhen(ctx => ctx.Request.Path.StartsWithSegments("/shipping"),
    b => b.UseProxyTo("http://shipping-service"));   // migrated → new service

app.MapWhen(ctx => ctx.Request.Path.StartsWithSegments("/payments"),
    b => b.UseProxyTo("http://payments-service"));   // migrated → new service

app.Run(LegacyMonolithHandler);                      // everything else → legacy
// Move one slice at a time; each is canaryable and reversible.

Key takeaways

🎯 Remember these three

  • Replace a legacy system incrementally — one slice at a time behind a façade — instead of a risky big-bang rewrite.
  • A routing layer/gateway sends migrated functionality to new services and the rest to the legacy, shifting over time.
  • It's incremental, reversible, and ships value throughout; the hard parts are shared data during transition and finishing the last slices.

🗣️ When this comes up

Name the pattern and contrast it with the failure-prone big-bang rewrite: a façade reroutes slices to new services incrementally. Calling out the shared-data challenge during transition shows you've thought past the happy path.
