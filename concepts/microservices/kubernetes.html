Kubernetes Concepts

Containers are great; running hundreds of them across many machines — scaling, healing, networking, updating — is the hard part. That orchestration is what Kubernetes does.

#kubernetes #orchestration #pods #declarative

The idea in plain English

Kubernetes (K8s) is a container orchestrator. You tell it the desired state — “run 5 replicas of the orders service, expose it on this address” — and it continuously works to make reality match: scheduling containers onto machines, restarting crashed ones, replacing failed nodes, and wiring up networking. It’s declarative: you describe the end state in YAML, K8s figures out how to get and keep you there. The core objects:

  • Pod — the smallest deployable unit: one (or a few tightly-coupled) containers sharing a network/storage. Pods are disposable and get new IPs when rescheduled.
  • Deployment — declares the desired number of pod replicas and manages rolling updates and rollbacks. This is what you usually write.
  • Service — a stable name + virtual IP that load-balances across a set of pods (since pod IPs change). This is service discovery, built in.
  • Ingress — HTTP(S) routing from outside the cluster to Services (host/path rules, TLS).

The platform splits into a control plane (the brain: API server, scheduler, controllers that drive desired→actual) and worker nodes (machines that run the pods). On top, K8s gives you self-healing (restart failed pods, reschedule off dead nodes), autoscaling (the Horizontal Pod Autoscaler adds pods under load), and zero-downtime rolling updates — exactly the operational machinery microservices need.

Real-world example

🏢 How this works in production

A team deploys 30 microservices to a Kubernetes cluster. Each is a Deployment with a few replicas; if a pod crashes (OOM, bug), K8s restarts it within seconds, and if a whole node dies, its pods are rescheduled elsewhere — self-healing with no pager at 3am. A Black-Friday spike trips the autoscaler, which adds pods; afterward they’re removed. Shipping a new version is a rolling update: K8s brings up new pods, waits for their readiness probes, then retires the old ones — zero downtime, instant rollback if health checks fail.

The cluster

A Deployment + Service (YAML)

YAML 
apiVersion: apps/v1
kind: Deployment
metadata: { name: orders }
spec:
  replicas: 3                       # desired state: keep 3 pods running
  selector: { matchLabels: { app: orders } }
  template:
    metadata: { labels: { app: orders } }
    spec:
      containers:
        - name: orders
          image: registry/orders:1.0
          readinessProbe:           # don't send traffic until healthy (rolling update)
            httpGet: { path: /health/ready, port: 8080 }
---
apiVersion: v1
kind: Service                       # stable name + LB across the 3 pods
metadata: { name: orders }
spec:
  selector: { app: orders }
  ports: [ { port: 80, targetPort: 8080 } ]

Key takeaways

🎯 Remember these three

  • Kubernetes orchestrates containers declaratively: you describe desired state, it reconciles reality to match.
  • Core objects: Pod (smallest unit), Deployment (replicas + rolling updates), Service (stable endpoint + LB), Ingress (HTTP routing).
  • You get self-healing, autoscaling, and zero-downtime rolling updates out of the box.
