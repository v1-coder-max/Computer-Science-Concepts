Docker & Containers

A container packages your app and everything it needs to run into one portable, isolated unit — so it behaves the same on your laptop, in CI, and in production.

#docker #containers #images #dockerfile

The idea in plain English

“Works on my machine” happens because apps depend on a specific runtime, libraries, and OS config that differ between environments. A container bundles the app with all of that — its dependencies, runtime, and config — into one isolated package. Run it anywhere with a container engine and you get the same behavior. Two terms to keep straight:

  • Image — an immutable, layered template (the “class”). Built once, stored in a registry (Docker Hub, GHCR), pulled anywhere.
  • Container — a running instance of an image (the “object”). You can run many containers from one image.

You define an image with a Dockerfile — a recipe of steps (base image, copy files, restore, build, set the entry point). Each step is a cached layer, so rebuilds only redo what changed (order your Dockerfile to maximize cache hits — copy project files and restore before copying source).

Containers are often confused with VMs but they’re lighter: a VM virtualizes a whole machine and ships a full guest OS; a container shares the host kernel and isolates just the process (via namespaces/cgroups). So containers start in milliseconds and pack far more densely onto a host — which is exactly why microservices ship as containers.

Real-world example

🏢 How this works in production

A team builds each .NET microservice into a Docker image in CI, tags it with the git commit, and pushes it to a registry. The exact same image a developer ran locally is what gets deployed to staging and production — no “but it worked on my box.” Because containers are lightweight and identical, a CI pipeline can spin up the service plus a Postgres and a Redis container to run integration tests in seconds, then tear them down. This reproducibility and density is the foundation that orchestration (Kubernetes) builds on.

Image → containers (vs VM)

A Dockerfile for a .NET service

Dockerfile 
# Build stage (multi-stage keeps the final image small)
FROM mcr.microsoft.com/dotnet/sdk:8.0 AS build
WORKDIR /src
COPY *.csproj .
RUN dotnet restore          # cached layer: only re-runs if the .csproj changed
COPY . .
RUN dotnet publish -c Release -o /app

# Runtime stage: just the runtime + published output (smaller, fewer CVEs)
FROM mcr.microsoft.com/dotnet/aspnet:8.0
WORKDIR /app
COPY --from=build /app .
ENTRYPOINT ["dotnet", "Orders.dll"]
Bash 
docker build -t orders:1.0 .       # build the image from the Dockerfile
docker run -p 8080:8080 orders:1.0 # run a container from it
docker push registry/orders:1.0    # share via a registry

Key takeaways

🎯 Remember these three

  • A container packages an app + its dependencies into a portable, isolated unit — same behavior everywhere.
  • Image = immutable layered template; container = a running instance. A Dockerfile is the build recipe; layers are cached.
  • Containers share the host kernel (lightweight, fast) — unlike VMs, which each ship a full guest OS.

Interview tip

🗣️ When this comes up

Two staples: image vs container (template vs running instance) and container vs VM (shared kernel/process isolation vs full guest OS). Mention layer caching and multi-stage builds for small images — it shows you’ve actually written Dockerfiles.
