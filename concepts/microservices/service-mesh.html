Service Mesh

Once you have dozens of services, every one needs retries, timeouts, mTLS, and tracing. A service mesh provides all of that in the infrastructure, so your code doesn't have to.

#microservices #service-mesh #networking #sidecar

The idea in plain English

As a system grows to many services, the same cross-cutting networking concerns recur in every one: retries with backoff, timeouts, circuit breaking, mutual TLS, load balancing, traffic splitting, and request tracing. Implementing them in each service (and each language) is duplicated, inconsistent effort.

A service mesh extracts that into the platform. It deploys a sidecar proxy (e.g. Envoy) next to every service instance; all traffic flows through the proxy, which transparently applies the policies. The application just makes a normal call to another service — the mesh handles the resilience, security, and telemetry around it.

It splits into a data plane (the sidecar proxies doing the actual work) and a control plane (which configures them centrally). So you declare 'mTLS everywhere, retry these calls 3×, send 5% of traffic to v2' once, and the control plane pushes it to every proxy — no code change, no redeploy of services.

The big wins are uniform mTLS (encrypted, authenticated service-to-service traffic by default), consistent resilience and traffic management (canary/blue-green at the network layer), and automatic observability (every hop is traced and measured). The cost is real operational complexity and per-hop latency from the extra proxies — so a mesh earns its keep at scale, not for three services.

Real-world example

🏢 How this works in production

Istio and Linkerd are the common meshes, deploying Envoy sidecars across a Kubernetes cluster to give every service mTLS, retries, and tracing without touching application code. Large platforms use the mesh to do canary rollouts and enforce zero-trust networking centrally. Smaller systems often skip it — the operational overhead and added latency outweigh the benefit until you have enough services that doing this per-service is genuinely painful.

Sidecars carry the cross-cutting concerns

In C#

YAML 
# A service mesh moves this OUT of your code into the sidecar.
# Declare policy once (Istio) — applied to every proxy, no app change:
apiVersion: networking.istio.io/v1
kind: VirtualService
spec:
  http:
    - retries: { attempts: 3, perTryTimeout: 2s }   # resilience, no code
      route:
        - destination: { host: orders, subset: v1 }
          weight: 95                                  # canary: 5% to v2
        - destination: { host: orders, subset: v2 }
          weight: 5
# Plus automatic mTLS + tracing for every hop.

Key takeaways

🎯 Remember these three

  • A service mesh moves cross-cutting networking (retries, timeouts, mTLS, tracing, traffic shifting) out of app code into sidecar proxies.
  • Data plane (proxies) does the work; control plane configures them centrally — change policy without redeploying services.
  • Wins: uniform mTLS, consistent resilience/observability; costs: operational complexity and per-hop latency — worth it at scale.

Interview tip

🗣️ When this comes up

Frame it as moving cross-cutting networking concerns into sidecars (data plane) configured by a control plane. The balanced take — uniform mTLS/resilience/observability vs operational overhead and latency — shows you'd adopt it at the right scale, not reflexively.
