API Gateway

One front door for many services. Clients talk to the gateway; it routes to the right service and handles the cross-cutting concerns in one place.

#api-gateway #routing #bff #microservices

The idea in plain English

With dozens of microservices, you don’t want clients calling each one directly — they’d need to know every address, handle auth everywhere, and make many round trips. An API gateway is a single entry point that sits in front of the services and does the shared work:

  • Routing — map incoming paths to the right backend service.
  • Authentication / authorization — validate tokens once, at the edge.
  • Rate limiting & throttling — protect the whole system.
  • TLS termination, compression, caching.
  • Request aggregation — fan out to several services and combine the responses, so the client makes one call.
  • Protocol translation — e.g. external REST/JSON ↔ internal gRPC.

A common refinement is the Backend-for-Frontend (BFF): a dedicated gateway per client type (web, mobile) tailored to that UI’s needs, so a mobile app gets compact, mobile-shaped responses.

The caveat: the gateway is a single point of failure and a potential bottleneck, so it must be highly available and kept thin — routing and cross-cutting concerns, not business logic (that belongs in the services). It pairs with a load balancer in front of it.

Real-world example

🏢 How this works in production

Netflix built Zuul as its gateway, fronting hundreds of services and handling auth, routing, and resilience at the edge. Your phone’s Netflix app hits one endpoint; the gateway authenticates you, routes to the right services, and often aggregates (profile + continue- watching + recommendations) into a single response so the app isn’t making ten calls over a mobile network. In .NET, teams use YARP or Ocelot for the same role.

One entry point, many services

In config (YARP reverse-proxy gateway)

JSON — YARP (ASP.NET Core) 
{
  "ReverseProxy": {
    "Routes": {
      "orders":  { "ClusterId": "orders",  "Match": { "Path": "/api/orders/{**c}" } },
      "catalog": { "ClusterId": "catalog", "Match": { "Path": "/api/catalog/{**c}" } }
    },
    "Clusters": {
      "orders":  { "Destinations": { "d1": { "Address": "http://orders:8080/" } } },
      "catalog": { "Destinations": { "d1": { "Address": "http://catalog:8080/" } } }
    }
  }
}

Key takeaways

🎯 Remember these three

  • An API gateway is the single entry point that routes to services and centralizes auth, rate limiting, TLS, aggregation, and protocol translation.
  • The BFF pattern gives each client type (web/mobile) its own tailored gateway.
  • Keep it thin (no business logic) and highly available — it’s a potential bottleneck and single point of failure.

Interview tip

🗣️ When this comes up

List the gateway’s jobs (routing, auth, rate limiting, aggregation, TLS) and why clients shouldn’t call services directly. Then show balance: it’s a single point of failure, so keep it thin and HA, and don’t leak business logic into it. Mentioning the BFF pattern is a strong add.
