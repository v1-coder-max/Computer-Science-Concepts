Retry, Timeout & Bulkhead

The resilience patterns you reach for daily. Used well they hide transient faults; used carelessly they amplify an outage.

#resilience #retry #bulkhead #timeout

The idea in plain English

Networks fail in small, temporary ways all the time. These patterns make a service tolerate that:

  • Retry — many failures are transient (a brief blip, a dropped packet). Just try again. But do it right: exponential backoff (wait longer each attempt) plus jitter (randomize the delay) so a thousand clients don’t retry in lockstep and create a thundering herd. Only retry idempotent operations, and cap the attempts.
  • Timeout — never wait forever. Every remote call gets a bounded timeout, so a hung dependency can’t pin your threads/connections indefinitely. A missing timeout is the root of most cascading failures.
  • Bulkhead — named after a ship’s watertight compartments: isolate resources so a flood in one can’t sink the whole vessel. Give each dependency its own limited pool (connections/concurrency); if one dependency goes slow and saturates its pool, the others keep working instead of every thread piling onto the slow one.

These compose: a typical resilient call has a timeout, bounded retries with backoff+jitter, a bulkhead limiting concurrency, and a circuit breaker for sustained failure — applied in a sensible order.

Real-world example

🏢 How this works in production

A service calling several downstreams gives each its own bulkhead (say, max 20 concurrent calls). When the recommendations service goes slow, its 20 slots fill and further recommendation calls are shed — but the orders and catalog calls, in their own compartments, sail on. Meanwhile a flaky network call is wrapped in a 2-second timeout and 3 retries with exponential backoff + jitter, so a momentary blip is invisible to users without hammering the dependency. The cautionary tale is retries without backoff: during an outage they multiply load and turn a small failure into a full meltdown.

Bulkhead isolation

In C# (Polly)

C# 
builder.Services.AddHttpClient<CatalogClient>()
    .AddResilienceHandler("catalog", b =>
    {
        // Retry transient faults with exponential backoff + jitter (idempotent calls!).
        b.AddRetry(new()
        {
            MaxRetryAttempts = 3,
            BackoffType = DelayBackoffType.Exponential,
            UseJitter = true                         // avoid synchronized retry storms
        });
        // Bulkhead: cap concurrency so a slow catalog can't exhaust all threads.
        b.AddConcurrencyLimiter(permitLimit: 20, queueLimit: 0);
        // Always bound the wait.
        b.AddTimeout(TimeSpan.FromSeconds(2));
    });
// ⚠ Never retry non-idempotent writes blindly, and never retry without backoff.

Key takeaways

🎯 Remember these three

  • Retry transient faults with exponential backoff + jitter, only for idempotent operations, with a cap.
  • Put a timeout on every remote call — unbounded waits cause cascading failures.
  • Bulkhead isolates resource pools per dependency so one slow service can’t exhaust everything.

🗣️ When this comes up

The retry trap: a naive retry loop amplifies outages — say backoff + jitter, idempotent only, and a cap. Add that every remote call needs a timeout, and explain bulkhead isolation with the ship analogy. Together with the circuit breaker, these are the four resilience patterns.
