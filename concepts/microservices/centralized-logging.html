Centralized Logging

Twenty service instances each writing logs to their own container is twenty haystacks. Centralized logging pulls them into one searchable place.

#logging #observability #elk #correlation-id

The idea in plain English

In microservices, the log lines for a single user request are spread across many services and many replicas — and containers are ephemeral, so a crashed pod takes its local logs with it. SSHing into boxes to grep doesn’t scale. Centralized logging fixes this: every service ships its logs to a central aggregation store where you can search, filter, dashboard, and alert across the whole system.

The common stacks:

  • ELK / Elastic Stack — Elasticsearch (store/search) + Logstash or Beats (ship/parse) + Kibana (visualize). The classic.
  • Loki + Grafana, Seq, Splunk, or a cloud service (CloudWatch, App Insights).

Two things make it actually useful. First, structured logging — emit key/value properties, not flat text — so you can query by field (status=500 AND service=orders). Second, a correlation/trace ID stamped on every log line and propagated across services, so you can pull up every log for one request in one search. That trace ID is the same one your distributed tracing uses, which is why logs and traces are sibling pillars of observability (with metrics).

Real-world example

🏢 How this works in production

A request errors out somewhere in a chain of services. An engineer copies the correlation ID from the error response, pastes it into Kibana, and instantly sees the ordered log lines from every service that touched that request — the exact exception, in the exact service, with the exact inputs. Dashboards show error rates per service, and alerts fire when 5xx spikes. This is table stakes at any company running more than a couple of services; without it, debugging distributed systems is guesswork by timestamp.

Ship → aggregate → search

In C# (Serilog → central sink)

C# 
builder.Host.UseSerilog((ctx, cfg) => cfg
    .Enrich.FromLogContext()
    .Enrich.WithProperty("service", "orders")           // tag which service emitted it
    .WriteTo.Console()
    .WriteTo.Elasticsearch("http://elasticsearch:9200")); // central, searchable store

// Stamp a correlation/trace id on every log of a request (middleware):
app.Use(async (ctx, next) =>
{
    var traceId = Activity.Current?.TraceId.ToString() ?? Guid.NewGuid().ToString();
    using (LogContext.PushProperty("TraceId", traceId))  // now every log line carries it
        await next();
});
// Search "TraceId: abc123" in Kibana → every service's logs for that one request.

Key takeaways

🎯 Remember these three

  • Ship logs from every service to one searchable store (ELK/Loki/Seq) — per-instance container logs are ephemeral and useless alone.
  • Use structured logging so you can query by field, and stamp a correlation/trace ID to follow one request across services.
  • Logs + metrics + traces are the three pillars of observability; the shared trace ID links logs to traces.

Interview tip

🗣️ When this comes up

“How do you debug across services?” — centralized logging with a correlation ID + structured logs in a store like the ELK stack. Connect it to distributed tracing (same trace ID) and name the three observability pillars (logs, metrics, traces).
