Circuit Breaker

When a downstream service is failing, the worst thing you can do is keep calling it. A circuit breaker fails fast — protecting both you and the struggling service.

#circuit-breaker #resilience #fail-fast #polly

The idea in plain English

Named after the electrical kind, a circuit breaker sits between your service and a dependency and trips when that dependency starts failing. Instead of every request piling up on a dead service — exhausting your threads/connections and cascading the outage upward — calls are rejected immediately (or routed to a fallback) until the dependency looks healthy again. It has three states:

  • Closed — normal. Requests flow through; the breaker counts failures.
  • Open — the failure threshold was crossed. Requests are rejected instantly (fail fast / fallback) for a cool-down period, giving the dependency room to recover.
  • Half-Open — after the cool-down, let a trial request through. Success → close the circuit (back to normal); failure → re-open and wait again.

The point is twofold: fail fast so your service stays responsive (a quick error beats a 30-second hang), and stop the stampede so the dependency can recover instead of being kept down by relentless retries. It pairs naturally with retries, timeouts, and a sensible fallback (cached data, a queued action, a friendly “try later”).

Real-world example

🏢 How this works in production

If the payment provider starts timing out during a sale, a circuit breaker around those calls opens after a handful of failures: checkout instantly shows “payments are temporarily unavailable, we’ll retry” instead of every shopper’s request hanging for 30 seconds and draining the thread pool. After a cool-down it sends one trial charge; if it succeeds, normal service resumes. Netflix’s Hystrix popularized this; in .NET it’s built into Polly and the standard resilience handler. Without it, one slow dependency takes the whole system down.

The state machine

In C# (Polly)

C# 
// Attach a circuit breaker to a typed HttpClient (Microsoft.Extensions.Http.Resilience / Polly).
builder.Services.AddHttpClient<PaymentClient>()
    .AddResilienceHandler("payments", b =>
    {
        b.AddCircuitBreaker(new()
        {
            FailureRatio = 0.5,                       // open if >50% of calls fail...
            MinimumThroughput = 10,                   // ...over a sample of 10+ calls
            BreakDuration = TimeSpan.FromSeconds(15)  // stay open 15s, then half-open
        });
        b.AddTimeout(TimeSpan.FromSeconds(2));        // pair with a timeout
    });

// When open, calls throw BrokenCircuitException immediately → catch it and return a fallback.
try { return await _payments.ChargeAsync(order); }
catch (BrokenCircuitException) { return PaymentResult.TryAgainLater; }

Key takeaways

🎯 Remember these three

  • A circuit breaker stops calls to a failing dependency, failing fast to keep your service responsive and let the dependency recover.
  • Three states: Closed (pass + count), Open (reject/fallback), Half-Open (one trial, then close or re-open).
  • It prevents cascading failures; combine it with timeouts, retries, and a sensible fallback.

Interview tip

🗣️ When this comes up

Describe the three states and the goal: fail fast to avoid cascading failures and give the dependency time to heal. Contrast it with retry (retry handles transient blips; the breaker handles sustained failure), and mention a fallback. Polly is the .NET answer.
