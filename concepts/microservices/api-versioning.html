API Versioning

Once others depend on your API, you can't just change it — you'd break them. Versioning and backward compatibility let you evolve without breaking existing clients.

#microservices #api #versioning #compatibility

The idea in plain English

An API is a contract with its consumers. Any change that breaks that contract — removing a field, renaming one, changing a type, tightening validation — breaks clients you may not control or be able to update in lockstep. Versioning is how you ship change anyway.

Prefer backward-compatible (additive) changes whenever possible: adding a new optional field or endpoint doesn't break anyone, so it needs no new version. Reserve a new version for genuinely breaking changes. This 'tolerant reader / additive change' discipline avoids most version proliferation.

When you do need versions, the common strategies are: URL path (/v1/orders — explicit, easy to route and cache, most popular); query string (?api-version=2); custom header (api-version: 2 — keeps URLs clean); and media type / content negotiation (Accept: application/vnd.api.v2+json — 'purist' REST). Pick one and be consistent.

Versioning also needs a deprecation policy: run old and new versions in parallel, announce timelines, signal deprecation (a Sunset header, docs), and give clients time to migrate before removing the old one. Internally, smaller teams lean on additive changes and consumer-driven contract tests to evolve safely without heavy versioning at all.

Real-world example

🏢 How this works in production

Public APIs (Stripe, GitHub, Twilio) version explicitly and keep old versions alive for years with clear deprecation timelines, because their customers can't all upgrade at once — Stripe famously pins each account to the API version it was created with. Internally, microservice teams favor additive, backward-compatible changes plus contract tests so services can deploy independently; a breaking change to a shared API is treated as a migration project, not a casual edit.

Evolve without breaking clients

In C#

C# 
// Asp.Versioning — declare versions; clients pick via URL/header/query.
builder.Services.AddApiVersioning(o => {
    o.DefaultApiVersion = new ApiVersion(1, 0);
    o.ReportApiVersions = true;          // advertise supported versions
});

[ApiVersion("1.0")] [ApiVersion("2.0")]
[Route("v{version:apiVersion}/orders")]   // URL versioning
public class OrdersController : ControllerBase
{
    [HttpGet, MapToApiVersion("2.0")]
    public IActionResult GetV2() => Ok(/* new shape */);
}
// Rule of thumb: add fields freely (no bump); version only on breaking changes.

Key takeaways

🎯 Remember these three

  • An API is a contract — prefer additive, backward-compatible changes (no version needed); version only on breaking changes.
  • Strategies: URL path (most common), query string, custom header, or media type — pick one, stay consistent.
  • Run old + new in parallel with a clear deprecation policy; internally, additive changes + contract tests often suffice.

Interview tip

🗣️ When this comes up

Lead with 'prefer additive changes, version only when you must break the contract,' then list the strategies (URL is most common). Mentioning a deprecation policy and parallel-running versions shows you think about consumers you can't force to upgrade.
