Distributed Tracing

When one user click fans out across a dozen services, “where’s the slowness?” is impossible to answer from logs alone. Tracing follows the request end to end.

#tracing #observability #opentelemetry #spans

The idea in plain English

In a monolith you read one stack trace. In microservices a single request hops Gateway → Orders → Payments → Inventory → … and there’s no shared stack. Distributed tracing stitches it back together. The model:

  • A trace represents the whole request and carries a unique trace ID.
  • A span is one operation within it (an HTTP call, a DB query) with a start/end time, a parent span, and tags. The spans form a tree.

The magic is context propagation: each service passes the trace ID (and current span ID) to the next via a header — the W3C standard traceparent — so every hop is recorded against the same trace. A tracing backend (Jaeger, Zipkin, Tempo) then renders the trace as a waterfall: you instantly see which span took 4 seconds and which service owns it. OpenTelemetry (OTel) is the vendor-neutral standard for producing this telemetry; tracing is one of the three pillars of observability alongside logs and metrics.

Real-world example

🏢 How this works in production

A user reports “checkout is slow.” Without tracing you’d grep logs across eight services hoping to correlate timestamps. With tracing, you open the trace for that request in Jaeger and the waterfall shows it plainly: the request spent 3.8 seconds in a single span — the inventory service waiting on a slow database query — while everything else was fast. That’s the difference between an afternoon of guessing and a 30-second diagnosis. The same trace ID also appears in the centralized logs, so you can jump straight to the relevant log lines.

A trace is a tree of spans

In C# (OpenTelemetry)

C# 
// Wire up OpenTelemetry tracing → exports spans to Jaeger/Tempo/etc.
builder.Services.AddOpenTelemetry()
    .WithTracing(t => t
        .AddAspNetCoreInstrumentation()   // auto-span for incoming requests
        .AddHttpClientInstrumentation()   // auto-propagates traceparent on outgoing calls
        .AddEntityFrameworkCoreInstrumentation()
        .AddOtlpExporter());              // ship to the collector/backend

// Add your own span around a unit of work:
using var activity = MyActivitySource.StartActivity("ReserveInventory");
activity?.SetTag("order.id", order.Id);
await _inventory.ReserveAsync(order);     // its child spans nest under this one
// The trace id flows across services automatically via the W3C 'traceparent' header.

Key takeaways

🎯 Remember these three

  • A trace follows one request across services; it’s a tree of spans (each span = one timed operation).
  • Context propagation via the trace ID (W3C traceparent) ties all the hops to one trace.
  • OpenTelemetry is the standard; a backend (Jaeger/Zipkin) renders the waterfall to pinpoint the slow service. Tracing + logs + metrics = observability.

Interview tip

🗣️ When this comes up

“How do you debug a slow request across microservices?” — distributed tracing. Define trace vs span, explain context propagation (the trace ID in traceparent), and name OpenTelemetry + Jaeger. Tie it to the three pillars of observability (logs, metrics, traces).
