Saga Pattern

A business transaction spanning several services can’t be one ACID transaction. A saga chains local transactions and, when a step fails, runs compensating actions to undo the earlier ones.

#saga #distributed-transactions #compensation #consistency

The idea in plain English

In a monolith, “place order” is one database transaction — all or nothing. In microservices the order, payment, and inventory live in separate services with separate databases, so there’s no single transaction to wrap them. Two-phase commit (2PC) could span them but it’s slow, locks resources, and couples services tightly — so it’s avoided.

A saga solves it differently. Each service does its own local transaction and signals the next step. If a later step fails, the saga doesn’t roll back (you can’t roll back a committed local transaction) — instead it runs compensating transactions that semantically undo the completed steps. “Reserve stock” is compensated by “release stock”; “charge card” by “refund.” You get eventual consistency across services instead of strict atomicity.

Two coordination styles:

  • Choreography — each service listens for events and emits the next; no central controller. Simple for short sagas, but the flow is implicit and hard to follow as it grows.
  • Orchestration — a central orchestrator (often a state machine) tells each service what to do and triggers compensations on failure. Clearer and easier to manage for complex sagas, at the cost of a coordinator.

Real-world example

🏢 How this works in production

An e-commerce “place order” saga runs: create order → reserve inventory → charge payment → schedule shipping. If the payment is declined at step 3, the saga fires compensations in reverse: release the reserved inventory and mark the order cancelled — so the customer isn’t charged and stock isn’t stuck. Travel bookings (flight + hotel + car) are the canonical saga: book all three, and if the car rental fails, cancel the flight and hotel. This is how distributed systems stay consistent without a global lock.

Steps forward, compensations back

In C# (orchestrated saga sketch)

C# 
// Orchestrator drives the steps and compensates on failure.
async Task PlaceOrderSaga(Order o)
{
    var done = new Stack<Func<Task>>();          // compensations to run if a later step fails
    try
    {
        await _orders.CreateAsync(o);    done.Push(() => _orders.CancelAsync(o.Id));
        await _inventory.ReserveAsync(o); done.Push(() => _inventory.ReleaseAsync(o.Id));
        await _payments.ChargeAsync(o);  done.Push(() => _payments.RefundAsync(o.Id));
        await _shipping.ScheduleAsync(o);
    }
    catch
    {
        while (done.Count > 0) await done.Pop()(); // run compensations in reverse order
        throw;
    }
}

Key takeaways

🎯 Remember these three

  • A saga replaces a cross-service ACID transaction with a sequence of local transactions.
  • On failure, run compensating transactions to semantically undo completed steps (eventual consistency, not rollback).
  • Choreography (events, decentralized) vs orchestration (a central coordinator/state machine).

Interview tip

🗣️ When this comes up

“How do you handle a transaction across services?” Say you avoid 2PC and use a saga: local transactions + compensating actions, giving eventual consistency. Compare choreography vs orchestration, and note compensations must be idempotent and that some steps (sending an email) can’t truly be undone — design for it.
