Service Discovery

In a world of autoscaling containers, service instances appear and vanish with ever-changing IPs. Service discovery is how one service finds a healthy instance of another.

#service-discovery #service-registry #microservices #kubernetes

The idea in plain English

Hardcoding http://10.0.3.14:8080 works until that instance is rescheduled, scaled, or replaced and its IP changes — which, with containers and autoscaling, is constantly. Services need a dynamic phone book. The core piece is a service registry (Consul, Eureka, etcd): each instance registers itself (name + address) on startup and is deregistered when it stops or fails a health check. Consumers look up the registry to get a current, healthy address.

Two models for the lookup:

  • Client-side discovery — the calling service queries the registry, gets the list of instances, and picks one itself (often load-balancing client-side). More control, but every client needs discovery logic.
  • Server-side discovery — the client calls a stable endpoint (a load balancer / gateway) that queries the registry and forwards. Clients stay dumb; the platform handles it.

On Kubernetes this is mostly invisible: a Service gives a stable virtual name and IP, and the platform’s DNS resolves orders-service to a healthy pod — built-in, DNS-based, server-side discovery. Health checks are what keep the registry honest by removing dead instances automatically.

Real-world example

🏢 How this works in production

On Kubernetes, a service just calls http://payments — the cluster DNS resolves that name to one of the currently-healthy payment pods, even as pods are killed, rescheduled, and autoscaled minute to minute. No service ever learns a real pod IP. Before Kubernetes made this built-in, teams ran Consul or Netflix’s Eureka as an explicit registry that instances registered with on boot and that health checks pruned. Either way, the point is the same: addresses are dynamic, so you look them up by name, never by IP.

Register, then look up

In practice

C# — call by service name, not IP 
// On Kubernetes the service NAME is the stable address; DNS + the platform do discovery.
builder.Services.AddHttpClient("payments", c =>
    c.BaseAddress = new Uri("http://payments"));   // resolves to a healthy pod, not a fixed IP

// The calling code never knows or cares which instance answered:
var http = factory.CreateClient("payments");
await http.PostAsJsonAsync("/charge", request);
// (Outside K8s, a Consul/Eureka client would resolve "payments" → an address list instead.)

Key takeaways

🎯 Remember these three

  • Instances have dynamic IPs, so services find each other by name via a registry, not hardcoded addresses.
  • Client-side discovery (caller queries the registry) vs server-side (a stable LB/gateway does it).
  • Health checks register/deregister instances; Kubernetes provides DNS-based discovery out of the box.

Interview tip

🗣️ When this comes up

State the problem first: dynamic IPs make hardcoding impossible. Then describe a registry with health-based register/deregister, and contrast client-side vs server-side discovery. Knowing that Kubernetes Services + DNS give this for free shows you’ve run it.
