Feature Flags

A feature flag is a runtime switch around new code. It separates deploy (the code is live) from release (users can see it) — the key to safe, gradual delivery.

#devops #feature-flags #release #experimentation

The idea in plain English

A feature flag (toggle) wraps a feature in a conditional checked at runtime: if the flag is on, the new path runs; if off, the old one. The code is deployed either way — you control exposure with configuration, not a deployment.

This decouples deploy from release, which unlocks a lot. You can ship dark: merge incomplete work to main behind an off flag (enabling trunk-based development without long-lived branches). You can do a gradual rollout: turn the feature on for 1%, then 10%, then everyone — a canary at the application layer.

It's also your kill switch: if a new feature misbehaves in production, flip the flag off instantly — no redeploy, no rollback, seconds not minutes. That's often the fastest mitigation available during an incident.

Flags power experimentation too (A/B tests, targeting specific users or regions). The cost is discipline: flags are debt — remove them once a feature is fully rolled out, or you accumulate a tangle of stale conditionals and untested code paths.

Real-world example

🏢 How this works in production

Companies like Facebook and Netflix release continuously behind flags: code ships to production constantly but features are 'dark' until enabled for a cohort, then ramped while metrics are watched. Platforms like LaunchDarkly exist entirely for this. The kill-switch value is real — when a launch goes wrong, flipping a flag off is the incident mitigation that beats waiting for a rollback deploy to roll through the fleet.

Deploy ≠ release

In C#

C# 
// The feature is deployed; the flag decides who sees it.
if (await _flags.IsEnabledAsync("new-checkout", user))
    return NewCheckout(cart);     // ramp 1% → 100%, or flip OFF instantly
else
    return LegacyCheckout(cart);

// Enables: trunk-based dev (merge dark), gradual rollout, A/B tests,
// and an incident kill switch — no redeploy. Then DELETE the flag when done.

Key takeaways

🎯 Remember these three

  • A feature flag decouples deploy from release — ship code dark, expose it by config.
  • Enables gradual rollout (app-layer canary), trunk-based dev, A/B tests, and an instant kill switch.
  • Flags are debt — remove them after full rollout to avoid stale, untested branches.
