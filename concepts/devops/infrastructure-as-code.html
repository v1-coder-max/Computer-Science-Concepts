Infrastructure as Code

Stop clicking in cloud consoles. Infrastructure as Code describes your servers, networks, and databases in version-controlled files you apply repeatably.

#devops #iac #terraform #automation

The idea in plain English

Infrastructure as Code (IaC) means defining your infrastructure — VMs, load balancers, databases, DNS, permissions — in text files that you commit to Git and apply with a tool. The infrastructure becomes reproducible, reviewable in pull requests, and diff-able over time, just like application code.

It's usually declarative: you describe the desired state ('three web servers, this database, that firewall rule') and the tool figures out the changes to get there. That makes it idempotent — applying the same config twice yields the same result, no duplicate resources.

The big problems it solves are snowflakes and drift. A hand-clicked server nobody can reproduce is a snowflake; when reality diverges from what you think is deployed, that's drift. IaC kills both: the code is the source of truth, and you can detect and correct drift by re-applying.

It also enables disposable, identical environments — spin up a staging that exactly mirrors prod, then tear it down — and disaster recovery (rebuild from code). Tools: Terraform/OpenTofu and Pulumi (cloud-agnostic), plus cloud-native ARM/Bicep and CloudFormation.

Real-world example

🏢 How this works in production

Cloud teams define their entire stack in Terraform: every VPC, subnet, database, and IAM role is code in a repo, changed only through reviewed pull requests and applied by a pipeline. New environments are a terraform apply away and are identical by construction. The anti-pattern it replaces — a production server someone configured by hand two years ago that nobody dares touch — is exactly the snowflake IaC exists to prevent.

Declare desired state, apply repeatably

In C#

HCL 
# Terraform — declare the desired state; the tool reconciles reality to it.
resource "aws_instance" "web" {
  count         = 3                    # exactly three, idempotently
  ami           = "ami-0abc123"
  instance_type = "t3.small"
  tags = { Name = "web", Env = "prod" }
}

# Workflow: edit → 'terraform plan' (preview the diff) → PR review →
# 'terraform apply'. Running apply again with no changes does nothing.

Key takeaways

🎯 Remember these three

  • IaC = infrastructure as version-controlled, declarative code — reproducible, reviewable, diffable.
  • Declarative + idempotent: describe desired state, apply repeatably with no duplicates.
  • Eliminates snowflakes and drift; enables identical environments and rebuild-from-code recovery.

Interview tip

🗣️ When this comes up

Define IaC as declarative, version-controlled infrastructure and name what it kills: snowflakes and configuration drift. Mentioning idempotency and 'the code is the source of truth, re-apply to fix drift' shows you've operated it, not just heard of it.
