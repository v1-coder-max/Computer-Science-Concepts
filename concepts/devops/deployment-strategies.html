Deployment Strategies

How you roll out a new version decides your downtime and your blast radius. The main strategies — rolling, blue-green, canary — trade cost for safety differently.

#devops #deployment #blue-green #canary

The idea in plain English

Rolling deployment replaces instances a few at a time: spin up some new-version instances, drain and remove old ones, repeat. No extra fleet needed and no downtime, but during the rollout both versions serve traffic, and rollback means rolling back across the fleet.

Blue-green runs two full environments — blue (current) and green (new). You deploy and test green privately, then flip all traffic to it at once. Rollback is instant (flip back to blue), but you pay for double the infrastructure during the switch.

Canary releases the new version to a small slice of traffic (say 1–5%), watches its error rate and latency, and gradually ramps to 100% only if it's healthy. It catches problems with minimal blast radius — most real users never see a bad version — at the cost of more orchestration and good monitoring.

Underpinning all of them: feature flags decouple deploy from release (ship code dark, turn it on later), and every strategy needs health checks + fast rollback. The trade-off axis is blast radius and rollback speed vs infrastructure and complexity.

Real-world example

🏢 How this works in production

Big platforms lean on canary: a new version goes to a tiny percentage of traffic (often one region or 1% of users), automated checks compare its error/latency against the baseline, and it auto-ramps or auto-rolls-back. Blue-green is common for databases-light services that want instant rollback, and rolling is the default for stateless Kubernetes deployments. The unifying goal is the same — never expose all users to an unverified version at once.

In C#

YAML 
# Kubernetes rolling update — replace pods gradually, no downtime.
spec:
  strategy:
    type: RollingUpdate
    rollingUpdate:
      maxSurge: 25%            # how many extra new pods during rollout
      maxUnavailable: 0        # never drop below desired capacity
  # readinessProbe gates traffic; a failing new version stalls the rollout.

# Canary (concept): route 5% to v2, watch metrics, then ramp 5 → 25 → 100%.
# Blue-green (concept): deploy green, test, switch the load balancer atomically.

Key takeaways

🎯 Remember these three

  • Rolling = gradual replace (cheap, no downtime); blue-green = two environments, instant flip + rollback (2× infra); canary = small slice first, ramp on health (smallest blast radius).
  • All require health checks and fast rollback; feature flags decouple deploying code from releasing it.
  • The trade-off is blast radius / rollback speed vs infrastructure cost and orchestration.
