Containers & Images

A container image bundles your app and everything it needs to run into one portable, immutable artifact — so it runs the same on your laptop, CI, and production.

#devops #docker #containers #images

The idea in plain English

A container is an isolated process: it shares the host's kernel but gets its own filesystem, network, and process view (via Linux namespaces and cgroups). That makes it far lighter than a VM — milliseconds to start, megabytes not gigabytes — because there's no guest OS, just your app and its dependencies.

An image is the immutable, read-only template a container runs from. Built from a Dockerfile, it packages the app, runtime, libraries, and config together, so 'works on my machine' becomes 'works as a unit everywhere' — the build-once-run-anywhere promise.

Images are made of layers — each Dockerfile instruction adds one — and layers are cached and shared. Change only your app code and just that layer rebuilds; ten images on the same base share that base once. This is why ordering your Dockerfile (dependencies before source) matters for build speed.

Images live in a registry (Docker Hub, ECR, GHCR) addressed by name + tag (or digest). CI builds and pushes an immutable, versioned image; deployment pulls that exact image — the same artifact that passed your tests is what runs in production.

Real-world example

🏢 How this works in production

Container images are the universal deployment unit now: CI builds myapp:1.4.2, pushes it to a registry, and Kubernetes pulls that identical image across hundreds of nodes. Because the image is immutable and self-contained, the thing tested in CI is byte-for-byte the thing in production — eliminating the environment drift that used to cause 'passed staging, broke in prod' incidents.

Layered image → running container

In C#

Dockerfile 
# Dockerfile — order matters: restore deps BEFORE copying source so the
# dependency layer is cached and only your code layer rebuilds on a change.
FROM mcr.microsoft.com/dotnet/sdk:8.0 AS build
WORKDIR /src
COPY *.csproj .
RUN dotnet restore                 # cached unless the .csproj changes
COPY . .
RUN dotnet publish -c Release -o /app

FROM mcr.microsoft.com/dotnet/aspnet:8.0   # small runtime-only final image
WORKDIR /app
COPY --from=build /app .
ENTRYPOINT ["dotnet", "MyApp.dll"]

Key takeaways

🎯 Remember these three

  • A container is an isolated process sharing the host kernel (light); an image is the immutable template it runs from.
  • Images are layered and cached — order the Dockerfile so dependencies precede source for fast rebuilds.
  • CI builds an immutable, versioned image in a registry; the exact tested artifact is what runs in prod.

Interview tip

🗣️ When this comes up

Contrast containers with VMs (shared kernel, no guest OS → lightweight) and stress immutability: the same image flows from CI to prod. Layer caching and Dockerfile ordering is the practical detail that shows hands-on experience.
