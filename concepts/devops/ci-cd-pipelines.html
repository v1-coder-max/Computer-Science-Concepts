CI/CD Pipelines

A pipeline turns 'it works on my machine' into 'it's tested and deployable on every commit.' CI integrates and verifies; CD gets it safely to production.

#devops #ci-cd #automation #delivery

The idea in plain English

Continuous Integration (CI) means everyone merges to a shared branch frequently, and every push triggers an automated build + test. The point is fast feedback: you find integration bugs in minutes, not at a painful 'integration week' months later.

Continuous Delivery (CD) extends that — every change that passes the pipeline is automatically prepared for release, so deploying is a one-click (or zero-click) routine, not an event. Continuous Deployment goes one step further: passing changes ship to production with no human gate at all.

A pipeline is a series of stages, each a gate: compile → unit tests → build an immutable artifact (often a container image) → integration tests → deploy to staging → deploy to production. A failure stops the line, so broken code never reaches users.

The cultural payoff is small batches: tiny, frequent, automatically-verified changes are far less risky than big quarterly releases. This is what DORA's research links to elite performance — high deploy frequency and low change-failure rate go together precisely because automation makes each change small and safe.

Real-world example

🏢 How this works in production

GitHub Actions, GitLab CI, Azure Pipelines, and Jenkins all express the same shape: on every push, run the build and tests, produce a versioned artifact, and promote it through environments with automated checks. High-performing teams deploy many times a day through these pipelines with confidence, because each deploy is a small, fully-tested increment — the opposite of the big-bang release weekend that the practice was invented to kill.

Build → test → deploy, on every commit

In C#

YAML 
# A minimal CI pipeline (GitHub Actions): build, test, publish artifact.
name: ci
on: [push]
jobs:
  build-test:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      - run: dotnet build -c Release
      - run: dotnet test -c Release           # gate: fails the pipeline on red
      - run: docker build -t app:${{ github.sha }} .   # immutable, tagged artifact
      # later stages: push image → deploy to staging → promote to prod

Key takeaways

🎯 Remember these three

  • CI = integrate often + automated build/test on every push (fast feedback); CD = always-deployable, automated path to release.
  • A pipeline is a chain of gates producing an immutable artifact promoted across environments.
  • Small, frequent batches are safer than big releases — high deploy frequency and low failure rate go together.

Interview tip

🗣️ When this comes up

Distinguish CI (integrate + test continuously) from continuous delivery vs deployment (auto-prepared vs auto-shipped). The mature point: small batches lower risk, which is why deploy frequency and stability rise together, not in tension.
