HTTP Status Codes

A three-digit code is the server’s one-word verdict on every request. Returning the right one is half of good API design.

#http #status-codes #rest #api

The idea in plain English

The first digit puts every response in one of five buckets:

  • 1xx Informational — rare (e.g. 100 Continue).
  • 2xx Success — 200 OK, 201 Created (with a Location header), 204 No Content (success, nothing to return).
  • 3xx Redirection — 301 permanent vs 302/307 temporary; 304 Not Modified (use your cache).
  • 4xx Client error — you messed up: 400 bad request, 401 unauthenticated, 403 forbidden, 404 not found, 409 conflict, 422 unprocessable, 429 too many requests.
  • 5xx Server error — the server messed up: 500 internal error, 502 bad gateway, 503 service unavailable.

The pairs people confuse: 401 vs 403 (“who are you?” vs “you can’t do that”), 301 vs 302 (permanent move — caches/SEO update — vs temporary), and 400 vs 422 (malformed request vs well-formed but semantically invalid). Picking correctly lets clients, caches, and crawlers react properly without parsing your error text.

Real-world example

🏢 How this works in production

Status codes drive real behavior. A Google crawler treats a 301 as “update your index to the new URL” but a 302 as “keep the old one” — get it wrong and your SEO suffers. A CDN serves cached content on 304 to save bandwidth. A rate limiter returns 429 with a Retry-After header so well-behaved clients back off, and a load balancer pulls a node out of rotation when it starts returning 503. The number is an API contract, not decoration.

The five classes

In C# (ASP.NET Core)

C# 
app.MapPost("/orders", async (CreateOrder dto, IOrders orders) =>
{
    if (!dto.IsValid)        return Results.BadRequest("Missing fields");      // 400
    if (await orders.Exists(dto.Id)) return Results.Conflict("Already exists"); // 409

    int id = await orders.AddAsync(dto);
    return Results.Created($"/orders/{id}", new { id });                        // 201 + Location
});

app.MapGet("/orders/{id:int}", async (int id, IOrders orders) =>
{
    var order = await orders.FindAsync(id);
    return order is null ? Results.NotFound()      // 404
                         : Results.Ok(order);      // 200
});

app.MapDelete("/orders/{id:int}", async (int id, IOrders orders) =>
{
    await orders.DeleteAsync(id);
    return Results.NoContent();                    // 204 — success, nothing to return
});

Key takeaways

🎯 Remember these three

  • The first digit = the class: 2xx success, 3xx redirect, 4xx client error, 5xx server error.
  • Know the confusable pairs: 401 (who are you?) vs 403 (not allowed); 301 (permanent) vs 302 (temporary).
  • Use 201 + Location for creates, 204 for empty successes, 409 for conflicts, 429 for rate limits.

Interview tip

🗣️ When this comes up

The classics: 401 vs 403 and 301 vs 302. Add value by knowing 201 needs a Location header, 204 for empty success, and 429 pairs with Retry-After. Returning 200 with an error in the body is a red flag.
