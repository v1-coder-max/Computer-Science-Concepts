JWT & Stateless Auth

A JSON Web Token is a signed, self-contained credential: the server can trust it without looking anything up — which is its superpower and its weakness.

#jwt #stateless-auth #claims #bearer-token

The idea in plain English

A JWT is three base64url-encoded parts joined by dots: header.payload.signature.

  • Header — the signing algorithm (e.g. HS256, RS256).
  • Payload — the claims: statements about the user — sub (subject/user id), exp (expiry), iss (issuer), aud (audience), plus roles/scopes.
  • Signature — the header + payload signed with a secret (HMAC) or private key (RSA).

The magic: to authenticate a request the server just re-computes the signature and checks exp. If it matches, the token is genuine and untampered — no database lookup, no shared session store. That makes JWTs ideal for stateless, horizontally-scaled APIs.

Two things people get wrong: a JWT is signed, not encrypted — anyone can base64-decode and read the payload, so never put secrets in it. And because validation is self-contained, you can’t easily revoke a JWT before it expires (no central record to delete) — the usual fixes are short-lived access tokens plus a refresh-token flow, or a denylist (which reintroduces state).

Real-world example

🏢 How this works in production

When a single-page app talks to an API gateway fronting dozens of microservices, JWTs shine: the user logs in once, gets a signed access token, and every service can validate it locally with the issuer’s public key — no chatty calls to an auth server per request. That’s how OAuth/OIDC providers (Auth0, Azure AD, Google) issue access tokens. The trade-off bites on logout: since the token is valid until exp, “log everyone out now” means short lifetimes + refresh tokens, or a revocation list.

Three parts, one signature check

In C#

C# — ASP.NET Core 
builder.Services.AddAuthentication(JwtBearerDefaults.AuthenticationScheme)
    .AddJwtBearer(o =>
    {
        o.TokenValidationParameters = new()
        {
            ValidateIssuer = true,   ValidIssuer   = "https://login.example.com",
            ValidateAudience = true, ValidAudience = "devlens-api",
            ValidateLifetime = true,                 // checks exp
            ValidateIssuerSigningKey = true,         // verifies the signature
            IssuerSigningKey = signingKey
        };
    });

// Each request: the middleware validates the signature + exp locally — no DB hit —
// and turns the claims into HttpContext.User. (See the Auth Pipeline page.)

⚠️ Signed ≠ encrypted; revocation is hard

The payload is readable by anyone — never put passwords or PII in it. And you can’t un-issue a valid JWT before exp; use short-lived access tokens + refresh tokens (or a denylist) when you need real logout/revocation. Store browser tokens in an HttpOnly cookie rather than localStorage to resist XSS.

Key takeaways

🎯 Remember these three

  • A JWT is header.payload.signature; validating = re-checking the signature and expiry, with no server lookup.
  • It’s signed, not encrypted — readable by anyone, so no secrets in the payload.
  • Stateless and scalable, but hard to revoke before expiry → short-lived tokens + refresh flow.

Interview tip

🗣️ When this comes up

“JWT vs session — and can you revoke a JWT?” JWT = stateless, validated by signature, scales without a shared store; the catch is revocation (valid until exp), solved with short lifetimes + refresh tokens. Always add: signed, not encrypted — a detail that trips up many candidates.
