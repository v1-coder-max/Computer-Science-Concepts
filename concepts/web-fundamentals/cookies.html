How Cookies Work

A cookie is a small key/value the browser stores and automatically sends back on every request to the same site — the web’s way of carrying a little state.

#cookies #httponly #samesite #state

The idea in plain English

Because HTTP is stateless, the server needs a way to recognize a returning browser. Cookies are that mechanism. The server sends a Set-Cookie header; the browser stores the value and, from then on, attaches it via the Cookie header on every request to that domain — no JavaScript required. Most commonly the cookie holds a session ID (see the Sessions page) or a token.

The attributes are where security lives:

  • HttpOnly — JavaScript can’t read it (document.cookie), blocking theft via XSS.
  • Secure — only sent over HTTPS, so it can’t leak on plain HTTP.
  • SameSite — Strict/Lax/None controls whether it rides cross-site requests, the main defense against CSRF.
  • Domain / Path — scope of where it’s sent.
  • Expires/Max-Age — absent = a session cookie (dies with the tab); present = a persistent cookie.

Cookies are tiny (~4KB) and sent on every matching request, so don’t stuff data in them — keep an ID and store the rest server-side. For client-only data, localStorage exists, but it’s readable by JS (so not for auth tokens you want protected from XSS).

Real-world example

🏢 How this works in production

When you log into Gmail, Google sets an HttpOnly; Secure; SameSite session cookie. Every subsequent request silently carries it, so you stay signed in without re-entering your password — and because it’s HttpOnly, a malicious script on the page can’t read and exfiltrate it. “Remember me” simply switches the cookie from session-scoped to a persistent one with a long Max-Age. The cookie consent banners you click are themselves managing which non-essential cookies get set.

Set once, sent every time

On the wire & in C#

HTTP 
HTTP/1.1 200 OK
Set-Cookie: sid=abc123; HttpOnly; Secure; SameSite=Lax; Max-Age=1800; Path=/

GET /account HTTP/1.1
Cookie: sid=abc123
C# — ASP.NET Core 
Response.Cookies.Append("sid", sessionId, new CookieOptions
{
    HttpOnly = true,                         // JS can't read it → mitigates XSS theft
    Secure   = true,                         // HTTPS only
    SameSite = SameSiteMode.Lax,             // CSRF hardening
    MaxAge   = TimeSpan.FromMinutes(30)      // omit for a session cookie
});

string? sid = Request.Cookies["sid"];        // read it back on later requests

Key takeaways

🎯 Remember these three

  • The server sets a cookie via Set-Cookie; the browser auto-sends it on every request to the domain.
  • Always use HttpOnly + Secure for auth cookies, and SameSite to blunt CSRF.
  • No expiry = session cookie; with expiry = persistent. Keep cookies tiny (store an ID, not data).

Interview tip

🗣️ When this comes up

“How do you secure a cookie?” — say HttpOnly (blocks XSS theft), Secure (HTTPS only), and SameSite (CSRF). Bonus: explain session vs persistent (presence of an expiry) and why an auth token in localStorage is riskier than an HttpOnly cookie.
