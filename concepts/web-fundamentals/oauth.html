OAuth 2.0 & OpenID Connect

OAuth lets an app act on your behalf without ever seeing your password. OpenID Connect adds a standard way to prove who you are on top of it.

#oauth #oidc #authorization #pkce

The idea in plain English

OAuth 2.0 is about delegated authorization. You want a photo-printing app to read your Google Photos — but you don’t want to hand it your Google password. OAuth solves this: you log in at Google, approve specific access, and the app receives a limited access token instead of your credentials. Four roles:

  • Resource owner — you.
  • Client — the app wanting access.
  • Authorization server — issues tokens (Google, Auth0, Azure AD).
  • Resource server — the API holding your data.

The recommended flow is the Authorization Code flow with PKCE: the client redirects you to the auth server, you authenticate and consent, it hands the client a short-lived code, and the client exchanges that code (plus a PKCE secret it generated) for an access token (+ a refresh token to get new ones). Scopes limit what the token can do (photos.read, not photos.delete).

OAuth is for authorization, not login per se. OpenID Connect (OIDC) is a thin identity layer on top: it adds an ID token (a JWT describing the user) so apps can do “Sign in with Google” properly. Rule of thumb: OAuth = access; OIDC = identity.

Real-world example

🏢 How this works in production

Every “Sign in with Google/GitHub/Apple” button is OIDC over OAuth. You click it, get bounced to the provider, approve, and your app receives an ID token (who you are) and an access token (what it may call). A calendar-scheduling app like Calendly uses the access token + scopes to read your Google Calendar availability — and only that — without ever knowing your password, and you can revoke its access from your Google account at any time.

Authorization Code flow (with PKCE)

In C# (ASP.NET Core)

C# 
builder.Services.AddAuthentication(o =>
{
    o.DefaultScheme = CookieAuthenticationDefaults.AuthenticationScheme;
    o.DefaultChallengeScheme = "oidc";
})
.AddCookie()
.AddOpenIdConnect("oidc", o =>
{
    o.Authority = "https://accounts.google.com";   // the OIDC provider
    o.ClientId = clientId;
    o.ClientSecret = clientSecret;
    o.ResponseType = "code";                        // Authorization Code flow
    o.UsePkce = true;                               // PKCE — required for public clients
    o.Scope.Add("openid");                          // OIDC identity
    o.Scope.Add("email");                           // request only the scopes you need
    o.GetClaimsFromUserInfoEndpoint = true;
});

Key takeaways

🎯 Remember these three

  • OAuth 2.0 = delegated authorization: an app gets a scoped access token, never your password.
  • Use the Authorization Code flow with PKCE; scopes limit access, refresh tokens renew it.
  • OpenID Connect adds an ID token for identity — OAuth = access, OIDC = login.

Interview tip

🗣️ When this comes up

“Difference between OAuth and OpenID Connect?” OAuth = authorization (delegated access via tokens/scopes); OIDC = authentication (adds an ID token). Name the Authorization Code + PKCE flow and explain why you never send the password to the client — that’s the whole point of delegation.
