WebSockets vs Polling vs SSE

Four ways to push fresh data to a browser, from crude to elegant. Picking the right one is mostly about direction and frequency.

#websockets #sse #polling #real-time

The idea in plain English

Plain HTTP is client-initiated, so getting server updates to a page takes one of these:

  • Short polling — the client asks “anything new?” every few seconds. Dead simple, but wasteful and laggy (most requests return nothing).
  • Long polling — the client asks and the server holds the request open until it has data, then the client immediately re-asks. Lower latency, still one response per message and lots of connection churn.
  • Server-Sent Events (SSE) — a single long-lived HTTP connection over which the server streams events to the client. One-way (server→client), auto-reconnects, simple — great for feeds/notifications.
  • WebSockets — a persistent, full-duplex TCP connection (upgraded from HTTP). Both sides send anytime with low overhead — the right tool when the client also talks back a lot.

Rule of thumb: one-way server updates → SSE; two-way, chatty, low-latency → WebSockets; polling only when you can’t use either or updates are rare. (In .NET, SignalR picks the best transport for you.)

Real-world example

🏢 How this works in production

A Slack-style chat uses WebSockets because both sides are constantly talking — you type, others’ messages arrive, typing indicators flow. A stock ticker or a live sports score that only pushes updates fits SSE perfectly (one-way, simple, auto-reconnecting). An old dashboard that refreshes a count every 30 seconds can get away with polling. Choosing WebSockets for a one-way feed is over-engineering; choosing polling for chat feels broken.

Four patterns

In code

JavaScript 
// SSE — one-way, dead simple, auto-reconnects:
const es = new EventSource("/api/prices/stream");
es.onmessage = (e) => render(JSON.parse(e.data));

// WebSocket — full-duplex:
const ws = new WebSocket("wss://chat.example.com/socket");
ws.onmessage = (e) => addMessage(e.data);   // server → client
ws.onopen = () => ws.send("hello");          // client → server, anytime

In .NET you rarely hand-roll these — SignalR negotiates WebSockets with SSE/long-polling fallback automatically.

Key takeaways

🎯 Remember these three

  • Polling repeats requests (simple, wasteful); SSE is a one-way server→client stream; WebSockets are full-duplex over one persistent connection.
  • Choose by direction: one-way push → SSE; chatty two-way low-latency → WebSockets.
  • SignalR abstracts the choice with automatic transport fallback.

Interview tip

🗣️ When this comes up

“WebSockets vs SSE vs polling — when would you use each?” Key discriminators: direction (SSE is one-way; WebSockets are full-duplex) and frequency. Don’t reach for WebSockets on a one-way feed — SSE is simpler and auto-reconnects. Mention SignalR’s transport negotiation.
