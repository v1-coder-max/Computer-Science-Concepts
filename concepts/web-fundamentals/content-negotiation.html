Content Negotiation

One URL, many representations. The client says what it can handle via Accept headers, and the server returns the best-matching format, language, or encoding.

#content-negotiation #accept #compression #vary

The idea in plain English

The same resource — say /products/42 — can be served as JSON or XML, in English or German, compressed or not. Content negotiation is how the client and server agree on which. The client expresses preferences with request headers, each supporting weighted priorities (q values):

  • Accept — media types it understands: application/json, text/html;q=0.9.
  • Accept-Language — en-US, de;q=0.8.
  • Accept-Encoding — compression it supports: br, gzip.

The server picks the best option it can produce and announces its choice with response headers — Content-Type, Content-Language, Content-Encoding. Crucially, it also sends a Vary header (e.g. Vary: Accept-Encoding) so caches store a separate copy per variant and don’t hand a gzipped response to a client that asked for brotli.

The most universally felt form is compression: the browser advertises Accept-Encoding: gzip, br and the server returns a much smaller brotli/gzip body — a big, free bandwidth win on text assets.

Real-world example

🏢 How this works in production

Every page you load uses content negotiation for compression: your browser sends Accept-Encoding: gzip, br and servers/CDNs return brotli-compressed HTML/CSS/JS, often cutting transfer size 70–80%. Localized sites use Accept-Language to pick your locale, and some APIs return JSON or XML from the same endpoint based on Accept. The Vary header is what keeps a CDN from serving the German page to an English visitor or a compressed body to a client that can’t decode it.

Client asks, server picks

In C# (ASP.NET Core)

C# 
// Honor Accept for JSON *and* XML from the same endpoint:
builder.Services.AddControllers(o => o.RespectBrowserAcceptHeader = true)
    .AddXmlSerializerFormatters();   // now Accept: application/xml returns XML

// Response compression negotiated via Accept-Encoding (gzip / brotli):
builder.Services.AddResponseCompression(o =>
{
    o.EnableForHttps = true;
    o.Providers.Add<BrotliCompressionProvider>();
    o.Providers.Add<GzipCompressionProvider>();
});

var app = builder.Build();
app.UseResponseCompression();   // adds Content-Encoding + Vary: Accept-Encoding

Key takeaways

🎯 Remember these three

  • Clients state preferences with Accept/Accept-Language/Accept-Encoding; the server returns the best representation and names it in Content-*.
  • Always send Vary so caches keep a separate copy per negotiated variant.
  • The everyday case is compression (gzip/brotli) — a large, near-free transfer-size win.

Interview tip

🗣️ When this comes up

Often a follow-up to caching: “what does the Vary header do?” It tells caches which request headers the response depends on, so they don’t serve the wrong variant (e.g. a gzipped body to a non-gzip client). Connect Accept-Encoding to compression — the most impactful negotiation in practice.
