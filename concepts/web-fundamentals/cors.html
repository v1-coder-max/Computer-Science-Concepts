CORS

Cross-Origin Resource Sharing — the browser rule that decides whether JavaScript on one origin is allowed to read a response from another.

#cors #same-origin-policy #preflight #browser-security

The idea in plain English

Browsers enforce the same-origin policy: by default, JavaScript on https://app.com can’t read responses from a different origin (scheme + host + port) like https://api.other.com. This stops a malicious page from quietly reading your logged-in data on another site. CORS is the controlled exception: the server opts in by returning Access-Control-Allow-Origin headers that say “these origins may read my responses.”

For anything beyond a “simple” request (custom headers, PUT/DELETE, JSON content type), the browser first sends a preflight — an OPTIONS request asking “am I allowed to do this?” The server answers with the permitted origins, methods, and headers; only if that checks out does the browser send the real request.

The single biggest misconception: CORS is not server-side security. It’s enforced by the browser, and it only governs whether JS can read the response. The server still received and processed the request; non-browser clients (curl, mobile apps, your backend) ignore CORS entirely. So CORS protects users’ browsers, not your API — auth and authorization still do that.

Real-world example

🏢 How this works in production

Every SPA-plus-API setup hits this: a React app on app.acme.com calling api.acme.com gets the dreaded “blocked by CORS policy” error until the API adds the front-end’s origin to its allow-list. Public APIs like Stripe or mapping services configure CORS so browser code from approved domains can call them directly. Teams repeatedly “fix” it by slapping * on allowed origins — fine for a truly public, cookie-less API, but wrong (and disallowed) when credentials are involved.

The preflight dance

In C# (ASP.NET Core)

C# 
builder.Services.AddCors(o => o.AddPolicy("spa", p => p
    .WithOrigins("https://app.acme.com")     // explicit allow-list (NOT "*")
    .WithMethods("GET", "POST", "PUT", "DELETE")
    .WithHeaders("Authorization", "Content-Type")
    .AllowCredentials()));                    // needed for cookies; '*' is illegal with this

var app = builder.Build();
app.UseCors("spa");   // must run before endpoints

// ⚠ AllowAnyOrigin() + AllowCredentials() is rejected by the spec — pin real origins.

Key takeaways

🎯 Remember these three

  • The same-origin policy blocks cross-origin reads; CORS is the server opting specific origins back in.
  • Non-simple requests trigger a preflight OPTIONS the server must approve.
  • CORS is browser-enforced and protects users — it is not server security, and * can’t be combined with credentials.

Interview tip

🗣️ When this comes up

The trap: “is CORS a security feature for your API?” No — it’s browser-enforced and only controls whether JS can read a cross-origin response; the server still handled the request, and curl ignores it entirely. Explain the same-origin policy, the preflight, and why * + credentials is forbidden.
