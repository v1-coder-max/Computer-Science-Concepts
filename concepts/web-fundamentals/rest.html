REST API Design

REST is a style, not a framework: model your API as resources (nouns) acted on by HTTP methods (verbs), and let the protocol do the talking.

#rest #api-design #resources #stateless

The idea in plain English

REST (Representational State Transfer) organizes an API around resources — things like orders, users, products — each identified by a URI. You don’t invent action URLs; you use the HTTP method as the verb on a noun:

  • GET /orders — list; GET /orders/42 — read one.
  • POST /orders — create; PUT /orders/42 — replace; PATCH — partial update; DELETE /orders/42 — remove.
  • Nest relationships: GET /orders/42/items.

Its core constraints:

  • Resource-based URIs are nouns — /orders/42/cancel as a verb-URL is the classic anti-pattern (use a state change or a sub-resource).
  • Stateless — every request carries everything needed (auth token, etc.); the server keeps no per-client session between calls. That’s what lets you scale horizontally.
  • Uniform interface — consistent methods, status codes, and representations (usually JSON).

The Richardson Maturity Model grades how “RESTful” an API is (resources → HTTP verbs → hypermedia/HATEOAS). Most real-world APIs stop at level 2 (proper resources + verbs + status codes), which is perfectly fine.

Real-world example

🏢 How this works in production

The GitHub and Stripe APIs are textbook REST: predictable resource URLs (/repos/{owner}/{repo}/issues, /v1/charges/{id}), correct verbs and status codes, JSON representations, and statelessness so any server in their fleet can handle any request. That consistency is why you can guess a Stripe endpoint and be right — the uniform interface means once you’ve learned one resource, you’ve learned them all.

Methods map onto resources

orders.MapGet("/",        (IOrders o) => o.ListAsync());                 // GET  /api/orders
orders.MapGet("/{id:int}",async (int id, IOrders o) =>
    await o.FindAsync(id) is { } x ? Results.Ok(x) : Results.NotFound()); // 200 / 404
orders.MapPost("/",       async (CreateOrder dto, IOrders o) =>
    Results.Created($"/api/orders/{await o.AddAsync(dto)}", null));       // 201 + Location
orders.MapPut("/{id:int}",async (int id, Order dto, IOrders o) =>
    { await o.ReplaceAsync(id, dto); return Results.NoContent(); });      // 204 (idempotent)
orders.MapDelete("/{id:int}", async (int id, IOrders o) =>
    { await o.DeleteAsync(id); return Results.NoContent(); });

// ❌ Avoid action URLs:  POST /api/orders/42/cancel-and-refund-now
// ✅ Prefer:             POST /api/orders/42/cancellation   (a sub-resource)

Key takeaways

🎯 Remember these three

  • Model resources as nouns with URIs; use HTTP methods as the verbs — no action URLs.
  • Be stateless and use correct status codes — that’s “level 2” REST and what most great APIs ship.
  • Uniform, predictable interfaces make an API learnable; HATEOAS is optional and rarely used.

Interview tip

🗣️ When this comes up

“What makes an API RESTful?” — resources as nouns, HTTP verbs, statelessness, correct status codes, a uniform interface. Flag the verb-in-URL anti-pattern and statelessness (no server-side session between calls). Contrast REST (resource-oriented, public) with gRPC (RPC, internal) for bonus depth.
