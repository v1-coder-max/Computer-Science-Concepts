HTTPS & TLS

HTTPS is just HTTP inside a TLS tunnel. TLS gives you three things: encryption, integrity, and proof you’re talking to the real server.

#https #tls #certificates #encryption

The idea in plain English

On plain HTTP, anyone between you and the server — a router, a coffee-shop Wi-Fi, an ISP — can read and even alter the traffic. TLS (the successor to SSL) wraps the connection to provide:

  • Confidentiality — the data is encrypted, so eavesdroppers see gibberish.
  • Integrity — tampering is detected.
  • Authentication — the server proves its identity with a certificate signed by a trusted Certificate Authority (CA).

The clever part is the handshake. Asymmetric crypto (a public/private key pair) is secure but slow; symmetric crypto is fast but needs a shared secret. So TLS uses asymmetric keys just long enough to safely agree on a one-time symmetric session key, then encrypts the actual traffic with that fast symmetric key. The server’s certificate — vouched for by a CA your browser already trusts — is what stops an attacker from impersonating it (man-in-the-middle). TLS 1.3 streamlines this to a single round trip.

Real-world example

🏢 How this works in production

The padlock on your bank’s site means a valid certificate chained to a trusted CA — which is why typing your password on airport Wi-Fi is safe from the person two seats over. Let’s Encrypt made certificates free and automated, so HTTPS is now the default everywhere; modern browsers actively warn on plain HTTP. At scale, TLS is often terminated at a load balancer or CDN edge (Cloudflare) that does the handshake, then forwards to internal services — concentrating certificate management in one place.

The TLS handshake

In C# (ASP.NET Core)

C# 
var app = builder.Build();

app.UseHttpsRedirection();   // bounce any plain HTTP to HTTPS

// HSTS: tells browsers "only ever use HTTPS for this site" (after first visit).
if (!app.Environment.IsDevelopment())
    app.UseHsts();           // Strict-Transport-Security header

// In production TLS is usually terminated at the edge (load balancer / CDN);
// Kestrel can also bind a certificate directly for end-to-end TLS.

Key takeaways

🎯 Remember these three

  • TLS gives confidentiality, integrity, and server authentication; HTTPS = HTTP over TLS.
  • The handshake uses slow asymmetric keys to agree a fast symmetric session key, then encrypts traffic with it.
  • A CA-signed certificate proves the server’s identity — that’s what defeats man-in-the-middle attacks.

Interview tip

🗣️ When this comes up

“Walk me through what happens when you hit an HTTPS site.” After DNS, describe the TLS handshake: asymmetric keys to exchange a symmetric session key, certificate validated against a trusted CA. State plainly what the cert proves (identity) and what TLS prevents (eavesdropping + MITM).
