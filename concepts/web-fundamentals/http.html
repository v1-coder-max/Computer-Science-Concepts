HTTP Fundamentals

The request/response protocol the entire web runs on — stateless, text-shaped, and built around methods, headers, and status codes.

#http #methods #idempotency #http2

The idea in plain English

HTTP is a request/response conversation. A client sends a request — a method (verb), a path, headers, and optionally a body — and the server sends back a status code, headers, and a body. That’s it. Loading one web page fires dozens of these for the HTML, CSS, images, and API calls.

HTTP is stateless: each request is independent and the server doesn’t inherently remember the last one (which is why we add cookies/sessions/tokens). The methods carry meaning you’re expected to honor:

  • GET — read; safe (no side effects) and idempotent.
  • POST — create/act; not idempotent (calling twice may create two things).
  • PUT — replace; idempotent (same result if repeated). PATCH — partial update. DELETE — remove; idempotent.

The wire format evolved without changing that model: HTTP/1.1 is text with one request at a time per connection (head-of-line blocking); HTTP/2 is binary and multiplexes many streams over one connection with header compression; HTTP/3 runs over QUIC (UDP) to dodge TCP-level blocking.

Real-world example

🏢 How this works in production

When you open a media-heavy site like YouTube, the browser needs to fetch hundreds of resources. Over HTTP/1.1 they’d queue behind a handful of connections; over HTTP/2 they stream in parallel over a single connection, and HTTP/3 keeps things flowing even on a flaky mobile network where a dropped packet would stall TCP. The idempotency rules matter too: a payment gateway makes “charge” a POST (don’t retry blindly) but “get order status” a GET (retry freely).

// GET is safe + idempotent → fine to retry.

// POST is NOT idempotent → use an idempotency key if you might retry.

Key takeaways

🎯 Remember these three

  • HTTP is a stateless request/response protocol: method + path + headers + body ↔ status + headers + body.
  • Honor method semantics: GET is safe/idempotent, PUT/DELETE idempotent, POST is not.
  • HTTP/2 multiplexes streams over one connection; HTTP/3 runs on QUIC/UDP — same model, faster transport.

Interview tip

🗣️ When this comes up

Expect “which HTTP methods are idempotent?” and “HTTP/2 vs HTTP/1.1?” Nail safe vs idempotent (GET safe+idempotent, PUT/DELETE idempotent, POST neither) and explain HTTP/2 multiplexing solving head-of-line blocking. Mentioning HTTP/3 over QUIC marks you as current.
