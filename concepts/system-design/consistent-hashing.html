Consistent Hashing

A way to spread keys across servers so that adding or removing a server moves only a few keys, not almost all of them. The trick behind scalable caches and distributed databases.

#consistent-hashing #distribution #sharding #virtual-nodes

The idea in plain English

Say you spread cache keys across N servers with server = hash(key) % N. It works — until N changes. Add or remove one server and the modulus shifts, so almost every key now maps to a different server. For a cache that means a near-total miss storm; for a database, a massive data reshuffle. That fragility is the problem.

Consistent hashing fixes it with a ring. Imagine the hash space (0 to 2³²−1) bent into a circle. You place each server at a position on the ring (by hashing its id), and each key at its own position. A key belongs to the first server clockwise from it. Now when you add a server, it only steals the keys between it and its previous neighbor — about 1/N of the keys move, not all of them. Remove a server and its keys simply roll over to the next one clockwise. Everything else is untouched.

One wrinkle: with few servers, the ring can be lumpy (uneven load). The fix is virtual nodes — place each physical server at many points on the ring — which smooths the distribution and makes rebalancing more even.

Real-world example

🏢 How this works in production

Consistent hashing is everywhere data is partitioned: Amazon DynamoDB and Cassandra use it to assign rows to nodes, distributed Memcached clients use it so adding a cache node doesn’t blow away the whole cache, and many load balancers and CDNs use it to pin clients to backends. The payoff is operational: you can scale a cluster up or down — or lose a node — and only a small slice of keys relocate, so the system stays warm and available instead of stampeding.

Keys map to the next node clockwise

In code (conceptual)

C# — sketch 
// A sorted ring: hash position → node. Virtual nodes = many entries per server.
private readonly SortedDictionary<uint, string> _ring = new();

public void AddNode(string node, int vnodes = 150)
{
    for (int i = 0; i < vnodes; i++)
        _ring[Hash($"{node}#{i}")] = node;       // spread each server across the ring
}

public string NodeFor(string key)
{
    uint h = Hash(key);
    // first ring position clockwise >= h, wrapping around to the start.
    foreach (var pos in _ring.Keys) if (pos >= h) return _ring[pos];
    return _ring.Values.First();                  // wrapped past the end → first node
}
// Naive alternative — DON'T: nodes[ Hash(key) % nodes.Count ] remaps everything when
// the count changes.

Key takeaways

🎯 Remember these three

  • hash(key) % N remaps almost all keys when N changes — terrible for caches and sharding.
  • Consistent hashing puts keys and nodes on a ring; a key goes to the next node clockwise, so adding/removing a node moves only ~1/N of keys.
  • Virtual nodes spread each server across the ring for even load and smoother rebalancing.

Interview tip

🗣️ When this comes up

Lead with why % N is bad (mass remapping on resize), then describe the ring and the “next node clockwise” rule, and finish with virtual nodes for even distribution. Naming Cassandra/DynamoDB/distributed-Memcached as users shows you know where it’s applied.
