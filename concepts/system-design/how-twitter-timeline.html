How Twitter’s Timeline Works

Showing you tweets from everyone you follow sounds trivial — until it’s 500 million tweets a day and some users have 100 million followers. This is the fan-out problem.

#fan-out #timeline #feed #case-study

The idea in plain English

Your home timeline is a merge of recent tweets from everyone you follow, newest first. The naive approach — at read time, query each followee’s tweets and merge them — is too slow when you follow thousands of people and refresh constantly. The core design decision is when to do that merging work:

  • Fan-out-on-write (push) — when you tweet, the system immediately pushes that tweet ID into the precomputed timeline of every follower (stored as a list in Redis). Reads are then trivial (just read your list). Fast reads, expensive writes — and a disaster when a celebrity with 100M followers tweets (100M list writes).
  • Fan-out-on-read (pull) — store each user’s tweets; build the timeline at read time by pulling from everyone you follow and merging. Cheap writes, expensive reads.

Neither wins alone, so real systems go hybrid: push for normal users (fast timelines), but pull for the handful of mega-accounts — at read time you merge the precomputed list with the latest tweets of the celebrities you follow. Timelines are cached aggressively (Redis), and the actual tweet content is hydrated separately from the IDs in the list.

Real-world example

🏢 How this works in production

Twitter popularized exactly this hybrid: most users’ home timelines are materialized on write into Redis lists so opening the app is a single fast read, while tweets from celebrities are merged in on read to avoid the “write to 100 million lists” explosion. The tweet objects live in a separate store; the timeline holds only IDs, hydrated at read time. This write-vs-read trade-off is the heartbeat of every social feed.

The shape of it

C# — hybrid timeline (conceptual) 
// WRITE: push the tweet id to followers' Redis lists — but skip mega-accounts.
async Task OnTweet(long userId, long tweetId)
{
    if (await Followers(userId) < CelebrityThreshold)
        foreach (var f in await GetFollowers(userId))
            await redis.ListLeftPushAsync($"timeline:{f}", tweetId);   // fan-out on write
    // celebrities: do nothing on write; merged in at read time
}

// READ: precomputed list  +  live pull of the celebrities this user follows.
async Task<List<long>> HomeTimeline(long userId)
{
    var ids = await redis.ListRangeAsync($"timeline:{userId}", 0, 200);  // O(1)-ish read
    var celeb = await LatestFromCelebrities(userId);                      // fan-out on read
    return Merge(ids, celeb);   // then hydrate ids → full tweet objects
}

Key takeaways

🎯 Remember these three

  • A home timeline is a fan-out problem: precompute on write (fast reads) or assemble on read (cheap writes).
  • Pure push breaks for celebrities (millions of list writes per tweet) → use a hybrid: push for normal users, pull for mega-accounts.
  • Cache timelines in Redis as lists of IDs; hydrate the tweet content separately.

🗣️ When this comes up

“Design a Twitter timeline” is a classic. Lead with the fan-out write vs read trade-off, raise the celebrity problem as why pure push fails, and land on a hybrid with Redis-cached ID lists hydrated at read time. That arc is the whole answer.
