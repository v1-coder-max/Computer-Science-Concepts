Rate Limiting Algorithms

Cap how fast a client can hit your API — to stop abuse, prevent overload, and keep things fair. The algorithm you choose decides how bursts are handled.

#rate-limiting #token-bucket #throttling #429

The idea in plain English

Rate limiting enforces a quota like “100 requests per minute per user.” It protects you from abusive clients, runaway loops, and accidental DDoS, and ensures one heavy user can’t starve everyone else. When a client exceeds the limit you return 429 Too Many Requests with a Retry-After header. The four classic algorithms:

  • Fixed window — count requests per clock window (e.g. per minute). Simple, but allows a double burst at the boundary (full quota at 0:59 and again at 1:00).
  • Sliding window — track a rolling time window so the rate is smooth across boundaries. More accurate, a bit more state.
  • Token bucket — a bucket holds up to N tokens, refilled at a steady rate; each request spends one. Allows short bursts (spend saved tokens) while bounding the average rate. The most popular.
  • Leaky bucket — requests enter a queue that drains at a fixed rate, smoothing output to a constant pace (overflow is dropped).

Across multiple servers you need distributed rate limiting — a shared counter in Redis (atomic INCR + TTL, or a token-bucket Lua script) so the limit is global, not per-instance.

Real-world example

🏢 How this works in production

GitHub and Stripe rate-limit their public APIs and return 429 with headers telling you your remaining quota and when it resets — so well-behaved clients back off automatically. Login endpoints throttle attempts to blunt credential-stuffing, and an API gateway (or Cloudflare) rate-limits per IP/key to absorb abuse before it reaches your app. Token bucket is the common choice because it tolerates legitimate bursts (a page firing several calls at once) while still capping the sustained rate.

Token bucket

In C# (ASP.NET Core rate limiter)

C# 
builder.Services.AddRateLimiter(o =>
{
    o.RejectionStatusCode = StatusCodes.Status429TooManyRequests;   // 429
    o.AddTokenBucketLimiter("per-user", b =>
    {
        b.TokenLimit = 100;                       // bucket capacity (burst)
        b.TokensPerPeriod = 100;                  // refill amount...
        b.ReplenishmentPeriod = TimeSpan.FromMinutes(1);  // ...per minute
        b.QueueLimit = 0;                         // no queueing: reject immediately
    });
});

var app = builder.Build();
app.UseRateLimiter();
app.MapGet("/api/data", () => "ok").RequireRateLimiting("per-user");
// For multiple servers, back the limiter with Redis so the count is global.

Key takeaways

🎯 Remember these three

  • Rate limiting prevents abuse/overload and ensures fairness; over-limit → 429 + Retry-After.
  • Fixed window (simple, boundary bursts), sliding window (smooth), token bucket (allows bursts — most common), leaky bucket (smooths output).
  • For multiple servers, use a shared store (Redis) so the limit is global, not per-instance.

Interview tip

🗣️ When this comes up

Be ready to compare the four algorithms and pick token bucket as a sensible default (handles bursts, bounds average). Mention the fixed-window boundary-burst flaw, the 429 + Retry-After contract, and Redis for distributed limits — that’s the complete answer.
