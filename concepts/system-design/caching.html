Caching at Scale

Caching isn’t one box — it’s a stack of layers from the browser to the database. Each one you can serve from is latency and load you don’t pay below it.

#caching #redis #memcached #hit-ratio

The idea in plain English

In a real system a request can be answered from several places, and the goal is to satisfy it as high up the stack as possible:

  • Browser cache — the response never leaves the device.
  • CDN / edge — served near the user, never touches your servers.
  • Application (in-process) cache — in-memory on the app server (e.g. IMemoryCache): fastest, but per-instance and lost on restart.
  • Distributed cache — a shared store like Redis or Memcached all app servers hit: survives restarts, shared across the fleet, the workhorse cache layer.
  • Database — the source of truth, and its own buffer cache.

The metric that matters is the cache hit ratio — the fraction of requests served from cache. A few percentage points there can be the difference between a database that’s bored and one that’s on fire. What to cache: data that’s read far more than it’s written and expensive to compute (rendered pages, query results, session data, computed feeds).

Redis vs Memcached: Memcached is a simple, multi-threaded string cache — lean and fast. Redis adds rich data structures, persistence, pub/sub, and clustering, so it doubles as a session store, rate limiter, and leaderboard. Most teams reach for Redis for the extra power. (For how you read/write through the cache — cache-aside, write-through, etc. — see Caching Strategies.)

Real-world example

🏢 How this works in production

Facebook famously runs one of the largest Memcached deployments on earth, absorbing the overwhelming majority of reads so its databases handle only a sliver of the load. A typical app stacks all the layers: a CDN for assets, an in-process cache for hot config, and Redis for shared data and sessions in front of SQL. The engineering work is mostly about raising the hit ratio and keeping caches from going stale — because at scale the database simply cannot serve every read.

The cache layers

In C#

C# 
// In-process cache: fastest, but per-server and lost on restart.
builder.Services.AddMemoryCache();

// Distributed cache (Redis): shared across all app servers, survives restarts.
builder.Services.AddStackExchangeRedisCache(o => o.Configuration = "redis:6379");

public class Catalog(IMemoryCache local, IDistributedCache shared)
{
    public async Task<Product?> GetAsync(int id)
    {
        // L1: process memory → L2: Redis → (else) the database.
        if (local.TryGetValue($"p:{id}", out Product? p)) return p;
        var bytes = await shared.GetAsync($"p:{id}");
        // ... deserialize from Redis, or load DB + populate both layers ...
        return p;
    }
}

Key takeaways

🎯 Remember these three

  • Caching is layered: browser → CDN → in-process → distributed (Redis/Memcached) → DB. Serve as high as you can.
  • The hit ratio is the headline metric; cache read-heavy, expensive-to-compute data.
  • Memcached = simple fast string cache; Redis = data structures + persistence + extras (usually the pick).

Interview tip

🗣️ When this comes up

In a design question, add caching layers explicitly and justify each. Talk about hit ratio, what’s safe to cache, and Redis vs Memcached. Pair this with a caching strategy (cache-aside + TTL) and invalidation — that combination is what interviewers are listening for.
