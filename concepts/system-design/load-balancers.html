Load Balancers & Reverse Proxies

One server can’t serve everyone — and shouldn’t be a single point of failure. A load balancer spreads traffic across many servers and quietly routes around the dead ones.

#load-balancer #reverse-proxy #scaling #high-availability

The idea in plain English

A load balancer sits in front of a pool of identical servers and distributes incoming requests among them. It buys you two things at once: scale (add servers to handle more traffic) and high availability (if one server dies, traffic just goes to the others). It runs continuous health checks and pulls unhealthy servers out of rotation automatically.

It works at one of two layers:

  • Layer 4 (transport) — routes by IP/port without looking inside the request. Very fast, protocol-agnostic.
  • Layer 7 (application) — understands HTTP, so it can route by URL/header/cookie, terminate TLS, and do content-based routing. More flexible, slightly more work.

How it picks a server — the algorithm:

  • Round-robin — next server each time (simple, even).
  • Least connections — the least-busy server (good for uneven request costs).
  • IP hash — same client → same server (a form of stickiness).
  • Weighted — bigger servers get more traffic.

A closely related concept is the reverse proxy (nginx, HAProxy): it fronts your servers and can also do TLS termination, caching, compression, and routing — load balancing is one of its jobs. (Contrast a forward proxy, which sits in front of clients.) If your app keeps server-side session state, you’ll need sticky sessions or, better, a shared session store so any server can handle any request.

Real-world example

🏢 How this works in production

Every large site runs behind load balancers — an AWS ALB, nginx, or HAProxy — spreading millions of requests across fleets of stateless app servers. During a deploy, new servers are added to the pool and pass health checks before receiving traffic (zero-downtime rollout); a crashing server is ejected within seconds so users never notice. This is also why apps push session state into Redis: with no sticky requirement, the balancer is free to send each request to any healthy server.

One door, many servers

In config (nginx reverse proxy + LB)

nginx 
upstream app {
    least_conn;                      # algorithm: send to the least-busy server
    server 10.0.0.11:8080;
    server 10.0.0.12:8080;
    server 10.0.0.13:8080 weight=2;  # bigger box gets more traffic
}

server {
    listen 443 ssl;                  # L7: terminate TLS here
    location / {
        proxy_pass http://app;       # reverse-proxy + load-balance to the pool
        proxy_set_header X-Forwarded-For $remote_addr;
    }
}

Key takeaways

🎯 Remember these three

  • A load balancer gives scale + high availability and ejects unhealthy servers via health checks.
  • L4 (fast, IP/port) vs L7 (HTTP-aware: route by URL/header, TLS termination); algorithms: round-robin, least-connections, IP hash, weighted.
  • Keep servers stateless (shared session store) so you don’t need sticky sessions.

Interview tip

🗣️ When this comes up

In any “design a scalable system” question, the load balancer is your first box. Know L4 vs L7, name a couple of algorithms, and raise health checks and statelessness (shared session store vs sticky sessions). Distinguishing reverse vs forward proxy is a nice extra.
