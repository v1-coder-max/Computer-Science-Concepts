How Sessions Work

HTTP forgets you the instant a request ends. A session is how a server remembers who you are across many stateless requests.

#sessions #cookies #authentication #state

The idea in plain English

HTTP is stateless: every request stands alone. The server has no built-in way to know that the request asking for /account came from the same person who logged in two seconds ago. Each request is a stranger knocking on the door.

A session fixes this. When you log in, the server creates a little pile of memory about you — your user id, your cart, your permissions — and tags it with a long, unguessable session ID. It keeps the pile server-side and hands you only the ID, wrapped in a cookie. Your browser sends that cookie back on every subsequent request, the server looks the ID up in its store, and says “ah — it’s Alice.”

Three things people constantly conflate:

  • Session — the data, living on the server (RAM, Redis, a DB).
  • Cookie — the envelope that carries the opaque session ID to and from the browser.
  • Token (JWT) — a stateless alternative where the data lives in the client’s hands, signed so it can’t be tampered with.

Real-world example

🏢 How this works in production

When you sign into Gmail, Google sets a session cookie. Every click — opening an email, searching, switching labels — silently ships that cookie back, so Google knows it’s still you without asking for your password again. Amazon goes further: it gives even anonymous visitors a session so the cart you fill before logging in survives until you check out. At their scale the session data can’t live in one server’s memory — any of thousands of servers must be able to serve your next request — so it’s kept in a shared store like Redis, keyed by the session ID.

How it works

The login hands you an ID; every later request replays it so the server can re-hydrate your state from the store.

In C# (ASP.NET Core)

ASP.NET Core ships session middleware. Back it with a distributed cache (Redis) and any server in your fleet can resolve the same session.

C# 
// Program.cs — distributed sessions backed by Redis
builder.Services.AddStackExchangeRedisCache(o =>
    o.Configuration = "localhost:6379");

builder.Services.AddSession(o =>
{
    o.Cookie.Name = ".DevLens.Session";
    o.Cookie.HttpOnly = true;                       // JS can't read it → mitigates XSS theft
    o.Cookie.SecurePolicy = CookieSecurePolicy.Always; // send over HTTPS only
    o.Cookie.SameSite = SameSiteMode.Lax;           // basic CSRF hardening
    o.IdleTimeout = TimeSpan.FromMinutes(30);       // sliding expiry
});

var app = builder.Build();
app.UseSession(); // must run before any endpoint that touches the session

app.MapPost("/cart/add", (HttpContext ctx, string sku) =>
{
    // Only the opaque session id travels in the cookie; data stays server-side.
    var count = ctx.Session.GetInt32("cartCount") ?? 0;
    ctx.Session.SetInt32("cartCount", count + 1);
    return Results.Ok(new { items = count + 1 });
});

⚠️ Don’t put secrets or large blobs in the cookie

The cookie should hold only the session ID. Anything sensitive (roles, balances) lives server-side where the user can’t edit it. Always set HttpOnly and Secure, and regenerate the session ID on login to defend against session fixation.

Key takeaways

🎯 Remember these three

  • HTTP is stateless; a session adds server-side memory keyed by an opaque ID carried in a cookie.
  • To scale horizontally, move that memory into a shared store (Redis) so any server can resolve any session.
  • Sessions are stateful on the server — JWTs trade that for client-held, signed state (no server lookup, harder to revoke).

Interview tip

🗣️ When this comes up

Expect “sessions vs JWT — which would you pick and why?” A strong answer names the real trade-off: sessions are trivially revocable but need a shared store and a lookup per request; JWTs are stateless and fast but hard to revoke before expiry. Mention sticky sessions vs a distributed store the moment you say “multiple servers.”
