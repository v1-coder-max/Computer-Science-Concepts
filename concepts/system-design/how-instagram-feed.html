How Instagram’s Feed Works

Instagram’s feed is a Twitter timeline plus two hard extras: it’s ranked, not chronological, and it’s mostly heavy media that has to load instantly worldwide.

#feed #ranking #cdn #case-study

The idea in plain English

Feed generation starts like the Twitter problem — gather recent posts from accounts you follow (a fan-out, often pull/hybrid). But Instagram adds a ranking stage. Rather than newest-first, the feed is a pipeline:

  1. Candidate generation — collect a pool of eligible posts (followed accounts + suggested/explore).
  2. Ranking — an ML model scores each candidate by predicted engagement (likes, comments, dwell time, recency, relationship) and sorts by score.
  3. Serving — return the top N, with the actual photos/videos delivered from a CDN.

The media is the dominant cost. Each post’s image/video is transcoded into multiple sizes and pushed to a global CDN so the feed loads from a nearby edge, not the origin. The post metadata (likes, captions) is small and lives in sharded databases; IDs are generated by a sharded, time-sortable ID scheme (Instagram’s well-known approach embeds a timestamp + shard id + sequence) so posts sort by time and route to the right shard without a central bottleneck. Everything hot is cached.

Real-world example

🏢 How this works in production

Instagram moved from a purely chronological feed to a ranked one precisely because, as the network grew, chronological buried the posts people most wanted to see. The ranking model is the product. Meanwhile the engineering muscle goes into media delivery — transcoding every upload into multiple resolutions and serving them from CDN edges so a feed full of 4K photos scrolls smoothly on a phone in any country. Their famous sharded ID generator lets billions of posts get unique, time-ordered IDs across many database shards.

The ranking pipeline

Sharded, time-sortable IDs

C# — Instagram-style 64-bit id (conceptual) 
// 41 bits time | 13 bits shard | 10 bits sequence → sortable by time, routable to a shard.
long NewId(int shardId, int seq)
{
    long ms = DateTimeOffset.UtcNow.ToUnixTimeMilliseconds() - Epoch;
    return (ms << 23) | ((long)(shardId & 0x1FFF) << 10) | (long)(seq & 0x3FF);
}
// No central ID server: each shard mints its own unique, time-ordered ids.

Key takeaways

🎯 Remember these three

  • A modern feed is a pipeline: candidate generation → ML ranking by predicted engagement → serve.
  • Media dominates cost — transcode to multiple sizes and deliver from a CDN; metadata is small and sharded.
  • Time-sortable sharded IDs let billions of posts stay ordered and routable without a central bottleneck.

Interview tip

🗣️ When this comes up

Build on the Twitter answer: add a ranking stage (candidates → ML score → serve) and emphasize media delivery via CDN (transcoded sizes) as the real scaling challenge. Mentioning a sharded, time-sortable ID scheme for billions of posts is a strong differentiator.
