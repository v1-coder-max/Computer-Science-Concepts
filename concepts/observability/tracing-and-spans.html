Distributed Tracing & Spans

When one user request fans out across ten services, a trace stitches the whole journey into one timeline of spans — so you can see exactly where the time (or the error) went.

#observability #tracing #spans #microservices

The idea in plain English

A trace represents a single request's end-to-end path through your system. It's a tree of spans, where each span is one unit of work (an HTTP call, a database query, a function) with a start time, duration, and metadata. The root span is the incoming request; children are the downstream calls it makes.

The magic is context propagation. A trace ID is generated at the entry point and passed along with every downstream call (via HTTP headers like W3C traceparent), plus a span ID linking parent to child. So even though the work spans many processes and machines, the backend can reassemble it into one coherent trace.

This is the only pillar that answers 'where in the distributed call did the time go?' Metrics tell you a service is slow overall; a trace of a slow request shows the exact span — maybe a single downstream API taking 2 seconds — that metrics and logs alone can't isolate.

OpenTelemetry is the vendor-neutral standard: instrument once (much is automatic for HTTP/DB), export to any backend (Jaeger, Tempo, Honeycomb). Sampling keeps the volume affordable — you don't store every trace, just a representative and error-biased slice.

Real-world example

🏢 How this works in production

In a microservices request that touches gateway → orders → payments → inventory, a trace shows each hop as a span on a waterfall, instantly revealing that, say, the payments call dominates the latency. Teams instrument with OpenTelemetry and view traces in Jaeger/Tempo; during a latency incident it's the fastest way to pin the slow service. The same trace ID also appears in the logs, so you jump from span to root-cause log in one click.

A trace is a tree of timed spans

In C#

C# 
// OpenTelemetry: instrument once, export anywhere. Much is automatic.
builder.Services.AddOpenTelemetry().WithTracing(t => t
    .AddAspNetCoreInstrumentation()      // root span per incoming request
    .AddHttpClientInstrumentation()      // child span per downstream call
    .AddEntityFrameworkCoreInstrumentation()
    .AddOtlpExporter());                 // → Jaeger / Tempo / vendor

// Trace context propagates automatically via the W3C 'traceparent' header,
// so spans from every service reassemble into one trace.

Key takeaways

🎯 Remember these three

  • A trace is one request's journey as a tree of timed spans; the root is the entry request, children are downstream calls.
  • Context propagation (a trace id passed in headers, e.g. W3C traceparent) reassembles work across services into one trace.
  • It's the only signal that localizes latency/failure in a distributed call; standardize on OpenTelemetry, use sampling for volume.
