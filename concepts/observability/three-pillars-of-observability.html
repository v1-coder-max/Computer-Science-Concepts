The Three Pillars: Logs, Metrics, Traces

Observability is being able to ask any question about your running system from its outputs. Three complementary signals — logs, metrics, traces — answer different questions.

#observability #logging #metrics #tracing

The idea in plain English

Observability is the property that you can understand a system's internal state from its external outputs, without shipping new code to investigate. It's broader than monitoring: monitoring watches known dashboards for known problems; observability lets you debug the unknown ones. The signals come in three pillars.

Metrics are cheap numeric aggregates over time — request rate, error rate, latency percentiles, CPU. They answer 'is something wrong, and how much?' and power dashboards and alerts. They're tiny to store but lack detail (no per-request context).

Logs are discrete, timestamped event records. They answer 'what exactly happened here?' with rich detail, but are voluminous and hard to aggregate. Structured (key-value) logs make them queryable.

Traces follow a single request across services as a tree of timed spans. They answer 'where did the time go / where did it fail?' in a distributed call — the piece metrics and logs alone can't give you. Used together, you go metrics → traces → logs: alert on a metric, find the slow span in a trace, read the log for the cause.

Real-world example

🏢 How this works in production

Mature teams run all three — Prometheus/Grafana for metrics, a log platform (ELK, Loki, Splunk) for logs, and distributed tracing (OpenTelemetry → Jaeger/Tempo) — and increasingly correlate them with shared IDs. The debugging flow during an incident is exactly the drill-down: a metric alert fires, a trace shows which service/span is slow, and its logs explain why. A system with only one pillar leaves you guessing about the others.

Three signals, three questions

In C#

C# 
// One request often emits all three signals — correlate them by trace id.
// Metric: a counter/histogram (aggregated).
Metrics.CreateCounter("orders_total").Add(1);

// Trace: a span around the work (OpenTelemetry).
using var span = tracer.StartActivity("CreateOrder");
span?.SetTag("order.id", id);

// Log: a structured event carrying the trace id for correlation.
logger.LogInformation("Order {OrderId} created in {Ms}ms", id, elapsed);

Key takeaways

🎯 Remember these three

  • Observability = understand internal state from outputs (debug unknowns), broader than monitoring (watch knowns).
  • Three pillars: metrics (is it wrong?), traces (where?), logs (what exactly?).
  • Use them together — drill metrics → traces → logs, correlated by a shared trace id.

Interview tip

🗣️ When this comes up

Name the three pillars and the distinct question each answers, then describe the drill-down workflow (metrics → traces → logs). Distinguishing observability (debug unknowns) from monitoring (watch knowns) is the framing that stands out.
