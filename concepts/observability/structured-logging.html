Structured Logging

Plain-text logs read fine to a human and are miserable to a machine. Structured logging records events as key-value data you can filter, aggregate, and correlate.

#observability #logging #structured #diagnostics

The idea in plain English

Traditional logging concatenates everything into a string: $"Order {id} failed for {user}". It's readable line-by-line but unqueryable at scale — you can't reliably ask 'all failures for user 42 in the last hour' across millions of lines without brittle regex.

Structured logging instead records each event as a set of named fields (an order id, a user id, a duration, a level) — typically emitted as JSON. Now your log platform can index those fields and you can query: filter by userId = 42, group by error type, chart durations. Logs become data, not prose.

The .NET idiom is the message template: logger.LogInformation("Order {OrderId} failed for {User}", id, user). The named holes both render a readable message and capture OrderId and User as structured properties. Pair it with a correlation/trace ID in scope so all logs for one request are tied together.

Round it out with sensible levels (Trace/Debug/Information/Warning/Error/Critical) so you can dial verbosity, and with hygiene: don't log secrets or PII, and don't log so much that the signal drowns. Good structured logs are the 'what exactly happened' pillar that pairs with metrics and traces.

Real-world example

🏢 How this works in production

Production .NET services use Serilog or the built-in ILogger with message templates, emitting JSON to Elasticsearch/Seq/Loki where engineers query by field during incidents. The difference shows under pressure: with structured logs you filter to the failing request's correlation ID and read its exact path in seconds; with string logs you're grepping and guessing. Cloud log platforms are built around structured fields for exactly this reason.

Strings vs queryable fields

In C#

C# 
// ✅ Message template: readable message + structured properties.
logger.LogInformation("Order {OrderId} failed for {User} after {Ms}ms",
    orderId, user, elapsed);
// → emits { OrderId: 17, User: "alice", Ms: 240, Level: "Information", ... }

// Correlate every log of one request with a scope (or the trace id):
using (logger.BeginScope(new Dictionary<string, object> { ["TraceId"] = traceId }))
{
    // all logs in here carry TraceId → query one request end-to-end
}

// ❌ Avoid: logger.LogInformation($"Order {orderId} failed");  // string only

Key takeaways

🎯 Remember these three

  • Log events as key-value fields (usually JSON), not concatenated strings — so logs are queryable, not just readable.
  • Use message templates ({OrderId}) to get a readable message and structured properties, plus a correlation/trace id.
  • Use appropriate levels; never log secrets/PII, and avoid drowning the signal.
