Value vs Reference Types

The distinction that explains half of C#'s surprising behaviors: value types are copied by value, reference types are copied by reference — and that changes how assignment, equality, and mutation behave.

#csharp #value-types #reference-types #memory

The idea in plain English

Value types (all structs — int, bool, DateTime, your own structs) hold their data directly. Assigning one or passing it to a method copies the whole value, so the copy is independent — change one, the other is unaffected. Reference types (all classes — string, arrays, your classes) hold a reference to data elsewhere; assigning copies the reference, so two variables point at the same object and a change through one is visible through the other.

Where they live: value types are typically allocated inline — on the stack as locals, or embedded inside their containing object — and disappear with no GC work. Reference types live on the managed heap and are cleaned up by the garbage collector. (This is a useful mental model, though the precise location depends on context, e.g. a struct field of a heap object lives on the heap.)

This drives behavior you must know. Default equality differs: value types compare by their contents, reference types by identity (same object) unless overridden. Mutation differs: passing a struct to a method that changes it won't affect the original (it got a copy) unless you pass by ref. And large structs are expensive to copy around — a reason to keep them small.

Practical guidance: use a class by default; use a struct for small, immutable, value-like data (a coordinate, a money amount) where copy semantics are desired and you want to avoid heap allocations. record class vs record struct lets you get value-based equality either way — choosing the type kind is choosing copy and identity semantics.

Real-world example

🏢 How this works in production

This distinction is behind countless real bugs and optimizations: a method that 'modifies' a struct parameter but the caller sees no change (it mutated a copy), or two list elements that unexpectedly change together (they're the same reference). On the performance side, high-throughput .NET code uses small structs and Span<T> to avoid heap allocations and GC pressure — the value-type semantics are exactly why struct can be a deliberate performance choice, not just a modeling one.

Copy the value vs copy the reference

In C#

C# 
// Value type: copied by value — independent.
struct PointS { public int X; }
var a = new PointS { X = 5 };
var b = a;                       // COPY of the value
b.X = 9;                         // a.X is still 5

// Reference type: copied by reference — shared.
class PointC { public int X; }
var c = new PointC { X = 5 };
var d = c;                       // copy of the REFERENCE
d.X = 9;                         // c.X is now 9 too — same object

// Equality differs: structs compare by contents, classes by identity (unless overridden).

Key takeaways

🎯 Remember these three

  • Value types (structs) copy the whole value (independent); reference types (classes) copy the reference (shared object).
  • Value types are usually inline/stack (no GC); reference types live on the heap (GC-managed).
  • It drives default equality (contents vs identity) and mutation semantics — use structs for small, immutable, value-like data.
