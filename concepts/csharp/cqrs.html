CQRS

Command Query Responsibility Segregation: use different models for writing and reading. One side changes state; the other side, optimized separately, serves data.

#cqrs #modern-pattern #read-model #mediatr

The idea in plain English

Most apps use one model for everything — the same entities and DbContext to write and read. That’s fine until reads and writes pull in opposite directions: writes want a normalized, validated, transaction- friendly model; reads want denormalized, query-shaped data, and there are usually far more reads. CQRS splits them:

  • Commands change state (PlaceOrder, CancelOrder). They’re imperative, validated, and typically return nothing (or just an ack).
  • Queries return data and have no side effects. They can use lightweight, denormalized read models shaped exactly for the screen.

At minimum CQRS is just separate command and query handlers over the same database (a common, modest version, usually via MediatR). Taken further, the read and write sides use different data stores — a normalized write store plus denormalized read projections kept in sync (often via events). That lets you scale and optimize reads independently, but introduces eventual consistency between the stores.

Important: CQRS is not a default. The simple shared-handler form is cheap and useful; the separate-stores form is powerful but complex (sync, consistency, more moving parts). Apply it where read/write needs genuinely diverge — not everywhere.

Real-world example

🏢 How this works in production

A high-traffic e-commerce site writes orders into a normalized, ACID store (commands), but serves product and order views from a denormalized, pre-joined read model (queries) — often cached or in a separate store — so the millions of reads never hammer the transactional database or join six tables. The read model is rebuilt from OrderPlaced/OrderShipped events. Banking and analytics-heavy systems do the same. In .NET this is typically MediatR commands and queries; when paired with event sourcing, the events are the write model and projections build the reads.

Two sides, one system

In C# (MediatR)

C# 
// COMMAND: changes state, returns an ack.
public record PlaceOrder(Guid CartId) : IRequest<Guid>;
public class PlaceOrderHandler(IOrderWriteStore store) : IRequestHandler<PlaceOrder, Guid>
{
    public Task<Guid> Handle(PlaceOrder c, CancellationToken ct) => store.CreateAsync(c.CartId);
}

// QUERY: no side effects, returns a read-shaped DTO from the read model.
public record GetOrderSummary(Guid Id) : IRequest<OrderSummaryDto>;
public class GetOrderSummaryHandler(IOrderReadStore reads)
    : IRequestHandler<GetOrderSummary, OrderSummaryDto>
{
    public Task<OrderSummaryDto> Handle(GetOrderSummary q, CancellationToken ct)
        => reads.GetSummaryAsync(q.Id);   // denormalized, no joins
}
// Same DB at first; split stores only when reads/writes truly diverge.

Key takeaways

🎯 Remember these three

  • CQRS separates the write model (commands, state-changing) from the read model (queries, side-effect-free).
  • The light version is just separate handlers on one DB; the heavy version uses separate read/write stores synced by events (eventual consistency).
  • It’s not a default — apply it where read and write needs genuinely diverge; it adds real complexity.

Interview tip

🗣️ When this comes up

Define CQRS as separate read/write models and resist over-selling it: say the simple handler split is cheap, but separate stores bring eventual consistency and complexity, so use it only when justified (read-heavy, very different read/write shapes). Mention MediatR and the event-sourcing pairing.
