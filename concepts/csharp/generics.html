Generics

Write an algorithm or container once, parameterized by a type, and reuse it with full type safety — no casting, no boxing. List<T> is the everyday proof.

#generics #type-safety #constraints #reuse

The idea in plain English

Before generics, a reusable container stored object — which meant casting on the way out and boxing value types, with no compile-time safety (you could put a string in a list of ints and only find out at runtime). Generics let you parameterize a type or method with a placeholder (T) that the caller fills in: List<int>, Dictionary<string, Order>. You get:

  • Type safety — the compiler enforces the element type; no invalid inserts, no casts.
  • Performance — value types are stored directly (no boxing/allocation).
  • Reuse — one definition serves every type.

Constraints (where T : …) let a generic require capabilities of T — a base class, an interface, new(), struct/class, or notnull — so you can actually use T meaningfully inside.

Under the hood, the CLR specializes generics for each value type (a real List<int> with no boxing) but shares one implementation across all reference types (they’re all pointer-sized). That’s why generics are both fast and code-size-efficient, unlike C++ templates which stamp out a copy per type.

Real-world example

🏢 How this works in production

The entire collections library — List<T>, Dictionary<K,V>, HashSet<T>, Span<T> — is generic, which is why working with strongly typed data is effortless and fast. App code leans on it constantly: a Result<T> wrapper for success/error, a generic repository or cache, Task<T> for async results, IOptions<T> for config. Constraints power real APIs — EF Core’s DbSet<TEntity> where TEntity : class, or a factory using where T : new() to construct instances.

One definition, many concrete types

In C#

C# 
// Generic class: type-safe container, no casting, no boxing for value types.
public class Box<T>
{
    public T Value { get; set; } = default!;
}
var n = new Box<int> { Value = 42 };   // stores int directly (no boxing)

// Generic method with a constraint: T must be a class with a parameterless ctor.
public static T CreateAndInit<T>() where T : class, new()
{
    var item = new T();                 // 'new()' constraint allows this
    return item;
}

// Constraints unlock capabilities of T:
public static T Max<T>(T a, T b) where T : IComparable<T>
    => a.CompareTo(b) >= 0 ? a : b;     // can call CompareTo because of the constraint

// ❌ The pre-generics way: object → boxing + casts + no type safety.

Key takeaways

🎯 Remember these three

  • Generics give type-safe, reusable code parameterized by type — no casting and no boxing for value types.
  • Constraints (where T : …) let you require a base/interface/new() so you can use T meaningfully.
  • The CLR specializes generics per value type but shares one implementation for reference types — fast and compact.

🗣️ When this comes up

“Why generics over object?” — type safety (compile-time) plus no boxing/casting for value types. Know constraints (where) and the runtime detail that the CLR specializes value types but shares reference types — that distinguishes a deep answer.
