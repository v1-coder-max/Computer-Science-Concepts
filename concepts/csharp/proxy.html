Proxy Pattern

A stand-in object with the same interface as the real one, sitting in front of it to control access — for lazy loading, security, remoting, or caching.

#design-pattern #structural #access-control #gof

The idea in plain English

A Proxy implements the same interface as a real object (the subject) and stands in for it, intercepting calls to add access control before (or instead of) forwarding to the real thing. Because it shares the interface, callers can’t tell they’re holding a proxy. The common flavors:

  • Virtual proxy — defers creating an expensive object until it’s actually needed (lazy loading).
  • Protection proxy — checks permissions before allowing the call.
  • Remote proxy — represents an object living in another process/machine, handling the network underneath.
  • Caching proxy — returns cached results to avoid repeating expensive work.

Proxy looks structurally identical to Decorator (both wrap an object with the same interface), but the intent differs: a decorator adds new behavior to enrich the object; a proxy controls access to it (when/whether/by whom it’s reached). Same shape, different purpose.

Real-world example

🏢 How this works in production

EF Core lazy loading uses proxies: navigation properties return proxy objects that fetch the related rows from the database only when you first access them (a virtual proxy). A gRPC client is a remote proxy — you call client.GetPriceAsync(...) as if it were local, and the generated proxy marshals it over HTTP/2. Frameworks like Castle DynamicProxy generate proxies at runtime to inject logging, transactions, or authorization around method calls. Anywhere a call looks direct but secretly does access control, lazy init, or remoting, a proxy is at work.

Stand-in controls access

In C#

C# 
public interface IImage { void Display(); }

// Real subject: expensive to create (loads a big file).
public class HighResImage : IImage
{
    public HighResImage(string path) { /* load megabytes from disk now */ }
    public void Display() { /* draw */ }
}

// Virtual proxy: same interface, but defers the expensive load until first use.
public class LazyImageProxy(string path) : IImage
{
    private HighResImage? _real;
    public void Display()
    {
        _real ??= new HighResImage(path);   // create on first access (lazy)
        _real.Display();                     // then forward
    }
}

IImage image = new LazyImageProxy("huge.png"); // cheap; nothing loaded yet
image.Display();                                // NOW it loads, then displays

// Protection proxy: gate access before forwarding.
public class SecureDocProxy(IDocument inner, IUser user) : IDocument
{
    public string Read() => user.CanRead ? inner.Read()
                                          : throw new UnauthorizedAccessException();
}

Key takeaways

🎯 Remember these three

  • A proxy shares the subject’s interface and controls access to it — virtual (lazy), protection, remote, or caching.
  • Same structure as a decorator, different intent: proxy controls access; decorator adds behavior.
  • EF Core lazy loading and gRPC clients are real-world proxies (virtual and remote).

Interview tip

🗣️ When this comes up

“Proxy vs Decorator?” Same wrapping structure; the difference is intent — proxy controls access (lazy/security/remote/cache), decorator adds behavior. Name the four proxy types and cite EF Core lazy loading or a gRPC client as proof you’ve seen it.
