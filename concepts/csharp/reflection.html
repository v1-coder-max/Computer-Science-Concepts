Reflection & Attributes

Inspect and manipulate types at runtime — discover members, read attributes, create objects, and call methods you knew nothing about at compile time. Powerful, and slow if overused.

#reflection #metadata #attributes #runtime

The idea in plain English

Because .NET assemblies carry rich metadata describing every type, the runtime can let you read and use that metadata programmatically — that’s reflection. Starting from a Type (via typeof(X) or obj.GetType()) you can enumerate properties, methods, and fields, read their attributes, create instances (Activator.CreateInstance), set values, and invoke methods — all without referencing the concrete type at compile time.

Attributes are the other half: declarative metadata you attach to code ([Required], [JsonProperty], [Authorize]) that means nothing until something reflects over it and acts on it. Attributes + reflection are how frameworks read your intent from your code.

This is what makes much of .NET feel “automatic”: DI containers scan for services, JSON serializers map properties, ORMs map entities, validators read [Required], test runners find [Fact] methods. The cost: reflection is much slower than direct calls and AOT/trim-hostile (the trimmer can’t see reflective access). So cache reflection results, and prefer source generators where performance or Native AOT matter — they do the same work at build time.

Real-world example

🏢 How this works in production

When ASP.NET Core builds your app, it reflects to find controllers, read [HttpGet]/[Authorize] attributes, and discover endpoints; the DI container reflects over constructors to resolve dependencies; System.Text.Json reflects to map object properties; validation reads [Required]/[Range]. It’s everywhere — and it’s also why startup has a warm-up cost and why high-perf/AOT scenarios switch to the source-generated equivalents (e.g. JSON source-gen) to skip reflection entirely.

Metadata → inspect & invoke

In C#

C# 
[AttributeUsage(AttributeTargets.Property)]
public class RequiredAttribute : Attribute { }   // declarative metadata

public class Customer
{
    [Required] public string Name { get; set; } = "";
    public string? Nickname { get; set; }
}

// Reflect: discover properties + read attributes (a mini validator).
Type t = typeof(Customer);                         // or customer.GetType()
foreach (PropertyInfo p in t.GetProperties())
{
    bool required = p.GetCustomAttribute<RequiredAttribute>() is not null;
    if (required && p.GetValue(customer) is null)   // read the value reflectively
        Console.WriteLine($"{p.Name} is required");
}

// Create an instance & invoke a method without knowing the type at compile time.
object obj = Activator.CreateInstance(t)!;
t.GetMethod("ToString")!.Invoke(obj, null);

// ⚠ Reflection is slow + AOT-hostile → cache results; prefer source generators on hot paths.

Key takeaways

🎯 Remember these three

  • Reflection reads/uses type metadata at runtime — discover members, read attributes, create instances, invoke methods, with no compile-time reference.
  • Attributes are declarative metadata that only matter when something reflects over them; together they power DI, serializers, ORMs, validation, and test runners.
  • It’s slow and AOT/trim-hostile — cache results and prefer source generators for performance/AOT.

Interview tip

🗣️ When this comes up

Explain reflection as runtime metadata inspection (with attributes as the declarative side) and name what it powers — DI, serialization, ORMs, tests. The senior caveat: it’s slow and AOT-hostile, so cache it and reach for source generators when it matters. That contrast is the whole point.
