Strategy Pattern

Define a family of interchangeable algorithms behind a common interface and swap them at runtime — turning a sprawling switch into a clean, pluggable choice.

#design-pattern #behavioral #algorithms #gof

The idea in plain English

When the same task can be done several ways — sort by different criteria, price with different discount rules, compress with different algorithms — the naive code grows a big if/switch that you must edit every time a new way appears. The Strategy pattern extracts each algorithm into its own class behind a shared interface, and the context holds a reference to one strategy and delegates to it. Swap the behavior by swapping the object — even at runtime.

It’s composition over conditionals: instead of branching on a flag, you inject the right algorithm. Adding a new variant means adding a class, not editing the context (Open/Closed), and each algorithm is independently testable.

In C# there’s a lightweight twist: when a strategy is a single method, you usually don’t need a whole interface and class hierarchy — a Func<...> delegate is the strategy. Both LINQ’s OrderBy(keySelector) and List.Sort(comparison) are strategy in delegate form.

Real-world example

🏢 How this works in production

A checkout applies different discount strategies — percentage off, buy-one-get-one, loyalty tier — selected per cart at runtime; each is its own class, and a new promotion is a new strategy with zero changes to the pricing engine. A maps app swaps route strategies (fastest / shortest / avoid tolls). Shipping picks a rate strategy per carrier. The pattern is everywhere a system must choose among algorithms; .NET bakes the delegate form right into LINQ and sorting.

Context delegates to a strategy

In C#

C# 
// The strategy interface + interchangeable algorithms.
public interface IDiscount { decimal Apply(decimal total); }
public class PercentDiscount(decimal pct) : IDiscount { public decimal Apply(decimal t) => t * (1 - pct); }
public class LoyaltyDiscount         : IDiscount { public decimal Apply(decimal t) => t - 5m; }

// Context holds a strategy and delegates — no switch on discount type.
public class Checkout(IDiscount discount)
{
    public decimal Total(decimal subtotal) => discount.Apply(subtotal);
}
var checkout = new Checkout(new PercentDiscount(0.1m));   // pick at runtime

// C# shortcut: when the strategy is one method, a delegate IS the strategy.
Func<decimal, decimal> discountFn = t => t * 0.9m;
var price = discountFn(100m);

// LINQ's OrderBy and List.Sort take a strategy (the key/comparison delegate):
people.OrderBy(p => p.Age);                 // the "by age" strategy
list.Sort((a, b) => b.Score - a.Score);     // the "descending score" strategy

Key takeaways

🎯 Remember these three

  • Strategy encapsulates interchangeable algorithms behind one interface and swaps them at runtime — composition over conditionals.
  • It replaces big switch/if ladders and supports Open/Closed (new algorithm = new class).
  • In C#, a Func<> delegate is often the whole strategy — see LINQ OrderBy and List.Sort.

Interview tip

🗣️ When this comes up

Strategy is the go-to answer for “how would you remove this big switch on behavior?” Mention runtime swappability and Open/Closed, and score points by noting C# often expresses it as a delegate (LINQ is strategy everywhere). Be ready to contrast it with State (same shape, different intent).
