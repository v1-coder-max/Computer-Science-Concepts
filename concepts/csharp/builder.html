Builder Pattern

Construct a complex object step by step with a readable, fluent API — instead of a constructor with fifteen parameters you can’t tell apart.

#design-pattern #creational #fluent #gof

The idea in plain English

When an object has many optional parts, constructors get ugly. You end up with telescoping constructors (a dozen overloads) or one giant constructor where new Pizza(true, false, true, 2, …) is unreadable and easy to get wrong. The Builder pattern separates how an object is assembled from the object itself: you call small, named steps (.AddCheese(), .SetSize(Large)) and finish with .Build().

The wins:

  • Readable — each step is named, so intent is obvious.
  • Flexible — set only the parts you need; defaults handle the rest.
  • Safe — Build() can validate invariants and hand back an immutable result.

The classic GoF version adds a director that drives a builder through a fixed recipe, but in C# the everyday form is a fluent builder (each step returns this). Note that modern C# often makes a full builder unnecessary: object initializers, optional/named parameters, and records with init cover simple cases. Reach for a builder when construction is genuinely complex, needs validation, or builds in stages.

Real-world example

🏢 How this works in production

Builders are everywhere in .NET itself. StringBuilder assembles a string efficiently in steps; WebApplication.CreateBuilder(args) builds up your whole app (.Services.Add…, .Configuration…) then .Build()s it; EF Core’s ModelBuilder and the fluent configuration APIs are builders. Test code uses them too — a new OrderBuilder().WithItem(...).WithCustomer(...).Build() keeps test setup readable. Anywhere you see a long chain of .Something(...) ending in .Build(), that’s this pattern.

Step by step, then build

In C#

C# 
public class PizzaBuilder
{
    private string _size = "Medium";
    private readonly List<string> _toppings = new();

    public PizzaBuilder Size(string size) { _size = size; return this; }   // fluent: return this
    public PizzaBuilder Add(string topping) { _toppings.Add(topping); return this; }

    public Pizza Build()
    {
        if (_toppings.Count > 10) throw new InvalidOperationException("Too many toppings");
        return new Pizza(_size, _toppings.ToArray());   // validate, return immutable result
    }
}

var pizza = new PizzaBuilder()
    .Size("Large")
    .Add("cheese")
    .Add("mushroom")
    .Build();           // readable intent, only the parts you need

// For simple cases, modern C# may be enough — no builder needed:
var order = new Order { Id = 1, Customer = "Ada", Priority = true };  // object initializer

Key takeaways

🎯 Remember these three

  • Builder constructs a complex object step by step, replacing telescoping constructors with a readable fluent API.
  • Each step returns the builder; Build() validates invariants and returns the (often immutable) result.
  • For simple objects, C# object initializers / records with init often make a builder unnecessary.
