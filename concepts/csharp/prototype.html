Prototype Pattern

Instead of building a new object from scratch, copy an existing one and tweak it — handy when construction is expensive or you want a variant of something already configured.

#design-pattern #creational #cloning #gof

The idea in plain English

The Prototype pattern creates objects by cloning a prototype rather than calling a constructor. It’s useful when: building an object is expensive (lots of setup, a DB hit, a heavy parse), you need many near-identical objects, or you want a copy of an object whose exact type you don’t know — just “give me another one like this.”

The thing you must get right is how deep the copy goes:

  • Shallow copy — copies the top-level fields. Reference-type members (lists, nested objects) are shared with the original, so mutating the clone’s list also mutates the original’s. Often a bug.
  • Deep copy — recursively copies nested objects too, so the clone is fully independent.

In C#, the old ICloneable interface is discouraged because it doesn’t specify shallow vs deep — callers can’t tell what they’ll get. Prefer an explicit copy constructor or a typed Clone() method, or — for immutable data — a record with the with expression, which makes “a copy with these fields changed” a language feature.

Real-world example

🏢 How this works in production

A game spawns hundreds of enemies that share most properties — cloning a fully-configured prototype is far cheaper than re-running expensive setup per enemy. Document/template systems clone a base template and customize it. In everyday C#, the cleanest version is records: an event-sourced system holds an immutable state object and produces the next state with state with { Balance = newBalance } — a deep-ish copy with one field changed, no mutation, no shared-reference bugs. That’s the prototype idea baked into the language.

Clone, then tweak

In C#

C# 
// ✅ Explicit deep copy via a copy constructor — caller knows exactly what they get.
public class Enemy
{
    public string Type { get; set; } = "";
    public Stats Stats { get; set; } = new();          // reference type
    public Enemy() { }
    public Enemy(Enemy other)                           // copy constructor
    {
        Type  = other.Type;
        Stats = new Stats(other.Stats);                 // DEEP copy the nested object
    }
}
var template = new Enemy { Type = "Orc", Stats = new() { Hp = 100 } };
var clone = new Enemy(template) { Type = "Orc Captain" };  // independent variant

// ✅ Immutable data: records make "copy with changes" a language feature (prototype-ish).
public record AccountState(string Owner, decimal Balance);
var next = state with { Balance = state.Balance - 50 };   // new copy, original untouched

// ⚠ Avoid ICloneable — it doesn't say whether the copy is shallow or deep.

Key takeaways

🎯 Remember these three

  • Prototype creates objects by cloning a configured instance instead of constructing from scratch.
  • Mind shallow vs deep copy — shared nested references cause subtle bugs; deep-copy when you need independence.
  • Avoid ICloneable (ambiguous); prefer copy constructors or records’ with expression.

Interview tip

🗣️ When this comes up

The real question hidden in “Prototype” is shallow vs deep copy — be able to explain the shared-reference bug and how to deep-copy. Note that ICloneable is discouraged and that records’ with is the idiomatic modern way to “clone with changes.”
