Records

Records are C#'s answer to boilerplate-heavy data types: a concise, immutable type with value-based equality and copy-with-changes built in.

#csharp #records #immutability #value-equality

The idea in plain English

A record is a type designed to hold data. From one line — public record Money(decimal Amount, string Currency); — the compiler generates an immutable type with a constructor, init-only properties, value-based Equals/GetHashCode, a readable ToString, and deconstruction. It removes the dozens of lines you'd hand-write for a well-behaved data class.

The headline feature is value-based equality. Two records are equal if their contents are equal, unlike normal classes which compare by reference (identity). So two Money(10, "USD") instances are equal — exactly what you want for DTOs, domain values, and dictionary keys.

Records are immutable by default (init-only properties), which makes them ideal for safe sharing (including across threads). To 'change' one you use a with-expression — order with { Status = "Paid" } — which creates a new record copying the rest. This non-destructive mutation is the immutable-data way to evolve state without mutating the original.

record class (the default) is a reference type with value equality; record struct is a value type with the same conveniences. Use records for data-centric types — DTOs, API models, events, value objects — and keep regular classes for entities with identity and rich behavior. They pair naturally with pattern matching.

Real-world example

🏢 How this works in production

Records have become the default for DTOs, API request/response models, domain value objects, and immutable events/messages in modern C# — anywhere you want 'compare and copy by value, don't mutate.' They're especially common in functional-leaning and concurrent code (immutability removes whole classes of bugs) and in CQRS/event-sourced systems where commands and events are naturally immutable value-like records.

Value equality + non-destructive copy

In C#

C# 
// One line → immutable type with value equality, ToString, deconstruction.
public record Money(decimal Amount, string Currency);

var a = new Money(10, "USD");
var b = new Money(10, "USD");
bool same = a == b;              // True — value equality (not reference)

// Non-destructive mutation: copy with changes, original untouched.
var raise = a with { Amount = 12 };   // new Money(12,"USD"); a is unchanged

// record class (default, reference type) vs record struct (value type).
public record struct Point(int X, int Y);

Key takeaways

🎯 Remember these three

  • Records generate a concise, immutable data type with value-based equality, ToString, and deconstruction.
  • Equal by contents (not reference); 'change' via a with-expression that copies a new instance (non-destructive mutation).
  • Use for DTOs, value objects, events; record class (reference) vs record struct (value) — keep classes for identity-rich entities.

Interview tip

🗣️ When this comes up

Lead with value equality and immutability as the two reasons records exist, then show a with-expression for non-destructive mutation. Knowing when not to use them (entities with identity and behavior → regular classes) shows judgment, not just syntax.
