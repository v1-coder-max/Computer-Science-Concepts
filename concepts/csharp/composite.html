Composite Pattern

Treat a single object and a whole tree of objects the same way. A file and a folder full of files both answer “what’s your size?” — and the client doesn’t care which it’s holding.

#design-pattern #structural #tree #gof

The idea in plain English

Lots of data is a part-whole hierarchy: a folder contains files and other folders; a UI panel contains buttons and other panels. The Composite pattern lets you build these trees and treat leaves (individual items) and composites (containers of children) uniformly, because both implement the same component interface.

A leaf does its operation directly; a composite implements the operation by recursing over its children and aggregating. So folder.GetSize() sums its children’s sizes, each of which might be a file (returns its own size) or another folder (recurses again). The win: client code handles the whole tree with one method call and never writes if (it’s a folder) … else … — the recursion lives in the composite.

Real-world example

🏢 How this works in production

The file system is the textbook composite — files and directories share operations (size, delete, search), and “delete this directory” recurses through the tree. UI frameworks are composites: a control tree where a container renders by rendering its children (WPF/Blazor component trees, the browser DOM). Org charts, menu/submenu structures, and graphics scene graphs all use it. Any time you find yourself writing recursive code over a nested structure with a shared operation, you’re using composite.

Leaves and composites, one interface

In C#

C# 
// Component: the shared interface for leaf AND composite.
public interface IFileSystemItem { long GetSize(); }

// Leaf: does the work directly.
public class FileItem(long size) : IFileSystemItem
{
    public long GetSize() => size;
}

// Composite: holds children and recurses.
public class Folder : IFileSystemItem
{
    private readonly List<IFileSystemItem> _children = new();
    public void Add(IFileSystemItem item) => _children.Add(item);

    public long GetSize() => _children.Sum(c => c.GetSize());  // recursion is here
}

// Client treats the whole tree uniformly — no leaf-vs-composite branching.
var root = new Folder();
root.Add(new FileItem(100));
var sub = new Folder();
sub.Add(new FileItem(50));
root.Add(sub);
Console.WriteLine(root.GetSize());   // 150 — recurses through the tree

Key takeaways

🎯 Remember these three

  • Composite models part-whole trees; leaves and composites share one component interface.
  • The composite implements operations by recursing over children — clients treat the whole tree uniformly, with no type-checking.
  • File systems, UI/DOM trees, and org charts are classic composites.

Interview tip

🗣️ When this comes up

The keyword is tree / part-whole hierarchy treated uniformly. Use the file-system example, point out that leaf and composite share an interface and the composite recurses, and that the payoff is client code with no if (folder) … else … branching.
