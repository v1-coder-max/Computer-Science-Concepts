Mediator Pattern

Stop objects from referencing each other directly. Route their interactions through a central mediator, turning a tangled mesh into a clean hub.

#design-pattern #behavioral #decoupling #mediatr

The idea in plain English

When many objects need to talk to many other objects, they end up holding references to each other — a many-to-many mesh that’s hard to change because everything is coupled to everything. The Mediator pattern introduces a central object that all the “colleagues” talk to instead of to each other. A colleague notifies the mediator; the mediator decides who else needs to know and coordinates them. Each colleague now depends only on the mediator, not on its peers.

This collapses the mesh into a hub-and-spoke: the interaction logic lives in one place (the mediator) and the colleagues stay simple and decoupled. The trade-off is that a complex mediator can become a “god object,” so keep its responsibility focused.

In modern .NET, the name is most associated with MediatR: an in-process messaging library where you send a request object (a command or query) and it dispatches to the matching handler — so your controllers don’t reference services directly, they just send messages. It also supports notifications (one message, many handlers) and a pipeline of behaviors (logging, validation). MediatR is the backbone of most CQRS-in-.NET implementations.

Real-world example

🏢 How this works in production

A thin ASP.NET Core controller does await _mediator.Send(new PlaceOrderCommand(...)) and nothing else — it doesn’t know about the order service, validators, or the database; MediatR routes the command to its handler. This keeps controllers tiny and the app loosely coupled, which is why MediatR + CQRS is so common. The classic GoF example is a chat room: users send to the room (the mediator), which relays to the others — no user references another user. Air-traffic control coordinates planes that never talk directly.

Mesh → hub

In C# (MediatR)

C# 
// A request (command) — just data.
public record PlaceOrder(Guid CartId) : IRequest<OrderResult>;

// Its handler — the only thing that knows how to fulfil it.
public class PlaceOrderHandler(IOrders orders) : IRequestHandler<PlaceOrder, OrderResult>
{
    public Task<OrderResult> Handle(PlaceOrder req, CancellationToken ct)
        => orders.PlaceAsync(req.CartId);
}

// The controller depends ONLY on the mediator — not on IOrders, validators, etc.
[ApiController]
public class OrdersController(IMediator mediator) : ControllerBase
{
    [HttpPost("/orders")]
    public Task<OrderResult> Post(PlaceOrder cmd) => mediator.Send(cmd); // routed to the handler
}

Key takeaways

🎯 Remember these three

  • Mediator routes interactions through a central hub so objects don’t reference each other — N×N coupling becomes N spokes.
  • The coordination logic lives in one place; keep the mediator focused so it doesn’t become a god object.
  • In .NET it’s MediatR: send a request → dispatched to a handler (plus notifications + a behavior pipeline) — the basis of CQRS.

Interview tip

🗣️ When this comes up

Sell it as mesh → hub: objects talk to a mediator instead of each other, cutting many-to-many coupling. In .NET, immediately bring up MediatR (send → handler) and its role in CQRS and thin controllers. Note the god-object risk to show balance.
