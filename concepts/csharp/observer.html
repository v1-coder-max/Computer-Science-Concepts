Observer Pattern

A one-to-many relationship: when one object (the subject) changes, all its registered observers are notified automatically. In C#, this is events.

#design-pattern #behavioral #events #gof

The idea in plain English

Sometimes many parts of a system need to react when something changes — a price updates, an order is placed, a setting toggles. You could have the changing object call each interested party directly, but that couples it to all of them. The Observer pattern inverts it: the subject keeps a list of observers who subscribe, and when its state changes it notifies them all — without knowing who they are or what they do.

That’s loose coupling and the publish/subscribe idea at the object level: observers can come and go (subscribe/unsubscribe) without the subject changing. The subject pushes; observers react.

C# has the observer pattern built into the language. A delegate is a list of method references, and an event is a subject’s notification point — += to subscribe, raise to notify all. You rarely hand-roll the GoF interfaces in C#; you use events. For streams of values over time, IObservable<T>/IObserver<T> (Reactive Extensions) is the same pattern at scale.

Real-world example

🏢 How this works in production

UI data binding is observer: when a view-model property changes, it raises PropertyChanged and every bound control updates — the view-model doesn’t know which controls are listening. A stock ticker pushes price updates to every subscribed chart and alert. Domain events (OrderPlaced) let handlers react without the order code referencing them. It’s the same idea as the system-level event-driven architecture, just inside one process — and in C# it’s all expressed with event.

Subject notifies many observers

In C# (events are observer)

C# 
public class Stock
{
    public event EventHandler<decimal>? PriceChanged;   // the subject's notification point

    private decimal _price;
    public decimal Price
    {
        get => _price;
        set { _price = value; PriceChanged?.Invoke(this, value); } // notify all observers
    }
}

var msft = new Stock();
msft.PriceChanged += (s, price) => Console.WriteLine($"chart: {price}");   // subscribe (+=)
msft.PriceChanged += (s, price) => AlertIfAbove(price, 400m);             // another observer
msft.Price = 410m;   // every subscriber is notified automatically
// msft.PriceChanged -= handler;   // unsubscribe when done (avoids leaks)

// For streams of values over time, IObservable<T> (Rx) is observer at scale.

⚠️ Unsubscribe to avoid leaks

An event holds references to its handlers. If a long-lived subject keeps a subscription to a short-lived observer, the observer can’t be garbage-collected — a classic managed memory leak. Always -= when the observer goes away (or use weak-event patterns).

Key takeaways

🎯 Remember these three

  • Observer is a one-to-many dependency: the subject notifies all subscribers when it changes, without knowing who they are (loose coupling).
  • In C# it’s built in — delegate + event (+=/-=); IObservable<T> for value streams.
  • Unsubscribe to prevent the event-handler memory leak.

Interview tip

🗣️ When this comes up

The C#-specific insight: events ARE the observer pattern — you don’t implement the GoF interfaces, you use event. Mention subscribe/notify/loose-coupling, the unsubscribe leak, and IObservable<T> for streams. Distinguish it from Mediator (hub) and pub/sub at the system level.
