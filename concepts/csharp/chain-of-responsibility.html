Chain of Responsibility

Pass a request down a line of handlers; each one either deals with it or hands it to the next. The sender never needs to know which handler will respond.

#design-pattern #behavioral #pipeline #gof

The idea in plain English

You have a request and several possible handlers, and you don’t want the sender hard-wired to a specific one. Chain of Responsibility links the handlers into a chain: each handler gets the request, decides whether it can deal with it, and either handles it, passes it to the next handler, or does some work and then passes it along. The sender just drops the request at the head of the chain.

The benefits:

  • Decoupling — sender and receiver don’t know about each other; the sender knows only “the chain.”
  • Flexible composition — add, remove, or reorder handlers without touching the others.
  • Single responsibility — each handler does one focused check/step.

If this sounds familiar, it’s because the ASP.NET Core middleware pipeline is chain of responsibility: each middleware can act on the request, call next() to pass it on, or short-circuit. Validation chains, logging/auth filters, and Polly’s policy-wrapping use the same shape. (A handler can also do work on the way back out, like middleware touching the response.)

Real-world example

🏢 How this works in production

Every ASP.NET Core app is built on this pattern: a request flows through exception handling → authentication → authorization → routing → your endpoint, and any link can handle-and-stop (auth returns 401) or pass along. Support systems escalate tickets through tiers (bot → tier-1 → tier-2 → human) until one resolves it. Approval flows route a purchase request up the management chain until someone has the authority to approve the amount. The sender just submits; the chain figures out who acts.

In C#

C# 
public abstract class Handler
{
    private Handler? _next;
    public Handler SetNext(Handler next) { _next = next; return next; } // build the chain

    public virtual void Handle(Request req) => _next?.Handle(req);      // default: pass along
}

public class AuthHandler : Handler
{
    public override void Handle(Request req)
    {
        if (!req.IsAuthenticated) throw new UnauthorizedAccessException(); // handle & STOP
        base.Handle(req);                                                  // else pass on
    }
}
public class ValidationHandler : Handler
{
    public override void Handle(Request req)
    {
        if (!req.IsValid) throw new ValidationException();
        base.Handle(req);
    }
}

// Compose the chain; the sender only knows the head.
var auth = new AuthHandler();
auth.SetNext(new ValidationHandler()).SetNext(new BusinessHandler());
auth.Handle(request);

Key takeaways

🎯 Remember these three

  • A request travels a chain of handlers; each handles it or passes it on, decoupling sender from receiver.
  • Handlers can be added, removed, or reordered freely; any can short-circuit the chain.
  • The ASP.NET Core middleware pipeline is this pattern — the clearest real example.

Interview tip

🗣️ When this comes up

The instant-credibility move: “ASP.NET Core middleware is Chain of Responsibility.” Explain handle-or-pass, short-circuiting, and the sender/receiver decoupling. Distinguish it from Decorator (which always delegates and adds behavior, rather than possibly stopping).
