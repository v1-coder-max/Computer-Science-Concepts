Decorator Pattern

Add responsibilities to an object by wrapping it in another object with the same interface — stackable, at runtime, without touching the original class.

#design-pattern #structural #composition #gof

The idea in plain English

You want to add behavior — caching, compression, logging, retry — to an object, but subclassing for every combination explodes (CachedLoggedRetryingRepository…). The Decorator pattern instead wraps the object in another object that implements the same interface, adds its bit of behavior, and delegates the rest to the wrapped instance. Because the wrapper has the same type, callers can’t tell the difference — and you can stack decorators like layers of an onion.

This is composition over inheritance for extension: behavior is added at runtime by composing objects, not at compile time by editing classes. Each decorator does one thing (Single Responsibility), and you mix and match them freely — caching and logging, or just one, in any order, without writing a class per combination.

The catch versus the Adapter and Proxy patterns (all “wrappers”): a decorator keeps the same interface and adds behavior; an adapter changes the interface; a proxy keeps the same interface but controls access rather than enriching behavior.

Real-world example

🏢 How this works in production

.NET’s Stream family is decorators in the wild: you wrap a FileStream in a BufferedStream, then a GZipStream, then a CryptoStream — each a Stream that adds buffering/compression/encryption and forwards to the inner stream. You compose exactly the pipeline you need. The same pattern appears when teams wrap a repository or HTTP handler with caching and logging decorators (and DI containers/Scrutor make registering decorators a one-liner), so each concern stays in its own small, testable class.

Stackable wrappers

In C#

C# 
public interface IProductRepository { Task<Product?> GetAsync(int id); }

// The real component.
public class SqlProductRepository : IProductRepository
{
    public Task<Product?> GetAsync(int id) => /* hit the DB */ default!;
}

// A decorator: SAME interface, adds caching, delegates to the inner repo.
public class CachingProductRepository(IProductRepository inner, IMemoryCache cache)
    : IProductRepository
{
    public async Task<Product?> GetAsync(int id)
    {
        if (cache.TryGetValue(id, out Product? p)) return p;     // added behavior
        p = await inner.GetAsync(id);                            // delegate inward
        cache.Set(id, p);
        return p;
    }
}

// Stack them — order is yours, no new combined classes needed.
IProductRepository repo =
    new LoggingProductRepository(
        new CachingProductRepository(
            new SqlProductRepository(), cache), logger);
// (DI: services.Decorate<IProductRepository, CachingProductRepository>();  via Scrutor)

Key takeaways

🎯 Remember these three

  • Decorator wraps an object in another with the same interface, adding behavior and delegating the rest.
  • Decorators stack — compose caching, logging, retry, etc. without a class per combination (composition over inheritance).
  • Same interface + adds behavior distinguishes it from Adapter (changes interface) and Proxy (controls access).

Interview tip

🗣️ When this comes up

The Stream example sells it instantly: GZipStream wrapping a FileStream is a decorator. Stress same interface + added behavior + stackable, and that it’s composition-over-inheritance for extension (avoids the subclass explosion). Distinguish it from adapter/proxy.
