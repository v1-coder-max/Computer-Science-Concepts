Boxing & Unboxing

Boxing is the hidden cost of putting a value type where an object is expected: it gets wrapped on the heap. Death by a thousand boxes is a classic .NET performance trap.

#csharp #boxing #performance #memory

The idea in plain English

Boxing converts a value type (an int, a struct) into a reference-type object by allocating a box on the heap and copying the value into it. Unboxing reverses it — extracting the value back out (with a type check). It happens whenever a value type is used where a reference type is expected: assigning to object, putting an int in a non-generic collection, or passing one to a method taking object.

The problem is cost. Each box is a heap allocation plus a copy, and those boxes later create GC pressure. One box is trivial; boxing in a hot loop or per-item across a big collection quietly tanks performance and allocates megabytes of garbage.

It's often invisible — the compiler inserts it silently. Classic triggers: the old non-generic ArrayList (stores object, so every int boxes), calling an interface method on a struct, or string formatting that takes object[]. You won't see it in the code; you see it in the allocation profiler.

The fix is generics. List<int> stores ints directly — no boxing — which is the main reason generic collections replaced the non-generic ones. Generally: prefer generic APIs over object-based ones, and watch for accidental boxing (struct-to-interface, value types in object parameters) in performance-sensitive paths.

Real-world example

🏢 How this works in production

Boxing is a top finding in .NET performance reviews: a service that allocates excessively and spends too long in GC is frequently boxing value types somewhere hot — logging that passes int args as object, a struct enumerator that boxes, or legacy ArrayList/Hashtable usage. Switching to generics and struct-aware APIs (and tools like allocation profilers / Roslyn analyzers that flag boxing) is standard practice for high-throughput code where GC pauses matter.

Value → boxed object on the heap

In C#

C# 
// ❌ Boxing — value type used as object (heap allocation each time).
object o = 42;                       // box
int back = (int)o;                   // unbox (with a type check)

ArrayList list = new();              // stores object → every int boxes
for (int i = 0; i < 1_000_000; i++) list.Add(i);   // a million boxes 😱

// ✅ Generics — no boxing, value stored directly.
List<int> fast = new();
for (int i = 0; i < 1_000_000; i++) fast.Add(i);   // zero boxing

// Watch for hidden boxing: struct → interface, value types in object params.

Key takeaways

🎯 Remember these three

  • Boxing wraps a value type in a heap-allocated object (copy + allocation); unboxing extracts it back.
  • It's often invisible and cheap once, but boxing in hot paths causes allocations and GC pressure.
  • Generics (List<int>) avoid it — prefer generic APIs over object-based ones; watch struct-to-interface boxing.

Interview tip

🗣️ When this comes up

Define boxing as wrapping a value type on the heap and explain the cost (allocation + GC pressure), then give the fix: generics. Knowing the invisible triggers — non-generic collections, struct-to-interface calls — is what separates a real answer from the textbook definition.
