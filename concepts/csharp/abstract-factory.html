Abstract Factory

A factory of factories: create a whole family of related objects through one interface, so you never accidentally mix a Windows button with a macOS checkbox.

#design-pattern #creational #families #gof

The idea in plain English

A plain factory makes one kind of product. Abstract Factory makes a family of related products that are meant to go together. You define an interface with several create-methods (CreateButton, CreateCheckbox, CreateMenu), and each concrete factory produces a matching set — a Windows factory makes Windows widgets, a macOS factory makes macOS widgets. The client picks one factory and gets a consistent family; it’s impossible to end up with mismatched pieces.

The point isn’t just decoupling creation — it’s guaranteeing consistency across a set. Swap the whole family by swapping the factory (e.g. choose the platform’s factory at startup), and the client code, which only talks to the abstract products, doesn’t change.

Versus Factory Method: factory method creates one product via a virtual method; abstract factory creates several related products via an object you pass around. It’s heavier, so reach for it only when you genuinely have product families.

Real-world example

🏢 How this works in production

Cross-platform UI toolkits are the canonical case: a WinFactory and a MacFactory each produce a full set of platform-native controls, and the app picks one at startup, so a window is never a mix of Windows and Mac widgets. The same shape appears in data access — ADO.NET’s DbProviderFactory creates a matching family (DbConnection, DbCommand, DbParameter) for a given provider, so you don’t pair a SQL Server connection with an Oracle command. Themes (dark/light widget sets) work the same way.

One factory → a matching family

In C#

C# 
public interface IButton   { void Paint(); }
public interface ICheckbox { void Paint(); }

// The abstract factory: creates a whole related family.
public interface IUiFactory
{
    IButton   CreateButton();
    ICheckbox CreateCheckbox();
}

public class WinFactory : IUiFactory
{
    public IButton   CreateButton()   => new WinButton();
    public ICheckbox CreateCheckbox() => new WinCheckbox();
}
public class MacFactory : IUiFactory
{
    public IButton   CreateButton()   => new MacButton();
    public ICheckbox CreateCheckbox() => new MacCheckbox();
}

// Client depends only on the abstractions; one factory choice fixes the whole family.
public class App(IUiFactory ui)
{
    public void Render()
    {
        ui.CreateButton().Paint();      // always matches...
        ui.CreateCheckbox().Paint();    // ...the same platform
    }
}
var app = new App(OperatingSystem.IsWindows() ? new WinFactory() : new MacFactory());

Key takeaways

🎯 Remember these three

  • Abstract Factory creates families of related objects through one interface — guaranteeing they’re used together.
  • Swap the entire family by swapping the factory; the client (which uses only abstract products) doesn’t change.
  • Use it only when real product families exist — otherwise it’s heavier than a plain factory.

Interview tip

🗣️ When this comes up

The exam question is Abstract Factory vs Factory Method: factory method makes one product (subclass decides); abstract factory makes a family of related products and enforces their consistency. Cite cross-platform UI or ADO.NET’s DbProviderFactory as the example.
