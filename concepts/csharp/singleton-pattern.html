Singleton Pattern

Guarantee a class has exactly one instance and give everyone a single, shared way to reach it. Simple to state — surprisingly easy to get wrong.

#design-pattern #creational #thread-safety #gof

The idea in plain English

Some things should exist once for the whole process — a configuration loaded from disk, a connection pool, an in-memory cache. The Singleton pattern (one of the original GoF creational patterns) enforces that: it makes the constructor private so nobody can new it, holds the one instance in a static field, and exposes it through a static Instance property.

The tricky part is doing it thread-safely and lazily — creating the instance only when first needed, without two threads racing to build two of them. The classic “double-checked locking” solves it but is fiddly; in C# the idiomatic answer is Lazy<T>, which handles the locking for you.

One honest caveat up front: Singleton is the most controversial GoF pattern. A hand-rolled singleton is global state — it hides dependencies and makes unit testing painful. In modern .NET you usually want a DI container singleton instead (more on that below).

Real-world example

🏢 How this works in production

In a high-throughput service like Stripe’s payment API, you don’t want every request spinning up its own HttpClient or logging pipeline — that exhausts sockets and wastes memory. Those are created once and shared process-wide. .NET itself does this everywhere: HttpClient is designed to be a long-lived singleton, and a ILogger factory, metrics collector, and config snapshot all live as one shared instance behind the DI container.

One instance, many callers

In C# — the thread-safe way

Lazy<T> gives you lazy and thread-safe initialization in a few lines — no manual locking, no double-checked-locking bugs:

C# 
public sealed class AppConfig          // sealed: nobody can subclass to break the guarantee
{
    // Thread-safe, lazy: the factory runs once, on first access to .Value.
    private static readonly Lazy<AppConfig> _instance =
        new(() => new AppConfig());

    public static AppConfig Instance => _instance.Value;

    private AppConfig()                // private ctor: only this class can construct it
    {
        // expensive one-time setup (read file, parse env, etc.)
        Environment = System.Environment.GetEnvironmentVariable("ASPNETCORE_ENVIRONMENT")
                      ?? "Production";
    }

    public string Environment { get; }
}

// Usage anywhere in the process — always the same object:
string env = AppConfig.Instance.Environment;

But in modern .NET, prefer letting the DI container own the lifetime:

C# — DI singleton (preferred) 
// One instance for the whole app — but injected, not globally reached.
builder.Services.AddSingleton<IAppConfig, AppConfig>();

// Consumers depend on the abstraction and stay testable (swap a fake in tests):
public class PricingService(IAppConfig config)   // primary constructor
{
    public bool IsProd => config.Environment == "Production";
}

⚠️ Singleton’s dark side

A classic X.Instance singleton is global state: callers reach for it directly, so dependencies become invisible and tests can’t substitute a fake. It also breaks the Dependency Inversion principle. Reach for the DI container’s singleton lifetime instead — same “one instance,” but injected through an interface. Also: a singleton holding mutable state must be internally thread-safe, since every thread shares it.

Key takeaways

🎯 Remember these three

  • Singleton = private constructor + one static instance, exposed through a single access point.
  • Use Lazy<T> for correct lazy, thread-safe initialization — don’t hand-roll double-checked locking.
  • Prefer a DI container singleton (AddSingleton) over a global X.Instance — you get one instance and testability.

Interview tip

🗣️ When this comes up

“Write a thread-safe singleton” is a staple. Reach for Lazy<T> and explain why it’s thread-safe (the runtime guarantees the factory runs once). Then earn the senior nod: mention that you’d usually prefer a DI-managed singleton because the hand-rolled version is global state that hurts testability.
