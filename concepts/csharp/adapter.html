Adapter Pattern

A translator between two interfaces that don’t fit. Your code expects one shape; a third-party library speaks another — the adapter bridges them.

#design-pattern #structural #integration #gof

The idea in plain English

Like a travel power adapter that lets a US plug fit a European socket, the Adapter pattern wraps an existing class (the adaptee) so it satisfies the interface your code expects (the target). Your app keeps depending on its own clean abstraction; the adapter does the translating — mapping method names, reshaping arguments, converting return values.

It’s the go-to when you need to integrate incompatible code you can’t (or shouldn’t) change: a third-party SDK, a legacy component, an external API. Instead of letting that foreign interface leak through your codebase, you confine it behind one adapter that implements your interface.

Two forms:

  • Object adapter (preferred) — the adapter holds an instance of the adaptee (composition) and delegates to it. Flexible; can adapt subclasses too.
  • Class adapter — the adapter inherits from the adaptee. Limited in C# (single inheritance) and tighter coupling.

Real-world example

🏢 How this works in production

Your app logs through its own IAppLogger interface, but you adopt a third-party logging library with a totally different API. Rather than rewrite every call site (or marry your code to the vendor), you write one SerilogAdapter : IAppLogger that forwards to the library. Swap vendors later? Write a new adapter; nothing else changes. The same move adapts payment gateways (each SDK behind a common IPaymentGateway), and .NET itself adapts constantly — e.g. StreamReader adapts a byte Stream into a text reader.

Translate between interfaces

In C#

C# 
// Target: the interface YOUR app depends on.
public interface IAppLogger { void Info(string message); }

// Adaptee: a third-party logger with an incompatible API (can't change it).
public class AcmeLog { public void Write(int level, string text) { /* ... */ } }

// Object adapter: implement YOUR interface, hold the adaptee, translate the call.
public class AcmeLogAdapter : IAppLogger
{
    private readonly AcmeLog _acme;
    public AcmeLogAdapter(AcmeLog acme) => _acme = acme;

    public void Info(string message) => _acme.Write(level: 1, text: message); // map shapes
}

// The rest of the app never sees AcmeLog — only IAppLogger.
IAppLogger logger = new AcmeLogAdapter(new AcmeLog());
logger.Info("order placed");   // swap the vendor → swap the adapter, nothing else changes

Key takeaways

🎯 Remember these three

  • Adapter converts an incompatible interface (adaptee) into the one your code expects (target).
  • It’s for integrating third-party/legacy code you can’t change — confine the foreign API behind one adapter.
  • Prefer the object adapter (composition + delegation) over the class adapter (inheritance).

Interview tip

🗣️ When this comes up

Be ready to separate the wrapper patterns: Adapter changes an interface to fit; Decorator adds behavior while keeping the same interface; Facade simplifies a complex subsystem behind one interface; Proxy controls access with the same interface. Adapter’s keyword is incompatible interface integration.
