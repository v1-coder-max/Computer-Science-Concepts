Factory Pattern

Create objects without hard-coding their concrete class. The caller asks a factory for “a payment processor” and gets the right one — without ever writing new StripeProcessor().

#design-pattern #creational #factory-method #gof

The idea in plain English

When code does new ConcreteThing(), it’s welded to that exact type — to swap or extend it you must edit every call site. The factory patterns move that decision into one place so the rest of your code depends only on an abstraction. Two common forms:

  • Simple Factory — a single method (often static) that picks and returns a concrete type behind an interface, usually based on a parameter. Not a “real” GoF pattern but extremely common.
  • Factory Method (GoF) — a base class defines a virtual creator method, and subclasses decide which concrete product to make. The base class works with the product through its interface, never knowing the concrete type.

The payoff is decoupling creation from use and honoring Open/Closed: add a new product type by adding a class (and a branch/registration), without touching the code that uses it. (DI containers are essentially configurable factories — that’s the modern default in .NET.)

Real-world example

🏢 How this works in production

A checkout system needs a payment processor chosen at runtime — card, PayPal, gift card. Instead of if (method == "paypal") new PayPalProcessor()… scattered everywhere, a PaymentProcessorFactory.Create(method) returns the right IPaymentProcessor. Adding “Apple Pay” means one new class and one factory entry — the checkout flow never changes. .NET’s own HttpClientFactory and ILoggerFactory are exactly this idea, and the DI container is the general-purpose version.

Caller asks, factory decides

In C#

C# 
public interface IPaymentProcessor { Task PayAsync(decimal amount); }

// Simple Factory: one place owns the concrete choice.
public static class PaymentProcessorFactory
{
    public static IPaymentProcessor Create(string method) => method switch
    {
        "card"    => new CardProcessor(),
        "paypal"  => new PayPalProcessor(),
        "giftcard"=> new GiftCardProcessor(),
        _ => throw new NotSupportedException(method)
    };
}
// Adding "applepay" = a new class + one arm here; callers below never change.
var processor = PaymentProcessorFactory.Create(order.Method);
await processor.PayAsync(order.Total);

// Factory Method (GoF): subclasses decide the product.
public abstract class Dialog
{
    protected abstract IButton CreateButton();   // the factory method
    public void Render() => CreateButton().Paint(); // base works via the interface
}
public class WindowsDialog : Dialog
{
    protected override IButton CreateButton() => new WindowsButton();
}

Key takeaways

🎯 Remember these three

  • A factory decouples object creation from use — callers depend on an interface, not new ConcreteType().
  • Simple Factory = one method picks the concrete type; Factory Method (GoF) = subclasses decide via a virtual creator.
  • It supports Open/Closed (add a type without editing callers); DI containers are the general-purpose factory.

Interview tip

🗣️ When this comes up

Distinguish Simple Factory (a method picks a type) from Factory Method (subclasses decide) and from Abstract Factory (families of products). The “why” is decoupling + Open/Closed. Bonus: note that in modern .NET the DI container usually replaces hand-written factories.
