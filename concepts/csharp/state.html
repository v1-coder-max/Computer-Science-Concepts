State Pattern

Let an object change its behavior when its state changes — so it acts like it switched class. Each state becomes an object that knows what’s allowed and where to go next.

#design-pattern #behavioral #state-machine #gof

The idea in plain English

Many objects behave differently depending on what state they’re in — an order that’s Pending allows cancel, but a Shipped order doesn’t. The naive version is a giant switch (status) in every method, repeated and fragile. The State pattern instead makes each state its own class implementing a common interface, and the context delegates behavior to its current state object. When an action causes a transition, the current state hands the context a new state object.

The result is a clean, object-oriented state machine: each state encapsulates its own behavior and its valid transitions, illegal actions are rejected in one obvious place, and adding a new state is a new class rather than edits to a dozen switch statements (Open/Closed).

State and Strategy have the same structure (a context delegating to a pluggable object), but different intent: strategies are independent algorithms chosen by the client; states know about each other and drive their own transitions over time. (In simple cases C#’s pattern matching / an enum can suffice; reach for the pattern when behavior and transitions get rich.)

Real-world example

🏢 How this works in production

An order’s lifecycle — Pending → Paid → Shipped → Delivered (with Cancelled branches) — is the textbook case: each state allows different actions and only legal transitions. Document approval workflows (Draft → Review → Published), vending machines, TCP connection states, and game character modes all model this way. Workflow engines and libraries like Stateless (and MassTransit’s saga state machines) are the State pattern formalized, so the rules live in one place instead of scattered conditionals.

States and transitions

In C#

C# 
public interface IOrderState { IOrderState Ship(); IOrderState Cancel(); }

public class PendingState : IOrderState
{
    public IOrderState Ship()   => throw new InvalidOperationException("Pay first");
    public IOrderState Cancel() => new CancelledState();          // legal transition
}
public class PaidState : IOrderState
{
    public IOrderState Ship()   => new ShippedState();           // transition to next state
    public IOrderState Cancel() => new CancelledState();
}
public class ShippedState : IOrderState
{
    public IOrderState Ship()   => throw new InvalidOperationException("Already shipped");
    public IOrderState Cancel() => throw new InvalidOperationException("Can't cancel shipped"); // illegal
}

// Context delegates to its current state — no giant switch anywhere.
public class Order
{
    private IOrderState _state = new PendingState();
    public void Ship()   => _state = _state.Ship();
    public void Cancel() => _state = _state.Cancel();
}

Key takeaways

🎯 Remember these three

  • State turns each state into an object; the context delegates to its current state, which drives transitions.
  • It replaces sprawling switch (status) logic with a clean, extensible state machine (illegal actions rejected in one place).
  • Same structure as Strategy, different intent: states know each other and transition; strategies are independent.

Interview tip

🗣️ When this comes up

When you see a method full of switch (status), the answer is often State. Explain it as an OO state machine and nail the State vs Strategy distinction (same shape; states transition and know each other, strategies are independent). The order lifecycle is your go-to example.
