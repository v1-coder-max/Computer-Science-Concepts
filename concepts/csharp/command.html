Command Pattern

Turn a request into a first-class object. Once an action is an object you can store it, queue it, log it, pass it around — and undo it.

#design-pattern #behavioral #undo #gof

The idea in plain English

Normally you call a method and it runs immediately. The Command pattern wraps that call — the receiver, the method, and its arguments — into a command object with an Execute() method (and often an Undo()). Because the action is now data, you gain superpowers:

  • Queue / schedule it — run later, on a background worker, or in order.
  • Log / persist it — replay or audit what happened.
  • Undo / redo — keep a history stack and reverse commands.
  • Decouple the invoker (the button, the queue) from the receiver (the thing that does the work) — the invoker just calls Execute().

The classic structure: an invoker holds commands and triggers them; each command knows its receiver and how to act on it. This is also the conceptual root of CQRS and libraries like MediatR, where a request object is dispatched to a handler.

Real-world example

🏢 How this works in production

Undo/redo in editors (text, design tools, IDEs) is the killer use: each edit is a command pushed onto a history stack, and Ctrl+Z pops one and calls Undo(). Task queues serialize commands and process them on workers (a job system). UI frameworks bind buttons/menus to command objects (WPF’s ICommand) so the same action works from a toolbar, a shortcut, or a context menu. And in modern .NET, MediatR commands (IRequest → handler) are the command pattern powering CQRS.

Invoker → command → receiver

In C#

C# 
public interface ICommand { void Execute(); void Undo(); }

// A concrete command knows its receiver and how to act + reverse.
public class AddTextCommand(Document doc, string text) : ICommand
{
    public void Execute() => doc.Append(text);
    public void Undo()    => doc.RemoveLast(text.Length);
}

// Invoker keeps a history → undo/redo for free.
public class Editor
{
    private readonly Stack<ICommand> _history = new();
    public void Run(ICommand cmd) { cmd.Execute(); _history.Push(cmd); }
    public void Undo() { if (_history.Count > 0) _history.Pop().Undo(); }
}

var editor = new Editor();
editor.Run(new AddTextCommand(doc, "Hello"));   // do
editor.Undo();                                   // undo (Ctrl+Z)

Key takeaways

🎯 Remember these three

  • Command wraps a request (receiver + action + args) as an object with Execute() (and often Undo()).
  • Because the action is data, you can queue, log, schedule, and undo it — and the invoker is decoupled from the receiver.
  • It’s the basis of undo/redo, task queues, UI ICommand, and MediatR/CQRS commands.

🗣️ When this comes up

“How would you implement undo/redo?” — Command, with a history stack and an Undo() per command. Stress that turning an action into an object is what enables queuing, logging, and undo, and that it decouples invoker from receiver. Tie it to MediatR/CQRS for modern .NET relevance.
