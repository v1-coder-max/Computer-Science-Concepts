Covariance & Contravariance

If a Cat is an Animal, is an IEnumerable<Cat> an IEnumerable<Animal>? Sometimes — and variance is the rule that says when.

#variance #generics #covariance #contravariance

The idea in plain English

Generics are invariant by default: List<Cat> is not a List<Animal>, even though Cat : Animal — because you could then add a Dog to your list of cats. Variance describes the cases where the substitution is safe, decided by whether T is produced (output) or consumed (input):

  • Covariance (out) — preserves the direction. An IEnumerable<Cat> is an IEnumerable<Animal>, because the interface only ever produces T (you take items out). Reading a cat as an animal is always safe.
  • Contravariance (in) — reverses the direction. An Action<Animal> can be used where an Action<Cat> is expected, because it only consumes T (you pass items in) — a method that handles any animal can certainly handle a cat.

The mnemonic: out = producer (covariant), in = consumer (contravariant). Library authors mark interface/delegate type parameters with out/in to enable it; that’s why IEnumerable<out T> is covariant and Action<in T>/ IComparer<in T> are contravariant. (A famous wart: arrays are covariant but unsafely — Animal[] a = new Cat[2]; a[0] = new Dog(); compiles but throws at runtime. Generics fixed that.)

Real-world example

🏢 How this works in production

Variance is the quiet reason a lot of code “just works.” You can pass a List<Cat> to a method taking IEnumerable<Animal> (covariance) — handy for LINQ and any read-only API. You can register one IComparer<Animal> and use it to sort Cats (contravariance). Event handlers and Action<T> callbacks rely on contravariance so a general handler accepts specific arguments. You mostly consume variance rather than declare it, but understanding it explains otherwise-mysterious “cannot convert” compiler errors.

out preserves, in reverses

In C#

C# 
class Animal { }  class Cat : Animal { }  class Dog : Animal { }

// Covariance (out): IEnumerable<out T> only PRODUCES T → Cat seq usable as Animal seq.
IEnumerable<Cat> cats = new List<Cat>();
IEnumerable<Animal> animals = cats;          // ✅ covariant

// Contravariance (in): Action<in T> only CONSUMES T → handler of Animal handles Cat.
Action<Animal> feedAnything = a => { /* feed any animal */ };
Action<Cat> feedCat = feedAnything;           // ✅ contravariant

// Invariant by default — and for good reason:
// List<Animal> bad = new List<Cat>();        // ❌ won't compile (could Add a Dog)

// ⚠ Arrays ARE covariant, but UNSAFELY (a historical wart):
Animal[] arr = new Cat[2];
// arr[0] = new Dog();                          // compiles, throws ArrayTypeMismatchException

Key takeaways

🎯 Remember these three

  • Generics are invariant by default; variance allows safe substitution based on producer/consumer roles.
  • Covariance (out) = output/producer, preserves direction (IEnumerable<out T>); contravariance (in) = input/consumer, reverses it (Action<in T>).
  • Mnemonic: out = producer, in = consumer. Arrays are covariant but unsafely (runtime exception) — generics fixed that.
