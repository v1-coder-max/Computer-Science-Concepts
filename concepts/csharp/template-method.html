Template Method Pattern

Pin down the shape of an algorithm in a base class, but leave specific steps for subclasses to fill in. The base method calls your overrides — not the other way around.

#design-pattern #behavioral #inheritance #gof

The idea in plain English

When several processes follow the same overall steps but differ in the details, the Template Method pattern captures the fixed sequence once in a base-class method (the “template method”), and defines the variable steps as abstract or virtual methods that subclasses override. The template method calls those steps in order; subclasses can’t change the structure, only fill in the blanks.

This is the Hollywood Principle: “don’t call us, we’ll call you.” Your subclass doesn’t drive the flow — the framework’s base method does, calling back into your overridden hooks at the right moments. It eliminates duplicated skeleton code and guarantees every variant runs the steps in the correct order (e.g. always validate before save).

Template Method is the inheritance-based cousin of Strategy: both vary part of an algorithm, but Strategy injects the variable behavior via composition (a strategy object/delegate), while Template Method varies it via subclassing. Composition is usually preferred — reach for Template Method when there’s a genuine is-a base with a fixed lifecycle.

Real-world example

🏢 How this works in production

ASP.NET Core’s BackgroundService is template method in the framework: the base class owns the start/stop lifecycle and calls your ExecuteAsync — you fill in the work, it controls the flow. Data-import pipelines do it too: a base Importer runs open → parse → validate → save → close in a fixed order, and a CsvImporter vs XmlImporter only override Parse. Test frameworks (Setup/ TearDown around your test) and ADO.NET base providers use the same shape.

Fixed skeleton, pluggable steps

In C#

C# 
public abstract class DataImporter
{
    // The TEMPLATE METHOD: fixed sequence, sealed so the structure can't change.
    public void Import(string path)
    {
        var raw = Open(path);
        var rows = Parse(raw);     // ← the variable step (hook)
        Validate(rows);
        Save(rows);
    }

    protected string Open(string path) => File.ReadAllText(path);   // fixed
    protected abstract IEnumerable<Row> Parse(string raw);          // subclass fills in
    protected virtual void Validate(IEnumerable<Row> rows) { }      // optional hook
    protected void Save(IEnumerable<Row> rows) { /* fixed */ }
}

public class CsvImporter : DataImporter
{
    protected override IEnumerable<Row> Parse(string raw) => /* split CSV */ default!;
}
// new CsvImporter().Import("data.csv");  // base controls the flow, calls Parse()

Key takeaways

🎯 Remember these three

  • Template Method fixes an algorithm’s skeleton in a base method and defers variable steps to subclass hooks.
  • It’s the Hollywood Principle — the base calls your overrides; subclasses can’t change the structure, only the steps.
  • It’s the inheritance-based counterpart to Strategy (composition); prefer Strategy unless there’s a real is-a lifecycle.

Interview tip

🗣️ When this comes up

“Template Method vs Strategy?” Same goal (vary part of an algorithm), different mechanism: Template Method uses inheritance + overridable hooks; Strategy uses composition. Mention the Hollywood Principle and BackgroundService.ExecuteAsync as a real framework example.
