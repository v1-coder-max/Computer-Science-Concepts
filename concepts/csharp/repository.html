Repository Pattern

A collection-like abstraction over your data store, so the domain says orders.Add(order) and never knows whether SQL, an ORM, or an in-memory list is behind it.

#repository #data-access #abstraction #testability

The idea in plain English

A Repository sits between your domain/business logic and the data-mapping layer, exposing a small, collection-like interface: Add, GetById, Remove, List. The domain works with objects and never sees SQL, connection strings, or ORM types. The usual goals:

  • Decoupling — business logic depends on IOrderRepository, not on EF Core/Dapper/SQL, so persistence can change without touching the domain.
  • Testability — unit tests inject an in-memory repository instead of hitting a real database.
  • Centralized data access — query logic lives in one place rather than smeared across services.

The important caveat in .NET: EF Core’s DbContext + DbSet<T> are already a Repository and Unit of Work. DbSet is a collection-like repository and SaveChanges is the unit of work. Wrapping them in another generic repository often adds a leaky, low-value layer that just forwards calls — and can hide EF’s power (Include, projections, IQueryable). So use it deliberately: a repository per aggregate with intent-revealing methods (GetActiveOrdersForCustomer) is valuable; a generic Repository<T> over EF is usually redundant.

Real-world example

🏢 How this works in production

In a domain-driven codebase, an IOrderRepository exposes meaningful operations (GetByIdWithLines, GetOpenOrders) and hides EF Core behind them, so the application layer reads like business language and tests run against a fake repository — no database needed. The anti-pattern teams regret is a generic IRepository<T> with GetAll() over every entity: it leaks IQueryable (or pulls whole tables into memory), defeats EF’s query translation, and adds a layer that earns its keep nowhere.

Domain ↔ store, abstracted

In C#

C# 
// ✅ Repository per aggregate with intent-revealing methods (worth it).
public interface IOrderRepository
{
    Task<Order?> GetByIdAsync(Guid id);
    Task<IReadOnlyList<Order>> GetOpenOrdersAsync(Guid customerId);
    void Add(Order order);
}

public class EfOrderRepository(AppDbContext db) : IOrderRepository
{
    public Task<Order?> GetByIdAsync(Guid id) =>
        db.Orders.Include(o => o.Lines).FirstOrDefaultAsync(o => o.Id == id);

    public async Task<IReadOnlyList<Order>> GetOpenOrdersAsync(Guid customerId) =>
        await db.Orders.Where(o => o.CustomerId == customerId && o.IsOpen).ToListAsync();

    public void Add(Order order) => db.Orders.Add(order);   // SaveChanges = unit of work
}

// ⚠ Often redundant: a generic repo over EF that just forwards to DbSet<T>.
// public interface IRepository<T> { IQueryable<T> GetAll(); ... }  // leaky, low value

Key takeaways

🎯 Remember these three

  • Repository gives a collection-like interface over data access, decoupling the domain from the DB/ORM and aiding testability.
  • EF Core’s DbContext/DbSet already are a repository + unit of work — wrapping them generically is often redundant and leaky.
  • When you do use it, prefer a repository per aggregate with intent-revealing methods over a generic Repository<T>.
