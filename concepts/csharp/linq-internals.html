LINQ Internals

LINQ feels like magic, but it’s two simple ideas: deferred execution and the split between IEnumerable (runs in memory) and IQueryable (translated to SQL).

#linq #deferred-execution #iqueryable #expression-trees

The idea in plain English

A LINQ query doesn’t run when you write it — it builds a chain of operators. The work happens only when you enumerate the result (a foreach, ToList(), Count(), First()). This is deferred (lazy) execution: define now, execute later, element by element. It’s powerful (compose queries, stop early) and a common source of bugs if you forget it.

The other half is where the query runs, decided by the static type:

  • IEnumerable<T> (LINQ to Objects) — operators take compiled delegates and run in memory, pulling items via iterators (yield). Use it on collections already in memory.
  • IQueryable<T> — operators take expression trees (data describing the code, not the compiled code). A provider (EF Core) walks that tree and translates it to SQL executed by the database.

That boundary is why db.Orders.Where(o => o.IsOpen) becomes a SQL WHERE, but calling .ToList() first pulls the whole table into memory and then filters in C#. Two classic traps: crossing to IEnumerable too early (losing SQL translation), and multiple enumeration — iterating the same deferred query twice re-runs it (two DB trips); materialize once with ToList() if you’ll use it repeatedly.

Real-world example

🏢 How this works in production

A query like db.Orders.Where(o => o.CustomerId == id).OrderByDescending(o => o.Date).Take(10) ships to the database as a single efficient SQL statement — because it’s IQueryable and EF translates the expression tree. Accidentally calling .AsEnumerable() before the Where would instead download every order and filter in app memory — a performance landmine. And a deferred query enumerated inside a loop (or logged then used) silently hits the DB multiple times; adding ToList() once is the fix. Understanding deferred execution and the IEnumerable/IQueryable line is what separates fast LINQ from slow LINQ.

Build the chain, execute on enumeration

In C#

C# 
// Deferred: this line runs NOTHING — it builds a query object.
var q = numbers.Where(n => { Console.WriteLine($"test {n}"); return n > 2; });
// "test ..." prints only here, when we enumerate:
var list = q.ToList();

// IQueryable → translated to SQL by EF (efficient, filtered in the DB).
var open = await db.Orders.Where(o => o.IsOpen).Take(10).ToListAsync();

// ❌ Crossing to IEnumerable too early pulls the whole table, then filters in memory.
var bad = db.Orders.AsEnumerable().Where(o => o.IsOpen);   // downloads everything

// ⚠ Multiple enumeration: each use re-runs the query (two DB round trips here).
IEnumerable<Order> deferred = db.Orders.Where(o => o.IsOpen);
var count = deferred.Count();    // query #1
var first = deferred.First();    // query #2 — materialize once with ToList() to avoid this

Key takeaways

🎯 Remember these three

  • LINQ uses deferred execution — the query runs when enumerated, not when defined.
  • IEnumerable runs in memory via delegates/iterators; IQueryable builds expression trees a provider (EF) translates to SQL.
  • Don’t cross to IEnumerable too early (loses SQL translation), and avoid multiple enumeration (re-runs the query) — ToList() to materialize.
