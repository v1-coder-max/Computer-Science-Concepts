Delegates & Events

A delegate is a type-safe reference to a method — a function you can store in a variable and pass around. An event is a delegate a class exposes for others to subscribe to.

#delegates #events #func-action #lambdas

The idea in plain English

A delegate is a type-safe function pointer: a variable that holds a reference to a method with a matching signature, so you can pass behavior around like data and call it later. You rarely declare delegate types by hand anymore — the built-ins cover almost everything:

  • Action / Action<T> — a method that returns void.
  • Func<…, TResult> — a method that returns a value.
  • Predicate<T> — a Func<T, bool> (a test).

A lambda (x => x * 2) is just a concise way to create a delegate instance. Delegates are also multicast: one delegate can hold an invocation list of several methods and call them all (this is what events use).

An event is a restricted delegate that a class publishes. Outside code can only subscribe (+=) and unsubscribe (-=); only the declaring class can raise (invoke) it. That encapsulation is the whole difference between a public delegate field (anyone can clobber or invoke it) and an event (safe pub/sub). Delegates and events underpin callbacks, the observer pattern, LINQ, and async continuations.

Real-world example

🏢 How this works in production

Every UI button click, every PropertyChanged in MVVM, every OrderPlaced domain event is a C# event — a publisher raising it, subscribers reacting. LINQ is built on delegates: list.Where(x => x.IsActive) passes a Func as the filter. Background work uses Action callbacks, and Task.ContinueWith takes a delegate. Delegates are how C# treats behavior as a first-class value, which is the basis for most of the language’s expressiveness.

Delegate = invocation list; event = guarded pub/sub

In C#

C# 
// Delegates as values (Func/Action), created from lambdas:
Func<int, int> square = x => x * x;           // holds a method
Action<string> log = Console.WriteLine;        // method group → delegate
int four = square(2);

// Multicast: one delegate, many methods.
Action steps = () => Console.Write("a");
steps += () => Console.Write("b");
steps();                                        // calls both → "ab"

// Passing behavior to a method (this is exactly how LINQ works):
var actives = people.Where(p => p.IsActive);    // a Func is the filter

// An EVENT: subscribers can += / -=, but only Stock can raise it.
public class Stock
{
    public event EventHandler<decimal>? PriceChanged;
    public void Set(decimal p) => PriceChanged?.Invoke(this, p);  // only the owner raises
}
var s = new Stock();
s.PriceChanged += (sender, price) => Console.WriteLine(price);    // subscribe

Key takeaways

🎯 Remember these three

  • A delegate is a type-safe method reference (use Func/Action/Predicate); lambdas create them; they’re multicast.
  • An event is a restricted delegate: outsiders can only subscribe/unsubscribe (+=/-=); only the declaring class can raise it.
  • They make behavior a first-class value — the basis of callbacks, the observer pattern, and LINQ.

Interview tip

🗣️ When this comes up

“Delegate vs event?” A delegate is a method reference you can pass and invoke; an event wraps a delegate so only the owner can raise it while others subscribe — encapsulated pub/sub. Mention Func/Action, multicast, and that events are the observer pattern in C#.
