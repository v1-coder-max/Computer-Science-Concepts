Specification Pattern

Turn a business rule like “active premium customers” into a reusable, composable object you can test, combine, and hand to a query — instead of copy-pasting the same Where clause.

#specification #composable #query #modern-pattern

The idea in plain English

The same selection rule often appears in many places — a filter in a query, a validation check, an authorization decision. Duplicating x => x.IsActive && x.Tier == "Premium" across the codebase means a rule change has to be hunted down everywhere. The Specification pattern names that rule as a single object that answers “does this entity satisfy me?” — via an IsSatisfiedBy(entity) check and/or an expression the query layer can apply.

The two big wins:

  • One definition, reused — the rule lives in one class, used for both in-memory checks and database queries.
  • Composability — combine specifications with And, Or, Not to build complex criteria from simple, named parts.

For database use, the specification exposes a Expression<Func<T,bool>> so EF Core can translate it to SQL (not just filter in memory). It pairs naturally with the Repository pattern — repo.ListAsync(new ActivePremiumSpec()) — keeping query logic out of the data layer. The Ardalis.Specification library is the popular ready-made implementation in .NET.

Real-world example

🏢 How this works in production

An e-commerce backend defines specs like InStockSpec, PriceUnderSpec(max), and OnSaleSpec, then composes them per request — InStock.And(OnSale) — to build product filters without writing a new query each time. The same ActiveCustomerSpec guards a query and a business-rule check, so the definition of “active” can never drift between them. Teams using Ardalis.Specification hand specs straight to the repository, which applies the expression (translated to SQL) plus includes/ordering, keeping controllers and the data layer clean.

Compose simple specs into complex criteria

In C#

C# 
public interface ISpecification<T>
{
    Expression<Func<T, bool>> ToExpression();             // for EF → SQL
    bool IsSatisfiedBy(T entity) => ToExpression().Compile()(entity); // in-memory check
}

public class ActiveCustomerSpec : ISpecification<Customer>
{
    public Expression<Func<Customer, bool>> ToExpression() => c => c.IsActive;
}
public class PremiumSpec : ISpecification<Customer>
{
    public Expression<Func<Customer, bool>> ToExpression() => c => c.Tier == "Premium";
}

// Compose, then apply once — same rule for query AND validation.
var spec = new ActiveCustomerSpec().And(new PremiumSpec());     // And/Or/Not combinators
var premium = await db.Customers.Where(spec.ToExpression()).ToListAsync(); // translated to SQL
bool ok = spec.IsSatisfiedBy(someCustomer);                    // reused in-memory
// (Ardalis.Specification provides this plus Includes/ordering out of the box.)

Key takeaways

🎯 Remember these three

  • Specification encapsulates a business rule/query as a reusable object (IsSatisfiedBy + an expression).
  • Specs compose with And/Or/Not, eliminating duplicated Where predicates and keeping rules in one place.
  • Expose an Expression<Func<T,bool>> so EF translates it to SQL; pairs with Repository (Ardalis.Specification).
