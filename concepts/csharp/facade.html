Facade Pattern

One simple front door to a complicated house. A facade gives clients a single, easy interface and hides the tangle of subsystem classes behind it.

#design-pattern #structural #simplification #gof

The idea in plain English

Some tasks require orchestrating many classes in the right order. If every caller has to know all of them, your subsystem’s complexity leaks everywhere and everything is tightly coupled to its internals. A Facade wraps that subsystem in one class that exposes a few high-level methods (PlaceOrder(...)) and does the coordinating internally.

Crucially, a facade only simplifies — it doesn’t add new behavior (that’s a decorator) or change an interface to match an existing expectation (that’s an adapter). It’s a convenience layer. Clients can still reach the subsystem directly if they need fine control; the facade just covers the common cases and reduces coupling for everyone else.

It’s one of the most common patterns precisely because it’s so natural: any time you write a “service” or “manager” class that hides several lower-level steps behind a clean method, you’ve built a facade.

Real-world example

🏢 How this works in production

Placing an order really means: validate the cart, reserve inventory, charge payment, create a shipment, and send a confirmation — five subsystems. Rather than make a controller juggle all five, an OrderService facade exposes PlaceOrderAsync(cart) and coordinates them inside. The web layer calls one method and stays blissfully unaware of the choreography. .NET is full of facades: HttpClient hides sockets, TLS, and HTTP parsing behind GetAsync; the whole point is a small, friendly surface over a big, messy one.

One door, many rooms

In C#

C# 
// Facade: one friendly method hides the multi-step subsystem coordination.
public class OrderService(
    IInventory inventory, IPayments payments, IShipping shipping, IEmail email)
{
    public async Task<OrderResult> PlaceOrderAsync(Cart cart)
    {
        await inventory.ReserveAsync(cart);          // step 1
        var charge = await payments.ChargeAsync(cart); // step 2
        var shipment = await shipping.ScheduleAsync(cart); // step 3
        await email.SendConfirmationAsync(cart, shipment); // step 4
        return new OrderResult(charge, shipment);
    }
}

// The caller doesn't know (or care) about the four subsystems:
var result = await _orders.PlaceOrderAsync(cart);

Key takeaways

🎯 Remember these three

  • A facade exposes one simple interface over a complex subsystem, reducing coupling to its internals.
  • It only simplifies — unlike a decorator (adds behavior) or adapter (changes the interface).
  • Any “service”/“manager” class that hides several lower-level steps is a facade; clients can still go deeper if needed.

Interview tip

🗣️ When this comes up

Define facade as simplification of a subsystem, and contrast the wrapper trio: facade simplifies, adapter converts an interface, decorator adds behavior. Note that an API gateway is a facade at the system level — a nice cross-link to show range.
