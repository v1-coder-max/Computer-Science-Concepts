Unit of Work

Collect all the changes a business operation makes, then commit them as one atomic transaction — so several writes succeed or fail together.

#unit-of-work #transaction #data-access #ef-core

The idea in plain English

A single business action often touches several entities — place an order, decrement inventory, write a ledger entry. You don’t want those persisted one-by-one (a crash mid-way leaves inconsistent data). The Unit of Work pattern keeps a list of everything that changed during the operation and, on Commit(), writes them all out inside one database transaction — all or nothing. It also coordinates so multiple repositories share the same transaction.

The headline for .NET developers: EF Core’s DbContext is a Unit of Work. Its change tracker records every add/modify/delete, and SaveChanges() commits them together in a transaction. So you usually don’t implement the pattern yourself — you use it every time you call SaveChanges.

A custom IUnitOfWork is still occasionally useful: to let several repositories share one DbContext/transaction behind an abstraction, or to express “commit” explicitly at the application boundary. But like Repository, wrapping DbContext in a thin Unit of Work that just forwards to SaveChanges is often redundant ceremony.

Real-world example

🏢 How this works in production

When you place an order, the service adds the order, updates stock, and records a payment intent — all tracked by one DbContext — and a single SaveChangesAsync() commits them in one transaction. If the payment row violates a constraint, the whole thing rolls back; you never get an order with stock removed but no payment. That’s Unit of Work doing its job invisibly. Teams that add an explicit IUnitOfWork usually do it to keep the “when do we commit?” decision in the application layer rather than scattered across repositories.

Track changes, commit once

In C#

C# 
// EF Core: the DbContext already IS a unit of work.
public async Task PlaceOrderAsync(Order order)
{
    db.Orders.Add(order);                       // tracked: Added
    var stock = await db.Stock.FindAsync(order.Sku);
    stock!.Quantity -= order.Qty;               // tracked: Modified
    db.Payments.Add(new Payment(order.Id));     // tracked: Added

    await db.SaveChangesAsync();   // commits ALL three in ONE transaction (all-or-nothing)
}

// An explicit IUnitOfWork is mostly a thin wrapper — useful to share one
// transaction across repositories or to express commit at the app boundary:
public interface IUnitOfWork { Task<int> CommitAsync(CancellationToken ct = default); }
public class EfUnitOfWork(AppDbContext db) : IUnitOfWork
{
    public Task<int> CommitAsync(CancellationToken ct = default) => db.SaveChangesAsync(ct);
}

Key takeaways

🎯 Remember these three

  • Unit of Work tracks all changes in an operation and commits them as one atomic transaction (all-or-nothing).
  • EF Core’s DbContext + SaveChanges are a Unit of Work — you use it constantly without naming it.
  • A custom IUnitOfWork helps share a transaction across repositories, but a thin wrapper over DbContext is often redundant.

Interview tip

🗣️ When this comes up

Define it (one transaction for many changes), then the key .NET insight: DbContext + SaveChanges is the Unit of Work (and DbSet is the Repository). So you rarely build it yourself — and a thin custom wrapper adds little. That pairing with Repository is the expected answer.
