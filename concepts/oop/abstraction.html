Abstraction

Show the essential what, hide the messy how. Callers depend on a simple contract and never see the complexity behind it.

#oop #abstraction #interfaces #contracts

The idea in plain English

Abstraction is modeling a thing by the operations that matter, while hiding the implementation. You drive a car with a wheel and two pedals — you don’t think about fuel injection. The pedals are the abstraction; the engine is the hidden detail. In code, an interface or abstract class is that simplified surface: it declares what can be done (Pay(amount)) without saying how (Stripe? PayPal? cash?).

The practical win is that the rest of your system depends on the contract, not a concrete type. Swap the implementation, add a new one, or fake it in a test — callers don’t change. That’s abstraction enabling the Dependency Inversion and Open/Closed ideas from SOLID.

Don’t over-abstract, though: an interface with one implementation that will never have another is just ceremony. Abstraction earns its keep when there’s genuine variation to hide.

Real-world example

🏢 How this works in production

.NET’s Stream is abstraction done right: your code reads and writes bytes through one simple surface, whether the bytes live in a file (FileStream), memory (MemoryStream), or over the network (NetworkStream). A logging library exposes ILogger so Netflix-scale services can swap the sink (console, file, Elasticsearch) without touching a single call site. The complexity is real; the caller just never has to see it.

One contract, hidden details

In C#

C# 
// The abstraction: WHAT can be done, not HOW.
public interface IPaymentProcessor
{
    Task<Receipt> PayAsync(decimal amount, string currency);
}

// Implementations hide wildly different complexity behind the same surface.
public class StripeProcessor : IPaymentProcessor
{
    public async Task<Receipt> PayAsync(decimal amount, string currency)
    {
        // tokens, idempotency keys, HTTP calls, retries… none of it leaks out
        return new Receipt(/* ... */);
    }
}

// The caller depends only on the abstraction — swap or fake freely.
public class Checkout(IPaymentProcessor payments)
{
    public Task<Receipt> CompleteAsync(decimal total) =>
        payments.PayAsync(total, "USD");
}

Key takeaways

🎯 Remember these three

  • Abstraction exposes the essential operations and hides the implementation behind a contract.
  • Interfaces/abstract classes let callers depend on the “what,” so implementations are swappable and testable.
  • Only abstract where there’s real variation — an interface with one forever-implementation is noise.

Interview tip

🗣️ When this comes up

Be ready to distinguish abstraction from encapsulation crisply: abstraction hides complexity behind a model (the what); encapsulation hides state and bundles it with its guards. Give a concrete contract example (Stream, ILogger) and note it powers Dependency Inversion.
