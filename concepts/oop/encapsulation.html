Encapsulation

Bundle data with the code that protects it, and expose only a controlled surface — so an object can guarantee its own rules.

#oop #encapsulation #invariants #access-modifiers

The idea in plain English

Encapsulation is the first pillar of OOP: keep an object’s state private and let the outside world touch it only through methods/properties the object controls. The fields are the object’s business; the public surface is the contract. That control is what lets a class enforce an invariant — a rule that must always hold (a balance can’t go negative, an email is always valid).

In C# you express it with access modifiers (private, protected, internal, public) and properties that wrap fields with validation. If anyone could reach in and set the field directly, the rule could be broken from anywhere; with encapsulation, every change funnels through one place you can trust.

It’s often confused with abstraction. Encapsulation hides the data and how it’s protected; abstraction hides complexity behind a simpler model. They work together but answer different questions.

Real-world example

🏢 How this works in production

A banking core at any fintech models an account so that Balance can never be set directly — you call Deposit or Withdraw, which reject negative amounts and overdrafts in one place. Multiply that across a system: every rule lives with the data it guards, so a careless caller elsewhere physically can’t corrupt the balance. That’s the same reason .NET’s own types (List<T> hiding its internal array, DateTime being immutable) keep their fields private.

A guarded object

In C#

C# 
public class BankAccount
{
    private decimal _balance;                 // hidden state — no one can poke it directly

    public decimal Balance => _balance;       // read-only to the outside

    public void Deposit(decimal amount)
    {
        if (amount <= 0) throw new ArgumentOutOfRangeException(nameof(amount));
        _balance += amount;                   // the invariant is enforced in ONE place
    }

    public void Withdraw(decimal amount)
    {
        if (amount <= 0) throw new ArgumentOutOfRangeException(nameof(amount));
        if (amount > _balance) throw new InvalidOperationException("Insufficient funds");
        _balance -= amount;
    }
}

var acct = new BankAccount();
acct.Deposit(100);
// acct._balance = -50;   // ❌ won't compile: the field is private, the rule can't be bypassed

Key takeaways

🎯 Remember these three

  • Keep state private and expose a controlled surface so the object enforces its own invariants.
  • Properties + access modifiers funnel every change through one trustworthy place.
  • Encapsulation hides data; abstraction hides complexity — related but distinct.

Interview tip

🗣️ When this comes up

“Encapsulation vs abstraction?” Encapsulation = hiding state and bundling it with the code that protects it (the how and the data); abstraction = exposing a simple model and hiding complexity (the what). Mention that the real payoff is enforcing invariants in one place.
