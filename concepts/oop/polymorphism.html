Polymorphism

“Many forms”: call the same method on different types and each does the right thing — so you can write code against an abstraction, not a pile of ifs.

#oop #polymorphism #dynamic-dispatch #open-closed

The idea in plain English

Polymorphism lets one name mean different behaviors depending on the actual object. You hold a Shape reference and call shape.Area(); if it’s really a Circle, the circle’s formula runs; if a Square, the square’s. The calling code never branches on type — it just trusts the contract.

Two flavors:

  • Runtime (subtype) polymorphism — via virtual/override. The runtime picks the right override based on the object’s actual type (dynamic dispatch, implemented with a method table / vtable).
  • Compile-time polymorphism — method overloading (same name, different parameters) and generics, resolved by the compiler.

This is the mechanism behind Open/Closed: add a new Shape subtype and the loop that draws shapes keeps working unchanged. Eliminating switch (type) ladders in favor of polymorphic dispatch is one of the biggest readability and maintainability wins in OOP.

Real-world example

🏢 How this works in production

A reporting or rendering engine keeps a List<IReportElement> and calls Render() on each — tables, charts, and text all render themselves without the engine knowing their concrete types. Payment systems do the same with IPaymentMethod.Charge(). Even object.ToString() is polymorphism: Console.WriteLine(x) calls whatever ToString() override x’s real type provides. Adding a new element/method type never touches the loop that uses them.

One call, the right behavior

In C#

C# 
public abstract class Shape
{
    public abstract double Area();           // each subtype must define it
}

public class Circle(double r) : Shape
{
    public override double Area() => Math.PI * r * r;
}
public class Square(double s) : Shape
{
    public override double Area() => s * s;
}

// The consumer is written ONCE against Shape — runtime polymorphism does the rest.
double TotalArea(IEnumerable<Shape> shapes)
{
    double total = 0;
    foreach (Shape s in shapes)
        total += s.Area();                   // dispatched to Circle/Square at runtime
    return total;
}
// Adding a Triangle : Shape needs ZERO changes here (Open/Closed).

// Compile-time polymorphism = overloading (resolved by the compiler):
void Log(string s) { }
void Log(int n)    { }

Key takeaways

🎯 Remember these three

  • Polymorphism = one interface, many behaviors; the call resolves to the object’s actual type.
  • Runtime (virtual/override, dynamic dispatch) vs compile-time (overloading, generics).
  • It enables Open/Closed — add new types without editing the code that consumes them, killing switch(type) ladders.

Interview tip

🗣️ When this comes up

Distinguish runtime polymorphism (virtual/override, dynamic dispatch via a vtable) from compile-time (overloading, generics). The strongest answer connects it to Open/Closed: replacing type-checking switch statements with polymorphic dispatch so new cases don’t modify existing code.
