Law of Demeter

“Only talk to your immediate friends.” Don’t reach through one object to fiddle with another’s internals — it quietly couples you to a whole object graph.

#oop #law-of-demeter #least-knowledge #coupling

The idea in plain English

The Law of Demeter (a.k.a. the principle of least knowledge) says a method should only call methods on a small circle of “friends”:

  • the object itself (this),
  • objects passed in as parameters,
  • objects it creates,
  • its own direct fields/components.

What it warns against is the “train wreck”: order.GetCustomer().GetAddress().GetCity().ToUpper(). Each . reaches one level deeper into someone else’s structure, so your code now silently depends on Customer and Address and City. Change any of their shapes and this line breaks. It’s coupling by chaining.

The fix is usually “tell, don’t ask”: instead of asking an object for its parts and doing the work yourself, ask the object to do it. Add a method where the data lives (order.GetShippingCity()) so callers talk only to order. (Caveat: fluent builders and LINQ chains aren’t violations — those return the same kind of object, not someone else’s internals.)

Real-world example

🏢 How this works in production

A pricing service that digs cart.GetUser().GetAccount().GetTier().GetDiscount() is one refactor away from a cascade of breakages — rename anything in that chain and every caller shatters. Teams that respect Demeter expose intent-revealing methods (cart.GetApplicableDiscount()) so the traversal stays hidden inside the objects that own the data. The result is fewer ripple-effect changes and tests that don’t need to build a five-level-deep object graph just to exercise one rule.

Train wreck vs delegation

In C#

C# 
// ❌ Violation: this method knows the entire object graph's shape.
string city = order.Customer.Address.City.ToUpper();
// breaks if Customer, Address, or City changes structure or nullability

// ✅ Tell, don't ask: expose intent on the object that owns the data.
public class Order
{
    private readonly Customer _customer;
    public string ShippingCityUpper() => _customer.ShippingCityUpper();  // delegate inward
}
public class Customer
{
    private readonly Address _address;
    public string ShippingCityUpper() => _address.City.ToUpperInvariant();
}

// Caller now depends only on Order:
string city2 = order.ShippingCityUpper();

// Not a violation — LINQ/fluent chains return the same kind of thing:
var names = people.Where(p => p.IsActive).Select(p => p.Name).ToList();

Key takeaways

🎯 Remember these three

  • Talk only to immediate friends (self, parameters, things you create, your fields) — not their friends.
  • Train-wreck chains (a.GetB().GetC().Do()) couple you to a whole object graph; “tell, don’t ask” instead.
  • Fluent/LINQ chains returning the same type aren’t violations — it’s about reaching into others’ internals.

Interview tip

🗣️ When this comes up

Recognize a “train wreck” when asked to spot a smell: a long a.b.c.d chain. Explain it couples the caller to every type in the chain and fix it with “tell, don’t ask.” Knowing the LINQ/fluent exception shows you understand the why, not just the rule.
