Coupling & Cohesion

Two forces that decide how maintainable your code is: how dependent modules are on each other (coupling) and how focused each one is (cohesion).

#oop #coupling #cohesion #design

The idea in plain English

Coupling measures how tightly two modules depend on each other. When class A directly news up a concrete class B, knows its internals, and breaks when B changes, that’s tight coupling. Loose coupling means A depends only on a small, stable contract (an interface), so B can change freely behind it. Goal: low coupling.

Cohesion measures how well the parts of a single module belong together. A class that does one well-defined job — and everything in it serves that job — is highly cohesive. A “god class” that handles parsing, persistence, emailing, and formatting has low cohesion. Goal: high cohesion.

The mantra is “loose coupling, high cohesion.” They reinforce each other: a focused class naturally has fewer, cleaner dependencies. This is also why Single Responsibility (cohesion) and Dependency Inversion (coupling) from SOLID are so central — they’re these two forces with names.

Real-world example

🏢 How this works in production

Well-designed microservices are the macro version of this: each service owns one bounded capability (high cohesion) and talks to others only through versioned APIs/events (loose coupling), so a team can rewrite their service’s internals without coordinating a big-bang release. The same instinct at the class level — splitting a sprawling OrderManager into focused OrderValidator, OrderRepository, and OrderNotifier classes, wired by interfaces — is what keeps a codebase changeable as it grows.

The two dials

In C#

C# 
// ❌ Tight coupling + low cohesion: one class does everything and hard-wires concretes.
public class OrderManager
{
    private readonly SqlConnection _db = new("...");
    private readonly SmtpClient _smtp = new();
    public void Place(Order o) { /* validate + save SQL + send email + format receipt */ }
}

// ✅ High cohesion (each class, one job) + loose coupling (depend on interfaces).
public interface IOrderRepository { Task AddAsync(Order o); }
public interface INotifier        { Task NotifyAsync(string to, string msg); }

public class OrderService(IOrderRepository repo, INotifier notifier)  // depends on contracts
{
    public async Task PlaceAsync(Order o)
    {
        await repo.AddAsync(o);
        await notifier.NotifyAsync(o.Email, "Thanks!");
    }
}

Key takeaways

🎯 Remember these three

  • Coupling = inter-module dependence (want it low); cohesion = intra-module focus (want it high).
  • “Loose coupling, high cohesion” — focused classes wired by stable contracts are easy to change and test.
  • SRP attacks cohesion; Dependency Inversion attacks coupling — SOLID is largely about these two forces.

Interview tip

🗣️ When this comes up

Define both crisply and state the goal in one breath: low coupling, high cohesion. Give the “god class” as the low-cohesion smell and “depends on a concrete type” as the tight-coupling smell, then connect them to SRP and Dependency Inversion.
