DRY, KISS & YAGNI

Three short rules that keep code honest: one source of truth, the simplest thing that works, and don’t build what you don’t need yet.

#dry #kiss #yagni #pragmatism

The idea in plain English

  • DRY — Don’t Repeat Yourself. Every piece of knowledge should have a single, authoritative home. Duplicated logic means a bug fixed in one copy and missed in three others.
  • KISS — Keep It Simple, Stupid. Prefer the most straightforward solution that solves the actual problem. Cleverness is a cost; simple code is read, debugged, and changed far more often than it’s written.
  • YAGNI — You Aren’t Gonna Need It. Don’t build features, configuration knobs, or abstractions for a future that may never arrive. Add them when a real requirement shows up.

They’re heuristics, not laws, and they pull against each other — which is the whole point. Chasing DRY too hard couples unrelated code behind a “shared” abstraction and violates KISS; the saying “duplication is far cheaper than the wrong abstraction” is the warning. DRY is about removing duplicated knowledge, not every line that happens to look similar. Use judgment: a little repetition is fine until the pattern is real.

Real-world example

🏢 How this works in production

Every team has felt all three: a tax-calculation rule copy-pasted into five services, then changed in only four (a DRY failure that ships wrong invoices); a “flexible” plugin framework built for integrations that never materialized (a YAGNI failure that no one can maintain); and a clever one-line LINQ expression nobody can debug at 2am (a KISS failure). The senior move is balance — de-duplicate genuine business rules, keep the design no more complex than the requirements demand, and resist building for hypothetical futures.

Three heuristics in tension

In C#

C# 
// ❌ Not DRY: the discount rule (knowledge) is duplicated and will drift.
decimal a = price - price * 0.10m;   // here
decimal b = total - total * 0.10m;   // and here... change one, forget the other

// ✅ DRY: one authoritative home for the rule.
static decimal ApplyLoyaltyDiscount(decimal amount) => amount * 0.90m;

// ✅ KISS: a clear loop beats a cryptic one-liner you can't debug later.
// ✅ YAGNI: no IDiscountStrategyFactoryProvider until a SECOND discount type actually exists.

// ⚠️ Over-DRY: don't merge two rules that only *look* alike today.
//    If "loyalty" and "clearance" discounts are different *concepts*, keep them
//    separate even at 90% — coupling them behind one method is the wrong abstraction.

Key takeaways

🎯 Remember these three

  • DRY = one home for each piece of knowledge; KISS = simplest solution that works; YAGNI = don’t build for an imagined future.
  • They’re heuristics in tension — over-applying one hurts the others.
  • DRY targets duplicated knowledge, not look-alike code: the wrong abstraction is costlier than a little duplication.

Interview tip

🗣️ When this comes up

Define all three quickly, then show maturity by naming the tension: aggressive DRY creates premature/wrong abstractions, and YAGNI is the antidote to over-engineering. Quoting “duplication is cheaper than the wrong abstraction” signals you’ve been burned by it.
