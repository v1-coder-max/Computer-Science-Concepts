Composition vs Inheritance

The most repeated design advice in OOP: favor composition over inheritance. Here’s what it means and when each actually fits.

#oop #composition #has-a #design

The idea in plain English

Both are ways to reuse code, but they wire objects together differently:

  • Inheritance (is-a) — a class extends a base and is permanently bound to it. Reuse is fixed at compile time; a change to the base ripples into every child (the fragile base class problem).
  • Composition (has-a) — a class holds other objects and delegates to them. Behavior is assembled from parts and can be swapped at runtime.

The classic trap: you model Penguin : Bird because a penguin is a bird, but Bird has Fly() — and penguins can’t fly. Inheritance forced a behavior that doesn’t fit. With composition you give a bird an IFlyBehavior; a penguin gets the “can’t fly” one. No lie in the type system, and you can change a bird’s flight behavior on the fly.

“Favor composition” doesn’t mean “never inherit.” Use inheritance for genuine, stable is-a specialization (exceptions, streams); use composition when you’re really just reusing or combining capabilities — which is most of the time.

Real-world example

🏢 How this works in production

ASP.NET Core is composition all the way down: a service doesn’t inherit a logger or a repository — it has them, injected through the constructor (Dependency Injection). That’s why you can swap a real IEmailSender for a fake in tests, or a SQL repository for an in-memory one, without changing the class. The Strategy pattern (swapping an algorithm object) and most of the GoF patterns lean on composition precisely because it stays flexible where deep inheritance trees calcify.

Rigid tree vs assembled parts

In C#

C# 
// ❌ Inheritance forces a behavior that doesn't fit every subtype.
public class Bird { public virtual void Fly() => Console.WriteLine("flap"); }
public class Penguin : Bird { /* Fly() is a lie — penguins can't */ }

// ✅ Composition: the behavior is a part you plug in (Strategy pattern).
public interface IFlyBehavior { void Fly(); }
public class WingFly  : IFlyBehavior { public void Fly() => Console.WriteLine("flap"); }
public class CannotFly : IFlyBehavior { public void Fly() => Console.WriteLine("can't fly"); }

public class Bird2(IFlyBehavior fly)
{
    private IFlyBehavior _fly = fly;
    public void PerformFly() => _fly.Fly();
    public void SetFly(IFlyBehavior fly) => _fly = fly;   // swap behavior at runtime
}

var penguin = new Bird2(new CannotFly());   // no lie in the type system
var eagle   = new Bird2(new WingFly());

Key takeaways

🎯 Remember these three

  • Inheritance = is-a, fixed and tightly coupled; composition = has-a, assembled and swappable.
  • Composition avoids the fragile base class problem and lets you change behavior at runtime.
  • “Favor composition” isn’t “never inherit” — use inheritance for true, stable is-a specialization.

Interview tip

🗣️ When this comes up

“Why favor composition over inheritance?” Lead with the fragile base class problem and the Penguin-can’t-fly example, then note composition’s runtime flexibility and testability (inject a fake). Mention that DI and the Strategy pattern are composition in practice.
