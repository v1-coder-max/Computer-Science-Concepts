Inheritance

A class can derive from another, reusing its members and specializing its behavior — an “is-a” relationship.

#oop #inheritance #is-a #virtual-override

The idea in plain English

Inheritance lets a derived (child) class take everything a base (parent) class has and add to or change it. A Dog is an Animal, so Dog inherits Name and Eat() and adds Fetch(). The child can also override a base behavior marked virtual to specialize it, and call back into the base with base.Method().

Key C# rules:

  • Single class inheritance — a class has at most one base class (but can implement many interfaces).
  • virtual + override enable specialization; sealed stops further overriding/inheriting.
  • abstract members force derived classes to provide an implementation.

Inheritance is powerful but easy to overuse. Deep hierarchies become rigid, and a change to the base can break every child (the fragile base class problem). The modern guidance — “favor composition over inheritance” — says use inheritance for true is-a specialization, and assemble behavior from parts when you’re really just reusing code.

Real-world example

🏢 How this works in production

The .NET framework itself is built on inheritance hierarchies: every exception derives from System.Exception, every stream from Stream, and UI toolkits model Button → ButtonBase → Control. That shared base lets a catch (Exception) handle any error and a method accepting a Stream work with files, memory, or sockets. It’s genuine is-a specialization — exactly where inheritance shines.

A base and its specializations

In C#

C# 
public class Animal
{
    public string Name { get; init; } = "";
    public void Eat() => Console.WriteLine($"{Name} eats.");
    public virtual string Speak() => "...";   // virtual: children may override
}

public class Dog : Animal                      // Dog IS-A Animal
{
    public override string Speak() => "Woof";  // specialize
    public void Fetch() => Console.WriteLine($"{Name} fetches.");
}

public sealed class ServiceDog : Dog           // sealed: no further subclassing
{
    public override string Speak() => base.Speak() + " (on duty)"; // reuse + extend
}

Animal a = new Dog { Name = "Rex" };
a.Eat();                 // inherited from Animal
Console.WriteLine(a.Speak()); // "Woof" — dispatched to Dog's override

⚠️ Don’t inherit just to reuse code

If the relationship isn’t a real “is-a,” inheritance creates tight coupling and fragile hierarchies. Needing only a helper method from a base class is a sign you want composition (hold an instance) instead. See the composition-vs-inheritance page.

Key takeaways

🎯 Remember these three

  • Inheritance models is-a: a derived class reuses and specializes a base via virtual/override.
  • C# allows one base class but many interfaces; sealed stops the chain, abstract forces implementation.
  • Favor composition when you’re only reusing code — deep hierarchies are rigid and fragile.

Interview tip

🗣️ When this comes up

Expect “inheritance vs composition” and questions on virtual/override vs new (hiding). Stress that inheritance is for genuine is-a relationships and that overusing it leads to the fragile base class problem — which is why the industry leans toward composition.
