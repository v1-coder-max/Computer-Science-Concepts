SOLID Principles

Five design rules that keep object-oriented code easy to change — the thing software actually has to do well over its lifetime.

#oop #solid #design #maintainability

The idea in plain English

SOLID is five principles (coined by Robert C. Martin) that together fight the one thing that rots a codebase: rigidity — where a small change ripples into ten files and breaks two you forgot about. None of them are laws; they’re pressure that pushes you toward classes that are small, swappable, and safe to extend.

  • S — Single Responsibility: a class should have one reason to change. Mixing “calculate payroll” with “format a PDF” means a report tweak can break payroll.
  • O — Open/Closed: open for extension, closed for modification. Add behavior by adding a class, not by editing a working one (and re-testing everything).
  • L — Liskov Substitution: a subtype must work anywhere its base type is expected. If Square : Rectangle breaks code that sets width and height independently, the inheritance is a lie.
  • I — Interface Segregation: many small, focused interfaces beat one fat one. Don’t force a class to implement Fax() just because it’s a IMachine.
  • D — Dependency Inversion: depend on abstractions, not concretions. High-level policy shouldn’t hard-wire a low-level detail like SMTP.

Real-world example

🏢 How this works in production

Amazon’s checkout doesn’t hard-code a single payment provider. It depends on a IPaymentProvider abstraction (Dependency Inversion), and each new method — card, gift card, Pay-Later — is a new implementation dropped in without touching the order pipeline (Open/Closed). Because every provider honors the same contract (Liskov), the checkout code never needs an if (provider is PayPal) branch. That’s why they can launch a payment method in one region without risking the global cart.

The five at a glance

In C# — Dependency Inversion

The most impactful one day-to-day. Watch the high-level OrderService stop hard-wiring a concrete email sender and depend on an abstraction instead:

C# 
// ❌ Rigid: high-level policy is welded to a low-level detail (SMTP).
//    Impossible to unit-test without sending real email; can't swap channels.
public class OrderService
{
    private readonly SmtpEmailSender _email = new();
    public void Place(Order o) => _email.Send(o.CustomerEmail, "Thanks!");
}

// ✅ Both sides now depend on the abstraction, not each other.
public interface INotifier
{
    Task NotifyAsync(string to, string message);
}

public class OrderService                       // high-level policy
{
    private readonly INotifier _notifier;
    public OrderService(INotifier notifier) =>  // injected (Dependency Injection)
        _notifier = notifier;

    public async Task PlaceAsync(Order o)
    {
        // ... persist the order (its single responsibility) ...
        await _notifier.NotifyAsync(o.CustomerEmail, "Thanks for your order!");
    }
}

// New channels are *added*, never edited in (Open/Closed).
public class EmailNotifier : INotifier { /* ... */ public Task NotifyAsync(string t, string m) => Task.CompletedTask; }
public class SmsNotifier   : INotifier { /* ... */ public Task NotifyAsync(string t, string m) => Task.CompletedTask; }

Now tests inject a fake INotifier, and switching email→SMS is a one-line registration change — no edit to OrderService.

Key takeaways

🎯 Remember these three

  • SOLID exists for one reason: keep code cheap and safe to change. Every principle attacks rigidity from a different angle.
  • Dependency Inversion + Open/Closed are the workhorses — program to interfaces and add behavior by adding types, not editing old ones.
  • They’re heuristics, not commandments. Over-applying SOLID creates its own mess (a factory for every class) — use judgment.

Interview tip

🗣️ When this comes up

Don’t just recite the acronym — interviewers want “give me a real example where you applied one.” Have a Dependency-Inversion story ready (swapping an implementation behind an interface to make code testable). Bonus points for admitting where SOLID can be over-applied — it signals seniority, not dogma.
