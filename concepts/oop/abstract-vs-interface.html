Abstract Class vs Interface

Both define a contract — but an abstract class can also carry shared code and state, while an interface is a pure capability you can mix into any type.

#oop #interface #abstract-class #design

The idea in plain English

An interface says “a type that implements me can do these things” — historically with no implementation at all. A class can implement many interfaces, so interfaces are perfect for capabilities shared across otherwise-unrelated types (IDisposable, IComparable, IEnumerable).

An abstract class is a partial base class: it can hold fields and state, constructors, and a mix of concrete and abstract members. It’s for an is-a family that shares real implementation. But a class can derive from only one abstract class.

The decision boils down to:

  • Interface — a contract/capability, no shared code, mixable into many types.
  • Abstract class — a base with common implementation and/or state for a related family.

Modern C# blurs the line a bit: default interface methods (C# 8) let an interface ship a default body. Still, interfaces can’t hold instance state, so the rule of thumb holds — reach for an interface unless you need shared implementation/state.

Real-world example

🏢 How this works in production

.NET shows both clearly. IDisposable and IComparable are interfaces — capabilities a FileStream, a SqlConnection, and your own type can all advertise independently. Stream is an abstract class — it provides shared buffering/async plumbing and leaves the raw Read/Write abstract for FileStream/MemoryStream to fill in. Capability across unrelated types → interface; shared machinery for a family → abstract base.

Two kinds of contract

In C#

C# 
// Interface: a capability any type can advertise; can have a default method.
public interface INotifier
{
    Task SendAsync(string to, string message);
    bool IsAvailable => true;                 // default interface method (C# 8+)
}

// Abstract class: shared state + code, plus a member subclasses MUST implement.
public abstract class RepositoryBase<T>
{
    protected readonly DbContext Db;          // shared STATE — interfaces can't do this
    protected RepositoryBase(DbContext db) => Db = db;  // constructor

    public async Task SaveAsync() => await Db.SaveChangesAsync(); // shared code
    public abstract Task<T?> FindAsync(int id);                   // each must implement
}

// A class can implement many interfaces but inherit only one base class:
public class SmsRepo : RepositoryBase<Message>, INotifier, IDisposable
{
    public SmsRepo(DbContext db) : base(db) { }
    public override Task<Message?> FindAsync(int id) => Db.Set<Message>().FindAsync(id).AsTask();
    public Task SendAsync(string to, string msg) => Task.CompletedTask;
    public void Dispose() { }
}

Key takeaways

🎯 Remember these three

  • Interface = a contract/capability with no instance state, implementable by many unrelated types.
  • Abstract class = a partial base with shared code, state, and constructors — single inheritance only.
  • Default rule: prefer an interface unless you need shared implementation/state; default interface methods narrow but don’t erase the gap.

Interview tip

🗣️ When this comes up

“Interface vs abstract class — when do you pick which?” Interface for a capability mixed across unrelated types and to get “multiple inheritance” of contracts; abstract class for shared implementation/state in an is-a family. Bonus: note default interface methods (C# 8) and that interfaces still can’t hold instance state.
